# 章节

### 自由电子

回想我在 Borland 的日子，我们那时正在全力开发 Paradox for Windows。我是一名测试工程师，负责测试数据库的创建和修改功能。我在工程部的搭档杰瑞工作很努力，但毫无进展。

当时正处于 1.0 产品周期的中后期，大多数工程师都慢慢地从开发阶段进入修复 Bug 模式，但杰瑞不是。他仍在……反复实现功能。当你把一个关键任务交给一个完全无法完成的人时，你就麻烦了。

现在，我们先别责怪杰瑞。他是个不错的程序员，但他有两个致命的弱点。首先，杰瑞从未为 Windows 编程过，所以他是一边编码一边学习。其次，这也是一款 1.0 的产品。本书的第 19 章标题是“1.0”，但它真正的标题应该是：“1.0，念作‘一点零，天哪，我再也见不到我的家人了’。”我来总结一下：1.0 版本极其困难，加上他对 Windows 不熟悉，杰瑞陷入困境了。

然而，杰瑞是有自尊心的。杰瑞相信自己能成功，但作为他代码的接收方，我观察到一种令人不安的编码实践，我们称之为“在盘子里拨弄垃圾”。杰瑞修复 Bug 的方法是以各种有趣的方式重新排列他的代码，就像你小时候为了糊弄妈妈，证明你确实吃了甜菜，而把食物在盘子里推来推去一样。实际上没有任何实质性改变，只是看起来不同了。这种编码实践的另一个名字可能是“靠希望编程”。

杰瑞代码的最终结果是，每次他修复一个问题，我们就会发现该功能的另一个根本性问题。是的，每次修复 Bug 都取得了微小的、渐进的进展，但杰瑞正身处败局，因为他的基本架构就是一坨屎。当被问及进度时，他列出的借口长得惊人且令人信服。这些借口来自一个真心相信自己能成功，并愿意为此投入时间的人。但就算把全世界的时间都给他也帮不了杰瑞，因为他已经完全超出了自己的能力范围。



### 引言

如果你是这个场景中的经理，你必须做出重大改变，因为你不能发布垃圾产品。有些公司这么做了，最终还赚取了可观的利润。但你不会是他们那样的人，因为一旦你因发布垃圾而获得奖励，你就开始盲目地走上一条平庸之路，最终结局是在一家名叫“Computer Associates”的公司，维护一个没人听说过、也没人在乎的产品。

这是一个两步走的修复流程。我们需要先做一个“Jerry 式的调整”，然后我们需要一个奇迹。我先从简单的那步开始。

我们需要 Jerry。他是唯一一个知道那堆意大利面条式代码里到底发生了什么的人，并且他能修复那些无关紧要的小错误。工程经理让 Jerry 坐下，告诉他我们需要关注数量。到处都有大量被忽视的小修复，Jerry 可以处理这些。是的，他的自尊心受到了伤害，但几周后，Jerry 就开始高速运转了，因为当人们在有可能完成的任务上取得进展时，他们总是工作得更好。

### 自由电子

Jerry 投入工作后，我们不得不面对另一个事实：距离发布还有六个月，而我们有很大一部分功能是拼凑起来的，勉强能运行。在这种情况下，你需要一个独特的人才。你需要一个*自由电子*。

自由电子是你将要遇到的最具生产力的工程师。我甚至还没给出定义，但我猜一个符合条件的人已经浮现在你的脑海里了。

在代码方面，自由电子无所不能。他们可以从头开始编写一个完整的应用程序，用一个周末学会一门语言，最重要的是，他们可以一头扎进一大坨意大利面条式代码中，理清头绪，并让它真正运行起来。你可以围绕一个自由电子建立一整个业务。他们就是这么厉害。

自由电子分为两类：*高级电子*和*初级电子*。两者都有相似的生产力产出，但高级版本已经变得在政治上和社交上都很成熟。在技术型组织中，许多 CTO 都属于这一类。想想 Bill Joy。高级电子总是有风险会陷入技术的高地球轨道，他们的想法听起来简直疯狂，但是，每当我们发言时，你都应该倾听。

初级版本拥有全部能力，他们只是缺乏与人打交道的经验，因为他们年轻时花了很多时间编写自己的操作系统，把这当作一种有趣的心智锻炼。这些初级电子代表着一个招聘经理能做出的最优秀的单项招聘。如果你在 20 年里能遇到两个，那你就做对了一些事。

### 养护与激励

如果你有幸在自己的组织中拥有一个自由电子，你需要意识到你正在与一种特殊的工程师打交道，对这种工程师的养护和激励方式可能与团队其他成员不同。

**让他们保持参与。** 首先，工程组织中有两个主要任务：研究和开发。虽然自由电子完全有能力做开发，但他们在组织中的价值在于研究。他们定义了技术前沿。如果你让一个自由电子在开发岗位上待得太久，他们会消失，仅仅是因为他们感到无聊。所有工程师都喜欢站在技术前沿，但自由电子必须定义它。一个离职的自由电子会永久性地损害你团队的生产力。

**被误导的自由电子精力可能产生奇怪的结果。** 在一个项目中，我把几个棘手的内存损坏 bug 分配给了自由电子，他只是安静地点点头，然后立刻消失了一个星期。当他回来时，bug 已经修好了，而且整个数据库层都被重写了。一段由两个工程师花了大约六个月设计的代码，在七天内被完全重做。听起来这是个好主意，直到你意识到我们当时正在做一个小更新，根本没有资源或时间去测试一个全新的数据库层。唉。

**自由电子有时不会参与，而且他们不会解释原因。** 自由电子是高功能的，对每件事都有强烈的意见……但他们可能永远不会告诉你这些意见。如果你要求他们做一些他们不认同的事情，他们是不会去做的。随便你怎么要求。最糟糕的情况是，当你要求一个自由电子做出一个救场式的挽救，而他们点点头……然后立刻回到你那个无用的请求打扰他们之前正在做的事情上。一周后，你还在期待奇迹，但自由电子会说：“还没弄到那个。”

又过了一周，你的头发基本上都要被揪光了，然后你会意识到，一开始你根本就不需要奇迹，不作为才是正确的选择。你的自由电子两周前就知道了。他们只是不想花两个小时为你画那张图。很烦人，是吧？你会习惯的。

**这是一个团队。** 所有这些建议都是针对你的自由电子的，但你需要记住，尽管他们生产力惊人，但他们也是团队的一部分。我的建议不应被理解为给自由电子特殊待遇，正如你不会对团队中的每个人给予你专注的关注一样。没必要刻意引起别人对你团队中有个自由电子的注意。相信我，每个人都已经知道了。

### 回到 Jerry

Bernard 出场了，Borland 公司常驻的自由电子。在他开始在代码里捣鼓之前，我根本不知道 Bernard 到底是做什么的。他有个办公室，里面堆满了书。他说了很多话，但产出却很少。“吹牛大王”就是我当时的想法。

Bernard 在一个周五的下午开始摆弄 Jerry 的代码。接下来的周一，我有史以来第一次能够完整运行我的功能测试矩阵。到那一周结束时，Bernard 已经关闭了大部分高严重性 bug，并开始涉足原本专属于 Jerry 的修复领域。接下来的一周，我拼命地提交 bug，就为了让 Bernard 保持参与。

这就是工作中的自由电子。

### 章节：重组的规则

你一定经历过这种事。

晚上 10 点过后，你收到老板一封随机邮件说：“我们需要谈谈。”除了 30 秒后出现的、标题为“回复：需要谈谈”的会议邀请之外，没有任何额外信息。你坐在那里盯着屏幕，想知道发生了什么。空气中没有裁员的传闻；公司运营良好。你想不出自己做了什么、说了什么或写了什么值得管理层跟进的事情，那么到底是怎么回事？

等到第二天开会时，你已经焦虑了整整 12 个小时，这意味着你反思了过去六个月里自己做的每一件事，以便为你老板可能抛出的任何刁钻问题做好准备。当你走进他的办公室时，他已经面朝着你，组织结构图就摆在他面前。

所以你现在知道了。你即将经历一次重组。

你的老板处理得相当不错。他开门见山地告诉你，你的一个小组将被调到别处。他解释了理由，并询问你的想法。你们又闲聊了一会儿，然后他彻底搞砸了：“我一直没告诉你，是因为事情变化太快，我不想折腾你。现在已经完全稳定下来了。”

哈。好吧。



在解释你的上司刚才如何对你撒谎之前，我们先准确理解“重组”（`reorg`）到底是什么，然后我为你提供一些应对混乱的建议。首先，重组不是裁员。裁员可能作为重组的一部分发生，但它们是副作用，而非原因。重组是为了应对公司战略的转变而对团队和产品进行重新调配。是什么样的转变？谁知道呢。也许你的产品市场发生了变化，或者经济形势糟糕。关键在于，执行层的某位决策者认为“我们需要对组织结构进行调整”，而这意味着一段整整一个月的混乱期。

以下是重组混乱期间需要注意的一些有用规则，以及一些渡过难关的窍门。

**规则 #1：明确你的角色**

当你第一次听到重组的风声时，你有一个选择：你打算如何参与？你是打算袖手旁观，看这场好戏，还是打算积极投身于混乱之中？

在你做出选择之前，可能需要先评估一下重组的规模。你尤其要关注那些与你所在部门相关的人事运作，但关键是要记住，重组代表着机遇。即使这次重组不涉及你的团队，也不意味着你不能向你的上司提议解决你所在小组长期存在的组织问题。

机遇在于，重组让组织变得非常灵活。整个组织的管理者们都在和你思考同一件事：“好吧，既然我们现在要解决 A 问题，那不如顺手把 B 问题也一起搞定，反正我们也要把一切搅个天翻地覆。”如果你有任何议程，有任何想要推动的改变，现在是时候考虑把它提出来了，因为在重组期间，你促成改变的可能性要大得多。

如果你满足于袖手旁观，看着其他人拉线操控，那也还有其他事值得你留意。

**规则 #2：人们会变得多疑**

从组织中的大多数人知道重组这件事，到实际重组发生，这中间有一段痛苦且漫长的时期。这段时期之所以痛苦，是因为它会促使员工开始提出关于公司的基本问题。谁要去哪里？为什么？会有裁员吗？为什么会发生这种情况？一切结束后我还能保住工作吗？

在团队得知重组的前一天，没有人问这些问题。团队当时正处于一种愉快的无知状态，他们最大的担忧是下一个截止日期。而现在，他们却在担心明天组织会变成什么样，而事实很简单：没有人知道。

让我们回到你上司对你说的话：“我之前没有告诉你，是因为事情变化太快，我不想让你跟着瞎折腾。现在一切都已经稳定下来了。”

再读一遍那句话：“事情变化太快”和“现在一切都已经稳定下来了”。他自相矛盾了，你听他这么说时就意识到了。是的，他试图在沟通，但他所做的只是让你变得多疑。走出那个会议室时，你会想：“他其实也不知道接下来会发生什么”，你是对的。他确实不知道。

没有人真正知道明天会发生什么，这一事实会滋生一种多疑的文化，这意味着你需要开始仔细倾听。

**规则 #3：小道消息疯狂传播**

重组期间谣言混乱的一个主要推手就是小道消息。简单来说，最初作为事实的信息，在人与人之间传播时会逐渐变得越来越像谣言。我们来看一个例子……

工程副总裁对她的员工说：“他们正在 Ted 手下组建一个新的硬件部门。目前还不清楚所有的人员编制从哪来，但其中一个选项是从其他部门削减编制。”

她说了什么？相当清晰的事实陈述。她试图给她的员工提个醒。我们继续往下看。

工程经理对他的下属说：“Ted 有了一个新部门。我们部门很可能会失去一些编制。”

好的，他说了什么？他先从事实开始，然后加上了一个观点——他会失去人手。他为什么这么说？也许他在公司待了很多年，知道这类事情通常会如何发展，又或者他只是在猜测。谁知道呢？小道消息正式生效了。再看……

高级工程师对他的朋友说：“我们部门要失去一些编制了，这些编制会转到 Ted 的部门去。天啊，我讨厌 Ted。”

欢迎来到一个为小道消息供应而准备成熟的完整谣言。

这是一个简单的例子，但它说明了基本的人性。我们想知道发生了什么，当我们不知道时，我们很可能会利用手头的任何事实来编造信息，以给人一种我们知道的假象。当你把观点和偏见添加到这个信息创造过程中时，最终你会看到源源不断的、引人入胜的虚构故事出现在你的面前。

在重组之外，我对小道消息相当信任，因为我觉得信息在人与人之间传递时的变异较少。但当饭碗受到威胁时，小道消息就会失控，一些离谱的垃圾信息会传到你的耳朵里。我建议采取一种耐心的、记者式的策略：在你相信任何事情之前，先用独立的信源去核实零散的信息片段。

**规则 #4：重组耗时漫长**

从你听说重组到重组真正完成，所需的时间会是你预期的四倍。重组会耗费极长时间。计划被制定出来，与利益相关者确认，根据反馈进行调整，与预算平衡，上报给大老板，然后再回到设计阶段。在这个官方流程进行的同时，还有走廊里的各种暗中活动，以及个别玩家为争夺人员编制而利用小道消息以实现其不可告人的目的。所有这些都会导致更多信息被注入官方流程，从而迫使流程进行更多的迭代。

我在这里的建议基于你选择的角色。显然，如果你深涉其中并有议程要推动，你需要在整个过程中保持参与。不要相信你的上司对你说“我们搞定了”，因为那只是他自以为搞定了。他不知道平台工程部的 Phil 还有几手准备。

如果你选择了旁观者的角色，我建议你坐回椅子上，享受这幕繁忙景象。让自己安心于你仍有工作这一事实，而且，嘿，如果这次重组影响到了你，换个环境或许对你有好处。别忘了在你的新办公室要个有窗户的位置。

重组并没有结束，直到某个重要人物打印出了一张新的组织架构图，并在整个公司面前展示出来。

**规则 #5：大多数人喜欢重组（但不愿承认）**

重组对那些对当前组织状态不满的人来说代表着机遇。如上所述，一旦利益相关者听说有重组的苗头，他们就会开始运作小道消息，以引导重组朝着对自己有利的方向发展。当你把这个事实与人们对八卦的喜爱结合起来，就肯定会迎来一场规模宏大、勾心斗角、旷日持久的重组。

如果你是个旁观者，你可能会对所有的走廊闲聊和闭门会议感到恼火，但事实是，大多数人就是喜欢这破事。谁被调走了？真的？哇哦。不可能吧。他是个白痴！太糟了！出于某种原因，关于重组的对话听起来很像关于出轨的对话。人们就是闭不上嘴。

具有讽刺意味的是，在重组周围制造最多噪音的群体，恰恰是对重组最终结果影响最小的群体。当管理团队正在战略、政治议程和财务责任中艰难跋涉，寻找一个能让公司不再纠结于谁为谁工作、转而专注于打造产品的计划时，这些人正徘徊在黑暗中。

**唯一规则：耐心**
