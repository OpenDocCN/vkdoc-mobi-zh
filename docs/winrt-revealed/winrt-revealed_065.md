# 章节

## 工程师思维

在兰兹管理守则中，新任管理者的"必做事项"清单非常简短。之所以简短，是因为"必须"意味着绝对，而面对人时几乎不存在绝对。适用于某人的巧妙管理方法，用到另一人身上可能就是灾难。这引出了管理必做事项中的第一条：

*保持灵活。*

认为自己已看透一切是种危险想法。当"持续变化"成为唯一恒常时，唯一该采取的姿态就是保持灵活。

矛盾的是，清单第二条却出奇地不灵活——这也是我个人偏爱的一条，因为它为管理成长奠定了基础。这条写道：

*停止写代码。*

其理论是：如果你想成为管理者，就必须学会信任下属承担编码工作。这条建议可能难以消化，尤其对新手管理者而言。他们成为管理者的原因之一很可能是出于高产开发者的身份，而当事情搞砸时，他们的第一反应往往是重拾树立自信的技能——那就是写代码。

当我看到新管理者退回代码领域时，我会告诉他们："我知道你能写代码。问题在于，你能管理吗？你不再只为自己负责——你要为整个团队负责。我希望看到你如何在不写代码的情况下带领团队解决问题。你的任务是找出让自己'规模化'的方法。我需要更多个你，而不只是孤军奋战的你。"

不错的建议，对吧？规模化、管理、责任。非常契合流行词汇。可惜我错了。

错了？

没错。错了。并非完全错误，但错得足以让我需要给前同事打电话道歉。"我那个'停止写代码'的论调？错了。是的。重新开始编程吧。从 Python 或 Ruby 开始。我是认真的。你的职业前途取决于此。"

我在 Borland 开启开发者生涯时，曾是 Paradox for Windows 团队的一员——这可是个大项目。仅应用开发团队就有 13 名开发者。如果算上提供核心技术（如核心数据库引擎、图形引擎和核心应用服务）的其他团队负责人，直接为产品贡献代码的工程师多达 50 人。

此后我所在的团队规模从未接近这个数字。事实上，随着时间推移，参与我所在产品开发的工程团队规模持续缩小。这是怎么回事？难道开发者集体变聪明了？不，我们只是在分散工作负载。

过去 20 年间，我们开发者究竟在做什么？嗯，我们写了堆积如山的代码。多到我们认为，或许将其开源共享是个好主意。好在互联网的出现让这种分享变得轻而易举。如果你是开发者，现在就试试：在谷歌或 GitHub 上搜索你的名字，找找那些你遗忘但所有人都能看到的代码。很可怕吧？没想到你的代码会永垂不朽？它确实会。



代码永存。优秀的代码不仅能存活，还能在珍视它的人确保其不会过时的情况下不断成长。正是这堆高价值、维护良好的代码，正在帮助缩小工程团队的平均规模，因为它让我们更少地专注于编写新代码，更多地专注于集成现有代码，从而用更少的人和更少的时间完成工作。

这里有一条令人沮丧的推论，即我们都只是一群用胶带连接不同现有活动部件来创造相同事物略微不同版本的集成自动化工人。正是这种思路让许多高级管理团队对外包感到兴奋。“任何会用谷歌并有点胶带的人都能做这个，那我们为什么要为我们的本地自动化工人支付大笔费用？”

我们付给这些管理人员的钱可不少，让他们想出这些破主意。尽管如此，这引出了我的最后一点：地球上到处都是渴望学习、聪明伶俐的开发者，即使他们从未在认证大学里待过一分钟，他们依然渴望学习、聪明伶俐。哦，对了，而且这样的人会越来越多。

我不是说你应该担心你的工作，因为海外某个聪明的家伙在瞄准你；我是说你应该担心你的工作，因为软件开发方式的演变可能比你进步得更快。你在你的岗位上工作了十年，做了五年经理，然后你在想：“我知道如何开发软件。”你确实知道。至少现在如此。

**把自己从代码中抽离，但是……**

如果你遵循我最初的建议，把自己从代码中抽离，那么你就是在把自己从创造的行为中抽离。这种行为正是我真正不担心外包的原因。自动化工人不构建，他们只是处理。虽然好的流程可以节省大量资金，但它不会给世界带来任何新东西。

随着更小的团队用更少的资源做更多的事，把自己从代码中抽离在我看来是一个糟糕的职业举动。即使在一个充斥着政策、流程和办公室政治的庞大公司里，你也不能忘记如何开发软件。而如何开发软件正在改变。现在。就在你脚下，就在这一秒。

你有你的困扰。我理解。我们来听听。

*   *“Rands，我正走在总监的轨道上，如果我继续写代码，就没人会觉得我有大局观了。”*

我对你的第一个问题是：从你即将成为总监的位置上看，你看到软件开发在你的公司内部正在发生变化吗？如果答案是肯定的，我的下一个问题是：它是如何变化的，你打算对此做些什么？如果你的答案是否定的，那么你需要挪动你的椅子，因为我向你发誓，软件开发就在此刻正在改变。如果你慢慢忘记了软件是如何构建的，你怎么可能拥有大局观？

我的建议不是让你在下一个版本中给自己分配一大堆功能。我的建议是你要采取行动，以便与你的团队构建产品的方式保持联系。作为一个总监或副总裁，你完全可以做到这一点。稍后会详细说明。

*   *“呃，Rands，总得有人当裁判。总得有人要有远见。如果我写代码，我就会失去对工作的全局视角。”*

你仍然需要当裁判，你仍然需要协调决策，你仍然需要每周一早上花 30 分钟和那个需要通过每周的“我们要完蛋了”咆哮来发泄的工程师绕着街区走四圈，但你也需要保持工程思维，而要做到这一点，你不需要成为一个全职的编码者。

我对保持工程思维的建议：

1. 使用开发环境来构建产品。这意味着你必须熟悉你团队的工具，包括构建系统、版本控制和编程语言。这个任务会让你与你的团队用来讨论如何完成工作的语言保持联系。它也会让你能够继续使用你最喜欢的文本编辑器……这太棒了。



2.  必须能够随时在白板上画出描述产品的详细架构图。我说的不是那种一个方框加两个箭头的简易版本。你需要掌握那种复杂、不美观且难以解释的详细版本。这张图是理解产品几乎一切原理的地图。它会随着时间变化，而你应该能够理解这些变化发生的原因。

3.  **拥有**一个功能模块。写这句话时我甚至感到有些畏缩，因为这充满了风险，但我认为，不真正拥有一个属于自己的功能模块，你很难做到第 1 条和第 2 条。拥有一个功能模块不仅迫使你积极参与开发过程，还能将你的角色从“对一切负责的管理者”切换为“某个东西的拥有者”。这是一种谦逊、低调的视角，能时刻提醒你小决策的重要性。

我仍然感到不安。肯定已经有人在对我喊：“让管理者拥有功能模块？！？！”（而且我同意）。你依然是个管理者，所以让它成为一个小的功能模块，好吗？你还有很多事情要做。如果你无法想象拥有一个功能模块，我的备选建议是去修复一些 Bug。虽然你无法获得拥有感，但你会对产品的构建产生深刻理解，这是你在走廊里闲逛永远无法获得的。

4.  **编写单元测试。** 在产品周期后期，当大家都快崩溃时，我仍然会做这件事。把它看作是理解产品功能的一份清单。经常做。

下一个担忧是什么？

“兰兹，如果我去写代码，团队会感到困惑。他们不知道我到底是管理者还是开发者。”

很好。

我是认真的。我很高兴你即将通过潜入开发者群体来让团队感到困惑。简单的事实是，软件开发中那些过于明确的角色划分正在消失。界面工程师正在做那些只能被称为 JavaScript 和 CSS 开发的工作。开发者正在学习更多交互设计知识。每个人都在和他人交流，从彼此的错误中学习，互相抄袭代码，管理者没有理由不参与到这场全球性、大规模的、交叉授粉式的信息混乱狂欢中。

此外，你应该渴望成为一个由可互换部件组成的团队的一员。这不仅能让你的团队更灵活，还能让每个人有机会从截然不同的视角看待产品和公司。当你看到那位沉默寡言的构建工程师弗兰克那优雅简洁的构建脚本时，你难道不会对他更加尊敬吗？

我并不是希望你的团队陷入困惑和混乱。实际上，我是在希望团队能拥有更好的沟通。我相信，如果你参与产品构建并接触功能模块，你会与团队更加贴近。但更重要的是，你会更贴近软件开发在你的组织中不断变化的方式。

## 不要停止开发

在 Borland 工作时的某位同事曾因我称她为“码农”而对我进行言语攻击。

- “兰兹，‘码农’是没有思想的机器，是只猴子。码农除了写下无聊无用的代码行之外，什么都做不了。我是一名**软件开发者**。”

她是对的。她可能会讨厌我给新管理者“停止编码”的建议。不是因为我暗示他们是码农，而是因为我主动告诉他们开始忽略工作中最重要的部分之一：软件开发。

所以，我修改了我的建议。如果你想成为一名优秀的管理者，你可以停止日常编码，但是……

*保持灵活性，牢记作为工程师的意义，并且永远不要停止开发。*

## 三种超能力

菲尔的团队正在迷失方向。

菲尔很聪明，也喜欢开会，但他是个糟糕的人事管理者，这种糟糕正在慢慢毒害他的团队。你知道情况有多糟，因为菲尔团队里的明星员工总能找到机会和你开会，而说辞总是一样：“他很聪明，但他天生就不会管理人。如果情况没有改变，距离全面造反还有三周时间。”

在现场，菲尔的团队一直在等待……一直等到某个迫在眉睫的灾难全面爆发，他们才不情愿地说：“嗯，好吧，呃，我们大概应该……现在就处理那个？”

令人沮丧和困惑的是，与菲尔的一对一谈话感觉像是有进展。他说得头头是道。他承担责任，阐述令人信服的后续步骤，并向你保证情况会改善。但这些会议与菲尔日常管理判断之间存在着巨大的战略鸿沟。行动与言语不符，每个周一早上你都会意识到：*菲尔还是什么事都办不成，他团队里的明星员工快要辞职了，而我完全不知道该怎么办。*

你需要一种新的超能力。

## 三种超能力

你有一个天然的管理状态。这是你的默认状态，是你做决策的基础，是你主持会议时的基调，也是你在走廊里交流时展现的个性。这个状态一直对你很有利；你对此感到舒适，因为这就是你。

不幸的是，这个天然状态正是你困惑于菲尔问题的根源。你的天然状态让你对显而易见的行动方案视而不见。你的直觉是错误的。你需要变成一个你不是的人，才能彻底改变你的视角。

这里有三种人格，每种都有其独特的视角，更重要的是，都有其独特的超能力：

## 机器人的辩论

*机器人*：我们经典的机械师。^(1) 她的座右铭是：“没有计划，就没有希望。”她坚信整个世界都是可衡量的。她遵循流程，因为只有严格遵守流程，我们才能真正产生有意义的衡量标准，从而清楚地了解自己是否在获胜。机器人对流程及其必要性的理解是无与伦比的。机器人热爱辩论；她喜欢考虑所有选项，但当她做出决定，就结束了。无法改变她的想法。

当机器人想让某件事完成时，她会使用“辩论”。辩论从数据开始。*请你们每个人，用极其详尽的细节，向我解释菲尔最近十次作为无能人事管理者的具体事例，并且要知道，我会继续追问，直到我们完全理解为止。*

辩论继续以假设进行。*综合我们丰富的数据，我提出以下假设，标题为“改善菲尔问题的三件事”。我期望所有相关方挑战这些假设。但请注意，基于感觉而非事实的论据将被终止。*

如果一个假设被成功辩护，它就成为法律，辩论结束。如果失败，辩论继续。一次运作良好的辩论不仅会验证真相，更重要的是，会围绕这个真相建立共识，这听起来很令人愉快，但有以下需要注意的地方：

- 如果众人渴望行动，辩论可能会看起来像是在拖延。如果你的菲尔问题很严峻，对那些快要辞职的人来说，对菲尔问题进行为期四周的分析可能不是他们想要的回应。
- 如果问题涉及深刻的情感，辩论几乎是一项无法完成的任务。人是很混乱的——尤其是情绪化、愤怒的人群——虽然他们很乐意发泄对菲尔失败的不满，但将这种发泄转化为建设性的辩论可能需要难以忍受的时间。
- 辩论缺乏艺术性。虽然一场好的辩论会彻底验证假设的有效性，但它很少能引领你走向创新或大胆的飞跃。辩论的全部意义在于制定出关于如何行动的、尽可能最好的理论，而根据我的经验，这个过程会引导你得出一个可信、可执行的计划，感觉就像常识，而不是灵感。

这就是为什么……

## 绝地大师的微调



### *绝地大师*拥有神奇的待人技巧

*绝地大师*拥有神奇的待人技巧。他的口头禅是：“帮帮我，让我帮你。”他满心都是员工的福祉。他说：“员工是我们最宝贵的资源。”他厌恶冲突，因为冲突就像……扫兴的家伙。他想做你的哥们。他对冲突的回避让他产生了合理的多疑，但他把这种多疑埋藏在令人愉悦的关怀表象之下。他用心倾听，通常能如愿以偿，但你永远不太确定他是怎么做到的。

我来告诉你他是怎么做到的——*绝地大师*会使用“`轻推`”。

你甚至不会察觉到`轻推`的到来。事实上，你很难弄清楚它到底是什么时候发生的。*绝地大师*会说些无关痛痒的话：“菲尔，你觉得 X 项目的设计怎么样？”菲尔不会直接回答，而是坐在那里，茫然地凝视着。他在琢磨着……什么。如果你能听到菲尔在想什么，那会是这样的：*设计？设计两个月前就敲定了。已经完成了。为什么现在提起？是因为设计是我年度考核中谈到的事情吗？肯定是。为什么我感觉自己好像漏掉了什么？我得马上去找设计团队谈谈。*

`轻推`是你能给出的最小、最具传染性的建设性反馈。它小到看起来毫无威胁，但它有足够的意图，足够的暗示，能在接收者心中生根发芽，并且必然会不断演化。

处理一个`轻推`需要时间——这正是关键所在。成功接收`轻推`的人会感觉自己隐约捕捉到了什么，但没有具体细节。他们拥有了潜力，而且说不准什么时候他们会把这种潜力转化为行动，但当他们行动时，这个行动完全是他们自己的，因为所有工作都是他们完成的。而这正是关键所在。

一个精心设计的`轻推`，是你关于人、领导力、政治以及你打算`轻推`的那个人的全部认知的巧妙结合。它是精心挑选一个简单的想法，旨在向某个（在这种情况下，除了自己之外不会听信任何人真相的人）缓慢揭示一个更大的意图。

一个恰到好处的`轻推`简直是管理艺术的杰作。一个用错地方的`轻推`则是让你陷入团队麻烦的绝佳方式：

-   目前为止，`轻推`是促成改变最慢的方式。如果时间紧迫，`辩论`会比`轻推`更快。但如果你想促成永久性的改变，那就用`轻推`。
-   `轻推`不是游戏。其目标不是构建一个旨在混淆视听的小巧神秘的短语。`轻推`是你播下的一颗种子，希望它能朝着富有成效的方向生长。你要监控它，当它似乎迷失方向时，你就再次`轻推`。
-   完全依赖`轻推`、选择用巧言妙语说话的*绝地大师*，会让人极其恼火。总有人会恼火，并要求你直截了当地告诉他们。记住，教训是学到的，而不是被告知的。（啪！）

## 可能需要采取严厉措施，所以……
