# 第 2 章：Objective-C 概要

## 类别（Categories）

有时，与其编写自己的类，你只想给现有类添加一个方法。例如，假设你想要一种快速将字符串加倍（通过将其追加到自身）的方法。如果能编写如下代码就好了：

`NSString *myString = @"Hello, World!";`

`NSString *doubleString = [myString doubleString];`

不幸的是，`doubleString`方法并不存在于`NSString`上。实现它的一种方法是创建一个实现了该方法的`NSString`子类。

但这并不完美，因为之后我们必须在所有想要使用新方法的地方，将我们的`NSString`替换为这个子类。幸运的是，Objective-C 的类别（categories）允许我们将方法添加到现有类中！以下是创建类别的方法：

```
@interface NSString (DoubleString)

- (NSString *)doubleString;

@end
```

这看起来像在创建一个新类，但不同于指定父类，我们指定了一个类别名称——在本例中，是`DoubleString`。然后，在实现中，我们指定相同的名称：

```
@implementation NSString (DoubleString)

- (NSString *)doubleString

{

return [self stringByAppendingString:self];

}

@end
```

请注意，在类别中我们仍然可以访问`self`。类别使得扩展类变得非常容易，而无需修改原始类或创建其子类。

类别的另一个有用用途是使用它们来分离功能。例如，如果你有一个对象既可以用于 Mac OS X 应用，也可以用于 iOS 应用，你可能会有两个类别：一个用于在 Mac 上绘制它，另一个用于在 iOS 上绘制它。苹果公司这样做的一个例子是`NSString`上的`UIKitAdditions`类别，它包含许多用于确定字符串如何在 iOS 上绘制的方法。由于`NSString`在 Mac OS X 和 iOS 之间是共享的，而只有后者（iOS）才需要这些方法，因此使用类别是分离这些关注点的一个好方法。

然而，类别并非没有风险。在我们之前实现`doubleString`方法的例子中，我们做了一个重要的假设：`NSString`上尚未声明`doubleString`方法。如果已经存在，那么其中一个实现将会覆盖另一个，这显然不是我们想要的。更糟糕的是，我们并不总是清楚哪个实现会“胜出”，所以你甚至不知道哪段代码在运行！另一个我们希望避免的情形是，苹果公司也决定添加相同的功能。如果在未来的 iOS 版本中，苹果添加了`doubleString`方法，同样的问题也会发生。为了避免这种情况，最佳实践始终是使用前缀来声明你的类别方法。在本例中，我会将`doubleString`改为`LCT_doubleString`。虽然这不太优雅，但它起到了保护代码不被其他人的实现破坏的重要作用。类别文件的命名约定是`<ClassName>+<CategoryName>.[h/m]`。对于我们的字符串加倍类别，我们可以将文件命名为`NSString+Doubling.h`和`NSString+Doubling.m`。

**注意：** 类别的另一个注意事项是，你不能在类别中声明新的实例变量。这会改变对象的内部内存布局，并且由于原始类并不总能被重新编译，我们无法修改其内存布局。

## 类扩展（Class Extensions）

与其他语言不同，Objective-C 不提供私有方法（private methods）。任何方法都可以被任何人随时调用。实际上，可以在运行时查询一个对象，并获取可对其调用的所有方法的列表。尽管如此，向其他开发者隐藏方法仍然有一些用处。通常来说，在头文件中声明的任何方法都应该是可以从其他类在你的对象上调用的方法。如果你想声明一个只在类内部使用的方法，你可以使用一种特殊类型的类别：类扩展。由于我们不希望它是公开的，类扩展的接口声明应放在你的类的实现文件（`.m`）中，而不是头文件（`.h`）中。以下是声明类扩展的一个例子：

```
@interface MyClass()

- (void)superSecretMethod;

@end
```

如你所见，类扩展使用类名后跟一个空的圆括号来声明。与类别不同，你可以在类扩展中定义新的实例变量，这使得它们成为存储你不想暴露给类外部的内部变量的好地方。

**注意：** 与方法一样，没有办法真正使一个实例变量变得秘密。可以在运行时查询一个对象，并获取其实例变量的列表。

你在类扩展中声明的方法与类的常规方法一同实现。一个包含类扩展的典型实现文件可能如下所示：

```
#import "MyClass.h"

@interface MyClass() {

NSString *secretString;

}

- (void)mySuperSecretMethod;

@end

@implementation MyClass

- (void)mySuperSecretMethod

{

// 在此实现超级秘密方法。

}

…

@end
```

## 协议（Protocols）



## 多开发者编程时的协作方式

当多个开发者共同编程时，常常需要同时处理两个类，它们虽然各自独立，但必须协同工作才能完成任务。这就是头文件的用武之地：一位开发者创建类的头文件，另一位开发者通过查看头文件了解可用的方法。不过，很多时候你并不需要关心其他类的内部实现，只需知道如何与之交互即可。协议就是一种将一组方法从类中抽象出来的方式。当我们创建协议时，需要定义一系列方法，遵循该协议的对象必须实现这些方法。一个常见场景是：当某个类完成耗时任务后，向另一个类发送通知。

让我们创建一个包含耗时任务的`Worker`类：

```
@interface Worker: NSObject

- (void)longRunningTask;

@end
```

我们希望的工作流程是：任务完成后，`Worker`能通知其他对象。在 Objective-C 中，这类辅助对象通常被称为委托（delegate），这种编程实践称为委托模式，是 Objective-C 中最常见的设计模式之一。要创建协议，我们需要新建头文件`WorkerDelegate.h`：

```
// WorkerDelegate.h

@class Worker;

@protocol WorkerDelegate

@required

- (void)workerDidFinish:(Worker *)worker;

@end
```

**注意：** `@class`编译器指令被称为*前置声明*。通过它我们可以在不导入`Worker`头文件的情况下使用该类。通常情况下，应尽量少导入头文件：只要实现文件中包含的任何头文件发生变更，即使内容未改动，整个文件也需要重新编译。即便不考虑编译速度优化，保持代码整洁有序也应该避免引入不必要的头文件。

编译器指令`@protocol`用于创建协议，其后紧跟协议名称。命名规范采用首字母大写和驼峰式命名法。接着使用`@required`标记所有遵循`WorkerDelegate`协议的对象必须实现的方法。你也可以使用`@optional`标记可选实现的方法。目前的方法默认声明为可选，但这并非一直如此。由于历史限制，一些旧版类采用"非正式协议"，这种协议并非语言意义上的协议，而只是定义了一个或多个可供其他对象调用的方法，类似常规协议但不受头文件强制约束。

### 遵循协议

要使类遵循协议，需要在类声明后添加协议列表。我们创建新类`Delegate`来遵循该协议：

```
#import "WorkerDelegate.h"

@interface Delegate : NSObject <WorkerDelegate>

@end
```

语法要点：使用尖括号（`<`和`>`）包裹协议列表的开头和结尾；多个协议间用逗号和空格分隔。如果创建新类后未能实现必需的方法，编译器会发出警告，提示尚未实现`workerDidFinish:`方法。

**注意：** 行业惯例是在遵循协议的类的实现文件中，将协议的所有方法集中排列在一起。

接下来需要修改`Worker`类，添加指向委托的指针：

```
#import "WorkerDelegate.h"

@interface Worker : NSObject

@property (weak) id <WorkerDelegate> delegate;

@end
```

委托变量通常声明为`id`类型，因为我们只关心它是否遵循`WorkerDelegate`协议，而不在乎具体是何种对象。在耗时任务方法中，可以这样使用委托：

```
- (void)longRunningTask

{

// 执行任务

[[self delegate] workerDidFinish:self];

}
```

协议能帮助你保持代码整洁，将不同功能区域在类中分离。Apple 在 Cocoa Touch 中广泛使用协议，尤其是委托设计模式，因此这是每位 iOS 开发者都应精通的核心技术。

## 模型-视图-控制器编程：精良代码的设计哲学

Apple 代码（以及众多现代代码）遵循的设计理念称为模型-视图-控制器（MVC）。MVC 要求将代码分为三部分：模型（应用程序处理的数据）、视图（数据的呈现方式）以及控制器（负责协调两者）。以通讯录为例：模型是地址数据（联系人、地址、电话号码等），视图是应用界面（联系人列表、信息编辑按钮等）。

MVC 的核心思想是模型代码绝不应包含任何视图代码。例如，在开发跨平台通讯录时，Mac OS X 上绘制通讯录窗口的代码不应出现在模型代码中，iOS 上的绘制代码也同样如此。另一方面，影响模型的代码也不应出现在视图中；视图可以显示按字母排序的联系人列表，但不应包含修改姓名的功能。图 2-3 展示了 MVC 范式中各组件的独立性。

MVC 编程的优势往往不在首次开发时显现，而是在后续功能扩展时体现。通过将代码模块化，可以更轻松地进行修改或替换。模块化程度越高，操作越便捷。假如未来 Apple 发布 Apple TV 应用开发 API，你肯定不想从 iOS 按钮创建代码中费力提取通讯录的核心逻辑。通常应假定视图可以是任何形态：iOS 屏幕、Mac OS X 窗口、网站甚至物理 LCD 显示屏输出！只要模型代码专注于数据存储，视图代码专注于数据显示与用户输入接收，你的代码就已在通往卓越之路上取得先机。

MVC 通过控制器实现，这是模型与视图之间无形的协调者。控制器从模型读取姓名数据发送给视图，同时接收视图中的用户输入并相应更新模型。控制器是 MVC 代码中复用性最低的部分，因为它们通常同时绑定模型和视图代码。iOS 高度依赖控制器概念，Apple 甚至为此创建了专用类`UIViewController`。我们将在下一章构建第一个真实应用时深入探讨视图控制器。
