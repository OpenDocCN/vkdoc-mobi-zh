# 如何使用本书

答案取决于你当前对 Swift 知识的掌握水平。

如果你是 Swift 和 iOS/iPadOS 平台的新手，你需要先学习 Swift 基础知识。在开始阅读本书之前，请确保你能回答以下问题：

- 什么是类？类和结构体有什么区别？
- 如何根据不同的条件运行代码的某一分支？
- 如何多次运行同一段代码片段？
- 什么是 nib/xib 和故事板？
- 如何声明函数或变量？
- `let` 和 `var` 有什么区别？

你知道答案了吗？那么你就可以开始了！

尝试 Swift 代码最简单的方式是使用 Playground。你可以在 Xcode 中创建一个 Playground，而不是创建项目。Playground 可以包含一个或多个文件；你可以在其中声明类、变量、常量和函数。基本上，它和项目是一样的，只是它不会被编译成可执行文件。在 Swift Playground 中，你无需 iOS 设备或模拟器即可运行代码，并且无需设置断点就能看到所有变量的内容。

如果你正处于 Swift 学习的起步阶段，最好手动输入代码，而不是复制粘贴。在输入过程中，你是在主动学习。

如果你已经了解其他编程语言，学起来会容易得多。但即使不是这样，Swift 也是一个不错的入门选择。

如果你已经了解 Swift，但还没有编写过自己的应用，那么尝试去写一个。想一下：你希望在自己的 iPhone 或 iPad 上拥有什么应用？即使这类应用已经存在也没关系——制作你自己的版本！你甚至可以最终将其发布到 App Store 上。只要你没有违反他们的某些规则，这就没问题。让它免费，从用户那里获得反馈，修复错误，并上传更新。

你可以不受限制地使用本书中的技巧，但不要忘记编程更像是一门艺术而非科学。根据自己的需求修改代码；让它变成你自己的。虽然从头到尾阅读本书仍然是更好的方式，但这不是强制的。发挥创意，找到你现在需要的技巧，并加以利用。重要的是理解你所使用的内容，正因如此，每个技巧都附带了说明。

如果你是一位经验丰富的开发者，你可以直接从 GitHub 上挑选代码片段。你知道该怎么做。如果某些内容对你不适用，或者你知道更好的实现方法，请将你的反馈发送至 `swiftrecipes@foxicorn.com`。我们将不胜感激！

## 好代码

本书旨在为你提供优秀的代码片段，你可以将其用于自己的项目，或根据需要对其进行修改。但什么是好代码？好代码和坏代码之间有什么区别？

假设你有一个函数，它可以编译，不产生错误或警告，并且在生产环境中运行完美。这是好代码吗？

```
func f1(aaa: Int) -> Int {
var bbb: Int? = aaa
bbb = bbb! + 1
return bbb!
}
```

这段代码是做什么的？如果你了解 Swift，你会明白这是一个简单的自增函数。但看起来很难看……所以我们可以说这是坏代码。编程是优美的！

```
func inc(i: Int) -> Int {
i + 1
}
```

看到第二个代码片段，程序员（包括你自己）会立刻明白它是做什么的。而且它运行得更快，消耗的内存也更少。Swift 可能会优化第一个函数，使其无需临时变量也能工作，但它仍然是坏代码。

好代码必须：

- 对于所有类型的参数都能完成其应有的功能。如果参数对函数无效，必须正确处理。
- 绝不导致崩溃或其他关键情况。如果代码无法运行，必须抛出异常或返回 `nil`。
- 在实现其功能时，使用尽可能少的代码。仅在语言结构有意义时才使用它们。
- 函数、类型和变量的名称应能解释其用途。必须使其他开发者以及一年后回看代码的你能够阅读和理解。
- 使用最少所需系统资源：计算能力、内存、磁盘空间和网络。
- 可测试。每个函数应只完成一项任务，并且必须有测试覆盖。

如你所见，尽管第一个代码片段可以工作，但它不可读、不优化，并且使用了不必要的可选变量……对于这么小的函数来说，这可能不是关键问题，但如果函数至少 15–20 行，它最终会变成难以理解的代码，而你在将来需要它时，将不得不重写它而不是复用。

编写良好的代码不需要大量注释。有时注释是必要的，但代码应尽可能自解释。如果代码无需注释就能让其他程序员理解，那它很可能就是编写良好的代码。

我们这里说的好代码通常被称为整洁代码。还有更多原则能让你的代码变得更好（通常有有趣的名字）：

- DRY（Don't Repeat Yourself，不要重复自己）。每个函数只编写一次，并从代码的不同部分调用。如果代码不遵循 DRY，那就是 WET（Write Every Time, We Enjoy Typing, Waste Everyone's Time，每次重写，我们享受打字，浪费每个人的时间）。
- 两次或以上，使用循环。这条规则意味着你不应该编写同一行代码两次，应优先使用循环。
- SoC（Separation of Concerns，关注点分离）。这一设计原则建议在每个代码片段（函数、方法甚至类）中只完成一项任务。

后续章节提供的技巧严格遵循这些原则。


## NS 还是非 NS

许多编程指南都会用到 `String` 和 `Data` 类型。但同时还有 `NSString` 和 `NSData`。它们之间有什么区别，`NS` 又到底是什么意思呢？

前缀 `NS` 源自 NeXT 公司及其平台 NeXTSTEP。这家公司制造个人电脑并为其编写软件。听起来挺耳熟，对吧？

1997 年，苹果公司收购了 NeXT，并创建了新的操作系统 MacOS X，也就是我们现在熟知的 macOS。这个系统本应取代苹果电脑上的 MacOS 9，而在 2001 年，这个目标确实实现了。Mac OS X 与 NeXTSTEP 一样，都基于 Mach 内核，并且它与 NeXTSTEP 的关系比与 Mac OS 9 更为密切。

NeXTSTEP（以及后来的 MacOS X）的主要编程语言是 Objective-C。如果说 C++ 是“带有类的 C”，那么 Objective-C 就是“带有消息的 C”。这门语言并非由苹果或 NeXTSTEP 独立开发，但在 NeXT 公司为 GCC 编译器编写了扩展以支持 Objective-C，并开发了两个框架（Application Kit 和 Foundation Kit）之后，前缀 `NS` 就出现了。

Application Kit（AppKit）至今仍用于 macOS 开发，大家更熟悉的名字是 Cocoa。Foundation 则用于 iOS、watchOS、iPadOS、tvOS 和 macOS 平台。UIKit（也称为 Cocoa Touch）是 Application Kit 在移动平台上的适配版本。

那么，以 `NS` 开头和不以 `NS` 开头的类型有什么区别呢？由于继承了 NeXTSTEP 平台的类型系统，Objective-C 一直沿用至今。但当苹果推出 Swift 时，他们引入了全新设计的类型：`String`、`Data` 等等。这使得他们能够移除这些类中冗余的方法，并添加新的功能。与 Swift 不同，Objective-C 是向后兼容的，这意味着其标准类型的许多方法甚至可以追溯到 20 世纪 80 年代。

与此同时，Swift 与 Objective-C 完全兼容，并继承了后者所有的类型。这就是为什么你同时拥有 `String` 和 `NSString`、`Data` 和 `NSData`、`Array` 和 `NSArray` 等等。通常，最好使用不包含 `NS` 前缀的原生 Swift 类型。但在必要时，你也可以将一种类型转换为另一种类型。例如：

```
let nsData = NSData()
let data = nsData as Data
```

这种转换是非可选（强制）的，意味着它总是会成功。它们并非同一个类，也不完全是一个包装器（wrapper），但苹果就是这样设计的。他们称之为*桥接*。`Data` 拥有一个字节缓冲区和一些方法；`NSData` 也拥有一个字节缓冲区和一些其他（部分重复的）方法。因此，这种转换总是可行的。它们的内部结构非常相似。Swift 的 `Array` 桥接到 Foundation 的 `NSArray`，`String` 桥接到 `NSString`，以此类推。

## 框架

Swift 使用框架系统将外部代码引入你的项目。有两个标准框架，我们通常假设它们已被默认导入：

```
import Foundation
import UIKit
```

如果你尝试某些示例代码发现无法编译，请确保已导入这两个框架。通常，当你新建一个文件时，其中已包含一个导入语句。

**注意**

如果你导入了 `UIKit` 框架，则无需再导入 `Foundation`，因为这会自动完成。

如果代码使用了其他框架，则需要显式导入。`import` 语句将作为代码片段的一部分。

苹果公司提供了大量“开箱即用”的框架。包括：

* `AVFoundation` – 用于播放音频和视频文件及流的框架
* `CoreData` – 用于存储和访问数据的框架
* `WidgetKit` – 用于编写小组件的框架
* `ARKit` – 增强现实框架
* `MLKit` – 机器学习框架
* `CoreLocation` – 用于获取设备位置的框架
* `CryptoKit` – 加密框架
* 以及其他数十个框架

这些框架可以直接添加到项目中；它们有各自的要求，例如 iOS 版本或设备功能，因此某些框架需要在特定条件下使用。但将它们包含到项目中并从 iOS 代码中导入是直接了当的。

此外，还有许多第三方框架。如果你有它们的源代码或编译后的版本，可以直接包含它们。但更常见的方式是使用依赖管理器。Swift Package Manager (SPM) 已集成到 Xcode 中。它是在 Swift 3.0 中引入的官方依赖管理器。但历史上最流行的却是 CocoaPods。它出现得更早，并且大多数开源库都支持它。

除了广泛的支持和流行度外，CocoaPods 对所有类型的 Xcode 项目都有更好的兼容性。例如，包含框架和自定义 scheme 的项目在使用 SPM 构建时可能会遇到问题，而 CocoaPods 则能正确处理这种情况。

如果你不想设置 CocoaPods，也可以随时使用替代方法，例如 SPM 或直接将源代码包含到项目中。通过 SPM 添加依赖项可以直接在 Xcode 中完成，使用菜单 **File** ➤ **Add Packages…**。

CocoaPods 可以通过命令 `pod init` 进行初始化。你需要在项目根目录下的终端中键入此命令。如果你没有安装 CocoaPods，可以访问[官方 CocoaPods 网站](https://cocoapods.org)获取安装说明。

**注意**

CocoaPods 可能无法直接在搭载 Apple 芯片（M1 及其他）的 Mac 上工作。要在 Apple 处理器上运行 `pod` 命令，请在命令前加上 `arch -x86_64`。

CocoaPods 会创建一个配置文件——Podfile。为了在 Swift 中使用它，你可能需要添加这行代码：

```
use_frameworks!
```

这会为每个依赖项创建一个框架，使得在 Swift 中使用更加方便。所有库，即使是使用 Objective-C 编写的，都可以通过 `import` 语句导入。

另一行可能的代码是：

```
inhibit_all_warnings!
```

此行代码会移除所有通过 CocoaPods 添加的库的警告。这不是强制性的；甚至在某些情况下，它可能是有害的，所以请谨慎使用。另一方面，你无法修改通过 CocoaPods 导入的外部库的源代码，你可能也不喜欢在 Xcode 中看到那些你无法修复的警告。如果你最终还是决定使用它，最好在未添加此标志的情况下至少构建一次，看看是否存在重要的警告。如果有，你可能需要考虑更换另一个库或更新该库的版本。

另一部分描述了目标及其依赖项。每个目标可以有一套独立的依赖项。每个依赖项通过 `pod` 语句添加：

```
target 'MyApp' do
pod 'Alamofire'
end
```

你可以添加一个版本号：

```
target 'MyApp' do
pod 'Alamofire', '~> 5.0.0'
end
```

如果最新版本与你的代码不兼容，这就会很有用。默认情况下，会下载所有库的最新兼容版本。

兼容性是基于平台计算的。为了最大化外部库的效用，你需要指定与你的应用相同的平台，例如：

```
platform :ios, '12.0'
```

每当某个指南使用外部库时，都会提供该库的 pod 名称。

一个完整的 Podfile 示例：

```
platform :ios, '12.0'
use_frameworks!
inhibit_all_warnings!
target 'MyApp' do
# HTTP 库
pod 'Alamofire'
pod 'Kingfisher'
# Firebase 库
pod 'Firebase/Analytics'
pod 'Firebase/Crashlytics'
end
```



## Swift 版本与兼容性

与 Objective-C 不同，Swift 不向后兼容。用 Swift 1.0 编写的代码可能无法通过 Swift 2.0 编译，而且极有可能无法通过 Swift 5.0 编译。同时，Swift 5.0 的代码可能使用了 Swift 4.0 中不具备的语言特性。

> **注意**：不同版本的 Objective-C 也并非 100% 兼容，但在过去 37 年中，它只有两个主要版本。iOS 应用中使用的所有代码都是用 Objective-C 2.0 编写的。

与此同时，基本语言元素是相同的，因此代码要么能在另一个版本的 Swift 上直接工作，要么只需稍作修改即可。

本书中的所有代码均使用 Swift 5.5 编译，这是撰写这些代码示例时的最新版本。不过，这些代码应该能在所有 5.x 版本的 Swift 上编译，并且大多数示例甚至能在 4.x 版本上编译。
