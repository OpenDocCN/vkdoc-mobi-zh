# 6. 文本编辑

iOS 开发的另一个重要方面是文本编辑。在移动平台上，添加文本输入尤其困难，因为存在以下几个复杂因素：

*   当你进入 `UITextField` 时，屏幕键盘会弹出。它可能会遮挡其他 UI 元素，甚至文本输入框本身。
*   屏幕键盘不会自动消失。在某些情况下，它可能会跟随你进入另一个屏幕。
*   屏幕键盘在不同设备上有不同的高度。即使你知道了所有现有型号的尺寸，新的 iPhone 或 iPad 也终将出现，而它们的键盘高度可能不同。
*   物理键盘可以连接到 iOS 设备。除了 iPad 的专用键盘外，任何蓝牙键盘都可以连接到这些设备。当连接了物理键盘时，屏幕键盘不会出现。
*   即使设置了键盘类型，也不能保证用户不会输入一个*不允许*的符号。

除了这些移动设备特有的问题，我们还面临着所有允许用户输入的应用所共有的问题：

*   我们可能需要以特定方式格式化文本，例如电话号码或信用卡号。
*   我们通常需要浮动文本，它会跟随你的输入。或者需要前缀。这在需要输入金额时尤其常见。它可能同时需要格式化和带有货币符号的浮动文本。

我们将在本章中讨论所有这些问题及其解决方案。作为额外内容，我们将创建一个密码输入板，一个包含四个数字文本输入框的 UI 组件。

## 实时分析用户输入

许多初学者开发者不理解如何实时处理文本输入。`UITextFieldDelegate` 代理协议包含许多有用的函数，但并没有类似 `whatUserJustTyped` 或 `afterThisInputTextWillBeLikeThis` 这样的方法。

为什么我们需要这个？毕竟，我们总是可以捕捉到用户结束编辑的时刻，或者在用户点击某个按钮时访问 `UITextField` 的 text 属性。尽管这没错，但现代应用已不像 1990 年代的网页表单——它们是交互式的，能跟随你的操作并相应地调整 UI。以下是几个例子：

*   如果你可以通过手机号码或电子邮件地址登录，文本输入框可以在识别出你正在输入什么后改变其格式。应用还可以更改周围的文本和图标。
*   当用户输入银行卡号时，在输入前几位数字后显示卡组织（Visa、MasterCard 或其他）的徽标是一个好做法。有些应用甚至能显示发卡银行的徽标。当你使用这样的应用时，你会感到更加自信，并觉得方向正确。
*   当用户填写表单并出错时，应用通常会高亮显示有问题的字段。当用户开始输入时，应用应移除警告或错误提示。
*   当用户输入正确的电话号码、银行账号、纳税人识别号或其他任何可以验证的数据时，某些应用会显示成功图标和/或将焦点移到下一个输入框。
*   密码输入板。我们稍后会讨论，但密码输入板需要实时输入处理。

对我们来说，关键函数是：

```
optional func textField(_ textField: UITextField,
shouldChangeCharactersIn range: NSRange,
replacementString string: String) -> Bool
```

看起来有点复杂，但这个回调函数功能极其强大。它提供了我们所需的所有信息：

*   `textField` 是发生编辑的 `UITextField` 对象（或 `UITextField` 的子类）。
*   `range` 显示从文本输入中*移除*的符号范围。如果其长度为零，则仅是插入位置。
*   `string` 包含新的字符。如果用户点击了屏幕键盘上的一个键，它可能是一个字符；如果用户从剪贴板粘贴了文本，则可能是一个完整的文本字符串。

*方法*返回一个布尔值。如果返回 `true`，则“替换”（或插入）将发生；否则，输入将被阻止。

缺少两个部分：

*   更改前的文本
*   更改后的文本

*更改前的文本* 很简单，就是 `textField.text`。*更改后的文本* 则需要某种“计算”（*配方* *6-1* *）*。

请注意，你必须非常小心这个方法的返回值。例如，如果你只在验证通过时返回 `true`，那么用户甚至连第一个符号都无法输入。

通常，这个代理函数内部的代码应处理三种情况：

*   输入的文本无效，根本不应被输入。
*   输入的文本有效，但结果无效。
*   一切有效；我们可以向用户表明他们输入正确。

只有在第一种情况下才应返回 `false`，例如，当用户输入电话号码，而 `string` 参数包含字母或表情符号时。

第二个重要注意事项——*始终* 允许退格或任何其他删除操作。如果由于错误操作，你的文本输入框中出现了一个*非法*字符，用户应该能够将其删除。因此，如果 `string` 参数为空，它永远不应返回 `false`。
