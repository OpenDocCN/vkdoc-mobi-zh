# 你好，Flutter

请在你的脑海中想象这样一幅画面：你是一家新公司的 CEO。显然，你的使命是在最小化成本的同时最大化销售额。“嗯，”你思索着，“如果我把产品放到网上，确实能大幅提升销量。”于是你向朋友们请教如何制作一个网页应用，他们回答说……

“你需要雇一个网页开发者。他们必须掌握`HTML`、`CSS`、`JavaScript`，很可能还需要某个框架，比如`React`、`Vue`或`Angular`。”

这虽然开销不菲，但你还是照做了，并且你的冒险得到了回报。销售额显著增长。为了紧跟需求，你时刻关注社交媒体并与客户互动。你听到他们说这个网页应用很棒，但是“如果你们在 App Store 里有个应用，我们早就来了。”于是你找到你的团队——他们虽然是网页专家，但并非 iOS 开发者。他们告诉你……

“你需要雇一个 iOS 专家。他们必须精通 iOS、`Swift`或`Objective-C`、`Xcode`、`macOS`以及用于开发的`CocoaPods`。”

你的调研显示，这类人才比你的网页开发者更为*专业*，因此也更昂贵。但再一次，这似乎是正确的选择，所以你咬咬牙雇用了他们。然而，就在这个应用还在开发时，你发现客户的反馈并不仅仅针对 iOS 应用，而是关注所有移动设备。而且——哦，糟糕！——全球 85%的设备运行的是 Android，而非 iOS。你双手抱头，纠结着是否能够承受忽略 85%潜在客户的代价。你的顾问告诉你……

“你需要雇一个 Android 专家。他们必须了解 Android 操作系统、`Gradle`、`Android SDK`、`XML`、`Android Studio`以及`Java`或`Kotlin`。”

“真的吗？！还要再招一个开发者？”你问道。“是的，而且费用和你那位 iOS 开发者一样高，”他们回答。

难道就没有一个人能同时胜任这三项工作吗？有没有办法在这些不同的环境之间共享代码？那样你只需要雇用一个人。事实上，他们可以一次编写代码，然后将其部署到网页、App Store 和 Google Play Store。只需维护一套代码库。在一个地方进行改进和升级。在一个地方修复缺陷。

女士们，先生们，请允许我向你们介绍 Flutter！

## Flutter 是什么？

Flutter 是一套工具集，它使我们能够创建在 iOS、Android、网页、Windows、MacOS 以及 Linux 桌面系统上运行的漂亮应用。

Flutter 是……

*   免费（如同免费啤酒；不收费）
*   开源（这是“免费”这个词的另一种含义）
*   由谷歌支持并起源于谷歌
*   由谷歌的开发团队和全球数百名非谷歌贡献者共同增强和维护
*   目前被全球众多组织的数千名开发者用于生产环境应用
*   速度飞快，因为它编译成真正的原生应用，无需依赖像`WebViews`和 JavaScript 桥接这样的辅助工具
*   一次编写，然后编译成面向数十亿浏览器的网页应用、面向 iPhone 和 iPad 的 iOS 应用，以及面向其他所有手机和平板的 Android 应用

## 为什么选择 Flutter？

谷歌对 Flutter 的使命是……

> *为移动端开发打造更优方式*

请注意这个使命中*没有*提到什么。它既没有提及 Android（同样归谷歌所有），也没有提及 iOS 或网页。Flutter 的目标是为所有设备创造一种更好的开发方式。换句话说，Flutter 应该比`Swift`更适合创建 iOS 应用。它应该比`Kotlin`更适合创建 Android 应用。它应该比`HTML`/`JavaScript`更适合创建网页应用。而如果你能用一套代码库同时实现所有这些，那就更好了。

Flutter 团队已经出色地完成了这一使命。

作为证明，Eric Seidel 提供了这样一个例子^(¹)：谷歌 CRM 团队使用 Flutter 构建了一个内部 Android 应用，速度比使用他们传统的 Android 工具链快了**三倍**！

> **注意**
>
> Flutter 已经兑现了创建网页应用和桌面应用的承诺。但我们必须指出，相对而言，选择使用 Flutter 开发这些应用的人并不多。Flutter 真正成功的领域是 iOS 和 Android 应用。请将其他平台看作是锦上添花，而非主菜。

但事实证明，Flutter 并不是跨平台开发的唯一选择。你还有其他选项。

## 其他选项

跨平台开发通常分为三大类，如表 1-1 所示。

**表 1-1** 跨平台开发分类

| | 部分技术 | 缺点 | 优点 |
| --- | --- | --- | --- |
| 渐进式网页应用 (PWA) | `HTML`/`CSS`、`React`、`Vue`、`Angular` | 不是真正的应用。在网页浏览器中运行。不在应用商店中提供。难以创建桌面快捷方式。无法访问设备的许多资源，如后台处理、指南针、接近传感器、蓝牙、NFC 等 | 易于编写 |
| 混合应用 | `PhoneGap`、`Cordova`、`Sencha`、`Capacitor`、`Ionic` | 在`WebView`中运行，因此可能速度较慢。几乎不可能与网页应用共享代码 | 网页开发者更容易学习，因为它使用 HTML 和 JavaScript 作为语言和结构 |
| 编译为原生方案 | `React Native`、`NativeScript`、`Flutter`、`Maui` | 学习框架可能比较困难。掌握工具链绝对是个挑战 | 是真正的应用，可以在应用商店中找到并且运行速度快 |

如果你的用户群体别无选择，他们非常看重你的应用以至于愿意接受较差的用户体验，那么最便宜的方案是创建一个 PWA。如果你的应用极其简单且速度预计不成问题，那么混合方案可能适合。但如果速度、流畅度和复杂功能很重要，你将需要选择原生方案。

## 原生方案

截至目前，有四个相当流行的编译为原生方案（表 1-2）。

**表 1-2** 编译为原生的跨平台框架

| | Maui | NativeScript | React Native | Flutter |
| --- | --- | --- | --- | --- |
| 推出年份 | 2011 | 2014 | 2015 | 2018 |
| 支持方 | 微软 | Telerik | 脸书 | 谷歌 |
| 界面呈现语言 | XAML | 专有但类似 XML | 专有但类似 JSX | `Dart` |
| 过程式语言 | `C#` | `JavaScript` | `JavaScript` | `Dart` |

这些都是相当不错的选择。它们都可免费用于开发，且经过了充分测试，已有大量生产应用被创建出来。它们都已被大型组织所使用。

Flutter 是这些框架中最新发布的。正因如此，它在观察前驱者方面拥有独特的优势。Flutter 团队留意了其他框架中哪些做得好、哪些失败了。此外，Flutter 还加入了新的创新和理念——所有这些从一开始就内置其中，而不是事后作为改进修补上去的。

不过，我猜想既然你买了这本书，可能不需要我过多说服，那么我就此打住。只需说一句：Flutter 太棒了！它易于编写、优雅且设计精良——用它编码绝对是一种享受^(²)。
