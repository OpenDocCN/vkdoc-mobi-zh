## 批处理

尽可能多地批处理那些依赖相同状态的操作，因为改变系统状态（通过使用 `glEnable()` 和 `glDisable()` 调用）的代价很高。OpenGL 不会在内部检查某个特定功能是否已经处于你期望的状态。在本书的练习中，我经常比实际需要更频繁地设置状态，只是为了确保行为易于预测。但对于商业化的、性能要求高的应用，请尽量在发布版本中移除任何多余的调用。

同时，也请尽可能多地批处理你的绘制调用。

## 纹理

一些纹理优化技巧已经讨论过，例如第 5 章中的 Mipmap 技术以及利用 PVRTC 纹理。其他则是一些常识：纹理占用大量内存。请尽可能让它们保持小巧，并在需要时重复使用。此外，在加载纹理前就设置好图像参数，因为它们就像一个提示，能告诉 OpenGL 如何在将数据传输到硬件之前优化信息。

先绘制不透明的纹理，并避免使用半透明的 OpenGL ES 屏幕。

作为一个关于纹理如何影响帧率的简单例子，之前的 VBO 示例在完全移除纹理后，性能提升了大约 25%。

## 精灵图表

精灵图表（或纹理图集、字体图集）在第 8 章讨论 OpenGL 环境中显示文本时曾简要提及。图 9-4 展示了一个用于在屏幕上渲染文本的精灵图表是什么样子。

**图 9-4. 24 号铜板体的精灵图表**

这张特定的图片是使用一个名为 LabelAtlasCreator 的免费工具创建的。除了图像文件外，它还会生成一个便捷的 plist 文件，其中包含了所有易于转换为纹理空间的放置细节。另一个工具是基于 Java 的 Hiero，它可以处理带渐变的字体。不过，它的速度非常慢。也许最强大的工具是“Angelcode-Bitmap Font Generator”，这个工具仅适用于 Microsoft Windows，但它似乎非常受欢迎。

后两个工具会生成一个 `.font` 格式的字体度量文件。它不如 `.plist` 文件方便，但网上有一些免费的解析器可以用来处理它。

但不要局限于字体，因为精灵图表也可以应用于任何一组风格相似的图像。从一个纹理切换到另一个纹理可能会产生大量不必要的开销，而精灵图表则作为一种纹理批处理形式，节省了大量的 GPU 时间。可以试试与 OS-X 兼容的 Zwoptex 工具，或者用于通用图像的 TexturePacker。

## 纹理上传

将纹理复制到 GPU 的代价可能非常高昂，因为它们在能被使用之前，必须先经过 GPU 芯片的重新格式化。对于较大的纹理，这可能会导致显示时出现卡顿。因此，请确保在程序启动时就通过 `glTexImage2D` 加载它们。

## Mipmap

除了 2D 非缩放图像外，务必确保对其他所有图像都使用 Mipmap。是的，它们确实会多用一点内存，但当一个物体在远处时，较小的 Mipmap 可以为你的物体节省大量的处理周期。建议你使用 `GL_LINEAR_MIPMAP_NEAREST`，因为它比 `GL_LINEAR_MIPMAP_LINEAR` 速度更快（尽管图像保真度稍差一些）。
