# 第 1 章：计算机图形学：从过去到现在

20 世纪 90 年代带来了《终结者 2：审判日》中的 T1000“液态金属”终结者、首部完全由计算机生成的长篇动画电影《玩具总动员》、《侏罗纪公园》中逼真的恐龙动画，以及詹姆斯·卡梅隆的《泰坦尼克号》，所有这些都帮助巩固了 CG 作为好莱坞导演工具箱中常用工具的地位。

到 90 年代末，几乎很难找到哪部电影在制作过程中没有使用计算机图形学——无论是用于实际效果，还是后期制作中帮助清理各种场景。新技术仍在不断开发中，并以更加壮观的方式得到应用，例如迪士尼精彩的《飞屋环游记》或詹姆斯·卡梅隆绝美的《阿凡达》。

现在，再次拿出你的 i 设备，意识到它是一个多么小巧的技术奇迹吧。请随意用低沉而充满敬意的语气说出“哇”。

## 工具包

如果没有软件，前面提到的所有 3D 魔法都不可能实现。许多 CG 软件程序高度专业化，而其他一些则更为通用，例如本书的重点 `OpenGL ES`。那么，以下是众多可用工具包中的几个。

### OpenGL

开放图形库（`OpenGL`）源于高端图形工作站和大型机制造商 `SGI` 的开创性努力。其专有的图形框架 `IRIS-GL` 已发展成为整个行业的事实标准。为了在竞争加剧时留住客户，`SGI` 选择将 `IRIS-GL` 转变为开放框架，以巩固其作为行业领导者的声誉。`IRIS-GL` 被剥离了非图形相关功能和硬件相关特性，更名为 `OpenGL`，并于 1992 年初发布。截至撰写本文时，4.1 版本是最新版本。

随着小型手持设备变得越来越普遍，`OpenGL for Embedded Systems`（`OpenGL ES`，嵌入式系统开放图形库）被开发出来，它是桌面版的一个精简版本。它移除许多更冗余的 `API` 调用，同时简化了其他元素，使其能够在低功耗 CPU 上高效运行。因此，它已被广泛采用于众多平台，如 Android、iOS、任天堂 3DS 和黑莓（操作系统 5.0 及以上版本）。

`OpenGL ES` 主要分为两个版本：1.x 和 2.x。许多设备同时支持两者。1.x 是基于原始 `OpenGL` 规范的高级变体。2.x 版本（是的，我知道这容易混淆）则针对可以通过可编程图形硬件处理的更专门化的渲染任务。

### Direct3D

`Direct3D`（`D3D`）是微软为应对`OpenGL`而推出的解决方案，主要面向游戏开发者。1995 年，微软收购了一家名为 RenderMorphics 的小公司，该公司专门开发用于编写游戏的 3D 框架`RealityLab`。`RealityLab`后来被改造为`Direct3D`，并于 1996 年夏季首次发布。尽管它是 Windows 系统的专有技术，但在微软的所有平台（Windows、Windows 7 Mobile、甚至 Xbox）上拥有庞大的用户群体。`OpenGL`与`Direct3D`阵营之间一直争论不休，讨论哪个更强大、更灵活、更易用。其他影响因素还包括：硬件制造商更新驱动以支持新功能的速度、易理解性（`Direct3D`使用微软的`COM`接口，对新用户来说可能非常困惑）、稳定性以及行业支持。

## 其他竞争者

尽管`OpenGL`和`Direct3D`在采用率和功能方面稳居榜首，但图形领域还遍布着众多其他框架，其中许多在现代设备上仍受支持。

在计算机图形学领域，图形库大致分为两大类：以`OpenGL`和`Direct3D`为代表的底层渲染机制，以及通常在游戏引擎中发现的高级系统，后者专注于资源管理，并附带扩展到常见游戏元素（如音效、网络、计分等）的特殊功能。后者通常基于前者中的一种来构建 3D 部分。如果设计得当，高级系统甚至可能具备充分的抽象能力，从而能够兼容使用`GL`和`D3D`。

### QuickDraw 3D

高级通用库的一个例子是`QuickDraw 3D`（`QD3D`）。作为苹果 2D 库`QuickDraw`（用于前 OS-X 时代）的 3D 兄弟，`QD3D`提供了一种优雅的方式来以易于理解的层级结构（场景图）生成和链接对象。它同样拥有用于加载 3D 模型的自有文件格式和标准查看器，并且是平台无关的。`QD3D`的高级部分会计算场景，并确定每个对象乃至对象的每一个部分如何在 2D 绘图表面上显示。在`QD3D`之下，有一个名为`RAVE`的极薄层，负责处理这些位图的设备特定渲染。

用户可以使用标准版的`RAVE`，它会按预期渲染场景。但更有野心的用户可以编写自己的`RAVE`，以更具艺术性的方式显示场景。例如，有一家公司生成的`RAVE`输出效果，使其对象看起来像手绘在洞穴墙壁上。当你将这个现代版的洞穴壁画旋转起来时，效果非常酷。这种插件架构也使得`QD3D`具有很高的可移植性，能够移植到其他机器上。当潜在用户因`QD3D`在 PC 上缺乏硬件解决方案而拒绝使用时，人们开发了一个`RAVE`版本，该版本通过实际使用其竞争对手作为光栅化器，来利用`Direct3D`可用的硬件加速功能。遗憾的是，在史蒂夫·乔布斯回归后，`QD3D`几乎立即被砍掉，他认定`OpenGL`应该成为 Mac 未来 3D 标准。这个说法很奇怪，因为`QD3D`并非竞争对手，而是一个让程序员生活更轻松的附加组件。在乔布斯拒绝将`QD3D`开源后，Quesa 项目成立，旨在尽可能多地重建原始库，截至本文撰写时，该项目仍在维护。毫不意外的是，Quesa 使用`OpenGL`作为其渲染引擎。

需要声明的是：我编写了`QD3D`的`RAVE`/`Direct3D`层，结果项目在“金版”（准备发货）几天后就取消了。

### OGRE

另一个场景图系统是面向对象的渲染引擎（`OGRE`）。`OGRE`于 2005 年首次发布，它可以同时使用`OpenGL`和`Direct3D`作为底层光栅化解决方案，同时为用户提供一个稳定且免费的工具包，被用于许多商业产品中。其用户社区规模巨大。快速浏览论坛会发现，截至本文撰写时，仅“一般讨论”板块就有超过 6,500 个主题。

### OpenSceneGraph

最近发布的 iOS 版本`OpenSceneGraph`大致实现了`QuickDraw 3D`的功能：提供在更高层级创建对象、将它们链接在一起、管理场景以及在`OpenGL`层之上进行特殊效果处理的方法。其他功能包括导入多种文件格式、文字支持、粒子效果（用于火花、火焰或云朵），以及在 3D 应用程序中显示视频内容。强烈建议具备`OpenGL`知识，因为许多`OSG`函数仅仅是其对应`OpenGL`函数的薄封装。

### Unity3D

与`OGRE`、`QD3D`或`OpenSceneGraph`不同，`Unity3D`是一个成熟的游戏引擎。区别在于产品的范围。前两者专注于在`OpenGL`周围创建更抽象的封装，而游戏引擎则更进一步，提供了游戏通常所需的大部分（如果不是全部）其他支持功能，如音效、脚本、网络扩展、物理引擎、用户界面和计分模块。此外，一个好的引擎可能还会包含帮助生成资源的工具，并且是平台无关的。

`Unity3D`具备所有这些功能，因此对于许多小型项目来说可能过于臃肿。同时，作为商业产品，其源代码不可获取，使用也非免费，需要支付适中的费用（与过去一些产品可能收取 10 万美元或更多相比）。

## 还有其他

我们也不能忽视`A6`、`Adventure Game Studio`、`C4`、`Cinder`、`Cocos3d`、`Crystal Space`、`VTK`、`Coin3D`、`SDL`、`QT`、`Delta3D`、`Glint3D`、`Esenthel`、`FlatRedBall`、`Horde3D`、`Irrlicht`、`Leadwerks3D`、`Lightfeather`、`Raydium`、`Panda3D`（来自迪士尼工作室和卡内基梅隆大学）、`Torque`（可用于 iOS）以及许多其他框架。尽管它们功能强大，但使用游戏引擎的一个缺点是，你的世界通常是在它们的环境中执行的。因此，如果你需要某种特定的、不支持的细微行为，可能就无能为力了。这让我回到了本书的主题。

## 回到两个立方体的华尔兹

直到 iOS4，苹果都将`OpenGL`视为更通用的框架。但从 iOS5 开始，他们希望强调它是游戏开发的理想环境。这就是为什么，例如，向导中的项目图标被命名为“OpenGL Game”，而之前是“OpenGL ES Application”。这也解释了为什么示例练习推崇性能更好但相当繁琐的`OpenGL ES 2`环境，而忽略了本书主题所涉及的更简单的版本。

**注意** 同样从 iOS5 开始，苹果在其新的`GLKit`框架中添加了许多特殊的辅助对象，接管了开发者早期必须自行完成的一些常见任务。这些任务包括图像加载、3D 相关的数学运算、创建特殊的`OpenGL`视图以及管理特殊效果。

考虑到这一点，我会时不时地踏入 2.0 领域，比如通过下面描述的示例应用，因为目前我们只有这些。关于 2.0 的详细讨论将留到最后一章，因为它对于本书的范畴来说确实是一个相当高级的话题。

着色器文件是 OpenGL ES 2.0 所独有的，用于微调场景的外观。它们就像是运行在显卡自身上的小型且快速的程序，使用一种类似 C 语言的独特语言。它们赋予你精确指定光线和纹理在最终图像中如何呈现的能力。不幸的是，OpenGL ES 2.0 要求使用着色器，因此学习曲线相对陡峭；而更简单且使用更广泛的 1.1 版本则不使用着色器，仅采用一些标准的光照和着色效果（称为“固定功能”管线）。基于着色器的应用程序最有可能是那些视觉体验与任何其他方面同等重要的游戏，而更简单的 1.1 框架则非常适合简单的游戏、商业图形、教育软件或任何其他不需要完美大气建模的应用。
