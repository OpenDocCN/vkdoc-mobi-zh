# 高级事件处理

我想提及几个高级事件处理主题，并附上一些实用建议。先从建议开始。

**确保事件处理的及时性**。正如你现在所知，你的应用通过主线程的运行循环来“保持活跃”。该运行循环将所有内容传递给应用：触摸事件、通知、用户界面更新等等。应用处理的每个事件、操作和消息都必须在下一个事件处理之前执行并返回。这意味着，如果你编写的任何代码耗时过长，应用就会看起来像“死”了一样。如果代码耗时*非常非常长*才完成，应用就会彻底“死亡”——iOS 会终止你的应用，因为它已经停止响应事件了。

我敢肯定你遇到过应用“锁死”的情况：屏幕冻结，无法响应触摸、摇晃或任何操作。这就是当应用的运行循环忙于处理其他事务而不处理事件时发生的事情。这体验可不好。大多数可能耗时较长的 iOS 功能都提供了异步方法（比如你在 Shorty 中使用过的那些），这样耗时的任务就不会阻塞你的主线程。请使用这些异步方法，注意程序执行各项操作所需的时间，并准备好重构应用以避免“锁死”运行循环。我将在后续章节中演示所有这些技术。

其次，处理多点触摸事件可能很棘手，甚至令人困惑。iOS 会尽力理清触摸事件的复杂性，并以合理且易于理解的方式将它们呈现给你的对象。iOS 提供了五项功能来简化触摸事件处理：

-   手势识别器
-   过滤掉其他视图的触摸事件
-   禁止多点触摸事件
-   提供独占触摸事件处理
-   暂停触摸事件

手势识别器是特殊的对象，它们代表视图对象拦截触摸事件。每个识别器检测特定的触摸手势，从简单的点击到复杂的多指滑动。如果检测到它被编程识别的手势，它会发送一个动作——就像你在早期项目中使用过的按钮对象一样。你只需要将该动作连接到 Interface Builder 中的一个对象即可完成。仅此一项功能就已经为 iOS 开发者节省了数万行触摸事件处理代码。我将在后续章节向你展示如何使用手势识别器对象。

正如我之前所述，触摸事件方法（例如 `-touchesBegan:withEvent:`）仅在 `touches` 参数中包含相关的触摸对象——那些源自你的视图对象的触摸。你的代码不必担心其他视图中可能同时发生的其他触摸。在 Touchy 应用中，这实际上是一个缺点，你不得不从 `UIEvent` 对象中挖掘出全局的触摸对象集合。但通常来说，你只需关注自己视图中的触摸事件。

你也已经看到 iOS 如何使用 `UIView` 的 `multipleTouchEnabled` 属性来禁止多点触摸事件。如果此属性为 `NO`，即使有多根手指同时触摸你的视图，iOS 也只会向你的视图对象发送与第一个触摸相关的事件。为了让 Touchy 应用获取所有触摸事件，你必须将此属性设置为 `YES`。如果你的视图只解释单次触摸事件，请将此属性设置为 `NO`，这样你就不必编写任何需要担心同时有多个触摸的代码了。

如果你不希望 iOS 同时向两个视图对象发送触摸事件，可以将 `UIView` 的 `exclusiveTouch` 属性设置为 `YES`。一旦设置，一旦你的视图中开始了触摸序列，iOS 就会阻止向任何其他视图发送触摸事件（反之亦然）。

最后，如果你的应用需要，你可以临时暂停向特定视图甚至整个应用发送所有触摸事件。如果你想让某个单独的视图对触摸事件“失聪”，请将其 `userInteractionEnabled` 属性设置为 `NO`。你也可以向你的应用对象发送 `-beginIgnoringInteractionEvents` 消息，这样你应用的所有触摸事件都将被静音。通过发送 `-endIgnoringInteractionEvents` 消息可以重新启用它们。这对于防止触摸事件干扰正在进行的其他操作（比如一个短暂的动画）很有用，但不要长时间让它们处于关闭状态。

## 总结

至此，你应该对消息和事件如何进入你的应用以及它们是如何被处理的有了相当扎实的理解。你了解了事件队列和运行循环。你知道了队列中的事件会被分派给应用中的对象。你明白了其中一些事件会直接发送给你的对象，触摸事件使用命中测试，而其余事件则被发送给第一响应者。

你已经学到了很多关于响应者链的知识。除了传递事件之外，响应者链在 iOS 中执行着许多重要的任务。

你知道了如何配置一个对象来处理或忽略特定类型的事件。你编写了两个应用，一个处理高级事件，另一个追踪低级触摸事件。

或许更令人惊叹的是，你在真实的 iOS 设备上构建并运行了你的应用！你也可以随意在设备上运行其他任何项目。创建一款可以随身携带的、属于你自己的 iOS 应用，是一项非常了不起的成就！

### 练习题

根据魔法八球的说明，你不应该摇晃它；这会导致液体中产生气泡。当然，这从未阻止我和我兄弟把它晃得天翻地覆。相反，你应该把球（“8”面朝上）放在桌子上，问一个问题，轻轻翻转它，然后读取答案。

为了获得额外加分，请重写 EightBall 应用，使其使用设备方向事件（而不是摇晃运动事件）来让消息消失和出现。设备的方向可以是纵向、左侧横向、右侧横向、倒置、屏幕朝上或屏幕朝下。

设备方向的变化通过通知传递。你之前还未使用过通知，但可以将其视为另一种事件（至少在此上下文中如此）。与事件不同，你的对象必须显式请求它想要接收的通知。每当设备方向改变时（例如，当用户翻转他们的 iPhone 时），你的对象将收到一条通知消息。

请求和处理设备方向变化通知所需的全部代码，在《iOS 事件处理指南》的“使用 `UIDevice` 获取当前设备方向”部分中均已给出。在 Xcode 中，选择“帮助”>“文档和 API 指南”，然后搜索“Event Handling Guide”。

修改 EightBall 应用，使其请求设备方向通知，而不是处理摇晃运动事件。当你的应用收到方向变化通知时，检查当前 `UIDevice` 对象的当前方向。如果方向属性是 `UIDeviceOrientationFaceUp`，则使一条新消息出现。如果是其他任何方向，则使消息消失。现在你拥有了一个更“正宗”的魔法八球模拟器！你可以在 `EightBall E1` 文件夹中找到我对这道练习题的解答。
