# 实物教学

摘要

我想暂停一章应用开发。优秀的 iOS 开发需要具备超越仅会编写`for`循环或连接按钮到输出口的概念和设计技能。软件工程师将这些称为设计模式与设计原则。为了领会这些理念，我将从所有这一切的基础讲起：对象。

“嘿！”你说，“我一直在使用对象，这有什么难理解的？”你会惊讶地发现，有很多程序员无法准确描述什么是“对象”。如果你到目前为止对本书中使用的术语（类、对象、实例、方法、消息等）没有任何疑问，并且你已经熟悉设计模式和原则，可以跳过或略读本章。如果你有疑问，请继续阅读。

在本章中我会：

* 简要介绍对象及面向对象编程的历史
* 准确解释类、对象和实例是什么
* 描述继承和封装
* 解释委托及其他几种设计模式
* 简要触及几个关键设计原则

要理解对象，先了解它们出现之前的情况以及它们为何如此重要，会大有帮助。

## 两家世仇，同样尊贵

计算机内部存在两种基本类型的信息。**数据**是表示数值和数量的二进制值，例如你的名字、一个 URL，或者一只未载重燕子的空速。**代码**是表示指令的二进制值，这些指令告诉计算机执行诸如在屏幕上绘制你的名字、加载一个 URL 或选择一种喜欢的颜色等操作。

在计算机语言中很容易看到这种划分。像 Objective‑C 这样的编程语言的语法，主要分为定义、检索和存储值的语句，以及改变值、做出决策和调用其他方法的语句。你可以将它们视为计算机语言的**名词**和**动词**。

就像蒙太古家族和凯普莱特家族一样，¹ 编程的这两个方面在很长一段时间内都保持分离。随着计算机变得更大更快，计算机程序变得更长更复杂，一系列问题开始显现。

程序员遇到了更多需要将多条信息捆绑在一起的解决方案。一个人不仅有名字，还有身高、年龄、税号等等。为了将这些相关的事实组合在一起，他们开始将多个值合并到一个称为**结构体**的内存块中。在 C 编程语言中，一个结构体看起来像这样：

```
struct Person {
    char name[kLongestName];
    bool female;
    float birthdate;
    float height;
    int taxNumber;
};
```

这些结构体变得非常方便，以至于程序员开始像使用单个值一样使用它们。他们会将`Person`传递给函数，或者将`Person`存储到文件中。他们会编写专门对`Person`结构体（而非像日期这样的单个值）进行操作的函数，例如编写一个判断是否为该人生日的函数，如`IsBirthdayToday( struct Person *person )`。

程序员也开始遇到许多结构体彼此相似的情况。`Player`结构体拥有`Person`结构体的所有属性，只是它包含更多变量，例如玩家的总得分。他们很快发现，可以用结构体来创建结构体，像这样：

```
struct Player {
    struct Person person;
    int gamesPlayed;
    int totalScore;
};
```

真正让程序员兴奋的是，现在他们可以为`Person`结构体编写的函数同样适用于`Player`结构体！他们甚至给这个想法起了个名字：**子类型多态**。如果你在下一次聚会中能把这个概念融入对话，你会获得额外加分。

事情本该一帆风顺，但事实并非如此。结构体和函数的数量以令人眩晕的速度增长。项目中会有成千上万个独立的函数和几乎同样多的不同结构体。有些函数能处理`Person`结构体，但大多数不能。

问题在于，数据结构体与函数仍分属不同阵营；它们并未融合。试图弄明白哪些函数应该与哪些结构体一起使用变得难以管理。程序无法正常工作。大型软件项目接连失败。必须要做出改变——也确实发生了改变。
