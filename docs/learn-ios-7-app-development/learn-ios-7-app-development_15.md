# 运行循环

iOS 应用会静止不动，等待某些事件的发生。这是应用设计的一个非常重要的特性，因为它能使应用保持高效：你的应用代码仅在需要处理重要事务时才会运行。

这种看似无害的机制，对于保持用户满意度至关重要。运行计算机代码需要消耗电量，而移动设备中的电量是一种宝贵资源。防止代码在不必要时运行，可以让 iOS 节省电力。具体做法是，在 CPU 和其他硬件配件未被使用时，关闭或最小化它们的耗电量。这种电源管理每秒会发生数百次，但它对移动设备的电池续航至关重要，而用户热爱电池续航持久的移动设备。

你的应用代码位于两种机制的最末端：运行循环和事件队列。运行循环在事件发生时执行应用代码，在无事可做时则让应用停止运行。事件队列是一种数据结构，其中包含有待处理的事件列表。只要队列中有事件，运行循环就会依次将它们发送给应用。一旦所有事件处理完毕，事件队列为空，应用就会停止执行代码。

从概念上讲，应用运行循环的结构如下所示：

```
while ( true ) {
    UIEvent *event = [iOS waitForNextEvent];
    [yourApp processEvent:event];
}
```

其精髓在于 `-waitForNextEvent` 消息（这是个虚构方法，并不真实存在）。如果有事件等待处理，该事件会从队列中取出并返回。运行循环将其传递给应用进行处理。如果没有事件，该函数就不会返回；应用会挂起，直到有事可做。现在，我们来探讨这些事件是什么以及它们来自何处。

## 事件队列

待处理的事件会被添加到一个先进先出（FIFO）缓冲区中，这个缓冲区称为事件队列。事件有多种类型，并来自不同的来源，如图 4-1 所示。

图 4-1. 事件队列

我们追踪一个事件在应用中的完整路径。当你的手指触摸 iOS 设备的屏幕时，会发生以下步骤：

- 屏幕硬件检测触摸的位置。
- 利用这些信息创建一个触摸事件对象，该对象记录了触摸的位置、发生的时间及其他信息。
- 触摸事件对象被放入应用的事件队列中。
- 运行循环从队列中取出触摸事件对象，并将其传递给应用对象。
- 应用对象利用应用中活跃视图的几何信息，确定你的手指“触摸”了哪个视图。
- 一条包含触摸事件的消息被发送给该视图对象。
- 视图对象决定该触摸事件的含义以及它将执行的操作。它可能高亮一个按钮，或发送一条操作消息。

当你在第 3 章的 Shorty 应用中点击“缩短链接”按钮时，正是通过上述过程，触摸屏幕的物理行为才转化为视图控制器接收到的 `-shortenURL:` 消息。

不同类型的事件走不同的路径。接下来几个小节将介绍不同的传递方式，以及每种方式所对应的事件类型。

## 事件传递

事件传递是指事件如何从事件队列到达应用中的某个对象。不同类型的事件根据其用途，遵循不同的路径。实际的传递机制是 Cocoa Touch 框架、应用对象以及应用对象定义的各种方法中逻辑的组合。

大致来说，有三种传递方式：

- 直接传递
- 命中测试
- 第一响应者

接下来几个小节将分别描述这三种方式，以及通过每种方式传递的事件。

### 直接传递

直接传递是最简单的事件传递形式。许多事件类型会指定目标对象。这些事件知道哪些对象将接收它们，因此除了它们是由运行循环分派这一事实外，关于这些事件如何传递，并没有太多需要了解的。

例如，一个 Objective-C 消息可以被放入事件队列。当该事件从队列中取出时，消息会被发送给其目标对象。网页视图就是这样告诉你的 Shorty 应用网页已加载完成的。当网络通信代码（在其自己的线程中运行）确定页面加载完毕时，它会将一条 `-webViewDidFinishLoad:` 消息推送到主线程的事件队列中。当主线程从事件队列中取出事件时，其中一个事件会将该消息发送给你的网页视图代理对象，告知页面已加载完成。

**注意：** 这并非异步代理消息传递的确切方式。但对于你——应用开发者而言，从概念上讲是准确的；具体细节并不重要。

其他发送给特定对象或对象组的事件包括通知、定时器事件和用户界面更新。所有这些事件都直接或间接地知道它们将被发送给哪些对象。作为应用开发者，你只需知道：当这些事件在事件队列中排到队首时，运行循环会向一个或多个对象发送一条 Objective-C 消息。



### 命中测试

命中测试基于用户界面的几何形状来传递事件，并且仅适用于触摸事件。当触摸事件发生时，`UIWindow`和`UIView`对象会协同工作，以确定哪个视图对象对应于触摸的位置。然后，消息会被发送到该视图对象，该对象可以按照自己的方式解释这些事件；它可能翻转开关、滚动购物清单或炸毁宇宙飞船。让我们快速了解一下命中测试的工作原理。

当一个触摸事件从事件队列中取出时，它包含触摸发生时的绝对硬件坐标，如图 4-2 左侧所示。此示例将使用上一章中 Shorty 应用程序的程式化表示。

![A978-1-4302-5063-0_4_Fig2_HTML.jpg](img/A978-1-4302-5063-0_4_Fig2_HTML.jpg)

**图 4-2.** 对触摸事件进行命中测试

您的 `UIApplication` 对象使用事件坐标来确定负责屏幕该部分的 `UIWindow` 对象。该 `UIWindow` 对象会收到一条包含待处理触摸事件对象的 `-sendEvent:` 消息。

然后，`UIWindow` 对象执行命中测试。从其视图层次结构的顶部开始，它向其顶层视图对象发送一条 `-hitTest:withEvent:` 消息，如图 4-2 的第二个面板所示。

顶层视图首先判断事件是否在其边界内。确实如此，于是它开始寻找包含触摸坐标的任何子视图。顶层视图包含三个子视图：顶部的导航工具栏、中间的网页视图和底部的工具栏。触摸点在工具栏的边界内，因此它将该 `-hitTest:withEvent:` 消息传递给工具栏。

工具栏重复此过程，寻找包含该位置的子视图，如图 4-2 的第三帧所示。工具栏对象发现触摸发生在其最左侧的栏按钮项边界内。该栏按钮项被作为“命中”对象返回，这导致 `UIWindow` 开始向其发送低级触摸事件消息。

作为一个“按钮”，栏按钮项对象会检查事件以确定用户是否点击了该按钮（而不是滑动它或其他无关手势）。如果是，按钮会发送其操作消息，在此例中是 `-shortenURL:`，发送给它所连接的对象。

**提示：** 如果您有需要，命中测试是高度可定制的。通过重写视图对象的 `-pointInside:withEvent:` 和 `-hitTest:withEvent:` 方法，您可以真正重写确定触摸事件如何找到将要发送到的视图对象的规则。有关详细信息，请参阅 iOS 事件处理指南，该指南位于 Xcode 的文档和 API 指南中。

### 第一响应者

第一响应者是活跃界面（一个可见窗口）中的视图、视图控制器或窗口对象。可以将其视为未由命中测试确定的事件的指定接收者。我将在本章后面讨论一个对象如何成为第一响应者。现在，只需知道每个活跃界面都有一个第一响应者。

传递给第一响应者的事件包括：

- 摇动动作事件
- 远程控制事件
- 按键事件

摇动动作事件告诉您的应用用户正在摇晃他们的设备（来回快速移动）。此信息来自加速度计硬件。

所谓的远程控制事件在用户按下任何多媒体控件时生成，这些控件包括：

- 播放
- 暂停
- 停止
- 跳至下一曲
- 跳至上一曲
- 快进
- 快退

这些被称为“远程”事件，因为它们可能源自外部配件，例如许多耳机线材上的播放/暂停按钮。实际上，它们最常见的是来自您在屏幕上看到的播放/暂停按钮。

按键事件来自点击虚拟键盘，或通过蓝牙连接的硬件键盘。

回顾一下，直接传递将事件对象或 Objective-C 消息直接发送给它们的目标对象。触摸事件使用命中测试来确定哪个视图对象将接收它们，而所有其他事件都被发送给第一响应者。现在是处理这些事件的时候了。

## 事件处理

您现在进入了事件处理之旅的后半部分：事件处理。简单来说，如果一个对象包含解释该事件并决定如何处理它的代码，那么这个对象就处理或响应了该事件。

我首先处理直接传递的事件。接收直接传递事件的对象必须有一个方法来处理该事件、消息或通知。这是必须的。如果对象没有实现发送给它的消息，您的应用程序将出现故障并可能崩溃。这就是您需要了解的有关直接传递事件的全部内容。

**注意：** 当请求定时器、消息或通知事件时，请确保接收它们的对象实现了正确的方法。

其他事件类型则宽容得多。就像可选的委托方法一样，如果您的对象实现了处理事件的方法，它就会接收这些事件。如果它不感兴趣处理该类事件，您只需从实现中省略这些方法，iOS 就会寻找另一个想要处理它们的对象。

例如，要处理触摸事件，您需要在类的实现中添加以下方法：

```
- (void)touchesBegan:(NSSet *)touches withEvent:(UIEvent *)event
- (void)touchesMoved:(NSSet *)touches withEvent:(UIEvent *)event
- (void)touchesEnded:(NSSet *)touches withEvent:(UIEvent *)event
- (void)touchesCancelled:(NSSet *)touches withEvent:(UIEvent *)event
```

如果命中测试确定您的对象应该接收触摸事件，那么当硬件检测到您的视图中有物理触摸时，它将收到 `-touchesBegan:withEvent:` 消息；当位置发生变化（拖动手势）时，将收到 `-touchesMoved:withEvent:` 消息；当与屏幕的接触被移除时，将收到 `-touchesEnded:withEvent:` 消息。当用户点击并在屏幕上拖动手指时，您的对象可能会收到许多这样的消息，而且常常是连续快速的。

**注意：** `-touchesCancelled:withEvent:` 消息是这组方法中的特例。如果某些事情中断了触摸事件序列（例如，在点击手势过程中您的应用切换到另一个屏幕），此消息会发送给您的对象。仅当不完整的触摸事件序列（例如，收到了“开始”消息但没有收到“结束”消息）会使您的对象混淆时，您才需要处理取消事件。

如果您从类中省略所有这些方法，您的对象将不会处理任何触摸事件。

所有用于处理事件的方法都继承自 `UIResponder` 类，并且每种类型的事件都有一个或多个方法，如果您想处理该事件，就必须实现这些方法。`UIResponder` 类文档提供了事件处理方法的完整列表。

那么，如果命中测试或第一响应者对象忽略了该事件会发生什么？这是一个好问题，答案可以在响应者链中找到。
