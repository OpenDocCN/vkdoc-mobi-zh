# 练习

你认为，将 SunTouch 变成一个可以同时由三个甚至四个玩家玩的游戏会有多复杂？令人惊讶的是，工作量并没有那么大。或者，也许正因为它不难，才说明 SunTouch 是一个设计精良的软件。

如果你愿意从朋友或家人那里再借一台 iOS 设备，本章的练习就是让 SunTouch 能够支持最多四个玩家同时游玩。

应用中需要修改的部分，可能不是你立刻就能想到的那些。如果你将所有来自任何远程玩家的攻击和捕获的太阳都视为“对手”，那么发送攻击和处理来自远程玩家捕获太阳的逻辑只需稍作调整。视图类根本不需要修改——这设计得有多精良？

复杂的问题在游戏的开头和结尾。你现在需要在最多四组太阳位置之间进行仲裁，并且所有玩家都得到相同的一组位置。但即使这样，也可以用几行代码解决。真正复杂的问题是，现在有不止一个对手的分数，所以单一的 `-opponentScore` 方法就没有意义了。

我想过几种将 SunTouch 演变成多人游戏的不同方法。我最终编写的解决方案可以在 `Learn iOS Development Projects` ➤ `Ch 14` ➤ `SunTouch E1` 文件夹中找到。它为 `STSun` 对象添加了一个新的 `playerID` 属性。该属性记录了捕获该太阳的玩家。一个新的 `-scoreForPlayer:` 方法用于计算任何单个玩家的分数。如果你想让游戏更加丰富多彩，可以修改 `STGameView` 为每个玩家分配不同颜色的太阳。在项目中搜索字符串 `*Multi-Player*` 可以找到这些以及其他更改。

在这么多玩家的情况下，挑选一个领头者来开始游戏并就一组太阳达成一致听起来会很棘手，但这只需要一些小的代码更改。仍然通过最高的投掷硬币结果来决定要使用的太阳组，唯一的区别是每个应用都会等待，直到它收到来自所有其他玩家的“开始游戏”消息。另一种方法是使用 `GKMatch` 的 `-chooseBestHostPlayerWithCompletionHandler:` 方法，让 GameKit 选择其中一个设备作为主机。一旦选定，主机设备将选择一组太阳位置，并通过单条消息将其传达给所有其他玩家，从而开始游戏并消除了对投掷硬币的需求。（`-chooseBestHostPlayerWithCompletionHandler:` 唯一的问题是不保证能选出一个主机，所以你仍然必须提供一个备用机制。）

我尝试了几种不同的解决方案来完成这个练习，其中一种方案增加了一条新的“分数”消息，每当本地分数改变时就广播该消息。你能想到其他解决方案吗？虽然通常有很多糟糕的解决方案，但编写软件并没有唯一正确的方法。
