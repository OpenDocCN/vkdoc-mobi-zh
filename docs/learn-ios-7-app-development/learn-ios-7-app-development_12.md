# 第二章：Boom！应用

## 摘要

在本章中，你将创建一个能够真正做些事情的 iOS 应用。虽然功能不多——毕竟还处于早期阶段——但足以称得上实用。在这个过程中，你将：

-   使用 Xcode 的 **Interface Builder** 设计你的应用
-   为你的应用添加对象
-   将对象连接在一起
-   自定义对象以提供内容
-   向项目添加资源文件
-   使用 **storyboard** 创建**转场**
-   使用**约束**控制视觉元素的布局

令人惊讶的是，你将在不编写一行代码的情况下创建这个应用。这并不常见，但这将展示 Xcode 的灵活性。

你将创建的这个应用会展示一些关于二十世纪女性超现实主义者的有趣事实。让我们开始吧。

### 设计

在启动 Xcode 并疯狂敲击键盘之前，你需要有一个计划。这是应用开发的设计阶段。在你的应用生命周期中，随着不断改进，你可能会多次修改设计，但在开始之前，你需要对应用的外观和工作方式有一个基本构想。

你的设计可以是正式写出来的，也可以是画在餐巾纸上的草图，甚至可以只是存在脑子里。只要你有设计就行，形式并不重要。你至少需要能够回答一些基本问题：你的应用将在哪些设备上运行（iPhone/iPod、iPad，还是两者都支持）？应用是竖屏运行、横屏运行，还是两者都支持？用户会看到什么？用户如何导航？他们将如何与应用互动？

本应用的一个粗略草图如图 2-1 所示。这个应用非常简单，因此初始设计不需要太多内容。这个超现实主义者应用将有一个包含著名女性超现实主义者肖像的启动画面。点击其中一张肖像，画面将过渡到第二个屏幕，该屏幕会展示一幅代表性画作以及一个包含艺术家生平信息的可滚动文本字段。你已决定此应用仅能在 iPhone 或 iPod Touch 上运行，并且仅支持竖屏方向。这将简化你的设计和开发工作。

## 构建界面

点击项目导航器中的 `Main.storyboard` 文件。Xcode 的 **Interface Builder** 编辑器会出现在编辑区域，如图 2-5 所示。

Interface Builder 是 Apple 应用厨房中的**秘密武器**。简而言之，它是一个用于在应用内添加、配置和互连对象的工具——无需编写任何代码。你可以在 Interface Builder 中定义应用的大部分视觉元素。Interface Builder 编辑 `storyboard`、`xib` 和（旧版）`nib` 文件。

![A978-1-4302-5063-0_2_Fig5_HTML.jpg](img/A978-1-4302-5063-0_2_Fig5_HTML.jpg)

图 2-5. Interface Builder

> **注意**：现代 Interface Builder 文件的扩展名为 `xib` 或 `storyboard`。旧版 Interface Builder 文件的扩展名为 `nib`（读作“nib”），你仍然会听到程序员将它们统称为“nib”文件。Nib 这个缩写代表 **N**ext **I**nterface **B**uilder，因为 Xcode、Interface Builder 和 Cocoa Touch 框架的根源都可以追溯到 Steve Jobs 的“另一家”公司——NeXT。在本书后面，你会看到很多以 `NS` 开头的类名，这是 `NeXTStep` 的缩写，即 NeXT 操作系统的名称。

Interface Builder 在两种视图中显示文件中的对象。左侧（见图 2-5）是组织成层级列表的对象，称为**大纲**。一些对象可以包含其他对象，就像文件夹可以包含其他文件夹一样，大纲反映了这一点。使用展开三角形来显示包含的对象。

右侧的视图称为**画布**。在这里你可以找到 Interface Builder 文件中的可视化对象。只有可视化对象（如按钮、标签、图像等）才会出现在画布上。没有可视化方面的对象只会在大纲中列出。如果一个对象同时出现在两者中，你使用哪一个都没关系——它们是同一个对象。

> **注意**：如果你一直在学习面向对象编程语言，那么你就知道什么是“对象”。如果你不知道什么是“对象”，不要惊慌。现在，只需将对象视为乐高®积木；它是一个离散的模块，在你的应用中执行特定任务，并且可以与其他模块连接起来组成更大的部分。如果你想立即了解对象，可以跳到第 6 章。

### 删除和连接对象

库中的导航控制器对象实际上是一组对象的集合。顾名思义，导航控制器管理用户如何在多个屏幕之间移动，每个屏幕都由一个单独的视图控制器对象控制。导航控制器连接到将显示的第一个屏幕的视图控制器，这个屏幕被称为其根视图控制器。不必担心细节；你将在第 12 章中全面了解导航控制器。

为了方便起见，库中的导航控制器会同时创建一个导航控制器对象以及它初始化时附带的根视图控制器。这个根视图控制器恰好是一个表视图控制器。你不需要表视图控制器。相反，你希望这个导航控制器使用你已经拥有的、没有多余功能的普通视图控制器。

首先，丢弃多余的表格视图控制器。如图 2-8 所示，只选中连接到导航控制器的那个表视图控制器。按 `Delete` 键，或选择 `Edit ➤ Delete`。

![A978-1-4302-5063-0_2_Fig8_HTML.jpg](img/A978-1-4302-5063-0_2_Fig8_HTML.jpg)

图 2-8. 删除表视图控制器

现在，你需要将新的导航控制器连接到你的项目自带的那个普通视图控制器。拖拽视图控制器并将其放置在导航控制器的右侧（见图 2-9）。

附加在视图控制器上的未连接箭头表示你应用的初始视图控制器。你想让导航控制器成为第一个控制器，因此，如图 2-9 所示，将该箭头从简单的视图控制器上拖离，并放到导航视图控制器上。

![A978-1-4302-5063-0_2_Fig9_HTML.jpg](img/A978-1-4302-5063-0_2_Fig9_HTML.jpg)

图 2-9. 指定初始视图控制器

最后一步是重新建立导航控制器与其根视图控制器的连接。在 Interface Builder 中有多种建立连接的方法。我将向你展示两种最常用的方法。按住 `control` 键，点击导航控制器，然后从它上面拖出一条线连接到视图控制器，如图 2-10 所示。

松开鼠标时，会弹出一个菜单，列出了这两个对象之间所有可能的连接。点击 `根视图控制器` 连接。现在，当你的应用启动时，导航控制器将把此视图控制器显示为第一个屏幕。

我确实答应过要教你两种连接对象的方法。第二种方法是使用工具区域中的连接检查器。选择 `View ➤ Utilities ➤ Show Connections Inspector`，或点击工具面板中的小箭头图标，如图 2-11 所示。

![A978-1-4302-5063-0_2_Fig11_HTML.jpg](img/A978-1-4302-5063-0_2_Fig11_HTML.jpg)

要使用检查器，首先选择一个对象。在此例中，选择导航控制器。连接检查器将显示该对象的所有连接。找到标记为`根视图控制器`的连接。每个连接的右侧都有一个小圆圈。要设置连接，请点击并将该圆圈拖拽到你想连接的另一个对象上——在此例中，是视图控制器。要清除（或“断开”）连接，请点击连接字段中的小“x”。

到目前为止，你已经创建了一个新项目。项目模板包含了一个简单的视图控制器。你为你的应用添加了一个新的导航控制器对象（以及一个不需要的、被你丢弃的表视图控制器）。你已将导航控制器指定为应用启动时接管控制的那个控制器，并将该控制器连接到了那个空的视图控制器。现在，是时候往那个空的视图中放入一些内容了。

### 添加资源

应用运行所需的一切都必须是你所构建应用的一部分。如果你的应用需要一张图片，那么该图片必须包含在其资源中。图片文件、`Interface Builder` 文件、声音文件，以及其他任何非计算机代码的东西，统称为资源。

你几乎可以将任何文件作为资源添加到你的应用中。资源文件在构建时会被复制到应用的 bundle 中，并在应用运行时可供其使用。

`Xcode` 有一种特殊的方式来组织常用资源（如图片），将它们整合成一个称为资源目录的单一资源。要向资源目录添加新图片，请在项目导航器中选中该目录，如图 2-17 所示。在 `Finder` 中找到你想要添加的资源文件。找到你在第 1 章中下载的 `Learn iOS Development Projects` 文件夹。在 `Ch 2` 文件夹内，你会找到 `Surrealists (Resources)` 文件夹，其中包含八个图片文件。让你的文件和项目窗口都可见，然后将这些图片文件拖拽到资源目录的组列表（左侧）中，如图 2-17 所示。

![A978-1-4302-5063-0_2_Fig17_HTML.jpg](img/A978-1-4302-5063-0_2_Fig17_HTML.jpg)

图 2-17. 将资源文件拖入资源目录

这些文件将被复制到你的项目文件夹中，添加到项目导航器，并作为资源添加到你的应用中。资源目录使你可以轻松组织同一张图片的多个分辨率版本，尽管在这个例子中只添加了低分辨率图片。像你刚才那样将图片拖入组列表，会在目录中创建新的分组。你稍后可以通过将图片拖入分组的预览窗格（图 2-18 右侧所示），来添加同张图片的其他版本（高分辨率或特定平台）。`Xcode` 无法编辑这些文件，但预览窗格（见图 2-18）允许你查看它们的缩略图。

## 使用故事板

故事板允许你在界面生成器中规划应用的各种屏幕，并定义用户如何在它们之间导航，从而简化了应用开发过程。在故事板出现之前，你可以布局每个屏幕，但必须编写代码来实现屏幕间的切换。这些代码量不大，也不复杂，但很繁琐。借助故事板，你可以在界面生成器中完成大部分工作——完全无需编写任何代码。

**顺畅开发的体验**

重复的代码会拖慢开发进度。将时间花费在反复编写相同代码上，就意味着你没有时间去开发炫酷的新功能。你需要的是一个顺畅无阻的开发环境，由它为你处理简单的任务，让你能专注于开发那些让你的应用与众不同的内容。

苹果公司致力于让 iOS 和 Xcode 尽可能流畅无阻。每个新版本都会添加新的类和开发工具，使开发高质量的应用变得更加容易。例如，在手势识别器引入之前，编写代码来检测多点触控手势（比如捏合或三指轻扫）是一项复杂的任务，通常需要一页或更多页的代码。而如今——你可能已经猜到了——你的应用只需将一个手势识别器对象拖放到设计中，并将其连接到一个操作，就能检测到这些手势。

### 添加约束

当视图大小发生变化时，一个名为 `auto-layout` 的功能会重新定位其子视图。这些子视图根据一组约束进行重新定位。约束仅仅是关于某个对象的规则。以下是一些约束示例：

- 对象的高度必须至少为 40 像素
- 对象的底部边缘应距离其父视图底部边缘 20 像素
- 对象应水平居中

单个视图通常会有多个这样的约束。`iOS` 会评估所有约束，以确定应如何调整每个视图，从而满足所有约束。

您可以在 Interface Builder 中定义单个约束，或要求 `Xcode` 为您生成部分或全部约束。`Xcode` 根据您定位视图的方式以及视图与附近视图的关系来创建约束。通常，`Xcode` 的猜测正是您想要的。如果不是，您可以自行定义。

您的 `Surrealist` 应用需要约束，以便四位艺术家场景中的文本和图像能在不同设备上正确调整大小；并且您希望初始场景中的四个按钮能均匀地填满屏幕。先处理前者，因为这些是最简单的。

对其他三个场景重复此过程：选择场景的视图控制器，然后从 `Resolve Auto Layout Issues` 按钮中选择 `Reset to Suggested Constraints in View Controller` 项。

然而，初始场景中四个按钮的位置和大小并不符合 `iOS` 中按钮对象的典型尺寸或放置方式。因此，`Xcode` 的建议约束作用不大。您需要自行设计这些约束。思考您希望这些按钮具有怎样的尺寸和位置，然后制定出能够向 `iOS` 描述这一点的最简单规则集。以下是我想到的约束：

- 上面两个按钮的顶部边缘应位于导航栏的底部边缘。
- 下面两个按钮的底部边缘应位于父视图的底部边缘。
- 左边两个按钮的左侧边缘应位于父视图的左侧边缘。
- 右边两个按钮的右侧边缘应位于父视图的右侧边缘。
- 左侧按钮应与右侧按钮接触。
- 上方按钮应与下方按钮接触。
- 所有按钮应具有相同的高度和宽度。
