# 用户体验与界面设计指南

事实上，你开发的移动应用很可能不会面向更大尺寸的设备屏幕。虽然屏幕分辨率能提升用户体验，但移动设备受便携性限制，屏幕尺寸不太可能持续增大。iPad 带来了截然不同的体验——尽管屏幕更大，但不能视其为新型笔记本电脑，当然也绝不能当作放大版的 iPod touch。以下准则同样适用于 iPad 开发：务必确保所有文字内容清晰可读。宽屏上的小字体会让多数人反感——在移动设备上，这甚至可能彻底赶走用户。

当然，提供字体缩放功能（图 5-7）是一种解决方案。但归根结底，最佳实践是在真实设备上测试字体，并诚实评估自己的视力情况。图标设计也要遵循同样的限制原则。当你通过尺寸让交互区域便于点击时，请记住：图标的本质是用比文字更快、更通用的方式表达操作、快捷方式或链接——这在移动设备上尤为关键。

**图 5-7.** `Twitterific` 应用在导航栏提供了调整文字大小的按钮。

### 避免不必要的交互

我们反复强调：用户体验应尽可能流畅。在传统网页中，表单填写可能阻碍用户继续操作；而在移动端，这很快就会变成苦差事。任何情况下都应优先使用单选按钮、复选框或下拉菜单替代文本输入框，因为在 iPhone 上，根据实际场景，打字可能相当困难。iPad 开发同样遵循此准则：Apple 建议 iPad 应用应完全支持单手操作，而双手操作则应带来额外的舒适性和操作效率。

### 善用新型输入类型

如今，表单已是网络用户日常生活的组成部分。人们越来越习惯填写表单，也因此变得越来越缺乏耐心，对美观性和易用性有更高要求——这在移动设备上尤其明显，糟糕的浏览条件容易让用户焦躁，输入过程可能困难重重。

HTML5 引入了多种向后兼容的新型输入类型来解决这些问题。表 5-1 列出了 Mobile Safari 支持的类型。

**表 5-1.** Mobile Safari 支持的新型输入类型

| 类型 | 描述 |
|------|------|
| `tel` | 电话号码（通常为数字字符） |
| `search` | 搜索字段 |
| `url` | 绝对 URL 地址 |
| `email` | 一个或多个有效电子邮件地址 |
| `number` | 数值 |

核心理念是：浏览器应为用户提供输入界面，并在处理表单时向开发者提交格式化内容。Mobile Safari 会根据这些输入类型为调用的上下文键盘注入内容。例如，点击 `tel` 类型输入框会弹出数字键盘（如图 5-8 所示）。

**图 5-8.** 所有支持的键盘类型

这些功能通过减少操作步骤实现无缝交互，显然能提升用户体验。在 iOS 2.0 时代，iPhone 曾为命名为 `zip` 或 `phone` 的输入元素自动弹出数字键盘，部分实现了类似功能。但该特性在 iOS 2.0 中被移除，直至 3.1 版本才通过支持新型输入类型重新引入。

### 审慎对待用户输入信息

虽然可以简化用户输入，但请始终自问：你计划索要的信息是否真的必要？或者说，这对最终用户是否具有首要价值？如果你的应用要按不同条件返回书籍列表，那么询问用户是否偏好精装本、特定出版社或出版年份就显得次要了。此时最佳做法是像亚马逊（图 5-9）那样分阶段进行查询：先通过标题、作者或主题搜索——理想情况下只需一个文本输入框，由服务器端脚本判断相关性；然后提供按日期、出版社或封面类型筛选结果的选项；最后让用户对结果进行排序，例如按价格或书店库存。

**图 5-9.** 亚马逊不会在搜索区域堆砌筛选条件，而是将筛选功能延迟到结果页面。

加速此类应用的另一种方法是记录用户输入。你的忠实用户很可能总在查找同一批出版社的书籍。你可以通过传统 Cookie 或 HTML5 最新的客户端存储技术（本书第 15 章将介绍）保存用户偏好。这样，当用户再次访问你的 Web 应用时，就不必重新输入“墨菲斯托菲勒斯的古籍书架珍藏”这样的内容了。

用户输入的核心理念是：在你索取任何信息时，都应反馈提供更多附加功能。当然，收集用户信息能显著提升服务质量，关键是要让“显著”的程度与“收集”的投入相匹配。

### 避免增加操作步骤

在前述案例中，用户需要经过筛选、搜索、排序三个步骤，其中仅第一步是必需的。如前所述，用户想要的核心功能应通过最直接的流程尽快呈现。将图书搜索应用拆分为三个独立页面（如同传统网页常见的账户注册表单那样），只为减轻每个阶段的负担——这种做法在此并不合适。这种设计的好处（减少页面跳转、降低信息杂乱）会被用户被迫等待的弊端所掩盖，用户甚至可能不清楚流程耗时或下一步操作。对开发者而言，要赌网络连接能连续三次保持高效，风险也很大。

### 核心理念：专注

iPhone 不是电脑，iPad 也不是。不要期望用户像操作电脑那样使用它们，也不要用桌面应用的思维开发 Web 应用。iOS 应用应专注于为特定需求或问题提供具体且令人满意的解决方案，几乎只聚焦于以最直接的方式呈现用户期望的功能。承载你 Web 应用的视口，应像时钟应用（图 5-10）那样，成为一个容纳精选元素、整体协调、清爽简洁、层次分明的空间。

虽然这种设计理念可能源于硬件限制（高负载下渲染缓慢、屏幕空间有限），但它已成为用户的期待。当你为主流需求提供恰当解决方案时，用户的忠诚度将是你的回报。这一准则同样适用于 iPad——至少是继承而来的。所以，始终专注于提供一项精确的服务，而非多个粗略的功能。
