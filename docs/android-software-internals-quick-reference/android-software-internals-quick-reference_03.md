# 3. 基础

## Android 沙箱

Android 运行在一个多用户的 Linux 系统下，这意味着每个应用程序及其存储都在一个独立的用户下运行。这意味着在正常情况下，应用程序无法读取另一个应用程序的数据或内部存储。每个进程使用自己的虚拟机（VM）来隔离应用程序。在 API 级别 21（Android 5）之前，这将是 Dalvik 虚拟机，而在后续版本中则使用 Android 运行时（ART）。两者的运行方式类似，它们模拟设备的 CPU、寄存器和其他功能，同时运行应用程序编译后的 Dalvik 字节码。然而，`ART`被认为在性能上有许多改进。

在这些虚拟机中，应用程序只能访问其运行所需的组件（最小权限策略）。这些单独的进程虚拟机由`Zygote`¹（zai·gowt）创建。`Zygote`由 Android 运行时在启动时以 root 权限启动，并附带第一个虚拟机以及所有共享的 Java 类和资源。当一个新的应用程序想要启动时，会派生出一个新的`Zygote`进程，应用程序绑定到该新进程的线程，并在其中运行其代码。`Zygote`会丢弃不需要且未被请求的权限，以便应用程序只拥有必要的权限。

## 应用程序组件



### 活动（Activities）

`Activities` 是 Android 应用程序的主要入口点。类似于单个网页，一个活动就是一个界面，通常只有在前台运行时才会保持活跃。虽然并非所有活动都必须可见，但大多数标准应用程序的活动都是可见的。通过继承 `Activity` 类，可以在程序上实现一个活动。

#### 服务（Services）

广义上，服务是 Android 中用于在应用程序未在前台运行时提供后台功能的一种工具，例如音乐播放器、轮询电子邮件的邮件客户端或地图应用。多年来，Android 中用于任务服务的首选技术已从使用 `Services` 转变为使用 `JobSchedulers`。这些将在第 10 章中进一步讨论。

### 广播接收器（Broadcast Receivers）

应用程序的另一个入口点是系统、其他应用以及应用本身可以“广播”事件，随后应用接收这些事件。广播接收器的功能有限（通常，它们只能运行 10 秒，超过此时间便被视为无响应^(²)），因此通常会启动其他形式的长期运行服务，例如前台 `Activity` 或 `JobScheduler`。广播接收器通过继承 `BroadcastReceiver` 类来实现。这些将在第 4 章中进一步讨论。

### 内容提供器（Content Providers）

内容提供器用于管理应用数据集，使其能与设备上的其他应用共享。通过使用 URI，即使 URI 所属的应用当前未在运行，其他应用也可以查询或修改这些数据。示例包括图像、文本文件、SQLite 数据库等。

### 清单文件（Manifest）

应用程序的清单文件^(³) 在编译前创建，运行时无法编辑。这些类似 XML 的文件（称为 `AndroidManifest.xml`）详细描述了单个应用程序中的所有组件（活动、广播接收器、服务等）。清单文件还列出了应用所需的权限、最低 API 级别，以及应用使用的硬件和软件功能（例如摄像头）。虽然一个应用只有一个清单文件，但导入的库可能拥有自己的清单文件。因此，在构建过程中，Gradle 会合并所有这些单独的清单文件，最终结果称为合并清单。^(⁴) 这在逆向分析 Android 应用（`APK` 文件）时尤为明显，因为（如果使用了库）逆向得到的清单文件会比原始的未合并清单文件大得多，并且包含更多元素。
