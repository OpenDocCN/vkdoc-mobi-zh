# 13. 预处理器

## 摘要

Objective-C 配备了一个功能强大的预处理器，它可以简化将重复代码输入程序的过程。例如，预处理器可用于导入头文件，这些头文件用于使类定义对应用程序的其他部分可用。预处理器还用于定义宏，这些宏既可以作为常量使用，也可以根据参数执行简单的替换。

然而，尽管预处理器功能强大，如果使用不慎，它也可能成为众多编程陷阱的源头。由于预处理阶段执行的替换不会立即被编译器检查，因此存在通过预处理器引入的元素之间微妙的交互而导致产生错误代码的可能性。

在本章中，我将介绍预处理器的主要特性，并展示如何避免其一些粗糙的边缘问题。你将首先详细了解预处理器的工作原理以及它为 Objective-C 程序员提供的额外工具。在第一部分中，我将解释宏是如何工作的，以及如何使用它们来定义新符号和基于参数的替换。你还将了解宏的一些常见陷阱以及如何避免它们。接下来，你将看到 Objective-C 中用于包含头文件的工具。处理文件包含是预处理器执行的最重要任务之一，你将了解它是如何执行的，以及你有哪些选项来控制这个过程。

## 定义宏

预处理器的首要任务之一是为程序中常用的值定义符号名称。例如，假设你有一个处理文件路径的算法，并且在某个特定平台上，文件路径有 255 个字符的限制。这可以通过定义一个新宏来表示，如下所示：

`#define MAX_PATH_LENGTH 255`

该语法仅由 `#define` 和一个宏名称组成，后跟你想要使用的任何符号或表达式（稍后你将看到当表达式长于一行时该如何处理）。Objective-C 预处理器中的宏定义机制基于字面替换。这意味着，每当预处理器看到已定义的符号时，它都会用定义期间提供的值进行替换，如下所示：

```
- (BOOL)checkPathSize:(NSString *)path
{
    if ([path length] > MAX_PATH_LENGTH)
    {
        NSLog(@"错误：路径长度无效");
        return NO;
    }
    return YES;
}
```

如上面的示例所示，宏名称的行为类似于变量，但存在一些重要区别。首先，替换甚至在代码编译之前就发生了。也就是说，编译器本身从未见过符号 `MAX_PATH_LENGTH`，只看到最终的替换结果。因此，宏符号不与任何内存相关联，这与变量不同，变量通常存储在内存中，你可以使用诸如 `&` 之类的运算符操作其内存内容。

宏处理方式的另一个后果是，它们可以做的远不止替换常量值。例如，虽然不推荐，但以下类型的代码是有效的（甚至在早期曾被一些程序员使用过）：

```
#define BEGIN {
#define END }
void myFunction(int val)
BEGIN
    if (val > 0)
    BEGIN
        NSLog(@"值为正");
    END
END
```

这是可能的，因为处理器盲目地执行宏替换，而编译器只会在这些替换完成后处理其结果。

为了避免处理宏替换时出现意外，C 和 Objective-C 有一项约定：宏名称应全部大写，以便与普通符号区分开来。这可能无法解决所有由宏产生的问题，但至少可以提醒程序员你正在处理的是一个预处理符号，而不是普通变量。



### 带参数的宏

宏的另一个特性是可以带参数，其语法与函数调用类似。例如，要声明一个接收两个参数并返回较大值的宏，可以使用以下代码：

```
#define MAXIMUM(a, b) (a) > (b) ? (a) : (b)
```

这表示宏 `MAXIMUM` 有两个参数，其替换值位于声明之后。该宏使用三元运算符来检查作为参数传递的两个值。当第一个值大于第二个值时，返回第一个值；否则，宏返回第二个值。

带参数宏的通用语法是，在 `#define` 指令之后紧跟宏名，然后是放在括号中的参数。与函数不同，参数没有类型，因此你可以输入任何内容作为实参，例如变量、关键字或运算符。

宏的主体紧跟在参数列表之后。你可以在主体中使用任何表达式。与函数不同，宏在首次被看到时并不会被编译。宏的主体仅在进行替换时才被使用。当替换发生时，预处理器会替换参数，并将结果粘贴到宏出现的位置。只有在完成此过程之后，编译器才有机会检查宏的内容。

宏不能像函数那样跨越多行。然而，预处理器将反斜杠字符 (`\`) 识别为续行标志。因此，只要需要连接到下一行的每一行都以反斜杠结尾，你就可以编写多行宏。例如，使用相邻行连接规则，之前的宏也可以写成：

```
#define MAXIMUM(a, b) \
   (a) > (b) ? \
         (a) : \
         (b)
```

请注意，除了表示宏结束的最后一行外，每一行都以反斜杠结尾。

### 宏运算符

宏的另一个特性是提供了自己的运算符，这些运算符在用于带参宏时可以执行有用的转换。提供这些运算符是为了简化在组合宏替换时经常需要的常见任务。

第一个运算符是字符串转换转换，也称为字符串化（stringify），因为它将作为参数传递的任何值或表达式转换为 C 风格的以空字符结尾的字符串。

字符串转换运算符使用一个前导的 `#` 字符来指示接下来的宏参数将被转换为字符串。下面是一个例子：

```
#define EXP_TRUE(expr) \
if (expr) \
NSLog(@"the following expression is true: %s", #expr)
```

这个宏可以用来判断一个表达式的计算结果是否为真值。它通过简单地使用一个 `if` 语句来实现，其中测试参数被用作表达式本身。如果这样的表达式计算结果为真，则会调用日志记录语句，并且字符串化运算符将把作为参数传递的任何表达式转换为一个字符串，然后发送到日志终端。

你可以向这个宏传递任何计算结果为 `YES` 或 `NO` 的表达式。例如，考虑下面的函数：

```
- (void) useMacro
{
EXP_TRUE(2 + 3 > 4);
}
```

这里，预处理器使用的表达式由括号内的全部内容组成。预处理器会找到宏名 `EXP_TRUE`，用作为实参传递的表达式替换所需的参数，并将结果粘贴到输出源文件中的相同位置。然后，当预处理器看到表达式 `#expr` 时，会触发将其转换为一个以空字符结尾的字符串。宏处理的结果如下：

```
- (void) useMacro
{
if (2 + 3 > 4)
NSLog(@"the following expression is true: %s", "2 + 3 > 4");
}
```

正如你所见，预处理器将通用表达式转换为字符串的能力对于调试目的非常有用。例如，一个常用的标准宏是 `assert`；它包含在头文件 `assert.h` 中，用于判断一个给定的表达式是否为真。如果不是，程序将停止并打印一条断言失败的消息。`assert` 宏在异常情况发生时立即诊断它们非常有用。

```
- (double)squareRoot:(double) x
{
assert(x > 0);
// 在此处执行计算
}
```

这段代码在执行任何其他操作之前检查 `x` 是否为正数。如果参数不正确，程序将立即停止。

另一个不太常用的运算符是连接运算符 (`##`)。当你提供两个符号时，会用到此运算符，其目标是创建一个新符号，该符号是这两个符号连接的结果。连接运算符的主要用途是创建唯一名称，这在处理复杂宏时有时是必要的。以下是其用法的一个简单示例：

```
#define VARIABLE_NAME(partA, partB) \
partA ## _ ## partB
```

这个宏仅仅将两个参数（中间用下划线分隔）粘贴在一起，形成一个新的符号。然后，可以如下使用该宏：

```
- (void)useMacroConcatenation
{
int VARIABLE_NAME(my, variable) = 10;
NSLog(@"the value is %d", my_variable);
}
```

请注意，在第一行中，`VARIABLE_NAME` 宏创建了一个由两部分组成的新变量名。在宏定义中，这些部分被命名为 `partA` 和 `partB`。在这个示例应用中，结果是一个变量名，它由第一部分、后跟一个下划线、再连接到第二部分组成。当在下一行中将这个组合名称作为参数传递给 `NSLog` 时，你可以清楚地看到这一操作如预期般发生了，`NSLog` 将打印当前存储在变量中的值 10。

与 `#define` 指令类似，预处理器也提供了 `#undef`，它能够从内存中移除宏的定义。当你想确保在代码的某个特定段中不使用某个宏时，这很有用。另一个用例是重新定义现有宏的值，例如在通过预处理器执行以下行之后：

```
#undef VARIABLE_NAME
```

现在你可以确信符号 `VARIABLE_NAME` 是空闲的，如果需要，可以用不同的方式重新定义它。



### 常见陷阱

在某些情况下，使用带参数的宏可能是一种快速且有用的机制，但应该提醒你不要频繁地使用它。以下是几个原因。

首先，宏导致的问题难以诊断。由于宏的整个主体在真正的编译开始之前就被替换了，编译器无法准确地判断发生了什么。当宏替换内部发生编译错误时，你通常只会看到对某行的引用，而实际的错误与该行的内容关系不大。现代编译器可能在此问题上有所改进，但宏的诊断仍然比普通代码困难得多。

调试宏也是一项艰巨的任务，部分原因在于上述编译问题。例如，这限制了你在调试器中的选择，因为你无法精确定位宏中出错的准确行。一个至少能部分解决此问题的可能方法是生成预处理文件并验证其输出。你可以在 Xcode 中通过菜单 **Product** ➤ **Generate Output** ➤ **Generate Preprocessed file** 来完成此操作。生成的文件通常非常大，因为它包含了所有通过 `#include` 或 `#import` 指令导入的头文件。但它可能有助于定位预处理器引入的确切转换。

一个相关的问题是，宏操作其参数的方式可能会导致生成的代码不符合有效的 C 或 Objective-C 代码规范。例如，考虑上面介绍的 `MAXIMUM` 宏，以及当你以下列方式使用它时代码被替换的情况：

```
- (void)useMacro
{
    int i = 10;
    NSLog(@"the maximum is %d", MAXIMUM(11, i++));
}
```

这段代码的问题在于，宏 `MAXIMUM` 在其定义的两个地方使用了参数 `i++`。虽然对于上述参数，`MAXIMUM` 的值自然的预期结果是 11，但实际情况是操作 `i++` 被计算了两次。这意味着，在这种情况下，打印的结果将与预期不符。在其他情况下，不仅可能发生这种情况，甚至可能导致结果未定义，从而引发难以修复的 bug。

**注**

正如你在前一个例子中所见，宏增加了参数被错误计算的风险。虽然可以通过创建临时变量等方式来避免部分问题，但最佳策略是尽可能避免使用宏。如果必须使用，除非参数是为此目的设计的，否则应避免将表达式作为宏参数传递。

## 条件编译

宏的另一个重要用途是定义条件编译语句，这些语句只有当某些编译时条件为真时才会被编译器处理。这些条件操作与普通条件语句（如 `if` 语句）不同，因为它们在编译器开始工作之前就已经执行。

### #if 指令

例如，`#if` 指令可用于定义只有在特定编译时条件为真时才被处理的代码段。通常，条件涉及一个简单的算术比较，只有当条件评估为真时，其后的代码行才会被传递给编译器。所有这些行都会被包含在内，直到遇到下一个 `#endif` 指令。示例如下：

```
- (int) getSupportLevel
{
#if VERSION > 5
    return 2;
#endif
#if VERSION >= 2 && VERSION <= 5
    return 1;
#endif
#if VERSION < 2
    return 0;
#endif
}
```

在此方法中，应用程序的支持级别是基于 `VERSION` 宏的值来定义的。这些检查发生在编译时，并且只有一个选项会为真。因此，方法 `getSupportLevel` 将返回一个支持级别（0、1 或 2）。

`#if` 指令后面也可以跟一个 `#elif` 指令。类比 `#if`，`#elif` 指令类似于 `else if` 语句，但它在编译时求值。最后，`#if` 或 `#elif` 指令后面也可以跟一个 `#else` 指令，它包含了仅当之前的检查为假时才应该编译的行。考虑到这些额外的指令，你可以使用 `#elif` 和 `#else` 更清晰地重写之前的示例：

```
- (int) getSupportLevel
{
#if VERSION > 5
    return 2;
#elif VERSION >= 2 && VERSION <= 5
    return 1;
#else
    return 0;
#endif
}
```

### #ifdef 指令

在其他情况下，你想要测试的编译条件可能涉及其他已存在宏的定义。当宏用于表示某个特定功能是否存在时，这种情况很常见。例如，GCC 编译器预定义了宏 `__GNUC__`，以便你可以检测程序是否正在被 GCC 编译。其他编译器和编译器特性也类似地定义了可用于条件编译的宏。

你可以使用 `#ifdef` 和 `#ifndef` 指令编写仅在特定宏被定义时才会被编译的代码。前者类似于 `#if`，但它只检查紧跟其后的符号是否已在当前文件中被定义。类似地，`#ifndef` 检查其后的宏符号，如果该符号尚未被定义，则执行后续代码行。示例如下：

```
- (void) detectGCCCompiler
{
#ifdef __GCC__
    NSLog(@"This is the GCC compiler");
#endif
#ifndef __GCC__
    NSLog(@"This is not the GCC compiler");
#endif
}
```

另一种测试某个符号是否被定义的方法是使用 `defined` 宏运算符。当你想要创建更复杂的测试时，这个运算符非常有用。

```
- (BOOL) systemIsSupported
{
#if defined(__GCC__) && VERSION > 3
    return YES;
#else
    return NO;
#endif
}
```

**注**

要定义一个预处理器符号，你至少有两个选择：使用 `#define` 指令引入宏，或者在 GCC（或 LLVM）编译器中使用 `-D` 选项。例如，使用 `-DIS_DEBUG_VERSION` 将定义宏符号 `IS_DEBUG_VERSION`。Xcode 也允许在其项目设置界面中定义预处理器符号，但这最终会被转换为传递给编译器的 `-D` 选项。



## 导入机制

预处理器的主要用途之一是为特定源文件所使用的元素插入必要的定义。例如，你已经了解到，在创建类时，应该准备一个包含类接口定义的头文件（扩展名为`.h`），以及一个包含该类方法实现的实现文件。每当使用某个类时，都需要将该类的头文件导入到实现文件中。

将头文件导入实现文件有两种方式。第一种方式是使用`#include`机制。`#include`指令在 C 语言和 Objective-C 中都有提供。它的基本功能是将所引用文件的内容插入到当前源文件中。因此，你可以利用 Objective-C 标准库中提供的多个库，以及外部库。

`#include`机制主要用在 C 语言库中，因为它是 C（和 C++）语言唯一可用的方式，并且定义新的头文件需要额外的工作。要使用`#include`，你只需说明需要访问哪个头文件，如下所示：

`#include <stdio.h>`

这会将标准 I/O C 库的内容导入到你的程序中。使用这个库，你可以调用诸如`printf`（打印功能）或`fopen`（打开文件）等函数。

定义可以被`#include`到文件中的头文件需要稍多一些准备。以下是使用`#include`导入的头文件的标准组织结构：

```
#ifndef INCLUDED_MYFILE
#define INCLUDED_MYFILE
// 文件内容写在这里
#endif
```

这里，`INCLUDED_MYFILE`是一个仅在此头文件中使用的唯一标识符。起初，这种结构可能看起来不自然，但它的主要目的是避免同一源文件多次处理同一个头文件。为此，每个头文件在顶部定义了自己唯一的符号。通常，宏符号的名称源自文件名。因此，如果文件名为`Employee.h`，那么包含宏可能被命名为`INCLUDED_EMPLOYEE_H`。

这种方案的结果很容易解释：首先，检查唯一符号是否已定义。如果未定义，则先定义该符号，然后编译头文件的内容。但如果该符号已定义，则意味着这不是预处理器第一次访问当前文件。因此，头文件的内容将被跳过，从而避免多次包含。

第二种包含文件的方式是 Objective-C 独有的，它通过`#import`指令实现。`#import`和`#include`的主要区别在于，前者会自动处理多次包含的问题，因此无需使用上述条件宏包含方案。

使用`#import`时，预处理器有一个内部数据库，因此它已经知道哪些文件已被处理过。这种技术保证了文件只会被处理一次。因此，你不需要使用条件编译技巧，比如上面提到的那种。

`#import`和`#include`指令可以通过两种方式指定文件：文件名可以放在尖括号中，也可以放在双引号中。这两种定义导入文件方式的区别在于文件名的搜索方式。当使用尖括号时，隐含的假设是预处理器将在所有系统包含目录中搜索。这些目录可以通过系统设置组合来确定。例如，你可以在 Xcode 的包含路径配置工具中列出它们。或者，你可以通过`–I`选项将这些目录添加到 GCC/LLVM 命令行界面中。

另一方面，你可以使用双引号来限定头文件名。在这种情况下，隐含的指示是预处理器将在当前目录中搜索具有指定名称的任何文件。因此，双引号版本可用于输入你自己的头文件，这些头文件将构成你自己项目中的类列表。

通常，双引号括起来的头文件名意味着该头文件是用户创建的。类似地，尖括号括起来的头文件名表明该文件是系统或某些其他外部库的一部分。保持使用这两个选项的一致性很重要，这样你就能始终导入正确的头文件，从而避免错误。



## 头文件最佳实践

鉴于 Objective-C 中头文件必要且广泛的使用场景，开发者需要遵循一些约定来维护基本的系统组织并避免性能问题。这些最佳实践对于简单项目而言有时并非必需，但当项目规模增大，尤其是当有许多开发者协作开发同一应用程序时，它们的重要性便日益凸显。

应遵循的首要规则是：头文件只应包含类和函数的定义。在头文件中包含方法或函数的实现是不规范的做法，因为大多数头文件设计用于在系统的多个位置被引用。若同一函数定义在不同文件中出现两次，可能会导致链接错误，因为链接器无法确定应保留哪个版本的函数或方法。

另一条经验法则是：尽可能避免头文件的嵌套引用。原因在于，由于预处理器会递归地执行文件包含操作，在主头文件中包含过多头文件，将会强制增加该类的每个使用者的编译时间。换言之，在包含文件中，仅添加能够定义该类接口的绝对必要内容即可。

显然，有时确实需要传递性地包含头文件；例如，当声明某个现有类的子类时，就需要包含父类的头文件，以便编译器能够看到其定义。但在其他情况下，可以避免直接包含头文件，从而降低应用程序的复杂性。

有几种方法可以避免包含额外的头文件。例如，在 Objective-C 中，当某个类仅作为类接口的一部分使用时，可以使用 `@class` 关键字来引入该类名。在类接口定义中，除非需要访问该类中的数据或方法，否则通常只需要这样做。下面是一个常见示例：

```
// 文件 MyInternalClass.h
@class MyExternalClass;

@interface MyInternalClass : NSObject
{
    MyExternalClass *externalObject;
}

-(MyExternalClass*) getExtObject;

@end
```

请注意，这里的 `MyExternalClass` 仅作为 `MyInternalClass` 接口的一部分使用。在这种情况下，实际上并不需要使用 `#import` 指令；仅用 `@class` 声明就足以告知编译器 `MyExternalClass` 是何种符号。从那时起，编译器便会将 `MyExternalClass` 记录为一个类的名称，可用于，例如，定义实例变量或作为其他方法的参数。稍后，当你准备编写方法实现时，才需要导入 `MyExternalClass` 的头文件。

```
// 文件 MyInternalClass.m
#import "MyExtenalClass.h"

@implementation MyInternalClass

- (MyExternalClass*) getExtObject
{
    // 此处为具体实现……
}

@end
```

另一个尽可能减少其他文件传递性包含的选项是创建最小化头文件——即其中只定义一个类，而没有多余内容。特别是，要在接口中避免外部依赖，并将其移至实现部分。此技术的目标是通过仅在类接口中添加你真正需要的接口和函数，来尽可能避免外部依赖。

例如，假设你正在创建一个使用其他三个库头文件的新类。与其创建一个引用了其他三个库的头文件，不如定义一个没有外部依赖的小型接口。为了帮助避免依赖，你可以创建一个匿名协议，其中包含你所需的、与外部库相关的所有属性。这样，在实现文件中，你就可以自由地导入所使用每个库的头文件。这种审慎的接口与实现分离，将有助于其他使用者导入和使用你的类。这种审慎用法的附带效果是使项目更易于维护、构建速度更快。

## 总结

预处理是源文件被准备成为编译器输入时所执行的第一项任务。因此，这是一项需要掌握的重要操作，以便充分利用 Objective-C 编程环境。

例如，预处理器会根据使用 `#define` 工具定义的符号执行多项转换。这些宏可用于简单的替换，既可以是无参数的，也可以是带参数的。无参数宏的作用类似于编译时常量。而带参数的宏则能够执行更复杂的替换。你已经看到了一些使用宏的常见陷阱，以及为何在有其他可行选项时应避免过度使用宏。

预处理器还能基于编译时表达式执行条件编译，通过评估表达式的结果来决定某段代码是否会被编译。这种条件编译通常用于兼容性目的，例如，可用于创建能运行在多个平台上的应用程序。

最后，你已看到预处理器执行的另一项重要任务：为外部定义的类和函数包含头文件。这类头文件包含接口或函数定义，可以随时提供给其他使用者。包含机制可通过 `#include` 或 `#import` 指令实现，并且你已经了解了这两种选项各自的优缺点。
