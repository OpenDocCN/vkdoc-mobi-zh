# 第 1 章：持续集成简介

### Apple 历史 101

在撰写本书时，Apple 刚刚发布了 Xcode 5.1 的最终版本。对于 Apple 来说，5.0 版本是一个相当重大的发布。它与 iOS 7 同时推出，而 iOS 7 是一个搭载了全新范式的完整新操作系统。对我们而言更有趣的是，Apple 终于向持续集成世界迈出了一步，开始推广新的工具。在接下来的章节中，我们将尽力涵盖尽可能多的这类工具。

当然，iOS 开发者社区并没有等待 Apple 最终采取行动。在 iOS SDK 首次公开发布之前，持续集成的原则就已经被倡导了。对于 Cruise Control、Jenkins、Apache Continuum 等情况，这主要是将现有工具适配到新技术的问题。

Xcode 的第一个版本于 2003 年发布。十多年后，Apple 终于开始提供官方支持的工具，例如 Xcode Bots。毋庸置疑，我们一路走来，进步巨大！

### 优点与缺点

持续集成的优点众多。最重要的是，当你在处理应用程序的某一部分时，能立即获得反馈。这可能是因为你破坏了测试套件，或者没有遵循团队的编码规范。甚至可能仅仅是因为某个生成的报告告诉你，你编写的代码过于复杂。最终，这种反馈能为你节省大量时间，避免将这些时间花费（甚至浪费）在调试上。

同样重要的是，你和你的团队能够尽早诊断出损坏的代码、bug 或劣质代码，从而获得内心的平静。

最后，集成一个重要的新代码片段通常意味着需要部署一个应用程序的新版本，供你质量保证团队的成员进行测试。当作为团队一员工作时，总会有人走到你桌前，要求你在他的设备上安装你正在开发的最新版本应用；不要做那个人！做一个将所有事情都自动化的聪明人。

构建、自动化测试、代码覆盖率……所有这些概念都只是一条 shell 命令的执行距离；为什么不让他们为你工作呢？正如著名的 UNIX 贡献者道格拉斯·麦克罗伊所说：“你今天做的事情，明天就能自动化。”这在 iOS 开发领域是千真万确的。

让你的团队成员、客户或老板能够在他们的设备上下载你的应用程序的构建版本，这是一件很复杂的事情。对于移动应用来说，这实际上是一个多步骤的过程，包括构建、代码签名，最后是部署。

虽然在接下来的章节中我们会展示，完全靠自己也能实现这一点，但一些服务的诞生就是为了简化这个过程。例如 Testflight、Hockey 甚至 Cloudbees，这些都是众多填补空白的服务中的几个。有些是免费的，有些是收费的，有些是基于云的，有些是自托管的，但它们都有一个共同的目标。

本书的目的不是比较这些服务并告诉你哪个最好。它不会提供一份详尽的列表，当然也不会让你成为我们所选择要讨论的服务的专家。

当然，这一切都不是魔法。能够立即获得反馈虽然是一个显著的优势，但也需要付出代价。建立一个持续集成系统需要时间。这时间没有白费——请不要误解——但这是你必须考虑的因素。当你独自开发一个只有两个屏幕的应用程序时，你可能不需要一个功能完善的、每五分钟就构建一次应用的持续集成平台。有时候，你必须务实一点。

另外，你的工具只取决于你用它们的方式。如果你设置不正确，你可能无法想当然地接受来自 CI 平台的反馈。

### 未来的道路

每一章都会试图涵盖 iOS 持续集成原则中一个非常具体的部分。

阅读这本书有点像踏上一段旅程。你将从一个最简单的入门方法开始，学习如何构建一个 iOS 应用程序并将其发布给你的团队，最终以建立一个完整的持续集成环境结束。以下是我们将共同涵盖的各章概览。

**第 2 章 - iOS 和 Xcode 中的持续集成工具与功能** - 我们将创建一个简单的 iOS 应用程序，并将其作为我们持续集成流程的示例，并在本书的其余部分中以此为示例。在本章中，我们将看到 Xcode 提供了哪些工具来帮助你进行日常开发工作。

**第 3 章 - 使用 Xcode 在 App Store 之外发布应用程序** - 利用第 2 章中创建的应用程序，我们将看到如何创建一个可以发送给测试人员的构建版本。这将是最简单的方法，但你总得从某个地方开始，对吧？

**第 4 章 - 调用命令行的强大功能** - 按下许多按钮是一回事，但理解它们的含义是另一回事。在本章中，你将学习如何利用命令行的强大功能，以及 Xcode 对“构建”等操作的实际含义。

**第 5 章 - 使用 Jenkins 自动化构建** - Jenkins 很可能是持续集成的事实标准解决方案。你将学习如何启动它，以及它如何构建你的 iOS 应用程序。

**第 6 章 - 使用 Bamboo 自动化构建** - Atlassian 是一家澳大利亚企业软件公司，为软件开发人员和项目经理开发产品。他们的产品之一是 Jenkins 的商业替代品。你将学习如何启动它，以及如何使用它来构建你的 iOS 应用程序。

**第 7 章 - 空中（OTA）分发** - 持续交付与持续集成略有不同，通常是成功构建过程的最后一步。你将学习如何让你的生活以及测试人员的生活更轻松——只要新版本的应用程序可用，就立即发送给他们。
