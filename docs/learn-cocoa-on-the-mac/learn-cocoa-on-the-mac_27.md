# 14. Cocoa 中的绘图

## 摘要

到目前为止，我们已经学习了许多强大的 Cocoa 技术，用于处理数据、优化应用程序类的组织，以及使用 Cocoa 附带的丰富的屏幕控件和其他视图。现在，是时候开始学习如何利用 Cocoa 创建我们自己的视图类，从而完全掌控文本和图形的显示了。Mac 应用通常以其“视觉吸引力”而闻名——这不仅体现在绘制不必要的图形装饰上，更在于为用户创造更具沉浸感的交互体验。作为 Cocoa 程序员，我们可以利用图形技术，以令人惊讶的少量工作实现一些类似的效果。

Core Graphics 提供了丰富的功能，用于渲染路径、操控坐标系统等等。Core Graphics 的一个主要部分是被称为 Quartz 的一组 API。事实上，Quartz 在 Core Graphics 中占据了如此大的比重，以至于这两个术语有时会被混用。Core Animation 则更进一步，让我们能以极其简单的方式对视图进行动画处理。

本章将涵盖坐标系统和在 `NSView` 实例中绘图的一些基础知识，并演示如何将视图放入滚动视图中，以显示比窗口可用空间更大的视图。此外，还会简要介绍如何轻松地为我们的应用添加基本的打印支持。

## 基础

在本节中，我们将介绍 Cocoa 绘图 API 中使用的一些基本概念。如果你做过任何形式的图形编程，那么对 Cocoa 中可用的绘图方式可能会相当熟悉。毕竟，计算机图形学的大多数基本概念在几十年前就已经定型了。Cocoa 的突出之处在于其渲染质量（包括文本和图形基元的抗锯齿处理），以及为更复杂的概念提供的高层级抽象。

在本章中，我们将统一使用“点（point）”而非“像素（pixel）”。Cocoa 中的绘图通常是基于矢量且与分辨率无关的。随着高像素密度显示屏（也称为 HiDPI 或 Retina 显示屏）的出现，绘图模型中的一个点实际上可能由屏幕上的一个或四个像素表示。最好的做法是，尽可能让底层操作系统自行处理从绘图基元到实际像素的转换。这样一来，图形在高分辨率显示屏上也能自动呈现出色效果。如果你处理的是与像素相关的代码，那就不那么容易了，你需要关注点和像素之间的区别，以及如何适当地缩放位图。不过，就目前而言，我们只讨论“点”。

### 视图坐标系统

Cocoa 的绘图系统采用 x-y 坐标系统，在默认情况下，原点 (0,0) 位于左下角，x 轴指向右方（数值增大，位置向右移动），y 轴指向上方（数值增大，位置在屏幕上向上移动）。这与数学中传统的作图方式类似，但与许多其他 y 轴指向下方（数值增大，位置在屏幕上下移）的计算机图形系统相反。虽然在 Cocoa 中翻转 y 轴是可行且有时是必要的，但在我们的示例中不会这样做。

绘图通常由 `NSView` 子类的实例完成。视图存在于一个层级结构中。如果层级中的最顶层视图被设置为某个 `NSWindow` 的 `contentView`，那么该层级中的所有视图都会在该窗口内显示。窗口中的顶层视图（即 `contentView`）可以在整个窗口区域进行绘制。对于层级中的其他所有视图，绘图区域被限制在其 frame（框架）内，即其父视图坐标空间中的一个区域（见图 14-1）。

### 视图框（Frame Rectangle）与边界框（Bounds Rectangle）

窗口中的每个视图都有两个重要的矩形（通常简称为 rects），它们定义了大量绘图特性：视图框和边界框。视图框仅指定视图在其父视图坐标空间中的位置和大小。例如，假设窗口中有个顶级视图，称为 A。如果 A 有一个子视图 B，其坐标从点 (10,20) 延伸到点 (40,60)，那么 B 的视图框原点为 (10,20)，大小为 (30,40)。

除了决定其父视图中显示位置和范围的视图框外，每个视图还有一个边界框，用于定义其自身的内部坐标空间。例如，前面描述的视图 B 默认的边界框原点为 (0,0)，大小为 (30,40)。若需要，我们可以轻松更改这一点。例如，假设我们有一系列要绘制的点，其 x 和 y 值介于 0.0 到 10.0 之间。通过将 B 的边界框大小改为 (10,10)，这些点将恰好填满 B 所占用的空间，无论 B 在屏幕上实际占据多大空间（见图 14-2）。

### 矩形、点与尺寸

在 Cocoa 中，矩形由 C 结构体 `NSRect` 表示，该结构体包含一个表示原点的 `NSPoint` 和一个表示尺寸的 `NSSize`。原点和尺寸各自包含两个 `CGFloat` 类型的值，`CGFloat` 是 Cocoa 绘图 API 中广泛使用的基本 C 类型；原点包含 x 和 y，尺寸包含宽度和高度。

更令人困惑的是，Core Graphics 中的许多函数使用了等效的结构体 `CGRect`、`CGPoint` 和 `CGSize`，它们具有相同的结构布局，且均由 `CGFloat` 元素组成。这种划分的历史原因并不值得深究，只要我们不处理 32 位遗留应用，这些 C 结构体就具有相同的布局，并且可以通过内联函数（例如 `NSRectToCGRect`、`NSSizeFromCGSize` 等，这些实际上只是类型转换）相互转换。本章我们将主要使用 NS 前缀的版本。

### 路径基础

Cocoa 的绘图机制包含路径的概念，路径可以包含任意数量的直线或曲线段。每个路径不需要是连续的。我们可以在一个点“抬起笔”，在另一个点重新开始绘制。在实际绘制路径之前，我们可以指定线条粗细、颜色等值。这些设置将应用于路径的每个部分。路径各部分勾勒出的任何形状也都可以用单独的颜色或图案填充。

### 基本绘图方法 drawRect:

现在开始编写代码。Xcode 创建的 `MSSmileyView.m` 文件包含一个空的 `drawRect:` 方法，我们将从这里开始编码。`drawRect:` 方法通常不会直接调用，除非我们正在实现某个现有视图的子类，并通过调用 `[super drawRect:rect]` 让父类完成其绘图工作。相反，当应用的主运行循环确定视图需要重绘时（通常是在视图被创建或调整大小之后，或者我们的代码在视图上调用了 `setNeedsDisplay:` 并传入 `YES` 作为参数时），系统会自动调用 `drawRect:` 方法。`drawRect:` 方法接受一个参数，即一个矩形，用于指示视图的哪个部分被视为“脏区”需要重绘。这可以用于优化复杂视图的绘制，但在我们的示例中，我们将忽略这一点，而是直接使用视图的边界矩形进行绘图。

### 图形状态

Cocoa 中的所有绘图都发生在特定上下文中，该上下文由 `NSGraphicsContext` 类表示。根据上下文的不同，我们可能直接绘制到窗口缓冲区，或绘制到一块离屏内存中供以后使用。除此之外，上下文还含有一系列会随时间变化的状态信息，例如当前用于所有绘图命令的颜色。当视图需要绘制自身时，它应确保绘图结束后图形状态恢复到绘图开始前的状态。

幸运的是，`NSGraphicsContext` 为我们提供了一种简便方法来实现这一点。其 `saveGraphicsState` 方法会将所有相关状态信息压入栈中，而 `restoreGraphicsState` 方法会将状态从栈中弹出。我们可以像下面的代码片段那样，用这两个方法调用将实际绘图代码“括起来”，从而确保不会让图形上下文处于意外的状态。

```
- (void)drawRect:(NSRect)dirtyRect
{
  [NSGraphicsContext saveGraphicsState];
  // 此处编写绘图代码。
  [NSGraphicsContext restoreGraphicsState];
}
```
