# 17. 并发

**摘要**

软件开发中最大的挑战之一，是编写能够同时执行多个任务的软件。几十年来，计算机通过高速在任务间切换来给人一种并发执行的假象，使其看似在同时做多件事（而实际上它们只是在不停地从一个任务切换到另一个任务，一次只“关注”一个任务）。如今的计算机通常包含两个或更多计算核心，因此它们确实可以同时执行多件事，在所有核心上同时执行指令。

然而，即使计算机的硬件和操作系统能够与多核心协同工作，编写能够有效利用多核心的应用程序软件在技术上仍然具有挑战性。在大多数开发环境中，默认假设是我们编写的代码是顺序执行的，一个操作接着另一个操作，而将工作拆分以便可以并发执行可能是一项相当艰巨的任务。

本章将展示一个常见场景的示例，在该场景中 Cocoa 应用程序可以从并发中受益，并演示如何实现。你一定熟悉 OS X 的旋转磁盘光标（有时称为“旋转沙滩球死亡之兆”），每当应用程序对用户操作无响应时就会出现。这通常出现在我们点击某个按钮或菜单项，触发了任何需要几秒钟以上时间的处理时。OS X 会注意到进程没有处理输入事件，并显示旋转磁盘光标以提醒用户。当应用程序完全停止响应并即将崩溃（或被用户强制关闭）时，也会发生这种情况。因此，当应用停止响应并开始显示旋转磁盘光标时，许多用户的即时反应是恐慌，怀疑应用即将崩溃！

因此，为了我们的最佳利益，必须确保应用程序中任何长时间运行的操作都以这样一种方式处理：用户仍然可以与应用程序交互，并且不会看到旋转磁盘——即让操作在后台进行，同时让应用程序像平常一样处理用户输入事件。本章将演示两种以最少的麻烦向 Cocoa 应用程序添加此类并发的方法。我们将使用 Foundation 框架中可用的`NSOperation`和`NSOperationQueue`类所体现的概念，还将看到如何使用 Grand Central Dispatch（以下简称 GCD）来实现这一点，这是一个所有 OS X 应用程序（无论是否基于 Cocoa）都能使用的 C 语言 API。
