# 数据来源于视图的委托对象。`Finder`中的`Column`视图就是浏览器的一个示例。

你的数据源对象必须遵循非正式协议，可以是`NSTableDataSource`或`NSOutlineViewDataSource`。该协议定义了一些必需的方法，用于向视图提供单元格和节点数据。此外，还有许多可选方法，其中很多用于定义视图中的单元格是否可编辑。如果省略这些方法，集合将是不可变的，视图仅用于显示。

> **注意** 一个常见的错误是忽略`NSCell`遵循`NSCopying`协议这一事实。`NSCell`对象会被大量复制。如果你子类化`NSCell`，必须确保它仍然遵循`NSCoping`协议。你可能需要重写`-copyWithZone:`方法。

iPhone 开发者会发现一组类似但功能大大简化的视图类，用于显示表格和列表。主要的类是`UITableView`。它与遵循`UITableViewDelegate`和`UITableViewDataSource`的对象进行通信，其方式与`NSTableView`使用`NSTableDataSource`对象非常相似。`UITableView`将其查询分发给委托（提供显示和行为属性）和数据源（提供单个单元格的值）。在实践中，这两个来源通常是同一个对象——通常是视图的控制器。这些是 iPhone 表格视图唯一的数据源，因为 iPhone OS 不支持`Cocoa bindings`技术，而该技术允许下一节描述的集合控制器工作。

虽然传统的数据源对象被视为“遗留”接口，但它们仍然提供了一些优势。它们特别适合动态生成的数据。它们可以提供比现代数据模型接口更好的性能。而且它们的组织结构与其他框架（如`Swing`）相似，这可能使移植更加容易。

#### 集合控制器

填充`NSTableView`、`NSOutlineView`、`NSBrowser`或任何其他集合视图的“现代”方法是将视图的内容属性绑定到`NSArrayController`或`NSTreeController`。优点是你不必创建自己的数据模型或数据源类，只需将现有的兼容数据模型（通常是`NSArray`）绑定到控制器，再将控制器绑定到视图；框架会处理其余的工作。如何创建绑定并不完全是直观的。与连接到表格视图对象的传统数据源方法不同，表格绑定是为每一列设置的：

-   将表格的各个列绑定到一个数组控制器，指定要在该单元格中显示的对象属性的模型键路径。要显示对象本身，请将模型键路径设置为`"self"`。
-   将大纲视图的各个列绑定到一个树控制器，指定要在该单元格中显示的对象属性的模型键路径。要显示对象本身，请将模型键路径设置为`"self"`。

控制器的键——即被绑定的整个对象集合——通常对所有列都是相同的，确保一行中的单元格都反映同一个对象。但是，你的数据可以组织成多个数组或树——只需确保这些集合保持一致。

虽然提供数据的机制是为你完成的，但你仍然可以自由定制视图的外观和感觉，方法是提供自己的单元格渲染器、指定列属性以及自定义标题视图，就像使用传统数据模型一样。

`NSCollectionView`是一个现代视图，它很大程度上取代了`NSMatrix`，并且与绑定一起工作。

#### Core Data

Core Data 是一个数据建模和持久化框架，提供从端到端的数据建模和存储。它处理了“与对象生命周期和对象图管理相关的许多任务，包括持久化。”[¹] 如果你想使用 Core Data 对象作为应用程序的一个（或主要的）数据模型，通常需要遵循以下步骤：

1.  使用 Core Data 或基于 Core Data 文档的项目模板之一创建你的 Xcode 项目。这些模板包含对将你的 Core Data 对象编码为 XML 文档的通用支持。这并不难设置，但使用模板可以节省你的精力。

2.  默认的持久化存储（即 Core Data 编码和保存数据模型的地方）将是一个 XML 文档。你可以覆盖此设置，将对象存储在 SQLite 数据库或其他格式中；请注意，创建自定义存储格式可能相当复杂。

3.  使用 Xcode 中的数据建模工具设计你的数据模型。你定义具有属性（Properties）和关系（Relationships）的实体（Entities）。当你构建应用程序时，该模型会被编译成一个在运行时加载的管理对象模型（`.mom`）文件。此模型用于创建你的数据模型对象。

4.  如果你正在创建基于文档的 Core Data 应用程序，请确保使用或子类化`NSPersistentDocument`。它会为你处理文档数据模型的大部分文档级管理。

5.  使用由 Core Data 创建的`NSManagedObject`实例，就像使用你自己的自定义数据模型对象一样。你可以获取和设置它们的属性，将它们添加到集合中，将视图绑定到它们，观察它们，等等。Core Data 框架会自动处理你的对象的获取、缓存和编码。

一个最初令人困惑的方面，特别是对于 Java 开发者来说，是`NSManagedObject`类的动态特性。数据模型中的每个实体都会在运行时实例化一个`NSManagedObject`；由 Core Data 管理的每个对象都必须是`NSManagedObject`或其子类的实例。虽然在特殊情况下可以子类化`NSManagedObject`，但大多数时候你不需要这样做。你在数据建模工具中分配给实体的属性在运行时会成为 Objective-C 属性——而且无需子类化即可实现。`NSManagedObject`利用 Objective-C 响应其未实现的消息和属性的能力，并在运行时“合成”这些属性。因此，如果你定义了一个包含名为`tag`的整数属性的实体，你可以在运行时将该对象视为`NSManagedObject`的子类，并拥有一个`int`类型的`tag`属性：`if (entity.tag>0) entity.tag -= 1`。更多细节请参见第 6 章和第 10 章。

Core Data 工具和框架使得在应用程序中开始使用 Core Data 变得非常容易。但 Core Data 实际上是一个深度且复杂的架构，具有高度的可定制性和可扩展性。如需完整解释，请查阅 Core Data 编程指南。[²]

#### 自定义数据模型对象

通常，最好的数据模型解决方案是简单地定义一个类并用数据填充它。`TicTacToe`项目中的`TicTacToeGame`类就是一个自定义数据模型。

在设计应用程序时，首先查看现成的类，如`NSArray`、`NSArrayController`和 Core Data，看看它们是否能有效地为你的数据建模。如果可以，那么请务必使用已经解决好的方案。如果不能，那么创建你自己的类或子类化现有的类，如`NSManagedObject`和`NSObjectController`。

[¹]: Apple Inc., Core Data 编程指南, http://developer.apple.com/documentation/Cocoa/Conceptual/CoreData/, 2009.
[²]: 同上。
