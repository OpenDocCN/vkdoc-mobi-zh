## 模型-视图-控制器模式

模型-视图-控制器设计模式是计算机科学中最重要的设计模式之一。大多数模式解决的是特定问题，而模型-视图-控制器（MVC）描述的是一个对象系统的架构。它可以应用于独立的子系统或整个应用程序。模型-视图-控制器设计模式的定义也比许多其他模式更模糊，为不同的实现方式留下了很大的空间。它更像是一种理念，而非一种配方。

清晰理解 MVC 对于使用 Cocoa 至关重要，因为整个 Cocoa 框架（在适用的情况下）都是围绕模型-视图-控制器设计模式设计的。如果说 Java 在许多任务中采用了 MVC，那么 Cocoa 则热切地拥抱了它。MVC 是 Cocoa 的灵魂。采用 MVC 设计将有助于你的开发，因为许多框架对象都被设计为能与 MVC 设计轻松集成。

作为一个广泛的设计模式，本章内容涵盖广泛，涉及众多类和技术。（如果仅仅描述你可能已经熟悉的 MVC 设计模式，那本章将毫无用处）。为了实用起见，每节不仅描述设计理念，还会深入探讨你将用于实现这些理念的类、协议、框架和开发工具。例如，本章从基本的 MVC 设计原则开始，接着介绍绑定（用于实现 MVC 通信的技术）和 Interface Builder（用于连接 MVC 对象的开发工具）。关于视图对象的部分解释了视图对象的作用，然后进一步说明如何使用预构建的视图对象、如何创建自己的视图对象、这些视图对象如何绘制自身，以及 Cocoa 图形环境的基础知识。关于数据模型对象的部分不仅描述了数据模型对象的概念性角色，还介绍了一些可用于创建复杂数据模型的类和开发工具。简而言之，这是一章非常长的内容，所以请静下心来慢慢阅读。你应该在充分理解每个概念之后再进入下一个概念。

### 理解模型-视图-控制器

模型-视图-控制器设计模式的表述很简单：

*   数据模型对象封装信息。
*   视图对象向用户显示信息。
*   控制器对象执行操作。
*   视图对象观察数据模型对象，并在数据变化时更新其显示。
*   视图对象收集用户输入并将其传递给执行操作的控制器对象。

成功实现 MVC 设计的关键在于密切关注对象的作用以及这些对象之间的通信。前三条规则定义了对象的作用。

*   数据模型对象存储、封装和抽象化你的数据。它们不应包含特定于使应用程序运行的方法或逻辑。例如，数据模型类应实现序列化其数据的方法，但不应实现“另存为…”命令。即使这两个功能几乎相同，处理抽象数据转换的代码应在数据模型对象中实现，而实现用户命令的代码应在控制器对象中实现。
*   视图对象向用户显示数据模型中的信息。视图类的设计范围从极其通用到非常具体；框架提供了通用的视图类来显示几乎任何类型的字符串、数字或图像，而你则更有可能实现针对应用程序设计的非常具体的视图对象。视图对象还通过与用户交互并通过解释用户发起的事件（如鼠标移动和按键）来启动操作。它将那些事件转换为要传递给控制器对象执行的*操作*。事件和产生的操作通常非常简单；在按钮对象上单击鼠标会向控制器发送一个操作消息。复杂的交互，如拖放，则更为复杂。
*   控制器对象实现应用程序的操作。操作通常由视图对象响应用户事件而发起。

当用户执行某些操作时，例如键入键盘快捷键，视图对象会对其进行解释并向控制器对象发送*操作消息*。控制器对象执行该操作，这通常涉及向数据模型对象发送消息。当数据模型发生变化时，它会通过向视图对象发送消息来通知其观察者，视图对象随后更新显示。
