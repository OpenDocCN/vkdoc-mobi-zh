# 第 26 章 ■ 运行时

Objective-C 应用程序的运行时环境具有一些您应该了解的属性。这些包括构成运行中应用程序的 Mac OS X 进程的信息、其环境变量、与其父进程的连接等。其中一些继承自操作系统，而另一些则是 Objective-C 和 Cocoa 框架的特性。许多内容已在其他章节中描述过。我们将在此与运行时环境的其他重要方面一起简要回顾它们。

### 进程

在 Mac OS X 中，每个 Objective-C 程序都作为一个进程执行。从技术上讲，Java 字节码在抽象虚拟机中执行；但大多数虚拟机在进程中运行，并具有几乎相同的属性。每个进程包括：

- 一个内存地址空间
- 一个或多个执行线程
- 一组键/值对形式的环境变量
- 一个命令行参数数组

在 Java 中，虚拟机可以访问进程的本地地址空间，但您（程序员）不能。在 Objective-C 中，您可以直接访问分配给进程的逻辑地址空间。硬件在物理上禁止直接访问进程本地地址空间之外的内存地址。任何访问尝试都会导致内存地址违规，这通常会导致进程终止。访问进程外部的任何资源都需要通过向内核发送消息来实现。您几乎不会直接这样做，但您应该知道，所有与硬件或操作系统交互的 API 最终都会向内核发送消息。

在 Mac OS X 中，进程与主线程是同义词。只要主线程存在，进程就存在，并在主线程终止时终止——或者使用 `exit()` 等函数有意终止进程。您可以创建或销毁其他线程，如第 15 章所述。

并非所有 Objective-C 代码都在其自己的进程中运行。Objective-C 还可以用于开发框架和插件。这些是可以由其他应用程序加载并在该应用程序环境中执行的代码和资源包。例如，您可以通过向操作系统提供 Spotlight 插件包来为文档提供自定义索引。该包包含在 Spotlight 搜索引擎想要索引您文档类型内容时执行的代码。您的 Spotlight 插件中的代码在元数据导入器进程中运行，而不是在您的应用程序中。

除了环境变量、命令行参数和进程属性之外，还有一些零散的进程级属性，例如当前工作目录。所有这些都将在以下部分中讨论。

#### 环境

每个进程都继承其父进程（创建您的进程的进程）定义的环境。环境是一组键/值字符串对，允许设置和其他信息从一个进程传递到另一个进程。在 Java 中，您可以使用 `java.lang.System.getenv()` 或 `java.lang.System.getenv(String)` 获取环境变量。在 Objective-C 中，向单例 `NSProcessInfo` 对象发送 `-environment` 消息，该对象通过 `[NSProcessInfo processInfo]` 获取。

### 版本控制

创建与框架或操作系统的多个版本兼容的应用程序可能是一项棘手的工作。Cocoa 框架提供了许多版本控制和兼容性特性，使这变得更容易，或者至少是可管理的。

#### 控制开发与部署版本

应用程序项目的 Xcode 构建设置包含两个重要的设置：

- **基础 SDK (`SDKROOT`)** 定义了用于编译程序的头文件和框架的版本。
- **部署目标 (`MACOSX_DEPLOYMENT_TARGET`)** 定义了允许你的应用程序运行的最早操作系统版本。

基础 SDK 选择了一组源头文件和库，你的应用程序将与之编译和链接。每个操作系统版本都定义了一组新的类、常量和数据类型。通过指定一个特定的基础 SDK，你将自己的代码限制在该 SDK 发布时可用的 API 上。你的应用程序不能使用新的类、方法或函数，因为（至少对编译器而言）这些函数不存在，你的代码将无法编译。

部署目标仅适用于 GUI 应用程序。它决定了允许你的应用程序启动的最早（最旧）操作系统版本。此构建设置的值存储在应用程序的包中，并在启动时由操作系统进行测试。

#### 测试类、方法和函数

综合来看，上述两个构建设置定义了一个你的应用程序可以运行的操作系统版本范围。这称为跨版本开发。在这个范围内，较新的操作系统将实现一些早期系统没有的特定函数、类和方法。要使用其中任何一个，你的代码必须在运行时测试它们是否存在：

- 通过调用 `NSClassFromString()` 并检查结果是否为 `Nil` 来测试类的存在性。
- 使用 `-respondsToSelector:` 测试方法的存在性。
- 通过将函数指针与 `NULL` 进行比较来测试函数的存在性。

前两种技术使用了 Objective-C 的内省机制。最后一种测试适用于 C 函数，并通过一个称为弱链接的特性工作。弱链接是对一个符号的引用，该符号被假定为由操作系统定义。如果在加载应用程序时该符号实际上不存在，则该引用被设置为 `NULL`。与强链接相反，弱链接的引用不会阻止你的应用程序加载。你可以在运行时测试符号是否存在，如下所示：

```
if (NSDisableScreenUpdates != NULL) { NSDisableScreenUpdates(); }
```

请记住，C 函数本身的名称（不带任何括号）会计算为该函数入口点的地址。关于跨版本开发设置和技术的完整描述，请参见《跨版本开发编程指南》。¹

### 软件包和捆绑包

**软件包** 是相关文件的集合。从用户的角度来看，软件包就像一个单一的文件系统对象：一个应用程序、一个文档或一个插件组件。实际上，软件包是一个可以包含任意数量文件或子目录的目录。软件包的使用方式与 JAR 文件非常相似——用于组织一组相关文件并将它们保持在一起——区别在于在 Mac OS X 中，软件包的内容没有被压缩或归档。

> **提示：** 你可以直接从 Finder 浏览软件包的内容。选择一个软件包，然后 Control/右键单击该项目。在上下文弹出菜单中选择 **显示包内容**。

**捆绑包** 是一种具有预定义结构的软件包。Cocoa 应用程序、浏览器插件、自动操作、框架和 Core Image 滤镜只是操作系统定义的几种捆绑包类型。这些捆绑包类型中的任何一种都可以包含任意数量的支持文件：NIB 文件、字符串属性、图像、声音、电影、HTML 文档、脚本、可执行实用程序，甚至其他捆绑包。你的捆绑包中包含哪些额外文件完全由你决定。

你可以使用 `NSBundle` 类来访问捆绑包并与之交互。基本的捆绑包功能及其组织方式已在第 20 章讨论过。要了解更多关于捆绑包的信息，请参考 `NSBundle` 类文档和《捆绑包编程指南》。²

并非所有程序类型都包含在捆绑包中。例如，命令行工具生成一个单一的可执行文件。额外的资源文件不能打包在命令行工具中。

### 框架

捆绑包的一种类型是**框架**。框架包含可供多个应用程序共享的编程资源的部署。一个框架可以包含共享库、Objective-C 类、图像、声音、NIB 文件和其他资源。在这方面，框架大致相当于用于部署 Java 类文件、属性文件和其他资源库的 JAR 文件。它们可以包含以下任意或全部内容：

- 已编译函数和类的共享库
- 定义框架 API 接口的头文件
- 文档
- 任意资源：NIB 文件、图像、本地化字符串、脚本、帮助文件等。

在运行时，框架会自动从多个预定义位置（最显著的是 `/System/Library/Frameworks` 和 `/Library/Frameworks`）加载，这与 Java 使用其类路径变量查找类的方式非常相似。或者，你也可以将框架嵌入到应用程序的包中，或通过编程方式定位和加载框架。要直接加载框架，请为框架的包目录创建一个 `NSBundle` 对象，并向其发送 `-load` 消息。这大致相当于创建一个 `java.lang.ClassLoader`。与 Java 不同，框架代码不会被垃圾回收。如果需要释放它，请向该包发送 `-unload` 消息。

大多数时候你使用的是他人开发的框架，但你也可以开发并部署自己的框架。框架使得开发可重用的解决方案变得容易，这些解决方案将成为新应用程序的构建块，并与这些应用程序一起部署。如果你正在考虑开发自己的框架，请参考《框架编程指南》。

### isa 交换

Objective-C 的一些“魔法”是通过一种称为 isa 交换的技巧实现的。它的名字来源于它搅动或交换了定义对象行为的 `isa` 变量。

通常，对象的 `isa` 变量在 `+alloc` 期间设置，并且永远不会改变。更改对象的 `isa` 变量实际上会更改对象的类，这定义了其结构以及它响应的所有消息。显然，您可以发现任意更改它可能会产生严重后果。

有几种技术，特别是键值观察，利用对象类的可变性来精确替换特定方法的行为。以下是当您请求观察特定属性时，键值观察所做的操作：

1.  获取对象的类。
2.  创建对象类的一个新子类。
3.  用一个返回原始对象 Class 的方法覆盖 `-class` 方法。
4.  覆盖原始类中所需的 setter 方法。
5.  将对象的 `isa` 变量设置为步骤 2 中创建的新子类。

Objective-C 类是使用 `objc_allocateClassPair()` 以编程方式创建的（尽管 KVO 可能使用一个名为 `objc_duplicateClass()` 的私有函数）。一旦定义了空类，您就可以使用 `class_addMethod()` 在类中添加或覆盖方法，使用 `class_addProtocol()` 使其符合某个协议，或者使用 `class_addIvar()` 定义实例变量。事实上，您可以以编程方式执行任何可以在 Objective-C 语言中定义的操作。

完成后，对象的行为与之前完全相同，只是步骤 4 中覆盖的消息会调用 KVO 定义的方法，而不是原始类的方法。由于 KVO 也覆盖了 `-class` 方法，因此该对象仍然显示为其原始类。您需要查看原始的 `isa` 实例变量才能看到对象中的差异。

Isa 交换是一种强大的技术，但它相当技术性，不适合胆小的人。本节旨在提供信息，而不是鼓励您自己尝试。但是，如果您确实遇到强烈需要涉及此类对象变异的解决方案，它并非遥不可及。

### 64 位编程

如果您已经涉足编程一段时间，您会看到几十年来微处理器寄存器的大小从 4 位增长到 32 位。我们现在不可避免地发现自己正处于从 32 位处理器向 64 位处理器过渡的过程中。希望向 128 位处理器的过渡还需要一段时间。

对于 Java 和 Objective-C 程序员来说，这一变革预示着新的内存容量和性能提升的前景。大多数面向对象的程序员在过渡期间几乎无事可做——除了更新他们的项目设置以生成应用程序的 64 位版本。如果您完全使用 Objective-C 类、方法和属性进行编程，您会发现这两个环境之间几乎没有什么区别。对于 Objective-C 程序员来说，最大的变化是某些整数变量类型的大小变化，如 `表 26-2` 所列。

**表 26-2.** 32 位和 64 位变量大小

| 类型 | 32 位大小 | 64 位大小 |
|------|-----------|-----------|
| `char` | 1 字节 | 1 字节 |
| `short` | 2 字节 | 2 字节 |
| `int` | 4 字节 | 4 字节 |
| `long` | 4 字节 | 8 字节 |
| `NSInteger` | 4 字节 | 8 字节 |
| `pointer` | 4 字节 | 8 字节 |
| `long long` | 8 字节 | 8 字节 |
