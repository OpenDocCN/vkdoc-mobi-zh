## 第 26 章：运行时

**提示** 一个常见错误是使用错误的格式字符串说明符来格式化 `int` 值。当编译为 64 位时，语句 `[NSString stringWithFormat:@"%d",(NSInteger)x]` 将无法正常工作；整数格式说明符（`"%d"`）期望的是 32 位整数。务必将格式说明符（`"%ld"`）与其期望的类型（`long int`）匹配。

指针变量大小的变化对于 Objective-C 程序员来说通常不是问题。所有指针（对象指针、C 类型指针、结构体指针、`CFTypeRef` 等）都始终存储在指针变量中。这种大小变化是透明的，不同指针类型之间的强制类型转换也同样透明。

除了大小变化之外，64 位的 `long int`、`long long int` 和指针变量都从 4 字节对齐变为 8 字节对齐。变量的对齐方式决定了该变量倾向于被放置到的地址位置。8 字节对齐意味着变量的地址应能被 8 整除。这主要影响结构体的组织，例如代码清单 26-3 所示的结构体。

### 代码清单 26-3. 结构体中的变量对齐

```
@interface Marker : NSObject {
    int tag;
    long long int position;
}
@end
```

在 32 位架构中，代码清单 26-3 中 `tag` 和 `position` 变量的相对地址偏移量分别为 0 和 4。当编译为 64 位架构时，相对偏移量将变为 0 和 8。变量的大小没有改变，但 `position` 整数的首选地址对齐方式从 4 字节变为 8 字节。编译器在这两个变量之间插入了 4 个未使用的字节，以便将第二个变量对齐到 8 字节地址边界。这意味着，当编译为 64 位时，该对象在内存中的大小增加了 4 个字节，尽管变量的尺寸没有变化。

以下是一个简短的清单，列出了让你的代码在 32 位和 64 位架构中都能顺畅运行的最佳实践：

*   切勿将指针值存储或传递给整型变量，反之亦然。
*   将返回的 `NSInteger` 值存储在 `NSInteger` 变量中。具体来说，`NSUInteger count = [collection count]`。
*   使用 `NSInteger` 值时，应使用对应的“整型”对象消息：使用 `-[NSCoder encodeInteger:forKey:]` 而非 `-[NSCoder encodeInt:forKey:]`，使用 `-[NSString integerValue]` 而非 `-[NSString intValue]`，依此类推。
*   确保格式字符串说明符与正确的变量类型匹配。
*   不要依赖或假设结构体的大小或组织方式。
*   通常，应使用对象和属性进行设计，而非操作结构体和指针。
