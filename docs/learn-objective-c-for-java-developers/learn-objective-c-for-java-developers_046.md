# 第 13 章 ■ 近程与远程通信

一旦调度完成，运行循环就会像分发其他类型的事件一样，向你的委托发送流事件消息。使用流事件是处理流数据最高效且首选的方法。你也可以通过在一个紧凑循环中反复向流发送`-hasBytesAvailable`消息来轮询流，但这很可能会造成 CPU 资源的巨大浪费。

## 高级通信

高级通信框架是一种面向对象的接口，用于交换消息和对象。主要有两种高级通信机制：

- 分布式通知
- 分布式对象

这些框架使用上一节中描述的底层通信类来执行实际的数据交换，从而使你摆脱大部分繁琐的细节。高级框架都具备某种形式的注册表或自动发现机制，用于连接发送方和接收方。大多数情况下，客户端和远程服务只需约定一个公共标识符；框架将负责连接两者。

### 分布式通知

分布式通知将在第 18 章中介绍，但这里也没什么可多说的。分布式通知与常规通知类似，但有两点关键区别：

- 通知会被广播到所有进程。
- 通知内容仅限于属性列表对象。

分布式通知使用 Mach 端口，这将其通信范围限制在同一引导命名空间内的进程。它们使用序列化将数据转换为适合进程间交换的形式，因此通知的名称、源对象和信息都必须为属性列表对象，详见第 12 章“Objective-C 序列化”一节。代码清单 13-3 展示了如何向运行在同一系统上的任意数量的观察者进程发送通知。

**代码清单 13-3.** 发送分布式通知

```objc
NSDictionary *info = [NSDictionary dictionaryWithObjectsAndKeys:
@"会议提醒", @"消息",
@"X-奖启动策略", @"描述",
[NSDate dateWithString:@"2009-04-02 10:00:00 -0700"], @"开始时间",
@"312 室", @"地点", nil];

[[NSDistributedNotificationCenter defaultCenter]
postNotificationName:@"PSEventReminder"
object:@"com.apress.schedule"
userInfo:info];
```

分布式通知的发布和订阅非常容易。对于独立的、低频率的、单向的广播式消息，这是目前为止最简单的进程间通信解决方案。
