## 第 24 章

## 内存管理

垃圾回收是 Objective-C 的一个近期特性，虽然这朝着正确方向迈出了一大步，但它并非普遍可用。替代方案是传统的 Objective-C 托管内存，其中你的代码负责销毁对象。

你选择使用托管内存而非垃圾回收的主要原因如下：

- 操作系统或编译器不支持垃圾回收（具体而言，iPhone OS 和旧版本的 Mac OS X 就是这种情况）。
- 你正在处理一个已经在使用托管内存的项目。
- 代码必须在托管内存环境中运行（例如，插件）。

在第一种情况下，你唯一的选择就是接受托管内存。本章将向你展示传统内存管理的基础知识。Objective-C 提供了一个非常合理且易于学习的对象管理模型。如果你认为这会像 C 或 C++内存管理一样，那你可以放心了。

在第二种情况下，将现有项目从托管内存转换为垃圾回收可能得不偿失。Objective-C 中的托管内存与垃圾回收并无根本性不同，因此采用现有方案可能是最便捷的解决方案。

> **注意**：在编译将在托管内存环境中运行的代码时，应关闭编译器对垃圾回收的支持。在 Xcode 中，通过将`Objective-C Garbage Collection`构建设置设置为`Unsupported`来完成此操作。这会关闭所有垃圾回收支持，生成与缺少垃圾回收的操作系统兼容的代码。

专用的组件（如插件、框架和服务）被加载到另一个应用程序的运行时环境中，必须与其宿主采用的内存管理方案共存。甚至可能需要编写代码使其既能运行在垃圾回收环境，也能运行在托管内存环境中。本章末尾有一些关于如何做到这一点的说明。

本章解释了内存分配和释放的基础知识。然后描述了 Objective-C 的引用计数方案和自动释放池。在检查每个概念时，会引入简单的编程规则。综合起来，它们定义了你在使用托管内存时应采用的编程实践。后续部分将涉及引用计数可能变得棘手的一些情况以及如何处理它们。

### C 语言内存分配

所有内存管理的核心是 POSIX 内存分配函数。它们被所有高级 API 使用，你也可以自己使用它们。它们极其简单、快速且高效。基本的使用模式如清单 24-1 所示。

**清单 24-1**. C 语言内存分配

```c
void *memory = malloc(100);

…

free(memory);
```

清单 24-1 中的`malloc()`函数分配了 100 个连续的内存字节，并返回第一个字节的地址。你可以以任何你认为合适的方式使用这 100 个字节。当你处理完该内存块后，`free()`函数会释放并回收它，之后你不应再访问该内存范围。

`malloc()`还有一些变体。最常见的两个是`calloc()`（分配内存并用零填充）和`realloc()`（更改先前分配的内存块的大小）。无论你使用哪个 POSIX 函数来分配内存，`free()`都会释放它。

尽管 POSIX 内存分配很简单，但它也非常笨拙：应用程序有责任在使用完内存后释放它，它只能被释放一次，并且之后不能再使用该指针。这带来了以下困难：

- 如果你未能释放内存，但“忘记”了指向它的指针，该内存块将永久不可用。这称为**内存泄漏**。
- 与其他代码共享指针会造成关于哪段代码应负责释放内存以及何时释放的歧义。
- 使用指向已释放内存块的指针可能是灾难性的。
