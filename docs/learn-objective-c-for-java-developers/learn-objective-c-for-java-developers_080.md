# 关键事件

关键事件沿着对象层级向下分发，通常是为了寻找第一响应者。事件的类型以及多层过滤会极大地影响关键事件的分发。关键事件首先传递给应用对象，然后传递给关键窗口，再传递给第一响应者，在某些情况下还会传递给菜单栏。

简而言之，关键事件会经历以下步骤：

1.  操作系统将硬件按键码转换为用户当前使用的语言。关键事件会经过系统级按键绑定进行过滤，然后传递给已注册拦截全局关键事件的系统组件。这就是“热键”的处理方式。
2.  关键事件被推送到活动应用的事件队列中。
3.  应用的主运行循环取出下一个关键事件，将其包装成一个 `NSEvent` 对象，并通过 `-sendEvent:` 消息发送给 `NSApplication` 对象。
4.  如果该按键可能是等效按键（通常是 Command+ 按键组合），应用会向关键窗口发送一条 `-performKeyEquivalent:` 消息，该窗口会解释该消息或将其传递给第一响应者。如果两者都没有响应此事件，则将其发送到应用的菜单栏。
5.  将 `-sendEvent:` 消息发送给关键窗口。
6.  如果该按键是一个界面控制键（如 Tab、右箭头、向下翻页、Home），则由当前活动视图定义，窗口和/或控件会将其转换为一个操作，并向窗口或其第一响应者发送相应的按键操作消息（分别是 `-setNextKeyView:`、`-forward:`、`-pageDown:`、`-moveToBeginningOfDocument:`）。
7.  如果该事件既不是等效按键也不是界面控制，则第一响应者会收到一条 `-keyDown:`、`-keyUp:` 或 `-flagsChanged:` 消息。
8.  如果没有第一响应者且窗口未处理该关键事件，则会将其传递给菜单栏。

大多数视图只需要重写 `-keyDown:`，可能还需要重写 `-keyUp:` 或 `-flagsChanged:`，即可自动接收常规的关键事件。如果你正在编写一个编辑文本的视图——我不建议你尝试这样做——你的视图对象必须遵循 `NSTextInput` 协议。

说实话，关键事件的处理过程比这要复杂得多，但除非你正在编写一个需要过滤按键或创建按键绑定的应用程序，否则你通常可以忽略关键事件处理中更深奥的方面。完整的描述可以在 Cocoa 事件处理指南中找到。

##### 鼠标事件

鼠标和触摸事件是利用用户界面的几何形状来分发的。鼠标事件总是与位置和窗口相关联，而位置和窗口则用于确定事件的接收者。

## 鼠标按下事件

鼠标事件处理通常始于鼠标按下事件：

1.  当鼠标被点击时，系统的 `WindowServer` 会确定哪个应用程序应该接收该事件。
2.  应用的主运行循环通过一条 `-sendEvent:` 消息将鼠标事件发送给 `NSApplication` 对象。
3.  `NSApplication` 使用事件的位置来确定事件发生在哪个窗口中，并通过另一条 `-sendEvent:` 消息将事件转发给该窗口。
4.  窗口的 `-sendEvent:` 方法使用该位置找到与点击相对应的子视图，并向该视图发送一条 `-mouseDown:` 消息。

视图中的第一次鼠标按下事件会试图使其成为第一响应者。这会使该视图成为关键事件的接收者；其窗口将成为关键窗口，并固有地重新定义响应者链，这将在本章后面描述。要符合条件，视图在收到 `-acceptsFirstResponder` 消息时必须返回 `YES`。默认情况下，`NSView` 返回 `NO`。如果视图同意，它将被指定为第一响应者，并收到一条 `-becomeFirstResponder` 消息。当窗口被停用，或另一个视图成为第一响应者时，该视图会收到一条匹配的 `-resignFirstResponder` 消息。

视图也可以控制“点击穿透”。通常，在非活动窗口中点击鼠标会激活该窗口，但不会对该事件进行任何其他处理。在非活动窗口中点击控件通常需要两次点击：第一次激活窗口，第二次向视图发送 `-mouseDown:` 事件。视图可以通过重写 `-acceptsFirstMouse:` 来改变此行为。当窗口通过鼠标点击被激活时，它会通过向点击坐标下方的视图发送一条 `-acceptsFirstMouse:` 消息来查询该视图。如果视图返回 `YES`，窗口将立即向视图发送一条 `-mouseDown:` 消息——通过一个单一事件同时激活窗口和点击视图。默认情况下，对于 `-acceptsFirstMouse:`，`NSView` 返回 `NO`。

## 鼠标拖动和鼠标松开事件

一旦视图收到一条 `-mouseDown:` 消息，它随后可能会收到 `-mouseDragged:` 和 `-mouseUp:` 消息。这些消息统称为鼠标跟踪消息。即使鼠标被拖到视图框架之外，同一个视图对象也会收到随后的 `-mouseDragged:` 和 `-mouseUp:` 消息。`-mouseDragged:` 仅在鼠标按钮按住的同时移动鼠标时发送。视图可能会收到多条 `-mouseDragged:` 消息，也可能一条都没有。视图在收到 `-mouseDown:` 消息之前，绝不会收到 `-mouseDragged:` 或 `-mouseUp:` 消息。

如果你希望你的视图在鼠标拖动手势期间执行一些有趣的操作，你可能需要重写全部三个方法。如果你只对实现点击行为感兴趣，请重写 `-mouseUp:` 并测试事件的位置，以确保鼠标仍然位于你的对象之上。

TicTacToe 项目中的 `-mouseUp:` 处理程序如代码清单 20-6 所示。

**代码清单 20-6.** `-mouseUp:` 事件处理程序

```
- (void)mouseUp:(NSEvent*)theEvent
{
    NSPoint loc = [self convertPoint:[theEvent locationInWindow] fromView:nil];
    SquareIndex index;

    for (index=0; index<9; index++) {
        if (NSPointInRect(loc,[ChalkboardView rectOfSquare:index])) {
            [[self document] playerClickedSquare:index];
            break;
        }
    }
}
```

代码清单 20-6 中的方法通过一个 `NSEvent` 对象接收鼠标事件信息。事件总是在窗口的坐标系中，因此它首先要做的是将位置转换为视图的局部坐标系。然后它进行搜索，看用户是否在游戏板的九个方格中的任何一个内部松开了鼠标。如果是，视图会向其控制器对象发送一条 `-playerClickedSquare:` 消息，控制器将执行移动并更新游戏板。

处理鼠标拖动事件的另一种方法是留在你的 `-mouseDown:` 方法中，并启动一个嵌套的运行循环，直到拖动完成。这实质上是一个“模态”鼠标拖动循环，它会阻止应用程序的其余部分执行，直到拖动结束。Cocoa 事件处理指南中包含一个鼠标跟踪循环的示例。

## 鼠标跟踪

拖动手势期间的鼠标移动是自动提供的。鼠标按钮松开时的鼠标移动通常被忽略。鼠标移动很常见，计算成本高，并且通常对应用程序没有意义。

但是，在某些情况下，你可能希望在界面中接收到普通鼠标移动的通知。有两种技术可以在鼠标按钮松开时跟踪鼠标。

第一种方法是向窗口请求鼠标移动事件：

1.  将包含该视图的窗口的 `acceptsMouseMovedEvents` 属性设置为 `YES`。
2.  在子视图中实现 `-mouseMoved:` 方法。

一旦设置了 `acceptsMouseMovedEvents`，窗口就会开始向其视图发送 `-mouseMoved:` 消息，这与其分发其他鼠标事件的方式非常相似。
