# 安抚提基神

你可能很清楚，将任何编程书籍的第一个项目命名为“Hello, World”已经成为一种传统。我们曾考虑打破这个传统，但又担心提基神会因为我们如此严重地违反礼仪而对我们施以痛苦的惩罚。所以，我们还是按规矩来，好吗？

在本章中，我们将使用 `Xcode` 创建一个小的 iOS 应用程序，它将显示文本“Hello, World!”。我们将探讨在 `Xcode` 中创建 iOS 应用程序项目所涉及的步骤，详细了解如何使用 `Xcode` 的 Interface Builder 来设计应用程序的用户界面，然后在 iOS 模拟器上运行我们的应用程序。之后，我们将为应用程序添加一个图标，使其更像一个真正的 iOS 应用程序。

我们有很多事情要做，那就开始吧。

## 在 Xcode 中设置你的项目

现在，你的机器上应该已经安装了 `Xcode` 和 iOS SDK。你还应该从 Apress 网站（`http://apress.com`）下载本书的源代码压缩包。同时，也可以看看 `http://forum.learncocoa.org/` 上的本书论坛。本书论坛是讨论 iOS 开发、解答疑问以及结识志同道合之人的绝佳场所。

**注意** 即使本书的源代码压缩包中提供了完整的项目文件，但如果你手动创建每个项目，而不是简单地运行下载的版本，你会从本书中获得更多收获。通过这样做，你将熟悉并掌握使用各种应用程序开发工具的技巧。

没有什么能替代实际创建应用程序的体验；软件开发不是一项旁观者的运动。

本章我们将要构建的项目位于源代码压缩包的 `02 - Hello World` 文件夹中。

在开始之前，我们需要启动 `Xcode`。`Xcode` 是我们在本书中完成大部分工作所使用的工具。从 Mac App Store 下载后，你会发现它安装在 `/Applications` 文件夹中，和大多数 Mac 应用程序一样。你会经常使用 `Xcode`，所以不妨考虑将它拖到你的 Dock 中，以便随时使用。

如果这是你第一次使用 `Xcode`，别担心；我们会逐步引导你完成创建新项目的每一个步骤。如果你已经是老手，但还没用过 `Xcode 6`，你会发现很多地方都变了（我们认为大多数是变好了）。

### 工具栏

Xcode 项目窗口的顶部称为**工具栏**（参见图 2-6）。在工具栏的左侧，有用于开始和停止运行项目的控件，以及一个用于选择要运行的样式的弹出菜单。一个**样式（scheme）**将目标（target）和构建设置结合在在一起，而工具栏弹出菜单让你可以快速轻松地选择特定的设置。

![9781484202005_Fig02-06.jpg](img/9781484202005_Fig02-06.jpg)

图 2-6：Xcode 工具栏

工具栏中间的大方框是**活动视图（Activity View）**。顾名思义，活动视图显示当前正在发生的任何操作或进程。例如，当你运行项目时，活动视图会实时显示构建应用程序所采取的各个步骤。如果遇到任何错误或警告，该信息也会在此显示。如果你点击警告或错误，你将直接进入**问题导航器（Issue Navigator）**，它提供了关于警告或错误的更多信息，如下一节所述。

在工具栏的右侧有两组按钮。左边的那组让你可以在三种不同的编辑器配置之间切换：

*   **编辑区域（Editor Area）**为你提供一个专用于编辑文件或项目特定配置值的单面板。
*   极其强大的**辅助编辑器（Assistant Editor）**将编辑区域分割为左右两个面板。右侧面板通常用于显示与左侧文件相关的文件，或者在编辑左侧文件时你可能需要参考的文件。你可以手动指定每个面板中显示的内容，或者让 Xcode 决定当前任务最合适的内容。例如，如果你正在编辑一个 Objective-C 类的实现文件（`.m`文件），Xcode 会自动在右侧面板中显示该类的头文件（`.h`文件）。如果你在左侧设计用户界面，Xcode 会在右侧显示用户界面可以与之交互的代码。你将在本书中看到辅助编辑器的实际应用。
*   **版本编辑器（Version Editor）**按钮将编辑器面板转换为一个类似时间机器的比较视图，可与 Subversion 和 Git 等版本控制系统配合使用。你可以比较当前版本的源文件与之前提交的版本，或者比较任意两个早期版本。

在编辑器按钮的右侧是一组开关按钮，用于显示和隐藏编辑器视图左侧和右侧的大面板，以及窗口底部的调试区域。分别点击这些按钮几次，看看这些面板的实际效果。你很快会学到更多关于它们如何使用的知识。

## Xcode 界面与功能详解

跳转栏左端那个看起来很有趣的图标实际上是一个弹出菜单，它会显示子菜单，列出最近的文件、对应文件、超类、子类、同级文件、分类、包含文件等等！这里显示的子菜单可以带你跳转到与当前编辑器中所打开代码相关的几乎所有其他代码。

在 *über* 菜单的右侧，有左右箭头，分别用于返回上一个文件和前进到下一个文件。

跳转栏包含一个分段弹出菜单，显示在项目中到达所选文件的层级路径。你可以点击任何显示组名或文件名的分段，查看位于层级结构中同一位置的所有其他文件和组。最后一个分段显示所选文件内项目的列表。在图 2-15 中，你会看到跳转栏的末端是一个弹出菜单，显示当前所选文件中包含的方法和其他符号。跳转栏显示文件 `AppDelegate.m`，并带有一个子菜单，列出该文件中定义的符号。

### Xcode 键盘快捷键

如果你更喜欢使用键盘快捷键而不是鼠标点击屏幕控件，那么你会喜欢 Xcode 提供的功能。你在 Xcode 中经常执行的大多数操作都分配了键盘快捷键，例如 ![image](img/flower.jpg) `B` 用于构建应用程序，或 ![image](img/flower.jpg) `N` 用于创建新文件。

你可以在 Xcode 的偏好设置中，在 **Key Bindings** 选项卡下更改所有 Xcode 的键盘快捷键，并为尚未分配快捷键的命令添加快捷键。

一个非常方便的键盘快捷键是 ![image](img/sym3.jpg)![image](img/flower.jpg) `O`，这是 Xcode 的 **Open Quickly** 功能。按下此快捷键后，开始输入文件、设置或符号的名称，Xcode 会显示一个选项列表。当你将列表缩小到所需文件时，按下 **Return** 键即可在编辑窗格中打开它，让你只需几次按键就能切换文件。

### 实用工具区

如前所述，Xcode 工具栏右侧倒数第二个按钮用于打开和关闭实用工具区。实用工具区的上半部分是一个上下文相关的检查器面板，其内容会根据编辑器窗格中显示的内容而变化。实用工具区的下半部分包含几种不同类型的资源，你可以将它们拖拽到项目中。你会在本书中看到相关示例。

### 界面生成器

早期版本的 Xcode 包含一个独立的界面设计应用程序，称为 **Interface Builder**，用于构建和自定义项目的用户界面。后续版本引入的重大变化之一是将 Interface Builder 集成到工作区本身。Interface Builder 不再是一个独立的应用程序，这意味着你不必在代码和界面演变时在 Xcode 和 Interface Builder 之间来回切换。这一转变已经过去几年了，但我们这些还记得独立版 Interface Builder 日子的人，现在对于 Interface Builder 直接集成到 Xcode 中的效果感到非常满意。

我们将在本书中大量使用 Xcode 的界面构建功能，深入探索其各个角落。事实上，我们将在本章稍后进行第一次界面构建。

### 新编译器和调试器

Xcode 4 带来的最重要的变化之一隐藏在底层：一个全新的编译器和一个低级调试器。两者都比其前身更快、更智能，并且之后的每个版本都进行了改进。

多年来，Apple 一直使用 GCC（GNU 编译器集合）作为其编译器技术的基础。但在过去几年中，它已完全转向 LLVM（低级虚拟机）编译器。LLVM 生成的代码远比传统 GCC 生成的代码快得多。除了生成更快的代码之外，LLVM 还更了解你的代码，因此它可以生成更智能、更精确的错误消息和警告。

Xcode 还与 LLVM 紧密集成，这赋予了它一些新的超级能力。Xcode 可以提供更精确的代码补全，并且在产生警告时，它可以对一段代码的实际意图进行有根据的猜测，并提供一个可能的修复弹出菜单。这使得拼写错误的符号名称、不匹配的括号和缺失的分号等错误变得容易发现和修复。

LLVM 还引入了复杂的 **静态分析器**，可以扫描你的代码以发现各种潜在问题，包括 Objective-C 内存管理问题。实际上，LLVM 对此非常智能，只要你编写代码时遵守一些简单的规则，它就能为你处理大部分内存管理任务。我们将在下一章开始研究名为 **自动引用计数**（ARC）的精彩新特性。

## 项目结构与界面构建器

### 项目导航器文件夹

**Hello World**：第一个文件夹始终以你的项目命名，你将在此投入大部分时间。你编写的大部分代码以及构成应用程序用户界面的文件都会放在这里。你可以自由地在 `Hello World` 文件夹下创建子文件夹来组织代码，如果你更喜欢不同的组织方式，甚至可以使用其他分组。虽然我们直到下一章才会涉及此文件夹中的大部分文件，但有一个文件我们将在下一节使用界面构建器时进行探索：

- `Main.storyboard` 包含特定于项目主视图控制器的用户界面元素。

- **Supporting Files**：此文件夹位于 `Hello World` 文件夹内，包含非 Objective-C 类但项目所需的源代码文件和资源。通常，你不需要花费大量时间在 `Supporting Files` 文件夹上。当你创建一个新的 iOS 应用程序项目时，此文件夹包含两个文件：
    - `Info.plist` 是一个属性列表，包含应用程序的相关信息，例如其名称、在运行的设备上是否需要任何特定功能等。
    - `main.m` 包含应用程序的 `main()` 方法。通常你无需编辑或更改此文件。事实上，如果你不清楚自己在做什么，最好不要碰它。

- **Hello WorldTests**：此文件夹包含如果你想为应用程序代码编写一些单元测试所需的初始文件。我们不会在本书中讨论单元测试，但 Xcode 能在你创建的每个新项目中为你设置这些内容，这一点很不错。与 `Hello World` 文件夹一样，此文件夹也包含自己的 `Supporting Files` 文件夹和 `Info.plist` 文件。

- **Products**：此文件夹包含项目构建后生成的应用程序。展开 `Products`，你会看到一个名为 `Hello World.app` 的项，即这个特定项目创建的应用程序。它还包含一个名为 `Hello WorldTests.xctest` 的项，代表测试代码。这两项都称为“构建目标”。由于我们从未构建过它们，两者都呈红色，这是 Xcode 用来告诉你文件引用指向不存在内容的方式。

**注意**：导航器区域中的“文件夹”不一定对应你 Mac 文件系统中的文件夹。这些是 Xcode 内部的逻辑分组，旨在帮助你保持一切井井有条，并在开发应用程序时更快、更轻松地找到所需内容。通常，这些项目文件夹中包含的项目直接存储在项目目录中，但你可以将它们存储在任何地方——甚至根据你的需要存储在项目文件夹之外。Xcode 内部的层次结构与文件系统层次结构完全独立，因此，例如在 Xcode 中将文件移出 `Supporting Files` 文件夹并不会改变该文件在硬盘上的位置。

可以使用实用工具面板将分组配置为使用特定的文件系统目录。然而，默认情况下，添加到项目中的新分组与文件系统完全独立，其内容可以存放在任何位置。

### 文件格式

界面构建器支持几种不同的文件类型。最古老的是使用 `.nib` 扩展名的二进制格式，其较新版本是使用 `.xib` 扩展名的基于 XML 的格式。这两种格式包含的文档类型完全相同，但 `.xib` 版本作为基于文本的格式具有许多优势，尤其是当你使用任何类型的 SCM 时。在界面构建器诞生后的前 20 年里，其所有文件都使用 `.nib` 扩展名。因此，大多数开发者习惯将界面构建器文件称为“nib 文件”。无论文件实际使用的扩展名是 `.xib` 还是 `.nib`，界面构建器文件通常都被称为 nib 文件。事实上，Apple 在其文档中仍使用术语 `nib` 和 `nib file`。

每个 nib 文件可以包含任意数量的对象，但在开发 iOS 项目时，每个 nib 文件通常只包含一个视图（通常是全屏视图）以及与之关联的控制器或其他对象。这使我们能够将应用程序模块化，只在需要显示视图时才加载其 nib 文件。最终结果是：当我们的应用程序在内存受限的 iOS 设备上运行时，可以节省内存。一个新创建的 iOS 项目包含一个名为 `LaunchScreen.xib` 的 nib 文件，其中包含应用程序启动时默认显示的屏幕布局。我们将在本章末尾详细讨论此文件。

界面构建器在过去几年中支持的另一文件格式是“storyboard”。你可以将 storyboard 视为一种“元 nib 文件”，因为它可以包含多个视图控制器，以及关于它们在应用程序运行时如何相互连接的信息。与 nib 文件一次性加载所有内容不同，storyboard 不能包含独立视图，也从不一次性加载所有内容。相反，你在需要时请求加载特定的控制器。Xcode 6 中的 iOS 项目模板都使用 storyboard，因此本书中的所有示例都将从 storyboard 开始。现在让我们回到界面构建器和 Hello World 应用程序的 `Main.storyboard` 文件（图 2-16）。

### Storyboard

你现在看到的是用于为 iOS 应用程序构建用户界面的主要工具。现在，假设你想创建一个按钮实例。你可以通过编写代码来创建按钮，但通过从库中拖出按钮并指定其属性来创建界面对象要简单得多，并且在运行时会产生完全相同的结果。

我们当前正在查看的 `Main.storyboard` 文件会在应用程序启动时自动加载（目前无需担心如何加载），因此它是添加构成应用程序用户界面对象的正确位置。当你在界面构建器中创建对象时，它们会在加载该 storyboard 或 nib 文件时在你的程序中被实例化。在本书中，你将看到许多此过程的示例。

每个 storyboard 都被模块化为一对或多对视图和控制器。视图是你可以直观地看到并在界面构建器中编辑的部分，而控制器是你将编写的应用程序代码，用于在用户与你的应用程序交互时执行操作。控制器是应用程序真正运作的地方。


在 IB 中，你经常看到一个由正方形表示的视图，当前的例子也不例外。这个正方形代表 iOS 设备的屏幕（实际上，它代表一个视图控制器——你将在下一章了解这个概念，但这个特定的视图控制器覆盖了设备的整个屏幕，所以它们几乎是一回事）。为什么屏幕是正方形？毕竟，这是一个 iPhone 项目，而 iPhone 看起来根本不像那样！在 iOS 早期，只有 iPhone 和 iPod touch。支持 iOS 开发的早期版本 Interface Builder 展示了一个 iPhone 形状的设计区域，而现在你看到的是一个正方形。当 iPad 出现时，Interface Builder 得到了增强，允许你设计 iPhone 形状和 iPad 形状的用户界面。为了构建能在两种设备上运行的应用程序（通用应用程序），你需要为 iPhone 和 iPad 分别构建一个故事板（或 nib 文件）。当处理 iPad 故事板时，Interface Builder 会为你提供一个 iPad 形状的轮廓来进行设计。在 Xcode 6 和 iOS 8 中，情况发生了变化。现在，Apple 希望鼓励你构建能够在任意尺寸屏幕上尽可能良好运行的应用程序。不再需要两个故事板，而应该只有一个。当你的应用程序在设备上启动时，它应该根据所遇到的屏幕形状进行自适应（实际上，Apple 将这样设计的应用称为**自适应**应用）。所以现在，Interface Builder 为你呈现一个正方形设计区域，以鼓励你不要考虑特定设备的屏幕尺寸。在本书前半部分，你将学习如何设计自适应应用程序，但我们需要先学会走再跑，我们最初的几个示例将针对 iPhone 和 iPod touch 尺寸的屏幕进行构建。

回到我们的故事板，点击正方形轮廓内的任意位置，你将在其顶部看到一行三个图标，如图 Figure 2-16 所示。将鼠标悬停在每个图标上，你会看到弹出提示显示它们的名称：`View Controller`、`First Responder` 和 `Exit`。暂时忽略 `Exit`，专注于另外两个真正重要的图标。

*   **`View Controller`** 代表一个控制器对象，它与关联的视图一起从文件存储中加载。视图控制器的任务是管理用户在屏幕上看到的内容。一个典型的应用程序有多个视图控制器，每个屏幕一个。编写只有一个屏幕（因此只有一个视图控制器）的应用程序是完全可能的，本书中的许多示例也只有一个视图控制器。
*   **`First Responder`** 简单来说，是用户当前正在交互的对象。例如，如果用户当前正在文本字段中输入数据，那么该字段就是当前的 `First Responder`。`First Responder` 会随着用户与用户界面的交互而改变，而 **`First Responder`** 图标为你提供了一种便捷方式来与当前 `First Responder` 的任何控件或其他对象通信，而无需编写代码来确定是哪个控件或视图。

从下一章开始，我们将更详细地讨论这些对象，所以如果你现在对何时使用 `First Responder` 或 `View Controller` 如何加载感到有点模糊，不用担心。

除了这些图标外，你在编辑区域看到的其余部分是可以放置图形对象的空间。但在此之前，关于 IB 的编辑器区域还有一件事你应该了解：它的层级视图——或者更准确地说，**`Document Outline`**。`Document Outline` 如图 Figure 2-17 所示。

实用工具视图的下半部分被称为 **`Library Pane`**，或简称为 **`Library`**。`Library` 是可以在你自己的程序中使用的可重用项目的集合。库窗格顶部栏中的四个图标将其分为四个部分：

*   **`File Template Library`**：此部分包含一组文件模板，当需要向项目添加新文件时可以使用。例如，如果要向项目添加新文件，一种方法是从 `File Template Library` 中拖入所需类型的文件。
*   **`Code Snippet Library`**：此部分包含一组代码片段，你可以将其拖入源代码文件中。记不住 Objective-C 快速枚举的语法？没关系——只需将该特定片段从库中拖出，你就不需要去查了。你是否编写了一些你认为以后可能会再次使用的内容？在文本编辑器中选中它，然后将其拖到 `Code Snippet Library` 中。
*   **`Object Library`**：此部分充满了可重用对象，例如文本字段、标签、滑块、按钮以及设计 iOS 界面所需的几乎所有对象。在本书中，我们将大量使用 `Object Library` 来构建示例程序的界面。
*   **`Media Library`**：顾名思义，此部分用于所有媒体文件，包括图片、声音和电影。

**注意** `Object Library` 中的项目主要来自 iOS UIKit，这是一个用于创建应用用户界面的对象框架。`UIKit` 在 Cocoa Touch 中的作用与 `AppKit` 在 Cocoa 中的作用相同。这两个框架在概念上相似；然而，由于平台的差异，它们之间显然存在许多不同之处。另一方面，Foundation 框架类，如 `NSString` 和 `NSArray`，在 Cocoa 和 Cocoa Touch 中是共享的。

### 模型-视图-控制器范式

在深入之前，我们先来了解一些理论。Cocoa Touch 的设计者遵循了一个名为**模型-视图-控制器**（MVC）的概念，这是一种将构成 GUI 应用的代码进行逻辑划分的方法。如今，几乎所有面向对象的框架都在一定程度上借鉴了 MVC，但很少有框架能像 Cocoa Touch 那样忠实于 MVC 模型。

MVC 模式将所有功能分为三个不同的类别：

*   **模型**：持有应用数据的类。
*   **视图**：由窗口、控件和其他用户可以看到并与之交互的元素组成。
*   **控制器**：将模型和视图结合起来的代码。它包含决定如何处理用户输入的应用逻辑。

MVC 的目标是尽可能让实现这三类代码的对象彼此区分开。你创建的任何对象都应能轻易识别为属于三个类别之一，并且很少或没有可归类为其他两个类别的功能。例如，实现按钮的对象不应包含处理该按钮点击时数据的代码，而银行账户的实现也不应包含绘制表格以显示其交易记录的代码。

MVC 有助于确保最大的可重用性。实现通用按钮的类可以在任何应用中使用。而实现一个点击时执行特定计算的按钮的类，则只能在其最初编写的应用中使用。

当你编写 Cocoa Touch 应用时，你将主要使用 Interface Builder 创建视图组件，尽管你也会通过代码修改，甚至有时创建用户界面。

你的模型将通过编写 Objective-C 类来持有应用数据，或使用名为 Core Data 的工具构建数据模型来创建，你将在第 13 章中了解这一点。在本章的应用中，我们不会创建任何模型对象，因为我们不需要存储或保留数据。不过，在后续章节中，随着应用变得越来越复杂，我们将引入模型对象。

### 理解输出口和动作



在 Chapter 2 中，您使用 Xcode 的 Interface Builder 设计了一个简单的用户界面。刚才，您看到了一个视图控制器类的框架。这个视图控制器类中的代码必须以某种方式与故事板中的对象进行交互，对吗？

当然可以！控制器类可以通过一种称为**出口**的特殊属性来引用故事板或 nib 文件中的对象。可以将出口视为指向用户界面中某个对象的指针。例如，假设您在 Interface Builder 中创建了一个文本标签（正如我们在 Chapter 2 中所做的那样），并希望从代码中更改该标签的文本。通过声明一个出口并将该出口连接到标签对象，您就可以在代码中使用该出口来更改标签显示的文本。您将在本章中了解如何实现这一点。

相反方向，故事板或 nib 文件中的界面对象可以设置为触发控制器类中的特殊方法。这些特殊方法称为**动作方法**（或简称**动作**）。例如，您可以告诉 Interface Builder，当用户点击按钮时，应调用代码中的特定动作方法。您甚至可以告诉 Interface Builder，当用户首次触摸按钮时，调用一个动作方法；然后，当手指从按钮上抬起时，调用另一个不同的动作方法。

Xcode 支持多种创建出口和动作的方式。一种方式是在源代码中指定它们，然后再使用 Interface Builder 将它们与代码连接。Xcode 的助理视图提供了一种更快、更直观的方法，让您可以在一步中创建并连接出口和动作，我们稍后将探讨这一过程。但在开始建立连接之前，让我们更详细地讨论一下出口和动作。出口和动作是您创建 iOS 应用时最基础的两个构建块，因此理解它们是什么以及如何工作非常重要。

#### 出口

出口是使用关键字`IBOutlet`声明的特殊 Objective-C 类属性。声明出口可以在控制器的类头文件中进行，也可以在控制器实现文件的一个特殊部分（称为**类扩展**）中进行。它可能看起来像这样：

```
@property (weak, nonatomic) IBOutlet UIButton *myButton;
```

这个例子是一个名为`myButton`的出口，它可以被设置为指向用户界面中的任何按钮。

`IBOutlet`关键字并非 Objective-C 语言的内置部分。它是系统头文件中的一个简单 C 预处理器定义，看起来像这样：

```
#ifndef IBOutlet
#define IBOutlet
#endif
```

感到困惑吗？`IBOutlet`对编译器而言绝对不做任何事。它的唯一目的是作为一个提示，告诉 Xcode 这是一个我们想要连接到故事板或 nib 文件中对象的属性。您创建并希望连接到故事板或 nib 文件中对象的任何属性，都必须以`IBOutlet`关键字开头。幸运的是，Xcode 现在会自动为我们创建出口。

##### 出口的变更

随着时间的推移，Apple 改变了出口的声明和使用方式。由于您可能在某时遇到旧代码，让我们看看出口是如何变化的。

在本书的第一版中，我们为出口同时声明了属性及其底层实例变量。那时，属性是 Objective-C 语言中的一个新结构，它们要求您声明一个对应的实例变量：

```
@interface MyViewController : UIViewController
{
    UIButton *myButton;
}
@property (weak, nonatomic) UIButton *myButton;
@end
```

当时，我们将`IBOutlet`关键字放在实例变量声明之前：

```
IBOutlet UIButton *myButton;
```

这是当时 Apple 示例代码的写法。这也是`IBOutlet`关键字在 Cocoa 和 NeXTSTEP 中的传统用法。

到了我们编写本书第二版时，Apple 已经不再将`IBOutlet`关键字放在实例变量之前，而是标准做法是将其放在属性声明中：

```
@property (weak, nonatomic) IBOutlet UIButton *myButton;
```

尽管这两种方法仍然有效（并且现在仍然有效），但我们遵循了 Apple 的指导，修改了书中的代码，使`IBOutlet`关键字位于属性声明而非实例变量声明中。

当 Apple 最近将默认编译器从 GNU C 编译器（GCC）切换到低级虚拟机（LLVM）时，为属性声明实例变量变得不再必要。如果 LLVM 发现一个没有匹配实例变量的属性，它会自动创建一个。因此，在本书的这个版本中，我们完全停止为出口声明实例变量。

所有这些方法都做完全相同的事情，即告诉 Interface Builder 出口的存在。将`IBOutlet`关键字放在属性声明上是 Apple 当前的推荐做法，因此我们将使用这种方式。但我们希望让您了解这段历史，以防您遇到旧代码在实例变量上放置`IBOutlet`关键字的情况。

您可以在 Scott Knaster、Waqar Malik 和 Mark Dalrymple 所著的*Learn Objective-C on the Mac*（Apress，2012）以及 Apple 开发者网站上的文档*Introduction to the Objective-C Programming Language*（位于`http://developer.apple.com/library/ios/#documentation/Cocoa/Conceptual/ProgrammingWithObjectiveC`）中了解更多关于 Objective-C 属性的信息。

#### 动作

简而言之，动作是使用特殊返回类型`IBAction`声明的方法，它告诉 Interface Builder 此方法可以由故事板或 nib 文件中的控件触发。动作方法的声明通常如下所示：

```
- (IBAction)doSomething:(id)sender;
```

它也可能看起来像这样：

```
- (IBAction)doSomething;
```

方法的具体名称可以是任何内容，但返回类型必须为`IBAction`，这与声明返回类型为`void`是一样的。`void`返回类型用于指定方法不返回值。此外，该方法要么不接收参数，要么接收一个参数，通常称为`sender`。当动作方法被调用时，`sender`将包含调用它的对象的指针。例如，如果该动作方法是在用户点击按钮时触发的，那么`sender`将指向被点击的按钮。`sender`参数的存在使您能够使用单个动作方法响应多个控件。它为您提供了一种识别哪个控件调用了该动作方法的方式。

**提示** 实际上还有第三种不太常用的`IBAction`声明类型，如下所示：

```
- (IBAction)doSomething:(id)sender
                   forEvent:(UIEvent *)event;
```

我们将在下一章开始讨论控件事件。

如果您用`sender`参数声明一个动作方法然后忽略它，这并不会造成任何问题。您很可能会看到大量代码正是这样做的。Cocoa 和 NeXTSTEP 中的动作方法无论是否使用`sender`，都需要接受它，因此很多 iOS 代码，尤其是早期的 iOS 代码，都是这样编写的。

现在您已经理解了动作和出口是什么，您将在设计用户界面时看到它们如何工作。然而，在开始设计之前，我们还有一项快速的内务处理要做，以保持一切整洁有序。

**自动引用计数 (ARC)**

如果您已经熟悉 Objective-C，或者阅读过本书的早期版本，您可能会注意到我们没有 `dealloc` 方法。我们并没有释放我们的实例变量！

警告！警告！危险，威尔·罗宾逊！

实际上，威尔，您可以放心。我们完全没问题。完全没有危险——真的。

不再需要手动释放对象。嗯，这并不完全正确。释放对象仍然是必要的，但 Apple 随 Xcode 提供的 LLVM 编译器非常智能，它会使用一个名为**自动引用计数**（简称 ARC）的新功能为我们释放对象。这意味着 `dealloc` 方法的使用频率降低，并且不再需要担心调用 `release` 或 `autorelease`。ARC 是一个巨大的改进，我们在本书的所有示例中都使用了它。ARC 在过去几年中一直是 Xcode 的一个选项，但现在对于您创建的每个新项目，它都是默认启用的。

ARC 仅适用于 Objective-C 对象，不适用于 Core Foundation 对象或使用 `malloc()` 等分配的内存，并且有一些注意事项和陷阱可能会让您出错。但大多数情况下，担心内存管理已经成为过去式。

要了解更多关于 ARC 的信息，请查看此 URL 上的 ARC 发行说明：

```
http://developer.apple.com/library/ios/#releasenotes/ObjectiveC/RN-TransitioningToARC/
```

ARC 非常酷，但它不是魔法。您仍然需要理解 Objective-C 中内存管理的基本规则，以避免在使用 ARC 时遇到麻烦。要复习 Objective-C 内存管理契约，请阅读此 URL 上的 Apple *Memory Management Programming Guide*：

#### 编写操作方法

到目前为止，我们已经设计了用户界面并连接了出口和操作。剩下的工作就是使用这些操作和出口在按下按钮时设置标签的文本。在项目导航器中单击 `ViewController.m` 以在编辑器中打开它，并找到 Xcode 之前为我们创建的空 `buttonPressed:` 方法。

为了区分两个按钮，我们将使用 `sender` 参数。我们将使用 `sender` 检索被按下按钮的标题，然后基于该标题创建一个新字符串，并将其指定为标签的文本。将以下粗体代码添加到您的空方法中：

```
- (IBAction)buttonPressed:(UIButton *)sender {
    NSString *title = [sender titleForState:UIControlStateNormal];
    NSString *plainText = [NSString stringWithFormat:@"%@ button pressed.", title];
    _statusLabel.text = plainText;
}
```

这非常简单。第一行使用 `sender` 检索被点击按钮的标题。由于按钮可以根据其当前状态（尽管在此示例中并非如此）具有不同的标题，因此我们使用 `UIControlStateNormal` 参数来指定我们希望获取按钮处于正常、未点击状态时的标题。这是您通常希望在向控件（按钮是一种控件）请求其标题时指定的状态。我们将在第 4 章中更详细地讨论控件状态。

下一行通过将文本附加到我们在上一行检索到的标题“button pressed.”来创建一个新字符串。因此，如果标题为 *Left* 的左侧按钮被点击，此行将创建一个字符串，内容为“Left button pressed.”。最后一行将新字符串赋值给标签的 `text` 属性，这就是我们更改标签显示文本的方式。

**消息嵌套**

Objective-C 消息通常是嵌套的。您可能会在旅途中遇到如下代码：

```
NSString *plainText = [NSString stringWithFormat:@"%@ button
    pressed.",
    [sender titleForState:UIControlStateNormal]];
```

这一行（逻辑上的）代码的功能与 `buttonPressed:` 方法的前两行完全相同。这是因为 Objective-C 方法可以嵌套，这本质上是用嵌套方法调用的返回值进行替换。

为了清晰起见，本书中的代码示例通常不会嵌套 Objective-C 消息，但 `alloc` 和 `init` 的调用除外，根据长期以来的惯例，这两个调用几乎总是嵌套的。

#### 添加一些样式

`NSAttributedString`类允许你将格式信息（如字体和段落对齐）附加到字符串上。此元数据可以应用于整个字符串，也可以将不同的属性应用于字符串的不同部分。如果你考虑一下文字处理器中格式如何应用于文本片段，那基本上就是`NSAttributedString`的工作模型。

然而，直到最近，Apple 提供的 UIKit 类中没有一个能够处理属性字符串。如果你想要呈现一个同时包含粗体文本和正常文本的标签，你只能使用两个标签，或者自己将文本直接绘制到视图中。这些方法并非不可逾越的障碍，但它们足够棘手，大多数开发者都不愿频繁使用。对于想要显示样式文本的人来说，iOS 6 带来了许多改进，因为大多数主要的 UIKit 控件现在都允许你使用属性字符串。对于像我们这里的`UILabel`来说，这就像创建一个属性字符串，然后通过其`attributedText`属性将其传递给标签一样简单。



所以，选择 `ViewController.m`，通过删除划掉的代码行并添加代码片段中加粗的行，来更新 `buttonPressed:` 方法：

```
- (IBAction)buttonPressed:(UIButton *)sender {
    NSString *title =  [sender titleForState:UIControlStateNormal];
    NSString *plainText =  [NSString stringWithFormat:@"%@ button pressed.", title];
    _statusLabel.text =  plainText;

NSMutableAttributedString *styledText =  [[NSMutableAttributedString alloc]
                                             initWithString:plainText];
    NSDictionary *attributes =
    @{
      NSFontAttributeName :  [UIFont boldSystemFontOfSize:_statusLabel.font.pointSize]
     };

NSRange nameRange =  [plainText rangeOfString:title];

[styledText setAttributes:attributes range:nameRange];
    _statusLabel.attributedText =  styledText;
}
```

这段新代码所做的第一件事，是基于我们将要显示的字符串创建一个属性字符串——具体来说，是一个 `NSMutableAttributedString` 实例。这里我们需要一个可变属性字符串，因为我们想修改其属性。

接下来，我们创建一个字典来存放要应用于字符串的属性。实际上，目前我们只有一个属性，所以这个字典只包含一个键值对。键 `NSFontAttributeName` 允许你为属性字符串的一部分指定一种字体。我们传入的值是所谓的粗体系统字体，其大小被指定为与标签当前使用的字体大小相同。从长远来看，以这种方式指定字体比通过名称指定字体更灵活，因为我们知道系统总是能合理地知道应该使用什么作为粗体字体。

**提示** 如果你使用 Objective-C 已有一段时间，可能对这种新的字典语法不太熟悉——但它其实相当简单。Xcode 附带的 LLVM 版本提供了一种简写形式，无需显式调用 `NSDictionary` 的类方法，使用起来更方便。它的基本形式如下：

`@{`     `key1 : value1,`     `key2 : value2`    `}`

除了省去每次创建字典时都要输入冗长的类名和方法名之外，它还把键和值按照“正确的顺序”排列——至少对于任何使用过内置字典的语言（如 Ruby、Python、Perl 或 JavaScript）的人来说是这样。

这种新的字典语法与数组和数字的类似语法一并于 2012 年引入。本书将通篇使用这些新语法。

接着，我们让 `plainText` 字符串给出包含标题的子字符串的范围（由起始索引和长度组成）。我们使用该范围将属性应用于属性字符串中对应标题的部分，然后将其传递给标签。

#### 查看应用程序委托

嗯，真酷！你的应用跑起来了！在我们进入下一个主题之前，花点时间看看我们尚未检查的两个源代码文件：`AppDelegate.h` 和 `AppDelegate.m`。这些文件实现了我们的应用程序委托。

Cocoa Touch 广泛使用委托，即负责代表另一个对象执行特定任务的对象。应用程序委托让我们能在某些预定义的时间代表 `UIApplication` 类执行操作。每个 iOS 应用程序都只有一个 `UIApplication` 实例，它负责应用程序的运行循环并处理应用程序级别的功能，例如将输入路由到适当的控制器类。`UIApplication` 是 UIKit 的标准组成部分，其工作大部分在后台进行，因此你通常无需担心它。

在应用程序执行的某些明确定义的时间点，如果委托存在并且实现了该方法，`UIApplication` 会调用其委托上的特定方法。例如，如果你需要在程序即将退出前执行代码，你可以在应用程序委托中实现 `applicationWillTerminate:` 方法，并将终止代码放在那里。这种委托方式允许你的应用程序实现常见的全局行为，而无需继承 `UIApplication`，实际上也无需了解 `UIApplication` 的内部工作原理。所有 Xcode 模板都会为你创建一个应用程序委托，并安排在应用程序启动时将其链接到 `UIApplication` 对象。

点击项目导航器中的 `AppDelegate.h` 查看应用程序委托的头文件。它应该看起来像这样：

```
#import <UIKit/UIKit.h>

@interface AppDelegate :  UIResponder <UIApplicationDelegate>

@property (strong, nonatomic) UIWindow *window;

@end
```

有一点值得指出的是这行代码：

```
@interface AppDelegate :  UIResponder <UIApplicationDelegate>
```
