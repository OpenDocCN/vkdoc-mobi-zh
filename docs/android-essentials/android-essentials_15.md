# 第 3 章：用户界面

## 在苏维埃谷歌中，界面渲染你

在移动 UI 开发架构的混战中，Android 的架构脱颖而出。如果你有一些移动开发经验，你会发现它是 Java ME 的 Canvas/Screen 对象与 BREW 控件层次结构的完美结合，并辅以一些 XML 布局工具。每个 Activity 从你的应用程序中启动时，会被放置在一个屏幕栈中。当你要求或用户按下返回键时，Android 已经配置好如何处理关闭顶部 Activity 并激活其下方的 Activity。这种设置让你可以将每个 Activity 视为单个屏幕的基础。

每个 Activity 可以包含一个层次树结构中的不同视图和视图组。你可以将视图组和布局对象想象为树干和树枝（因为视图组对象可以转换为视图），而视图或控件则是树叶。一个基本的**视图**是一个可绘制的矩形。一个基础的**视图组**是一个包含一个或多个视图的对象。这种对象层次结构允许你布置复杂的用户界面，而无需经历计算视图矩形和控件重叠图的易出错过程。另一方面，如果你喜欢那种方式，Android 也不会干涉，让你以 Java ME 的手绘游戏画布风格进行渲染。

在本章中，你将从基本的基于 XML 的屏幕布局开始，然后转向更复杂的自定义画布绘制。为了本书的叙述，我将把视图分为三大类：

- **XML 定义的控件/视图和视图组**：适用于基本信息显示和菜单
- **Android 原生视图**：`TextView`、`LayoutGroup`、`ScrollBar` 和文本输入
- **自定义视图**：游戏程序员的最佳朋友

## 简单快速的 XML 布局

一开始，使用 XML 布局可能看起来很简单，但它很快就会变得复杂。你将先从布局开始，然后逐步深入到各个元素。

### 布局

大多数 XML 屏幕会被包裹在一个布局对象中。布局对象有许多不同的类型，你将快速了解每一种，然后在接下来的章节中通过一个简单示例进行验证。

**注意**：在编译时，这些 XML 布局文件会被 Android 解析并打包成紧凑的二进制格式。这节省了启动时大量的解析时间。然而，这意味着在运行时你的代码不能更改这些 XML 文件。更具体地说，你也许可以在执行期间更改这些 XML 文件，但这对你的应用程序布局完全没有影响。此外，从 XML 中填充一个视图或视图组需要付出一定的性能代价。具体影响取决于 CPU 负载和 UI 复杂度。
