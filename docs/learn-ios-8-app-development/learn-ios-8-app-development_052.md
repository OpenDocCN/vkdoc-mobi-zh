# 高级事件处理

我想提及几个高级事件处理主题，并附上一些良好建议。先从建议说起。

请保持事件处理及时。如你现在所知，你的应用是通过主线程的运行循环（run loop）来“保持存活”的。该运行循环将所有内容交付给你的应用：触摸事件、通知、用户界面更新等等。你的应用处理的每一个事件、动作和消息，都必须在下一个事件被处理之前执行完毕并返回。这意味着，如果你编写的任何代码执行时间过长，你的应用就会像死掉一样。而且，如果编写的代码需要极长时间才能完成，你的应用*确实会*死掉——iOS 会因为应用停止响应事件而终止它。

我相信你肯定遇到过应用“卡死”的情况：屏幕冻结，对触摸、晃动或任何操作都没有反应。这就是当应用的运行循环在处理其他事务而非处理事件时发生的情况。这很不愉快。大多数 iOS 中可能耗时较长的功能都提供异步函数（就像你在 Shorty 中使用过的那些），这样这些耗时的任务就不会占用你的主线程。请使用这些异步函数，留意你的程序执行各项操作所花费的时间，并准备好重构应用，以避免“锁死”运行循环。我将在后续章节中演示所有这些技巧。

其次，处理多个触摸事件可能很棘手，甚至令人困惑。iOS 会尽力理清触摸事件的复杂性，并以合理且易于理解的形式呈现给你的对象。iOS 提供了五项功能，能使你的触摸事件处理更简单。

*   手势识别器（Gesture recognizers）
*   过滤掉其他视图的触摸事件
*   禁止多点触摸事件
*   提供独占触摸事件处理
*   暂停触摸事件

手势识别器是代表视图对象拦截触摸事件的特殊对象。每个识别器检测特定的触摸手势，从简单的点击到复杂的多指滑动。如果它检测到编程设定的手势，就会发送一个动作——这与你之前项目中用过的按钮对象完全一样。你只需要将该动作连接到 Interface Builder 中的一个对象即可完成。仅此一项功能就已为 iOS 开发人员节省了数万行触摸事件处理代码。我将在后续章节中向你展示如何使用手势识别器对象。

正如我之前所述，触摸事件（例如 `touchesMoved(_:,withEvent:)`）仅在 `touches` 参数中包含相关的触摸对象——即那些起始于你的视图对象的触摸。你的代码无需担心同一时间可能发生在其他视图中的其他触摸。在 Touchy 应用中，这实际上是一个缺点，你不得不从 `UIEvent` 对象中挖掘出全局的触摸对象集合。但通常，你只需要关注自己视图中的触摸。

你也看到了 iOS 如何使用 `UIView` 的 `multipleTouchEnabled` 属性来禁止多点触摸事件。如果该属性为 `false`，即使实际用户正在用多根手指触摸你的视图，iOS 也只会发送与首次触摸相关的视图对象事件。为了让 Touchy 应用获取所有触摸事件，你不得不将该属性设为 `true`。如果你的视图仅解释单点触摸事件，请将此属性设为 `false`，这样你就不必编写任何处理同时多点触摸的代码了。

如果你不希望 iOS 同时向两个视图对象发送触摸事件，可以将 `UIView` 的 `exclusiveTouch` 属性设为 `true`。如果设置，一旦触摸序列在你的视图中开始，iOS 就会阻止将触摸事件发送到任何其他视图（反之亦然）。

最后，如果你的应用需要，你可以暂时暂停所有向特定视图甚至整个应用发送的触摸事件。如果你想让某个单独的视图对触摸事件“充耳不闻”，请将其 `userInteractionEnabled` 属性设为 `false`。你也可以调用应用对象的 `beginIgnoringInteractionEvents()` 函数，这样应用的所有触摸事件都会被静默。通过调用 `endIgnoringInteractionEvents()` 可以重新开启它们。这对于防止触摸事件干扰正在进行的其他操作（例如，一个短动画）非常有用，但不要长时间保持关闭状态。
