# MVC 设计模式与 iOS 开发实践

`model-view-controller`（模型-视图-控制器）设计模式将提升你的代码质量，使应用更易于编写和维护，并赋予你令人愉悦的、健康的光彩。然而，切勿沉迷于其魔力而成为它的奴隶。

虽然使用设计模式能让你在成为优秀 iOS 开发者的道路上占据优势，但我告诫你不要为了使用而使用它们。实用主义者称这种做法为**过度工程化**。有时简单的解决方案才是最好的。看看这个例子：

```
class MyScoreController {
    var score = 0
    @IBOutlet var scoreView: UILabel!

    @IBAction func incrementScore(sender: AnyObject!) {
        score += 1
        scoreView.text = "\(score)"
    }
}
```

这个控制器有什么问题？MVC 纯粹主义者会指出这里没有数据模型对象。控制器同时扮演了数据模型的角色，存储并操作`score`属性。这既违反了 MVC 设计模式，也违反了单一职责原则。

你想知道我的看法吗？这个解决方案完全没有问题；它只是一个该死的整数！为了保存一个数字而创建一个新类毫无意义，而且这样做会浪费大量时间。

如果将来某天，你的数据模型增长到包含三个整数、一个字符串和一个本地化对象，那么当然可以：重构你的应用，将整数从控制器中提取出来，移到一个真正的数据模型对象中。但在那一天到来之前，不必担心。

有一种称为**敏捷开发**的编程纪律，它重视已完成、可工作的软件，而非计划和完美的设计。在这些情况下，我的建议是使用最简单的解决方案来完成工作。要意识到你何时在 MVC 设计中走捷径，并制定计划在问题出现时（如果确实会出现）修复它，但不要将自己束缚在一种设计哲学上。设计模式应该让开发更轻松，而不是更困难。
