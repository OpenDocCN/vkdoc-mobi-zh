# 总结

到现在为止，你对于消息和事件如何进入你的应用以及它们如何被处理，已经有了相当扎实的理解。你了解了事件队列和运行循环。你知道队列中的事件会被分发给应用中的对象。你知道其中一些事件直接发送给你的对象，触摸事件使用命中测试（hit testing），其余的事件则被发送给第一响应者。

你学到了很多关于响应者链的知识。在 iOS 中，响应者链除了传递事件之外，还执行许多重要任务。

你知道了如何配置一个对象来处理或忽略特定类型的事件。你编写了两个应用，一个处理高级事件，另一个追踪低级触摸事件。

甚至更令人惊叹的是，你构建并运行了应用在真实的 iOS 设备上！也可以随意在设备上运行其他项目。创建一款可以随身携带的你自己的 iOS 应用，是一项令人印象深刻的成就！

在下一章中，你将学习一些关于数据模型的知识，以及复杂的数据集如何转化为屏幕上的滚动列表。

## 练习题

根据魔术八球（Magic Eight Ball）附带的说明，你不应该摇晃它；这会导致液体中产生气泡。当然，这从未阻止过我和我兄弟把它晃得够呛。相反，你应该把球“8”面朝上放在桌子上，问一个问题，轻轻地将它翻转过来，然后读取答案。

作为附加题，请重写 EightBall 应用，使其使用设备方向事件（而非晃动运动事件）来让消息消失和出现。设备的方向可以是：竖屏、向左横屏、向右横屏、倒置、屏幕朝上或屏幕朝下。

设备方向的变化是通过通知传递的。你还没有使用过通知，但可以将它们视为另一种事件（至少在此上下文中如此）。与事件不同，你的对象必须显式请求它想要接收的通知。每当设备改变方向时，例如用户将 iPhone 翻转过来，你的对象就会收到一条通知消息。

请求和处理设备方向更改通知所需的所有代码，都在《iOS 事件处理指南》的“使用 UIDevice 获取当前设备方向”部分中。在 Xcode 中，选择“帮助”![图片](img/arrow.jpg)“文档和 API 参考”，然后搜索“事件处理指南”（*Event Handling Guide*）。
