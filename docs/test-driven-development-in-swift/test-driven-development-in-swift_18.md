# 当你需要使用 Fake 或 Dummy 时，可能并不会立刻显现出来，但对于实践 TDD 的开发者来说，这不成问题。迭代是测试驱动开发的核心。如果使用 Spy 是你为了让测试能编译通过而最先想到的方法，那就放手去做；一旦测试通过，你会有时间退一步思考，看看其他测试替身是否更合适。

## 关键要点

*   **着力保持测试套件的无状态、快速且简洁**。在 TDD 中，你在代码库的测试端花费的时间与在生产端一样多；测试套件内的工作越容易，效果就越好。

*   **Fake 测试替身通过将依赖替换为简化版本来绕过有状态或耗时的计算**。使用 Fake 来保持测试的隔离性和快速性。

*   **Dummy 测试替身将依赖替换为一个什么都不做的版本，以表明它不影响行为**。在测试中使用 Dummy 来明确那些代码构建所必需但对待测实际结果没有输入的依赖。
