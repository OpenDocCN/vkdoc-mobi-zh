# 1. 为什么选择测试驱动开发？

## 什么是测试？

《牛津英语词典》将名词 *test* 定义为：“*一种旨在确定某物的质量、性能或可靠性的程序，尤其是在其被广泛使用之前。*”在软件开发领域，我们可以将此定义调整为：“*一种旨在确定软件质量、性能或可靠性的程序，尤其是在其交付给用户之前。*”

测试我们的软件，本质上就是运行它并验证其行为是否符合预期。

以这个“Hello, world!”脚本为例：

```
#!/usr/bin/env xcrun swift
func main() {
    guard CommandLine.argc > 1 else {
        print("Hello, world!")
        return
    }
    print("Hello, \(CommandLine.arguments[1])!")
}
main()
```

一个测试可以是：无输入运行它，即 `./hello_world.swift`，并检查它是否输出“Hello, world!”。然后，我们可以用特定输入运行它，并验证它是否在问候语中使用了该输入。例如，`./hello_world.swift Ada` 应输出“Hello, Ada!”。

同样，你可以启动 iOS 模拟器中的应用，点击其 UI 检查其行为是否按编程设定运行。或者，你可以在 Web 应用中通过表单提交数据，然后检查数据库是否包含该数据。

以上所有例子都有一个共同特征：需要有人来运行软件并验证其行为。这些是*手动*测试。

要按计划持续交付高质量软件，仅靠手动测试是不够的。

## 手动测试效率低下

我们可以手动彻底测试像 `hello_world.swift` 这样的小脚本，因为其功能集有限，但实际程序并非如此简单。即使是最小的应用，也存在许多可能的代码路径、代码接受的输入排列组合以及用户可在系统中采取的步骤。要手动测试所有这些，将花费*大量*时间。

手动测试既耗时又成本高昂。无论执行测试的是谁，无论是软件或质量保证工程师、产品负责人还是第三方公司，真人需要支付薪水。他们还需要休息时间、会生病、可能忘记事情，而且不可避免地会犯错。

然而，为了节省时间和金钱而跳过测试将是一个糟糕的举动。如果我们不在交付前测试产品，用户就会替我们测试。这远非理想。虽然我们可能会为发现一个 bug 而庆祝，但用户却会感到沮丧，并可能转而使用竞争对手的产品。

必须有一种方法，能够高度可信地测试我们的软件，速度快，且无需人类完成所有工作。



## 检验代码的代码

软件开发者编写代码来自动完成原本需要手动执行的任务。正如投资人马克·安德森[在 2011 年的一篇文章](https://a16z.com/2011/08/20/why-software-is-eating-the-world/)中所言，过去几十年里，软件一直在“吞噬世界”。

过去人们用传真机发送签名文件，在车里放一本大地图册，用名片夹存储商业联系人。如今，我们使用带有数字签名的电子邮件、利用智能手机 GPS 的导航应用，以及 CRM 网络工具。

软件也可以实现其他软件的自动化，从通过读取与会者的日历来确定最佳会议时间、生成带视频通话链接的邀请，到按计划发布预先撰写的内容。

验证软件的行为只是另一个适合自动化的任务。你可以编写代码来检查你写的代码是否按预期运行。

检查其他代码的代码——这就是自动化测试的本质。

为了理解“编写检查其他代码的代码”的含义，让我们来看编程面试中经久不衰的经典——“fizz-buzz”算法：

- 给定一个整数，如果它能被 3 整除，输出“fizz”；如果能被 5 整除，输出“buzz”；如果能同时被 3 和 5 整除，输出“fizz-buzz”；否则，输出该数字本身。

以下是一种可能的实现：

```
func fizzBuzz(_ number: Int) -> String {
    let divisibleBy3 = number % 3 == 0
    let divisibleBy5 = number % 5 == 0
    switch (divisibleBy3, divisibleBy5) {
    case (false, false): return "\(number)"
    case (true, false): return "fizz"
    case (false, true): return "buzz"
    case (true, true): return "fizz-buzz"
    }
}
```

为了测试这段代码，我们可以用一个脚本，用不同的数字调用`fizzBuzz(_:)`，如果返回值正确则打印“PASSED”，否则打印“FAILED”：

```
func testFizzBuzz() {
    if fizzBuzz(3) == "fizz" {
        print("PASSED")
    } else {
        print("FAILED")
    }
    if fizzBuzz(5) == "buzz" {
        print("PASSED")
    } else {
        print("FAILED")
    }
    // 以此类推
}
```

为每个测试重复`if/else`结构会使测试变得冗长且乏味。让我们把检查值与预期值是否一致的逻辑提取到一个专门的函数中：

```
func test(value: String, matches expected: String) {
    if value == expected {
        print("PASSED")
    } else {
        print("FAILED")
    }
}

func testFizzBuzz() {
    test(value: fizzBuzz(1), matches: "1")
    test(value: fizzBuzz(3), matches: "fizz")
    test(value: fizzBuzz(5), matches: "buzz")
    test(value: fizzBuzz(15), matches: "fizz-buzz")
}
```

现在我们有了一个自动化测试脚本来验证我们的`fizzBuzz(_:)`函数。如果输出全部是“PASSED”，那么实现就是正确的——或者更准确地说，符合我们以单个测试形式列举的预期。

自动化测试将输入数据、读取输出并验证其正确性这些工作外包给了计算机。计算机执行这些任务的速度和准确性都远超我们人类。

由于速度快，我们可以更频繁地运行自动化测试，从而更有可能及早发现问题和捕获回归。

我们还可以自动化运行这些自动化测试，确保永远不会忘记执行它们。例如，如果你使用 Git，可以设置一个钩子，在每次提交前运行这些测试。

任何人都可以运行所有测试，因为脚本将这些测试编码化了。反之，在一个人能手动测试一个复杂产品之前，他需要接受相关培训，否则不知道预期结果是什么。而任何新员工都可以运行测试脚本，并立即判断所有检查是否通过。

为某个行为编写自动化测试可能比手动测试花费的时间更长。然而，一旦你支付了这笔一次性的设置成本，你就可以在手动测试所需时间的一小部分内，无数次地运行该测试。

自动化测试是你在软件随着新功能和变更而进化时，维护其质量所能做的最佳投资。相比机械的手动测试，它们是一种更快、更便宜且可扩展的替代方案。

## 单元测试、集成测试、UI 测试……天哪！

阅读关于软件测试的资料时，你会接触到各种自动化测试的名称。单元测试、集成测试、UI 测试、基于属性的测试、探索性测试、特征化测试……清单还很长。至少可以说，这令人眼花缭乱，更糟的是，不同的人或资料对某些名称的实际含义有着不同的解释。

一些学者将任何不涉及与应用程序实际 UI 交互的自动化测试都称为单元测试；另一些人则采用更严格的定义，只将在隔离状态下测试某个组件的测试视为单元测试。

在本书中，我们将遵循罗伊·奥谢罗夫在《单元测试的艺术》中对单元测试的定义：

- 单元测试是一段调用某个工作单元并检查该工作单元的一个特定最终结果的代码。

其中，工作单元是“从调用系统中的公共方法到系统测试能观察到的一个显著最终结果之间所发生的一系列动作的总和”。

另外两个值得给出的定义是集成测试和 UI 测试。

在进行测试驱动开发时，单元测试是最适合编写的测试，因为它们聚焦明确，能够实现更快的反馈循环。因此，本书只关注单元测试。

在本书中，除非另有说明，每次提到“测试”时，指的都是单元测试。



## 从编写测试到先编写测试

为已有的代码编写测试存在一个问题：你无法真正信任一个从未见过失败的测试。

要确保你的测试不会产生误报——即代码有误时测试依然通过——唯一的方法是修改代码的行为以强制其失败，然后观察测试能否捕获该错误。

为了验证测试的正确性而强制修改生产代码并非高效的做法。这需要更改你已写好的代码，而验证测试后又要立即撤销改动。

如何才能在不费心创建临时性失败的情况下，验证我们的测试是正确的呢？

答案是将流程反转过来，**先**编写测试。

通过为你尚未实现的功能编写测试，你可以确保该测试能够捕获错误。这是因为如果该行为的代码尚不存在，测试*必须*失败；如果没有失败，则说明测试编写有误。

让我们再看一下 fizz-buzz 的例子。这次，我们将从一个空实现开始，然后为预期行为编写一个测试：

```
func fizzBuzz(_ number: Int) -> String {
return ""
}
func testFizzBuzz() {
test(value: fizzBuzz(1), matches: "1")
}
```

运行 `testFizzBuzz()` 会打印 "FAILED"。为了让测试通过，我们只需修改实现，将输入的 `Int` 转换为 `String`：

```
func fizzBuzz(_ number: Int) -> String {
return "\(number)"
}
```

如果现在调用 `testFizzBuzz()`，它会打印 "PASSED"。

让我们继续逐个测试地实现 `fizzBuzz(_:)`，并添加一个测试来确保能被三整除的输入返回 "fizz"：

```
test(value: fizzBuzz(3), matches: "fizz")
```

为了让两个测试都通过，我们可以将 `fizzBuzz(_:)` 的实现更新为：

```
guard number % 3 == 0 else { return "\(number)" }
return "fizz"
```

然后我们可以继续处理下一个行为方面——能被五整除的输入应返回 "buzz"：

```
test(value: fizzBuzz(5), matches: "buzz")
```

为了让这个新测试和其他两个测试都通过，我们可以将 `fizzBuzz(_:)` 更新为：

```
switch (number % 3 == 0, number % 5 == 0) {
case (true, false): return "fizz"
case (false, true): return "buzz"
default: return "\(number)"
}
```

我们只剩下一个场景需要覆盖——能被十五整除的数字应返回 "fizz-buzz"：

```
test(value: fizzBuzz(15), matches: "fizz-buzz")
```

让我们更新 `fizzBuzz(_:)`，让这最后一个测试也通过：

```
switch (number % 3 == 0, number % 5 == 0) {
case (false, false): return "\(number)"
case (true, false): return "fizz"
case (false, true): return "buzz"
case (true, true): return "fizz-buzz"
}
```

我们通过一步步构建，始终从一个失败的测试开始，最终得到了一个可用的 "fizz-buzz" 实现。由于每个测试都代表了整个 "fizz-buzz" 行为的一个特定用例，迭代之间的代码改动始终是最小的。在每一步中，新行为的测试帮助我们验证新代码是否有效，而无需多次手动调用它，同时已有的测试则确保了新代码不会破坏其余逻辑。

这种先测试、逐步进行、快速反馈的编码方式极为高效。这是一种独特的软件开发方式，称为测试驱动开发，简称 TDD。

测试驱动开发是有意识且持续地利用测试反馈来指导软件实现的过程。但实践 TDD 不仅仅是在生产代码之前编写测试；其副产品远比仅仅知道软件行为符合预期更有价值。

## 从先编写测试到测试驱动开发

先编写测试会对你的软件设计施加有益的压力。环境约束和自然选择造就了构成我们世界动植物群的那些美丽而高级的物种。同样，强迫自己先编写测试会将你和你的代码库推向更高的标准。

当你先编写测试时，你立刻就能看出你的设计是否易于测试。直接且隔离的行为比复杂的行为更容易测试。一个具有许多依赖关系的大型对象比一个只有少数几个 `init` 参数的较小对象更难放入测试框架中。

如果代码易于测试，那么它在生产环境中的使用也会很简单。这是因为测试是代码公共 API 的消费者。在测试中使用代码与在生产环境中使用代码是相似的。

由于编写简单的测试比编写复杂的测试更容易，你自然会倾向于编写隔离且只做一件事的代码。你将构建由高内聚、低耦合的组件组成的软件。

TDD 还能帮助你更好地理解产品需求。要测试代码的行为，你首先需要知道它是什么：你需要具体地定义输入及其预期输出。要做到这一点，你需要恰当地吸收软件应满足的需求。这种前置分析使代码实现变得简单明了。先编写测试消除了跳过这一关键步骤的诱惑。

TDD 为你提供了心理优势。一次一个测试地开发软件设定了可持续的节奏，而当我们看到终点在望时，大脑能更好地实现目标。

一次一个测试地攻克大型复杂应用程序的开发，就像马拉松运动员对待长跑比赛的方式：一步一步，脚踏实地。

测试驱动开发是软件质量的关键习惯。当日复一日地应用时，它会对开发者的生产力、幸福感以及软件设计质量产生巨大的连锁效应。

正如测试驱动开发的创造者（或者像他自称的那样，再发现者¹）肯特·贝克在《测试驱动开发：实战》中所说：

> “如果你乐于拼凑出差不多能用的代码，并且乐于再也不看结果，那么 TDD 不适合你。TDD 建立在一个天真而可爱的极客式假设之上：如果你写出更好的代码，你就会更成功。TDD 帮助你在正确的时间关注正确的问题，这样你就能让设计更清晰，并在学习过程中不断完善设计。”

在本书中，你将学习如何使用 TDD 构建 Swift 应用程序，了解该语言强大的类型系统如何增强你编写测试的能力，以及如何使用 Apple 提供的 Xcode IDE 和 XCTest 框架高效地做到这一点。你将看到这种节奏稳定、先测试的方法如何帮助你应对任何规模和复杂性的问题。你将培养一种以反馈为先的思维方式，无论是对于软件还是产品开发，从而帮助你更快地交付，并更多地了解什么对你的业务有价值。

## 关键要点

*   **自动化测试是使用代码来测试其他代码**；它比手动测试更廉价、更有效，而且远不易出错。
*   **先编写测试比在实现完成后才编写更好**；通过观察测试在被测行为尚无实现时失败，你可以相信它将捕获未来的回归错误。
*   **测试驱动开发是一贯且有意识地以测试为起点编写软件的实践**；这一实践能够产生由高内聚、低耦合组件组成的更优秀的软件，每个组件负责一个微小且定义明确的行为。
