# 16. 结论

贯穿本书，我们使用测试驱动开发构建了一个功能完备的应用。我们在测试和编译器的反馈指导下，以小步快跑的方式前进。

我们从一个展示静态内容的单一屏幕开始，通过迭代，最终打造出一个动态的订购应用：它可从后端进行配置，将支付处理委托给第三方服务，并在本地存储中持久化状态。

在此过程中，我们学习了不同的 TDD 技术与方法。一切的基础是“红、绿、重构”工作流：从一个失败的测试开始，然后用你能想到的任何方式让它通过。最初的版本是否丑陋、非最优或直接复制粘贴并不重要，重要的是测试通过。不要害怕从硬编码值开始。*先造假，再实现（Fake It, Till You Make It）*。只有在测试变绿后，才着手消除重复，并将实现重构为令你自豪的代码。

我们多次运用了“分解问题，逐一解决”的技术，以进行小规模迭代并缩短反馈循环。问题可以被拆解成更小的组件，我们可以一次专注于一个组件，隔离处理。缩小工作范围能让我们更专注，因为我们可以将整个问题保留在工作记忆中。每一次通过的测试和成功的重构都是一次小小的胜利。沿着测试列表一步步推进，能给你带来进步感和动力。

我们学习了通过抽象让组件交互的价值，了解了依赖倒置原则，以及依赖注入——即让对象显式请求其依赖——是如何产生“诚实”的可测试代码的。

应用依赖倒置和依赖注入使我们能够构建测试替身，以应对被测试系统并非纯函数的情况。

我们可以使用 Stub 来控制依赖如何影响被测系统的行为，并使用 Spy 来处理依赖受被测系统影响的情况。Fake 将我们的测试与本地存储的状态性解耦，而 Dummy 则清晰地表明某个必需的依赖对测试结果没有贡献。

我们发现 SwiftUI 是 TDD 工作流的强大盟友。状态是唯一的真实来源，我们无法直接操作视图。这意味着我们可以完全专注于生成输入状态的业务逻辑，并相信框架会按指令渲染它。这形成了一个薄而谦逊的视图层，该视图层与一个从 SwiftUI 中解耦出来的展示与行为层交互。我们可以快速迭代这两者：前者使用 Xcode Previews，后者使用测试。

业务逻辑层的测试覆盖率接近 100%，因为我们为了构建它所编写的每一行代码都是由一个失败的测试驱动的。那些管理层强制设定测试覆盖率目标的团队，常常需要在功能完成后回头补充测试以达到目标；而我们无需任何额外努力就达到了这一点。

拥有广泛的代码覆盖率非常棒，但这只是冰山一角。TDD 给予我们的远不止这些价值。

## 不仅仅是测试

一旦你体会到为代码编写自动化测试的价值，首先编写单元测试就是下一个显而易见的步骤。它带给你信心和及时性，并让你免于为了验证测试而修改已完成的生产代码。

测试驱动开发不仅仅是为你代码编写测试的一种有效方法。事实上，它最重要的好处并不在于测试覆盖率。无数开发者采用 TDD 是因为他们关心测试，而坚持使用它则是因为获得了设计质量和生产力的提升。

## TDD 与软件设计

TDD 对我们编写代码的方式施加了一种有益的压力，这是一种自然选择，只让最简单的测试存活下来。

正如我们在本书中看到的，从测试开始编写代码使得处理小组件和引入抽象成为必要。否则，测试将难以编写，反馈周期也会很慢。不断地问自己：“我怎样才能获得更快的反馈？”你将构建出由松耦合、高内聚组件构成的代码库。

一个典型的例子：必须采用测试优先的方法处理 UI 代码，这迫使我们把布局声明与生成填充值的逻辑分离开。结果，我们得到了一个不含逻辑的精简 SwiftUI 层，可以轻松将其插入 Previews。视图将其智能委托给它们的 `ViewModels`，后者代表了视图本身的抽象，并且我们对其实现了完全的代码覆盖。理论上，我们可以将 ViewModels 插入不同的布局，甚至无需任何修改就能用它们来驱动一个命令行应用。

使用 TDD 也帮助我们处理了不受我们控制的依赖，例如网络和第三方支付 SDK。为了测试与它们交互的代码而必须引入的抽象，产生了一个积极的副作用：使我们的设计模块化——我们可以替换网络或支付组件的实现，而无需触及应用的其余部分。

模块化也使我们能够并行工作。一旦定义了抽象的 `protocol`，不同的团队成员就可以同时处理具体实现及其消费者。

你*并不需要*测试驱动开发来做出优秀的软件设计选择。然而，**TDD 会引导你走向优秀的设计**。

来自测试的实时反馈清晰地表明你是否走在正确的方向上。如果你发现为某些代码编写测试很麻烦，这是一个信号，表明这些代码在生产环境中也不会易于处理，是时候退一步重新审视设计了。



## 测试驱动开发（TDD）与生产力

也许人们对测试驱动开发最大的怀疑之一是，它看起来会让编写代码变得更耗时。毕竟，你得先写测试，然后只写刚好能让测试通过的代码，接着再回头进行重构。直接编写最终实现，避免所有这些来回折腾，不是更快吗？

当人们因为觉得 TDD 会降低编码速度而避开它时，他们最终会陷入一个*局部最优*的困境。他们以牺牲长期有效性为代价来优化短期生产力。是的，事后编写测试——或者根本不写测试——可能会更快，但随着时间的推移，在这样构建的代码库中工作会变得越来越耗时。

测试驱动开发有助于在代码库体量和复杂性增长时保持敏捷性——尤其是在增长时——因为它能产生广泛测试覆盖和精心设计代码的双重效果。测试网络能帮助你在更改代码时，无需担心遗漏某个副作用而导致破坏应用的其他部分。基于测试反馈产生的模块化、定制化设计，使得代码易于更新、重构和替换实现。

当你一次只通过一个测试时，你离稳定的绿灯状态总是一步之遥。你可以轻松地进行实验。如果某个方案行不通，抛弃最后一次改动，你就能回到安全状态。

红绿重构工作流会将看似无法完成的大量工作，分解成一个个小模块。终点始终在望：一旦你让一个测试通过，你就有了一个可工作的成果，尽管它可能尚未完成。

一旦你的代码进入绿灯状态，你就可以稍作休息，因为你知道你留下的是一个稳定的代码库，并且清楚地知道下一步该做什么——要么重构当前实现，要么进行列表中的下一个测试。当你的大脑知道代码能正常工作，并且知道清晰的下一步时，它就不会在后台浪费精力去担忧。用效率大师戴维·艾伦的话来说，你*关闭了一个开放式回路*。

**测试驱动开发通过保持代码库的可塑性，并让你以可实现且有成就感的小步骤进行工作，从而帮助你保持高效。**

## TDD 与产品开发

红绿重构；红绿重构；红绿重构。测试驱动开发是一个迭代过程。

以这种方式工作，并不断问自己“这个我目前需要吗？”以及“让这个测试通过的最简单方法是什么？”，这种做法是具有感染力的。你会发现自己在越来越广泛的范围内应用`分解问题`和`顺序求解`的思维。

从编写某个功能的代码放大到设计该功能本身，你将发现可以设计出一个简化版本来更快地实现和发布。这有双重好处：更早地向客户交付价值，并更快地了解工作系统。

再放大一步，到产品设计层面，你会发现自己在安全的小步骤中开始迭代和实验，使用诸如纸面原型和 A/B 测试等工具，在投入昂贵的开发阶段之前收集数据并验证假设。

再次声明，TDD 并不是这种产品开发方法的先决条件。尽管如此，**TDD 使以小周期思考和工作成为一种习惯**。它将你的工作流塑造成一个以收集反馈为最重要事项的模式。

正如 Spotify 首席执行官丹尼尔·埃克在[一份股东报告](https://www.businesswire.com/news/home/20190731005324/en/Spotify-Technology-S.A.-Announces-Financial-Results-for-Second-Quarter-2019)中所写，“迭代速度将胜过迭代质量。”1 在一个变化日益加速的世界里，知道如何快速行动以学习和调整你的路线，是一个获胜的优势。

测试驱动开发有一个学习曲线。你可能不清楚要写什么测试，如何塑造代码接口以注入`Test Double`，或者如何从让测试通过的虚假实现过渡到真实实现。

不要让这些阻碍你。

任何向前迈进都比原地踏步要好。持续前进并纠正航向。如果你不知道从什么测试开始，那就先写一个初始化`SUT`并断言它不是`nil`的测试。一旦你有了那个测试，尝试添加另一个断言，看看它会把你引向何处。

如果你持续应用我们共同见过的这个工作流，先写测试最终将成为你的第二天性。就像尼奥深入矩阵内部看到其代码源流一样，当你阅读代码的规格说明时，你就会开始看到测试你尚未编写的代码的方法。

发现测试驱动开发加速了我作为软件工程师的成长。通过这本书，我希望我也能帮助你成长，激发你对编写干净代码并以小步迭代前进的兴趣。

本书到此结束，但你的 TDD 之旅才刚刚开始。祝你玩得愉快！
