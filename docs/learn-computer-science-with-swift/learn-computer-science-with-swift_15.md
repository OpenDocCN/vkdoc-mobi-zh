# 13. 纳入人的因素

计算机科学领域通常被纯粹视为计算机硬件和软件的问题。在某种程度上，我们作为社会的一份子，需要审视人的参与方式，因为人是计算机科学项目的使用者、创造者、管理者和资助者。

本章从应用世界的角度探讨计算机科学中“人”的一面。尽管这里描述的问题和原则也适用于看似不涉及人的项目（例如星际空间任务、电网和发电机的控制软件），但这些项目仍然涉及人，至少体现在项目的管理和资助方面。

然而，本章的重点更多地放在你凭借计算机科学知识可能参与的项目上。首先快速概述一下“可计算”与“不可计算”的形式化概念；然后你会看到一份问题清单，用于询问和回答你可能正在开发的特定项目。这些问题都是必须解决的。

## 面向人的可计算性

计算机科学的核心是可计算性概念。简单来说，可计算性是指可以通过系统性步骤解决的问题。这与算法不同，因为算法往往关注比可计算性问题更小的领域。因此，一个可计算性问题的解决方案可能包含一个或多个算法。

> **注**
> 如果你与不同的人交谈，会发现计算机科学的“核心”有多个候选，可计算性只是其中之一。它目前特别有用，因为许多当前用于向年轻人（甚至学龄前儿童）教授计算机科学的工具都聚焦于可计算性。

可计算问题有两种基本类型：

- **判定问题。** 给定一组项目，某个项目是否属于该组？这张发票是否属于未付账单的一部分？用户 ID 是否有权登录？这些都是答案为“是”或“否”的问题。
- **函数或可计算问题。** 这些是需要计算的问题。X 乘以 Y 的乘积是多少？

其他类型的可计算问题包括优化问题（如何最好地做某事——通常是在给定特定数据的情况下，如何最好地计算出这两种基本类型之一的答案）和搜索问题。问题通常被分解为子问题，这样即使是最复杂的问题也可以拆解成可计算的片段。

思考可计算性问题的主要原因是能够（基于一些逻辑分析）有把握地说某个特定问题是可计算的还是不可计算的。毕竟，当人们考虑将自己的计算机科学知识付诸实践时，第一个问题就是：计算机科学能帮我解决这个问题吗？



## 开发问题

你在网络和各种出版物中可以找到许多关于应用开发的指南。以下六个问题是在任何计算机科学项目启动之初都需要回答的。如果你无法回答这些问题，你的项目迟早会遇到麻烦。期望那些有计算机科学使用经验的人能够帮助整合项目。这或许不是正式课程的一部分，但这确实是现实世界的一部分。（如果你与某人谈论一份将"计算机科学"列为必要条件的工作机会，那么熟悉开发流程通常与掌握专业术语同样重要。）

如果你是那种迫不及待想要编写代码的人，这六个问题可能看似过于简单，但在你和其他人投入时间、精力和金钱于一个尚未清晰成型的项目之前，必须先回答它们。每个问题的更多细节（包括如何回答）将在本章后面部分说明。

> **提示**  
> 当你开始回答这些问题时，请记住你随时可以改变想法，事实上，随着任何计算机科学项目的发展，你应该计划调整和演进你的答案。

以下是这六个问题：

-   你在做什么？
-   对象是谁？谁将构建、使用、管理、维护并资助它？
-   你为什么要构建它？人们为什么会使用它？
-   项目将在何时进行？何时开始，何时结束？
-   项目将在哪里运行？它将运行在什么设备上；是否需要其他计算机资源，如网络服务器或数据库？
-   你如何获知项目的结果？

### 你在做什么？

这个项目可能是关于构建一个应用，但也可能是构建一个开放数据资源，以便其他人可以构建运行在公共数据上的应用。许多人建议这个问题应该在 10 秒内回答出来（这有时被称为电梯演讲，意思是如果你和你的老板或其他重要人物一起在电梯里，你只有电梯运行一层楼的时间来说出你的答案）。

如果一个朋友问："你在做什么工作？"，大多数时候答案应该只需几秒钟。令人惊讶的是，许多人都觉得回答这个问题极其困难。如果你无法简单地描述项目，那么你很可能也无法实现它。即使是最复杂的项目也可以被概括：让无人驾驶航天器登陆火星。这就是对一个非常复杂（且成功）的一系列任务集合的简单描述。

### 谁将参与其中？

谁将负责项目的工作，谁将使用最终成果？你构建这个应用是为了自身利益，以便学习如何构建应用并丰富你的项目作品集吗？这是为你目前或想要从事的工作而做的项目吗？用户会是熟练使用移动设备的人（如果你的应用运行在移动设备上），还是那些在设备上安装了 400 个游戏却不知道如何更改设置的人？他们会是熟练使用移动设备但从未使用过台式机或笔记本电脑的人吗？（是的，没有个人电脑使用经验的人数正在增长。）

谁将管理或领导这个项目？在许多项目中，团队会协同工作。我们越来越多地看到跨学科协作项目受益于不同领域人员的有序互动。如果你不熟悉数据科学及其相关项目，你可以在网上查找数据科学的相关资料。有许多成功案例，其中大部分都涉及计算机科学技术的开发或应用。

### 人们为什么会参与进来？

这归结于人们将从项目中获得的益处。他们参与项目是为了获得学分、找到工作还是赚钱？项目使用者的动机通常与开发者的动机不同。

### 项目将在何时进行？

这实际上是最难回答的问题之一。在非数字世界中，判断项目何时完成相当容易：一位要人剪彩，乐队演奏，大桥通车。所有人都能看到这一时刻。

软件则不可见。在旧日的软件零售时代，人们可以看到商店里的软件包装盒，这无疑让项目看起来是完整的。确定项目何时完成，需要你结合"做什么"、"谁参与"和"为什么"的答案，以及你所做的任何变更。

现在，对于"你在做什么？"有了一个新的视角。结合之前问题获得的信息，现在这个问题可以更精确地表述为：你交付什么？

在产品开发（尤其是软件开发）领域，一个常见的短语是最小可行产品或`MVP`。这个短语有各种解释，但通常指的是一个能运行并执行你对最终产品所期望的基本功能的东西。

这可能意味着错误信息不完整（这可能导致如果不小心使用，应用会崩溃）。也可能意味着某些功能尚未实现。但这意味着你可以不借助幻灯片演示向别人展示"这个"东西：有一个可以运行的东西在。

最小可行产品的一种解释是，它可以被——或者正在被——销售。有时它会以入门版或预览版的形式销售（即使售价为零），但总归是有一个可工作的东西存在。

当你开始决定何时拥有一个基础产品时，现在又有一系列额外的问题需要回答。基本上，对于大多数计算机科学项目，你将要面对一系列的交付物。（项目早期可能有项目提案、幻灯片演示之类的交付物，但这些还不是可以运行的东西。）

根据你构建的内容，你有多种选择。以下是这些选择的一个概述，但实际上你还可以有数不清的其他选择。这里只是让你了解交付工作的可能性。

你可以规划应用或数据库（或任何其他计算机科学项目）的版本。通常，这些版本会编号为 1、2、3……中间版本也可以编号。第一个版本就命名为 1.0 版本并不罕见。它可能很快就会有 1.1 版本，其中包含一些对拼写错误等问题的修复。你可能会继续迭代，直到 1.0 版本稳定。此时，就该开始考虑 2.0 版本了。

随着项目推进，你可以开始制定路线图，以便对你的版本规划有个概念。这个路线图和日程表将成为你的项目管理工具。通常，主版本（1、2、3）以及可能是重要的子版本（1.1、2.3 等）会与市场推广活动相关联（如果适用的话）。这些里程碑也可能与学术日历、资助（拨款或投资）的里程碑或其他日期相关联。请记住，一旦里程碑与某个日期（如学术日历或付款日）挂钩，它就不再完全受计算机科学团队的控制了。

关于"何时"的考量还有另一个方面。一旦项目稳定下来，它会被定期使用吗？如果你正在构建一个需要为年底假期期间流行的产品进行维护的数据库，那么你更新数据库的关键时间窗口可能是 7 月（这是年终假期数据准备的典型前置时间）。请规划好你的应用、数据库或其他项目的时间安排，以及它们的后续发展。



### 项目将在何处运行？

尽管软件可能是虚拟的，但它需要运行在设备上。关于项目运行位置，你需要做出两个主要决定。如果项目涉及需要集中存储的数据，通常的做法是将其托管在 Web 服务器上；应用程序和网站可以通过通用的 `REST` 协议访问这些数据。这种存储通常需要付费，但一般价格较为合理。当然，如果你的项目是针对已有网络存储资源的组织（如公司、大学或私营企业），你可能无需为此担忧。

诸如亚马逊云服务（Amazon Web Services）、Azure 和 IBM 等云平台，让你能够轻松使用和配置云存储。这些平台旨在快速响应客户端存储需求的突发增长（或缩减）。

如果你在开发一个应用，则需要决定它将在哪个平台上运行。目前，主要有四大移动平台：

-   iOS
-   Android
-   Windows
-   网页（Web）

某些应用（如在线银行应用）必须在所有平台上运行，而其他应用可能只需在一个平台上运行。在许多情况下，会先选定一个平台作为起点（这类似于最小可行产品的变体）。应用上线后，就可以收集反馈，并在必要时为项目的后续开发争取更多支持。

微软的 `Xamarin` 产品吸引了许多人的兴趣。它基于 `.NET` 和 `C#`，目标是实现“一次编写，随处运行”的理想，让你编写的代码能够运行在 iOS、Android、Windows 或 macOS 上。这类策略面临的挑战是，当目标系统进行修改时，像 `Xamarin` 这样的整合系统往往会更新滞后。对许多应用场景来说，这是一种非常有用的工作方式；而对另一些场景而言，缺乏对最新原生操作系统功能的及时支持则是一个严重问题。

无论你持何种观点，如果你想在计算机科学领域为同事和客户提供指导，就必须理解跨平台和多平台的相关问题。

### 你将如何得知结果？

如果你已经决定了要做什么、为谁而做、为什么做，以及何时何地使用，那么计算机科学项目的计划就差不多拟定了。剩下的就是确定如何衡量项目的状态和成功与否。

有趣的是，这个对许多人来说看似显而易见的问题，回答起来却和第一个问题（要做什么？）一样困难。再次说明，在非数字化世界中，因为你可以看到劳动成果，所以这个问题并不那么重要。你可以衡量应用的销量或下载量，但这是你想要的吗？如果你开发的应用程序旨在提高实体店的销售额，那么应用下载量远不如店内销售额重要。如果你正在构建一个假日礼品数据库，最直接的衡量指标就是这些礼品的销售量。网络上充斥着电商网站上被遗弃的购物车（一个常被引用的数据是大约 67%的购物车被遗弃，但这是 2014 年的数据，在线购物网站正在尝试通过新的界面工具来降低这个数字）。

像应用商店（App Store）这样的在线应用分发平台，其主要优势在于能产生海量数据（你可以在 App Annie —— [`https://www.appannie.com`](https://www.appannie.com) 上找到大量数据）。许多人认为这些数据极其有用，但如果你真想好好利用它们，请查看应用商店能提供给你的数据，以及其他像 App Annie 这样的指标，以及网站和数据库的访客数据。了解可用的数据，并提前规划你的结果分析策略。不要等到拿到数据后才决定如何处理：从一开始就建立你的分析体系。

这样，你就能追踪你的成果。

## 本章小结

本章展示了计算机科学如何融入你在其他学科领域从事的项目。随着应用世界的发展（请记住，应用商店于 2008 年 7 月才开放，距本书写作时不到十年），人们期望掌握计算机科学的人既了解技术，也懂得应用技术的方法。
