# 问题 51：在编写 UI 时，你更倾向于使用 XIB 文件、Storyboards 还是通过代码编程创建 UIView？

支持不同模型的观点都有。假装拥有一个确切的答案是自大的。相反，这个问题的目的是与候选人讨论他/她**偏好**某一种方式（或哪种场景更适合它们）的思考过程和理由。一个可能的答案可以关联到每种机制的优缺点，例如：

### XIB 文件的优点

*   可以快速搭建 UI。
*   为屏幕数量极少的小型应用提供了直接简单的实现方式。
*   可以为不同的本地化（例如，语言或国家）创建单独的 XIB 文件。
*   非常擅长布局元素和直观地发现未对齐问题。便于对布局进行细微调整。

### XIB 文件的缺点

*   在团队环境中工作时，合并冲突很困难（难以进行差异比较、合并和阅读）。
*   高度动态的视图无法用 XIB 来描述。
*   从性能角度看，它们比通过代码创建视图慢，因为 XIB 必须从磁盘读取并进行解析。
*   XIB 缺少一些你可以在代码中完成的定制，例如 Quartz 效果（阴影、圆角）。
*   更难调试（例如，如果你在 Interface Builder 中忘记建立连接或建立了错误的连接）。

### Storyboards 的优点

*   Storyboards 适用于屏幕数量少到中等、视图间导航需求相对直接的应用。
*   你可以在几乎不写代码的情况下，模拟出应用程序的流程。

### Storyboards 的缺点

*   Storyboards 不兼容 iOS 5 之前的系统，因此无法支持 iOS 4.3。
*   在团队环境中很难并行工作，因为每个人都在修改同一个文件。
*   同样地，在 GIT 中合并冲突的 storyboards 会非常痛苦。
*   使用 storyboards 时，人们曾遇到 Xcode 的 bug（例如，由于不一致而不得不频繁清理 `DerivedData` 文件夹）。

### 编程创建视图的优点

*   相比 XIB 文件，合并冲突和对比代码行更容易。
*   在调试时可以跟踪代码，而不必查看 Interface Builder。
*   从性能角度看，它提供了比 XIB 更快的视图创建速度。
*   通过代码创建视图给你更多的控制权和自由度。

### 编程创建视图的缺点

*   如果你的所有 UI 创建代码没有集中在一个地方，那么就更难直观地想象 UI，并在脑海中形成它最终的样子。
*   你不能直观地放置元素，这使得布局视图更耗时。
*   项目团队的新成员需要更长的时间来理解应用程序的流程和导航。
