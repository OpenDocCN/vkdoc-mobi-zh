# 面向初级开发者的问题

本章提出的问题是关于 iOS 的基础问题。高级开发者应该能够轻松作答，而经验不超过一年的初级开发者，也应了解相关概念。

这绝不是什么圣经。你可以随意选择一个题目，与朋友或同事开始一次非正式的讨论，观察讨论会走向何方。根据讨论的分支情况，运用你自己的判断力和兴趣决定是否继续下一个问题。

## 问题 1：iOS 中的结构体是什么？

结构体是一种命名类型，允许将多个项分组到一个变量中。Swift 中的结构体示例如下：

```
struct SomeStruct {
var name: String
var attribute: String
}
```

而在 Objective-C 中，它看起来像这样：

```
struct City {
NSString *name;
NSString *attribute;
};
```

可以追问候选人的一个好问题是解释类和结构体的区别，以及它们分别在哪些场景下使用更高效。在 Swift 中，结构体是值类型，而类是引用类型。因此，对于大型对象，结构体会作为值传递，而非引用传递。Swift 甚至对 `String` 和 `Array` 对象也使用结构体以提高性能。在以下情况下，使用结构体更可取：

*   目的是封装少量数据值。
*   该结构不需要继承其他现有类型的值。

## 问题 2：iOS 中的框架是什么？

框架是一个打包成 Bundle 的“骨架”，可供其他应用程序使用和分发。框架可以包含各种各样的额外资源（图像、已本地化的字符串、数据文件、UI 对象等）。除非框架已公开发布，否则它通常也包含运行所需的 `.h` 文件。

一个可以深入探讨此主题的好问题是询问候选人框架和库的区别。库可以被定义为可供应用程序链接和使用的一系列功能的集合。框架可以是一个库，一个包含多个库的集合，一系列脚本，或者任何你创建应用所需的东西。它是一个更抽象的概念。如果要求候选人举例，库可以是一个网络库。框架则可以是一个插件管理器或一个 GUI 系统。

## 问题 3：如何在 iOS 应用中存储信息？

iOS 原生提供了几种不同的方式在应用程序中持久化数据。它们包括：

*   `Keychain`：用于以安全的方式存储敏感数据（常见的例子是登录名和密码）。
*   `SQLite` 数据库：如果要存储的数据结构复杂或需要大量结构化数据，通常使用 `SQLite` 数据库是最佳选择。
*   `Core Data`：这是苹果的持久化框架。它基于对象图来描述需要保存的对象。然后，你可以直接操作这些对象，而框架负责将其保存到数据库中。
*   `Files`：文件可以直接保存在应用程序内（出于安全原因，应用只能访问其自身的应用容器）。
*   如果你想深入了解，一个潜在的候选人可能会讨论选择 `SQLite` 而非 `Core Data` 的原因。例如，一个原因是 `SQLite` 可以在其他平台的不同应用程序之间共享，而 `Core Data` 是苹果专有的框架。
*   `User Defaults`：可用于存储少量数据。典型的例子是运行应用程序所需配置的一部分。`User Defaults` 可以持久化 iOS 中的基本数据类型（`String`、`Data`、`Number`、`Date`、`Array` 和 `Dictionary`）。

## 问题 4：字典是什么？它与其他编程语言中的类似结构有何异同？

字典在一个没有定义顺序的集合中，存储同一类型的键和同一类型的值之间的关联。每个值都与一个唯一的键相关联，该键在字典内充当该值的标识符。它类似于其他语言中所谓的 `HashMap`，但存在一些细微差别。

## 问题 5：什么是描述文件？

描述文件是一组数字实体的集合，它将开发者和设备唯一地绑定到一个经过授权的 iPhone 开发团队，并使设备能够用于测试。你必须在希望运行应用程序代码的每台设备上安装一个开发描述文件。

如果你想与候选人进一步讨论，可以询问他/她关于 iOS 应用程序的签名过程。

## 问题 6：什么是 ARC？

自动引用计数（`ARC`）是一项编译器功能，为你所有的 Objective-C 对象提供自动内存管理。无需手动执行保留和释放操作，`ARC` 让你可以专注于代码中更有趣的部分，即对象图以及应用程序中对象之间的关系。

讨论 `ARC` 可以转向更有趣的方向。如果候选人展现出扎实的背景，你可能想讨论传统垃圾回收方法之间的区别。对于 `ARC` 而言，它不像其他技术那样，有一个后台进程在运行时去分配对象。

## 问题 7：什么是自动布局？

`Auto Layout` 是开发者通过定义元素之间的关系来创建用户界面的一种方式。它提供了一个灵活而强大的系统，用于描述视图和 UI 控件如何相互关联。通过使用 `Auto Layout`，你可以获得对布局的极高控制度，拥有广泛的定制选项，从而实现完美的界面。

## 问题 8：在 iOS 中如何管理依赖项？

与其他平台不同，苹果官方并未提供任何处理 iOS 项目依赖项的方法。作为事实上的标准，`CocoaPods` 在 iOS 社区中被广泛使用。项目的依赖项在一个名为 `Podfile` 的文本文件中指定，`CocoaPods` 会用它来解析库之间的依赖关系，获取源代码，然后在 Xcode 工作区中将它们链接在一起以构建你的项目。

经验更丰富的候选人肯定会提到 `CocoaPods` 的另一个替代方案，即 `Carthage`，它相当流行，并且是用 Swift 编写的。

想进一步讨论这个问题吗？那么可以开始一场关于 `Carthage` 与 `CocoaPods` 各自优点的讨论。

## 问题 9：如何在 iOS 上进行调试和性能分析？

这是一个非常宽泛的问题，旨在引出候选人的单一回答。为了使讨论更有趣，候选人应该能够讨论 iOS 中的性能分析工具和崩溃日志。如果你的应用程序中存在已知的 bug，让开发者打开你的环境，看他/她如何浏览各种分析工具（CPU 调试等）会是一个不错的实践。

## 问题 10：iOS 中的 App ID 和 Bundle ID 有什么区别？

`Bundle ID` 用于精确标识单个应用程序。`Bundle ID` 字符串必须是统一类型标识符（`UTI`），且只能包含字母数字字符（A–Z， a–z， 0–9）、连字符（-）和句点（.）。该字符串应采用反向 DNS 格式。

`App ID` 是一个由两部分组成的字符串，用于标识来自单个开发团队的一个或多个应用程序。该字符串由一个 `Team ID` 和一个 `Bundle ID` 搜索字符串组成，中间用句点（.）分隔。`Team ID` 由苹果提供，对于特定的开发团队是唯一的，而 `Bundle ID` 搜索字符串则由你提供，用于匹配单个应用的 `Bundle ID` 或一组应用程序的 `Bundle ID` 集合。

苹果为何选择如此令人困惑的命名方式，至今无人知晓。



## 问题 11：代码签名是如何工作的？

代码签名是唯一标识一个应用程序以建立你自己的凭证（使用公私钥对）的过程。所使用的密钥存储在你的 Macintosh 电脑本地，当你想在其他工作站上进行开发时，也必须共享该身份标识。

有个老笑话是：“只需 75 步轻松签署你的 iOS 应用的教程”。这个过程并不简单，无论你经验多么丰富，总容易忘记其中一些步骤。然而，开发者应该能够概述该流程的关键步骤。

注意：此流程将来可能会改变，特别是具体步骤。

1.  你必须首先在你的本地计算机上创建一个 CSR 文件。为此，你必须使用钥匙串访问程序来创建 CSR 文件。在这个过程中，需要指定一些信息（如通用名称、电子邮件地址、加密类型等）。这个文件必须`安全存储`。

2.  完成这一步后，你必须在你的开发者计划网站上创建一个证书。在流程中的某个时刻，助手会提示你上传之前创建的 CSR 文件。然后会创建并下载一个证书。双击该证书，它会自动添加到你的钥匙串中。要使用它，此证书必须存在于你的机器上。

## 问题 12：Frame 和 Bounds 有什么区别？

在 iOS 中，`UIView` 的 Frame 是一个相对于其父视图位置的假想矩形。另一方面，Bounds 是相对于其自身坐标系的矩形。两者都表示为一个位置，以 `x`、`y` 为起点，以 `width` 和 `height` 表示大小。

## 问题 13：如何在类型之间进行转换？

对于 Objective-C 和 Swift，答案会有所不同。一般来说，由于 Objective-C 是 C 语言的超集，类型转换的工作方式类似。我们可以通过以下方式转换类型：

```
int i = (int)42.69f;
```

在 C 语言中，这种转换会丢失精度。

Swift 中的类型转换稍微复杂一些，但功能强大。我们使用所谓的类型检查运算符（`is` 和 `as`）来验证某个变量是否属于特定类型。以下是一个示例：

```
var appleCount = 0
var orangeCount = 0
for item in Shop {
    if item is Apple {
        appleCount += 1
    } else if item is Orange {
        orangeCount += 1
    }
}
```

在这个例子中，如果条件满足（item 是 Apple 或 Orange），运算符返回 `true`，评估成功，代码将继续执行。请注意，如果要考虑多种类型，情况会变得更复杂。

`as` 运算符非常相似。考虑以下代码片段：

```
if let _ = item as? Apple ...
if item is Apple ...
```

这两者非常相似，但在这种情况下，`as` 运算符将转换结果类以供后续使用（通常是这种情况）。

如果你想跟进这个问题，可以询问候选人使用 `as?` 和 `as!` 之间的区别。

## 问题 14：你会调用哪个方法来查找对象类型？

同样，根据我们使用的语言，我们有不同的选择。

在 Objective-C 中，我们必须调用以下函数：

```
[name isKindOfClass:[NSString class]]
```

而在 Swift 中，我们可以这样做：

```
if object is String
```

在 Swift 中，我们还有一堆额外的选项可以帮助我们：

```
type(of:)
isKind(of:)
isMember(of:)
conforms(to:)
```

## 问题 15：`#if` 和 `#ifdef` 有什么区别？

`#if` 的功能类似于普通的 if 语句。

```
#if __IPHONE_OS_VERSION_MAX_ALLOWED >= 30200
if (UI_USER_INTERFACE_IDIOM() == UIUserInterfaceIdiomPad) {
    return YES;
}
#endif
return NO;
```

`#ifdef` 意思是“如果定义了——某个值或宏”。

```
#ifdef    RKL_APPEND_TO_ICU_FUNCTIONS
#define RKL_ICU_FUNCTION_APPEND(x) _RKL_CONCAT(x, RKL_APPEND_TO_ICU_FUNCTIONS)
#else  // RKL_APPEND_TO_ICU_FUNCTIONS
#define RKL_ICU_FUNCTION_APPEND(x) x
#endif // RKL_APPEND_TO_ICU_FUNCTIONS
```

## 问题 16：iOS 编译器是什么？

编译器是一个庞大的主题，初级开发人员绝不可能详细解释它们。然而，如果你从事 iOS 开发，你应该知道在 iOS 中可以使用不同的编译器，并且它们经历了不同的成熟阶段。它们是 `GCC`、带 `Clang` 的 `LLVM` 以及带 `GCC` 的 `LLVM`。

这个话题可以无限延伸。如果被问及当前的技术现状，知识渊博的候选人会告诉你，简而言之，不再需要 `GCC`，而带 `Clang` 的 `LLVM` 是可以满足你所有需求的编译器。也可以给出细节：`LLVM` 编译速度比 `GCC` 快；生成的代码通常更快，并且在编译过程中提供更准确的错误信息。

## 问题 17：如何为生产版本和开发版本保留不同的变体？

这里可以采用不同的方法。一种初始方法可能是创建不同的目标，每个目标使用不同的 `Info.plist` 文件。每次选择一个目标时，将使用不同的 `Info.plist`，从而能够区分不同的变量（例如令牌、URL 等）。

也可以考虑使用 Bundle 标识符。定义不同的预处理器宏将控制各种代码块的编译条件。

或者，或者除此之外，你可以将构建配置设置（包括 `Info.plist` 文件的变更位置）放入 `*.xcconfig` 文件中，并在项目的项目、信息和配置区域引用这些文件。然后，你只需更改你的方案就可以构建应用程序的不同版本。将构建配置设置放入文件中对配置控制也有巨大的好处。

## 问题 18：`viewDidLoad` 和 `viewDidAppear` 有什么区别？你会使用哪一个从远程服务器加载数据并将其显示在屏幕上？

`viewDidLoad()` 只在视图控制器首次加载到内存时调用一次。通常，当我们想要实例化变量并构建在视图控制器的整个生命周期中都存活的视图时，我们在这里进行。然而，此时视图还不可见！

`viewDidAppear()` 在视图首次显示到屏幕上时被调用。实际上，在视图控制器的生命周期中，这个方法可能会被多次调用。例如，想象一下，当一个模态视图控制器被加载然后被关闭时。视图已经被加载过了，但它会因此出现两次。在这个方法中，我们通常执行布局相关的操作，比如绘制 UI、呈现模态屏幕等。

这是你想要实例化任何实例变量并构建在此视图控制器整个生命周期中都存在的任何视图的地方。然而，此时视图通常还不可见。

要回答这个问题，你应该在 `viewDidLoad` 方法中从远程服务器加载数据。这样，数据只加载一次，而不是每次视图出现时都加载。

为了跟进，你可以询问候选人，在 `viewDidAppear` 中保留内容会有什么影响。影响是，如果在视图消失后这些内容没有被释放，很可能会发生内存泄漏。

## 问题 20: 什么是 NSUserDefaults？

`NSUserDefaults` 是一个允许开发者保存应用程序或用户数据相关设置、属性和信息的类。`NSUserDefaults` 存储键及其关联值。可以使用 `NSUserDefaults` 存储以下类型：

*   `NSData`
*   `NSString`
*   `NSNumber`
*   `NSDate`
*   `NSArray`
*   `NSDictionary`

这里可能的后续问题如下：iOS 中与其他存储机制有何区别？是否可以用 `NSUserDefaults` 存储图片？答案是肯定的。图像可以转换为 Base64 并作为 `NSString` 存储。另一个问题是，这是否是使用 `NSUserDefaults` 的有效方式。

通常，应该引导候选人讨论使用 `NSUserDefaults` 相对于其他存储类型的优缺点。

## 问题 21: 你如何测试代码？如何使代码可测试？

这些也是重要的问题，可以引出许多有趣的讨论，我们希望借此了解候选人的思维方式及其理论知识。一个相关的讨论可以是手动测试和自动化测试的好处。每种测试类型都有不同的优缺点。让我们谈谈自动化测试。

*   测试可以运行得相当快，因为不需要人工干预。
*   如果能合理利用自动化工具，它们可能具有成本效益。短期内可能成本较高，但从长远来看肯定会降低成本。
*   每个人都能看到结果。测试可以重现、记录和保存。
*   另一方面，工具可能有局限性，我们可能会受到测试工具或 API 开发的限制。

手动测试也有其自身的优缺点。

*   短期内，它降低了成本。
*   它更有可能发现真实的用户问题。因为自动测试是自动化的，它们不会测试特殊的边缘用例或即兴测试脚本中未提供的内容。
*   另一方面，有些任务可能很难手动执行，最坏的情况是重复、枯燥且毫无激励性。

也可以向候选人提问：你使用哪些测试框架？`XCTest` 与 `Xcode` 紧密集成，因此大多数开发人员至少应该对该框架有最低限度的理论了解。其他测试框架包括 `Appium` 和 `Calabash`。为了深入探讨，你可以讨论每个框架的优缺点。这可能是将候选人的知识与公司知识对齐的便捷方式，尽管雇佣候选人不应仅仅基于他/她掌握的框架知识，而应基于他/她的总体态度和推理能力。今天的框架明天可能就在博物馆里了。

最后，关于这个话题一个有趣的讨论是如何使代码可测试。创建可测试代码的原则总是能引发有趣的对话，其中 `SOLID` 原则是一个很好的起点。

## 问题 22: atomic 和 nonatomic 属性之间有什么区别？合成属性的默认行为是哪个？你会在何时使用其中之一？

`atomic` 是默认行为。一个 `atomic` 属性将确保在另一个进程访问该变量之前，当前进程由 CPU 完成。显然，这并不快，因为它要确保进程完全完成。

另一方面，`nonatomic` 行为不是默认行为。对于合成代码（即使用 `@property` 和 `@synthesize` 创建的变量）来说它更快。它们不是线程安全的，当两个不同的进程同时访问同一个变量时，使用它们可能会导致意外行为。

## 问题 27: 什么是 blocks？

`blocks` 是添加到 C、Objective-C 和 C++ 中的语言级别特性，它允许您创建不同的代码段，这些代码段可以像值一样传递给方法或函数。

`blocks` 是 Objective-C 对象，这意味着它们可以添加到 `NSArray` 或 `NSDictionary` 等集合中。它们还能够从封闭作用域捕获值，这使得它们类似于其他编程语言中的闭包或 `lambda` 表达式。

愿意深入探讨这个主题的候选人也可以说，Swift 中等效于 `blocks` 的是 `closures`。`closures` 是一等对象，因此可以嵌套和传递（就像 Objective-C 中的 `blocks` 一样）。在 Swift 中，函数只是 `closures` 的一个特例。

## 问题 28: 如何插入一个在发布构建中会被禁用的合理性检查？

你可以使用一个仅在调试构建中运行的断言代码。以下代码片段是一个示例：

```
#if defined(NDEBUG)
{
// The assertion code below should be compiled out of existence in a release
// build.  Log an error and abort the program if it is not.
bool ok = true;
NSCAssert(ok = false, @"NS assertions should be disabled but are not");
if (!ok)
{
NSLog(@"Detected release build but NS_BLOCK_ASSERTIONS is not defined");
return -1;
}
}
#endif
```

## 问题 29: 在 Swift 中，何时适合使用 let 和 var？

`let` 关键字定义一个常量。

```
let theAnswer = 42
```

之后 `theAnswer` 不能被更改。这就是为什么任何可选或弱引用的东西都不能用 `let` 来写。前者必须在运行时更改，必须使用 `var` 来写。

`var` 定义一个普通变量。

```
var president = "Lincoln"
```

## 问题 30: 什么是协议，如何定义你自己的协议，何时使用它？

协议是一种指定一组方法的方式，如果一个类想要与你的某个类一起工作，就需要实现这些方法。委托和数据源，如 `UITableViewDelegate` 和 `UITableViewDataSource`，确实是协议。

```
@protocol MyProtocol 
- (void)aRequiredMethod;
@required
- (void)anotherRequiredMethod;
@optional
- (void)anOptionalMethod;
@end
```

`@required` 指定该方法*必须*实现，而 `@optional` 让开发者决定。

然后，你可以在类的接口中指定一个类“符合”某个协议（即实现了所需的方法），如下所示：

```
@interface MyClass 
@end
```



## 问题 31：什么是 MVC，它在 iOS 中是如何实现的，它是否有替代方案？

MVC 是一种众所周知的设计模式，它定义了每个组件必须如何实现、组件之间如何通信以及每个功能必须驻留在何处。默认情况下，Apple 提供了 MVC 模式的实现：`UIView` 是承载用户界面的视图；`UIViewController` 支持控制器，控制器监听事件，因此可以根据我们的需要更新视图；模型是应用程序使用的数据，可以驻留在我们为存储有用信息而创建的任何对象中。

MVC 有很多替代方案。一个常见的方案是结合使用 MVVM 和 Reactive Cocoa。其他替代方案可能包括 Viper 和使用函数式响应式代码。

一个经验丰富的候选人会告诉你模式和架构之间的区别。广义上说，模式是解决计算机科学领域中反复出现问题的模型，而架构是一组规则和组织方式，它定义了代码必须如何构建才能可维护和可扩展。

## 问题 32：在 UIView 中指定元素布局有哪些不同的方式？

在 iOS 应用中有几种不同的技术来指定布局。

*   我们可以使用 `InterfaceBuilder` 向项目中添加一个 XIB 文件。之后，可以从我们的应用程序代码中加载这个 XIB 文件。`InterfaceBuilder` 也允许我们创建故事板。
*   你可以编写自己的代码来使用 `NSLayoutConstraint`，让视图中的元素通过自动布局进行排列。
*   你可以创建描述每个元素确切坐标的 `CGRect`，并将它们传递给 `UIView` 的 `- (id)initWithFrame:(CGRect)frame` 方法。
