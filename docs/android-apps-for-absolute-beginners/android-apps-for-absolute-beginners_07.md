## XML 入门

可扩展标记语言，即 XML，是更早的标准 SGML（标准通用标记语言）的一个子集。XML 的目标是以标准化的结构来传递数据及其相关信息（即元数据），这种结构可以在不了解数据含义或结构规则的情况下被查询和使用。我知道，这是一个相当抽象的定义。维基百科提供了另一种定义：

> *可扩展标记语言（XML）是一种标记语言，它定义了一组规则，用于以人类可读且机器可读的格式对文档进行编码。*
> 
> ——[`https://en.wikipedia.org/wiki/XML`](https://en.wikipedia.org/wiki/XML)，2020 年 7 月

作为定义，这并没有好到哪里去。一个更实用的定义可能是：

> *XML 是一种标记语言，它定义了如何交换数据，以及如何描述被交换的数据，以便人类和机器都能理解数据、其结构以及与其他数据的关系。*
> 
> ——Grant Allen，2020 年 7 月

这仍然不完美，但已经很接近了。遗憾的是，这个评价不仅适用于我的简短描述，也适用于整个 XML。也许构建你对 XML 理解的更好方法是实际深入其结构和运作方式。

关于 XML 的一些关键点。严格来说，XML 不是一种编程语言……用 XML 编写的东西不会“执行”任何操作。XML 是一种标记语言，旨在为数据传达含义和结构，以便程序和编程语言可以用这些数据做事情。

当你创建 XML 时，你是在将文本放入一个文档中，该文档的某些方面受 XML 标准和规则约束——这被称为 XML 文档。无论 XML 文档变得多么复杂或令人费解，你始终可以退后一步，将其视为纯粹的文本——某些地方具有特殊含义、并遵循约定的结构和语法规则的文本，但归根结底仍是文本。

XML 没有预设定义 XML 文档中需要哪些预定义的组件。它确实定义了如何使用基本构建块在文档中定义你自己组件的规则和语法。XML 及其规则的总体目标是提供一种独立于机器/程序的方式来描述和打包数据，并使描述和结构在 XML 文档本身或相关联的 XML 模式中自描述。

让我们来看看关键的 XML 构建块。

### XML 元素

元素是 XML 的基本构建块，它们是为数据提供含义和结构的“标签”，但其本身并非数据。XML 元素是嵌套的，因此 XML 文档中的第一个、最外层的元素被称为**根元素**，它可以包含（在其内部嵌套）一个或多个**子元素**。一个 XML 文档中仅有一个根元素。

元素标签由 `<`（大于号）和 `>`（小于号）符号括起。对于我定义和使用的每个元素，都需要一个开始标签和一个名称相同、但在名称前添加了 `/`（反斜杠）的匹配结束标签。必须使用此方法才能保证 XML 在语法上正确——在 XML 术语中称为“格式良好”。例如，我可能决定创建一个 XML 文档来存储人员的联系方式，其数据结构类似于通讯录，并决定将根元素命名为 `<addressbook>`。

我的 XML 文档起初看起来像这样：

```
<?xml version="1.0" encoding="UTF-8"?>
<addressbook>
</addressbook>
```

这包含了版本和编码的特殊标签以及 `<addressbook>` 根元素的开始标签，并以匹配的 `</addressbook>` 结束标签正确闭合。到目前为止，一切顺利。

我可以为一个人包含一个子元素，并在该人员元素内包含名字和姓氏的子元素，如下所示：

```
<?xml version="1.0" encoding="UTF-8"?>
<addressbook>
    <person>
        <firstname>Jane</firstname>
        <lastname>Smith</lastname>
    </person>
</addressbook>
```

到目前为止，一切顺利。请记住，元素名称以及它们隐含的作用都是由我来决定的。

**注意：** 元素的结束标签不必总是独占一行。实际上，XML 的任何部分都不一定要有换行。但是，这样做确实能让 XML 更易读。对于可读性好的 XML，普遍接受的规范是：开始标签出现在新行；当某个标签没有子元素时，结束标签可以放在与开始标签同一行的末尾（如前面的 `<firstname>` 和 `</firstname>`）；或者，开始和结束标签各自独占一行，并将任何子元素嵌套在其中（如前面的 `<person>` 和 `</person>` 标签）。标签也会缩进以表明它们是子元素，但同样，这只是为了方便人类阅读的约定。

如果这是一个真实的通讯录，里面应该不止一个人。但您可能会注意到，对于读取此 XML 的计算机（甚至人类）来说，很难区分哪个人是哪个。如果我们执行以下操作，任何试图追踪人员对应关系的程序都会遇到问题：

```
<?xml version="1.0" encoding="UTF-8"?>
<addressbook>
    <person>
        <firstname>Jane</firstname>
        <lastname>Smith</lastname>
    </person>
    <person>
        <firstname>John</firstname>
        <lastname>Jones</lastname>
    </person>
</addressbook>
```

XML 对此的解决方案之一是使用**属性**，它们类似于元素的参数。

### XML 属性

任何 XML 元素都可以拥有零个或多个与之关联的属性。属性与元素本身提供的数据不同，它们出现在元素名称的 `< >` 标记内，紧跟在标签名称之后。属性采用以下形式：

```
attribute_name="value in double quotes"
```

继续我们的通讯录示例，我们可以通过向 `<person>` 元素添加一个 ID 属性，并使用数值为每个人分配不同的 ID，来区分 XML 数据中的不同人员。结果如下：

```
<?xml version="1.0" encoding="UTF-8"?>
<addressbook>
    <person id="1">
        <firstname>Jane</firstname>
        <lastname>Smith</lastname>
    </person>
    <person id="2">
        <firstname>John</firstname>
        <lastname>Jones</lastname>
    </person>
</addressbook>
```

请注意，即使您和我可能会将此处呈现的 ID 值视为整数，甚至编写程序将其作为整数处理，但 XML 总是将属性放在双引号内，使其看起来像是字符串。

如果某个元素有多个属性，只需用空格分隔即可。例如，我们可以使用 `dateadded` 属性来记录将某人添加到通讯录的日期：

```
<?xml version="1.0" encoding="UTF-8"?>
<addressbook>
    <person id="1" dateadded="2023-01-15">
        <firstname>Jane</firstname>
        <lastname>Smith</lastname>
    </person>
    <person id="2" dateadded="2023-02-20">
        <firstname>John</firstname>
        <lastname>Jones</lastname>
    </person>
</addressbook>
```

### XML 值

XML 文档中封装的实际数据是包含在元素开始和结束标签内的值。在我们一直使用的通讯录示例中，值包括 `<firstname>` 元素中的 "Jane" 和 `<lastname>` 元素中的 "Smith"。为传递这些信息而设置所有这些 XML 语法和元素标签可能显得有些多余（这种冗余性也是对 XML 的常见批评之一），但对于阅读 XML 的人类来说，它具有良好的自描述性，并且还能提供关于结构、完整性等方面的保证。

说到完整性，元素完全可以不包含任何数据，就像本章前面第一个通讯录 XML 文档所示那样。有几种选项可以让元素存在但不包含任何数据：既可以使用您已经见过的开始标签和结束标签方法，也可以使用简写的自闭合标签方法——这种方法下元素标签名称只出现一次，并在末尾添加一个 `/`（反斜杠）。让我们向 `<person>` 结构添加一个 `<middleinitial>` 元素，展示这两种选项，如下所示：

```
<?xml version="1.0" encoding="UTF-8"?>
<addressbook>
    <person id="1">
        <firstname>Jane</firstname>
        <middleinitial></middleinitial>
        <lastname>Smith</lastname>
    </person>
    <person id="2">
        <firstname>John</firstname>
        <middleinitial/>
        <lastname>Jones</lastname>
    </person>
</addressbook>
```

对于没有数据的元素，`<middleinitial></middleinitial>` 和 `<middleinitial/>` 这两种形式都是有效的。请注意，不能在自闭合标签下嵌套任何子元素。

### XML 命名空间

XML 的一大优势在于它极大地简化了开发者和使用数据的系统之间的数据交换。一旦您习惯了以 XML 格式共享数据，它就会变得像第二天性一样自然。但所有这些易于传输和共享的特性也可能带来问题。一个特殊的问题是：由于您可以定义自己的 XML 标签（甚至可能定义自己的 XML 模式），您可能会发现其他人也开发了自己的模式，并且使用的元素名称与您的冲突。如果您和他们从不共享数据，这没什么大不了的；但如果您成为分享 XML 格式数据的实践者，那么在尝试混合来自多个来源的数据时，几乎肯定会遇到名称冲突。那么该怎么办呢？

XML 标准提供了一种包含两个部分的方法来解决这种名称冲突问题。首先，您可以为任何喜欢的元素添加一个标识字符串作为前缀，这是一个标记，用于指示给定元素源自哪个来源。所以，如果我们知道，我们要将来自我们自己的 `<addressbook>` 以及我们自己定义的 `<person>` 元素的数据，与来自其他人的、可能具有完全不同结构和一组元素的数据进行混合，我们可以为我们的元素添加前缀以进行区分，例如：

```
<ab:addressbook xmlns:ab="http://www.example.com/addressbook">
    <ab:person id="1">
        <ab:firstname>Jane</ab:firstname>
        <ab:lastname>Smith</ab:lastname>
    </ab:person>
    <ab:person id="2">
        <ab:firstname>John</ab:firstname>
        <ab:lastname>Jones</ab:lastname>
    </ab:person>
</ab:addressbook>
```

这个前缀充当了这些元素的命名空间。而来自另一个 XML 文档、具有令人混淆的 person 版本的元素将使用不同的命名空间前缀，例如：

```
<other:person xmlns:other="http://www.othercompany.com/schema">
    <other:firstname>Judy</other:firstname>
    <other:lastname>Walsh</other:lastname>
</other:person>
```

XML 标准要求我们提供一个 URI 作为命名空间的唯一区分符——基本上，这是一个保证唯一并能反映我们为文档定义的 XML 结构的属性。这通过在元素的开始标签中使用 `xmlns` 属性来指定——通常您会在 XML 文档的根元素中看到它，如下所示：

```
<?xml version="1.0" encoding="UTF-8"?>
<ab:addressbook xmlns:ab="http://www.example.com/addressbook">
    <ab:person id="1">
        <ab:firstname>Jane</ab:firstname>
        <ab:lastname>Smith</ab:lastname>
    </ab:person>
    <ab:person id="2">
        <ab:firstname>John</ab:firstname>
        <ab:lastname>Jones</ab:lastname>
    </ab:person>
</ab:addressbook>
```

### XML 世界的其他部分

XML 标准中还有其他部分我们不会在本章中介绍，随着您知识的增长，这些内容可能会引起您的兴趣。



#### 1. CDATA

`CDATA` 值是特殊的 XML 值，它们会被视为逐字数据，并且不会被处理或解析以发现其中是否包含更多嵌套的 XML 标签。这起初可能不太容易理解，但你会发现它常用于在 XML 文档中传递其他形式的数据标记或展示标记（如 HTML）。其作用是防止那些旨在强制执行 XML 标准的 XML 解析器和程序，在遇到看起来像 XML、但实际并非 XML 且不遵循 XML 规则的内容时意外出错。

#### 2. XML 路径和 XQuery

`XML 路径`与 `XQuery` 提供了一种遍历 XML 文档、通过浏览标签层次结构、评估属性等方式来查找数据的方法。默认情况下，这些工具不会用于 Android 应用的基本设计和构建，不过，如果你构建的 Android 应用需要自行处理基于 XML 的数据，那么你当然可以使用它们。

#### 3. XSLT 和转换

`XSLT` 是 XML 样式表转换语言，它是一种编程语言，旨在查询和转换 XML 的结构与数据，并以此生成衍生数据作为输出。XSLT 转换的输出通常是另一个 XML 文档，但它也可以是不同格式的数据。

实际上，有成千上万个网站可以帮助你进一步学习 XML，但现在你已掌握了足够的基础知识，可以深入探究 Android 如何使用 XML，并开始为你的 Android 应用编写和编辑 XML 文件了。

### 使用 XML 进行资源定义

Android 利用 XML 的另一个主要领域是常量和引用值的定义与使用，在这些场景中，给定的数据以只读方式用于一系列用例。有大量的例子可以说明常量数据如何有助于简化应用构建过程，并减少所需的工作量，从确保所有活动和视觉元素具有一致的外观和感觉，到确保诸如珠穆朗玛峰的高度（8848 米）、你的应用名称（`"MyFirstApp"`）或其他不变的数据值可以在一个地方声明，但在许多其他地方使用。

一些关键的资源定义文件包括：

1.  `colors.xml`：一个提供有用名称（对你而言有意义的）的参考文件，对应各种颜色的红绿蓝（RGB）十六进制表示。
2.  `strings.xml`：一个用于抽象文本字符串的参考文件，例如简单的单词、短语、句子和段落，这样它们就可以在各种源代码和其他文件中被引用，而无需将文本复制到许多地方。
3.  `styles.xml`：一个参考文件，用于在自定义样式的名下收集一组颜色定义和其他样式元素，以便该样式可以在整个应用中用作常见的设计主题。
4.  `dimens.xml`：一个参考文件，用于提供不同尺寸和大小设置的抽象表示，允许你在一个位置控制密度/大小的变化，而无需在许多不同的文件中编辑许多值。

这些资源文件的定义对于高层次的概念理解来说还不错，但要理解它们*如何*被使用，则还欠缺一些。让我们深入实践，亲自动手，将这些资源文件的内容混搭并添加新的内容。

### 增加描述性

在 Android 世界中，`strings.xml`文件就像一个沉睡的巨人。你可以在此定义各种常量，这些常量基本上是很少变化，或者至少在你的应用上下文中初始值为固定值的东西。当你考虑常量时，可以超越我之前给出的幽默示例（为什么要把自己限制在珠穆朗玛峰以米为单位的高度上——为什么不用英尺、杆、弗隆或链呢？！）。更严肃地说，你应该时刻留意（此处非双关）任何在代码中输入字面字符串的情况，并问问自己，将单个条目放入`strings.xml`文件中是否会更好。程序员们几十年来艰难领悟到的教训，归结起来就是一句格言：“将字面字符串从你的代码中抽象出来——你以后会感谢我的。”
