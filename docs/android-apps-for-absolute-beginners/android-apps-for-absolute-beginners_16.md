# 4. 关闭

这是最终状态，此时所有持久化数据、视图层级结构、缓存数据等都将从操作系统内存中清除（并且，隐式地，任何不处于`Launched`、`Running`或`Killed`状态的应用程序都将如此）。

这些状态的含义不言自明。而乐趣的起点在于，是什么促使应用程序在这些状态之间切换——这正是生命周期回调方法的领域。

### 理解 Activity 生命周期回调方法

图 11-1 中展示的每个生命周期方法都具有特定的行为和用途，您应该熟悉它们。您可能不需要立即为应用程序的这些生命周期阶段添加自定义逻辑，但了解在何处添加内容，将为未来作为 Android 开发者的快速进步奠定基础。我们将逐一讲解每个方法，以突出其具体用途。

每个方法都具有与其他方法相同的共性，例如，几乎无一例外地，当任何回调方法被调用时，其第一个动作都是调用相应父类中的等效方法。例如，在本书前面的示例中，您会看到 `onCreate()` 方法的第一个动作是调用 `super.onCreate()`，这会调用父类的对应版本。当您不希望自动调用父类方法时存在例外情况，但我会在遇到这些情况时明确指出。目前，请默认遵循“调用父类”这一良好实践。

#### `onCreate()`

每个 Activity 的生命周期都始于 `onCreate()` 方法。无论用户是点击了 Android 主屏幕上的图标来启动您的应用程序，还是配置变更触发了重建显示的需求，`onCreate()` 都会被调用。该方法接收一个 `Bundle` 对象，用于应对后续的重建情况，因为 `Bundle` 会存储之前的状态、数据以及激活 Activity 所需的资源——这些统称为实例状态。

对于 Activity 的 `onCreate()` 方法，您应考虑执行以下操作：

1.  将所有计划使用的布局加载到内容视图中，以便 Android 在 `onStart()` 方法被调用时，能在显示 UI 之前创建好它。
2.  初始化您在 Activity 类定义中定义的所有 Activity 级别变量。

您可能还需要做一些工作来使用作为应用程序一部分而创建的全局资源或变量。我们将在稍后讨论 `onRetainNonConfigurationInstance()` 时，以及在本书本章及后续章节中介绍偏好设置时，探讨更多示例用法，届时再回到全局资源这个主题。

#### `onStart()`

一旦 `onCreate()` 完成了从布局定义中构建所有对象以及任何其他初始化工作，`onStart()` 的工作就开始了。`onStart()` 的职责是将最终的 UI 实际渲染到用户屏幕上。在前面的生命周期模型中，`onStart()` 方法也会在 `onRestart()` 之后被调用。

您很少需要重写 `onStart()` 并提供自己的逻辑。即使您想这样做，也需要意识到，如果您的 Activity 正沿着 `onRestart()` 路径执行，此时实例状态尚未完全恢复，并且直到随后调用 `onRestoreInstanceState()` 时才能恢复。

实现自定义 `onStart()` 重写的一个有用场景是，当您拥有某些长期存在的自定义资源，并且基于 Activity 不可见时，您已经优雅地冻结或暂停了这些资源——做一个好公民，在不需要时就不使用资源。如果您选择设置这样的资源管理，那么 `onStart()` 方法就是恢复或解冻此类自定义资源的地方。请务必调用 `super.onStart()` 方法来调用父类的等效方法，即使您知道父类中没有重写代码。在这种情况下，Android 本身仍然需要执行其内部工作。

#### `onRestoreInstanceState()`

如果用户通过返回按钮或其他方式结束一个 Activity，从当前 Activity 切换到另一个，则表示他们已结束与原始 Activity 及其所有关联状态的操作。这向作为应用程序开发者的您表明，除了需要维护的长期存在的资源外，您可以丢弃该 Activity。当 Android 自身终止一个 Activity 时，情况通常不同。Android 终止 Activity 主要是由配置变更引起的，并且它确实需要关心重建该 Activity 以满足用户需求所需的内容。这意味着不仅要重建 Activity 的视觉显示，还要重建其所有实例状态。

您可能希望在此处执行任何自定义的实例状态恢复，并请记住，实例状态与长期存储不同。存在其他机制来处理长期存储，包括偏好设置，我们将在本书后面介绍。如果您希望持久化的数据类型比偏好设置系统设计的更复杂或更大，那么文件和数据库等基础计算概念是最佳方法，我们将在第 18 章中介绍它们。

#### `onResume()`

在 Android UI 方面，`onResume()` 方法是关键时刻。`onResume()` 方法是使完全渲染和填充的布局变为可见的关键方法，将所有累积的对象、逻辑等置于前台。`onResume()` 与 `onPause()` 一起，在 Activity 的生命周期中，进出前台的转换会多次触发，因为其他 Activity 和应用程序会抢占它、获得焦点、被用户关闭等。

您通常应考虑添加到 `onResume()` 中的唯一逻辑是与改变 Activity 的实时视觉方面相关的，包括：

1.  动画
2.  视频
3.  自定义视图过渡效果

在所有其他方面，您应避免在 `onResume()` 中添加任何其他自定义逻辑。



#### `onPause()`

Android 从底层设计之初就支持多个应用程序同时运行，并依赖应用能够简单、轻松地暂停其活动，以便为用户决定使用的其他活动释放资源。为了支持这一点，应用程序会频繁地从前台状态切换到后台状态。在此之前，系统会立即调用 `onPause()` 方法。

在 `onPause()` 调用期间，最重要的任务包括：保存任何尚未放入 Activity 的 `Bundle` 中的状态，以及处理自定义动画、视频、音频或逻辑中的其他实时方面。此外，`onPause()` 调用也是一个临界点，超过此点后，Android 可以单方面终止你的 Activity 并回收其资源。这意味着一旦 `onPause()` 调用结束，就无法保证你的进程和 Activity 还能再次接收到任何事件——因此，请保存你需要的内容，并且不要假设在此之后，没有你的干预，Activity 的状态依然会持续存在。

#### `onStop()`

如果说 `onStart()` 负责管理 Activity 移入前台，那么你可以将 `onStop()` 视为其反向操作：负责将你的 Activity 移入后台。在 `onStop()` 完成后，你的 Activity 将不再可见，尽管相关的视图层级和属性仍然保持不变。

在 Android 积极的内存资源管理策略下，你应当时刻牢记，`onStop()` 可能永远不会在你的 Activity 中被调用，因此依赖于在此处编写自定义逻辑并不明智。你可能需要考虑重写默认行为的一个场景是，与通常不需要前台交互性的 Android 服务进行任何交互。

#### `onSaveInstanceState()`

`onSaveInstanceState()` 方法的名字及其在图 11-1 生命周期图中的位置，都非常清晰地表明了它的用途。该方法的作用是为任何即将被 `onDestroy()` 方法销毁的 Activity 保存其状态，从而避免用户发现重要的状态丢失，而他们却未意识到 Activity 可能已经结束并被重新创建。这种情况在发生任何配置变更事件时经常出现，因此需要为用户提供透明且自动的体验。

> **注意**
> 目前有一种趋势，即开始倾向于使用 `ViewModel` 的概念，并通过该技术保存任何临时状态。然而，`onSaveInstanceState()` 方法仍然得到全面支持。你可以在 `developer.android.com` 上阅读更多关于 ViewModel 的内容。

#### `onRestart()`

每当 Activity 从停止状态移动至启动状态时，都会调用 `onRestart()` 方法。该方法使你可以灵活地处理新启动的 Activity 版本，使其不同于那些曾被停止而现在又重新启动的版本。例如，在重新启动的情况下，可以利用保留的视图层级结构来加速某些初始化活动。

#### `onRetainNonConfigurationInstance()`

这是一个方法名非常长且奇怪的方法，仔细查看生命周期图会发现，它甚至没有出现在图上！原因是 `onRetainNonConfigurationInstance()` 严格来说并不与 Activity 的生命周期绑定。相反，其目的是在 Android 系统发生配置变更时，提供一个可被调用的回调机制。

你可能会问：“什么是配置变更？”有几种重要的操作，Android 将其视为整个设备配置的变更，必须通知所有正在运行的应用程序，并且应用程序需要执行必要的步骤来适应这种变更。这些配置变更操作包括：

1.  将设备从横屏旋转为竖屏，或反之亦然
2.  将设备连接到 USB 电源
3.  将设备放入底座
4.  添加或移除 SD 类存储设备
5.  更改 Android 操作系统的输入或显示语言

你的 Activity 可能需要处理部分或全部这些事件。最明显的例子是，需要根据设备的新方向重新绘制（或者严格来说是重新创建）布局。在这些情况下，Android 会保留 Activity 先前实例的所有资源，调用配套的 `getLastNonConfiguationInstance()` 方法将返回对这些资源的引用，使你能够执行所需的任何进一步变更处理。

随着无头 Fragment 的出现，这种方法在现代应用开发中已不常见。我们将在第 12 章中介绍 Fragment。

#### `onDestroy()`

我们已到达生命周期方法的终点，恰如其分的，这最后一个方法就是 `onDestroy()`，它本身完全关乎于结束 Activity。我们知道 Activity 是廉价且丰富的，在构建应用时，你应该随时使用它们并在需要时将其丢弃。你可以考虑添加到 `onDestroy()` 重写中的核心逻辑是任何以 Activity 为中心的清理工作。由于无法保证 `onDestroy()` 一定会被你的 Activity 调用，因此你不应依赖它，也不应期望在此处与其他资源或服务进行交互。

## 领悟 Activity 生命周期的目标

对于新开发者来说，掌握回调的整个概念对于更广泛地理解消息机制和基于事件的开发至关重要。然而，如果你仅仅将 Android 应用程序视为处于特定状态的 Activity，等待回调信号来触发行动，你就会错过一些更宏大的设计理念。

总的来说，廉价且易于操控的 Activity，以及精心封装的回调方法（意味着每个 Activity 都携带了如何从“这里”到达“那里”的指令），其最大的目标是给用户呈现一个无缝衔接的应用程序，能够响应用户的输入和需求。这种以用户为中心的观点对于提供连贯的体验至关重要，因为你的用户倾向于将你的应用视为一组或一系列层级的屏幕，而不是一堆松散关联、随时可以被轻易创建和销毁的 Activity。

当你开始编写包含越来越多 Activity 的应用程序时，牢记以下原则将对你有益：

1.  **Activity 应有重点。** 你创建的每个 Activity 应服务于一个单一的（但不一定简单的）目的或结果，而不是成为无关选项、操作和结果的大杂烩。
2.  **保存状态，然后再次保存。** 用户是不可预测的，他们会在使用你的应用时做出最意想不到的事情——从关闭手机到反复切换进出你的应用，甚至更多。为了让用户相信你的应用能够直观地处理任何意外情况，经常保存状态非常重要。可以将其视为一个自然的安全网，既廉价又有效。
3.  **为每个视图设置 ID。** 每个控件、每个视图、每个 `ViewGroup` 都应带有 `android:id` 属性。这有助于你帮助自己，因为当每个视图都有 ID 时，保存实例状态会变得容易得多。
4.  **应用和 Activity 可能消失。** 存在一些你无法控制的情况，尤其是无论你如何使应用高效和轻量，用户仍可能决定将其与那些占用大量内存和资源（“内存大户”和“资源恶霸”）的应用一同安装。如果用户在这种环境下暂停了你的应用，Android 操作系统可能会来终止你正在运行的应用程序，以释放急需的内存。



## 管理 Activity 配置变更

通过本章前面的描述，我强调了正常的 Activity 生命周期至少有一个例外情况，即 Activity 配置变更。配置变更通常被定义为影响整个应用、Android 用户空间或整个 Android 设备的事件。这可能会让你以为我在谈论地震事件——也许是 Android 地震。实际情况要平淡得多。被归类为配置变更的事件包括：更改设备的界面语言、将设备插入电源、与另一台设备配对，或者简单地将设备从竖屏旋转为横屏，反之亦然。

### 使用默认方法处理配置变更

首先有个好消息：Android 会通过向所有正在运行的应用发送通知回调来告知特定变更，从而为你跟踪设备发生的所有配置变更。除了这个基本通知之外，Android 还竭力避免让你承担过多工作来确定配置变更应如何在应用中体现。Android 通过利用程序中的现有结构来实现这一点。

Android 为你做的最有价值的操作之一就是监控发生的任何配置变更，并引用你为应用创建的所有可用布局，执行重新创建 Activity 所需的所有布局工作，使用与配置变更结果相匹配的布局——无论是将旋转匹配到相关的竖屏/横屏布局，还是从左到右到从右到左的语言变更，等等。

由于布局行为和重新创建已为你处理完毕，接下来只需要关注你为特定 Activity 创建或获取的任何资源。由于 Android 会销毁之前的 Activity 并重新创建它，因此你需要作为配置变更的一部分来保存和恢复这些资源。你的 Activity 有机会干净地保存其当前状态和资源，以便在重新创建时，新的 Activity 可以恢复它们，使 Activity 状态恢复到与销毁前相同的状态。以下回调涵盖了允许你在配置变更过程中保存和恢复状态与资源的流程：

1. `onSaveInstanceState()`：当触发任何配置变更时，在 Android 销毁当前 Activity 之前，会立即调用 `onSaveInstanceState()`。这是保存你需要保留的任何临时或正在使用的资源或数据的好时机。创建并使用一个 `Bundle` 对象来保存你想要保留的所有项目，以便在随后的 Activity 重新创建中使用。如果你回顾本书中已经展示的示例，可以看到 `Bundle` 对象在 `onCreate()` 时是如何被使用的（稍后会详细说明）。`onSaveInstanceState()` 的超类祖先还会在幕后为你执行一系列非常有用的工作，包括为任何已声明 ID 的对象（如小部件、UI 元素等）保存视图状态，以便在需要时随时重新创建。

2. `onCreate()`：当 Activity 即将启动时，它要么是全新创建的，要么（在当前讨论中）会接收到所需的 `Bundle` 对象，以便恢复已保存的资源和其他数据。你只需确保调用父类 `super.onCreate()` 方法，无需进行更复杂的操作。你始终可以选择添加自定义状态并在 `onSaveInstanceState()` 期间保存它——随后你还需要在 `onCreate()` 或下一个在（重新）创建 Activity 时调用的方法 `onRestoreInstanceState()` 中提供自定义的恢复逻辑。

3. `onRestoreInstanceState()`：你的 `Bundle` 对象也会传递给此方法，你可以选择此时检索和恢复你希望保存的任何其他资源。等到 Activity 生命周期的这个节点有一个好处，那就是你可以确保 Activity 的布局已经被填充（inflated）并设置了内容视图，从而使 Activity 的所有视觉方面都已就位。

## 总结

现在你已经对 Activity 生命周期以及你可用的、在需要时能被重写的回调方法的重要性有了扎实的了解。我们将在第 12 章中继续讨论生命周期，届时将介绍 Fragment，并展示一个综合的示例应用，说明在实际中如何以及在哪里使用生命周期回调方法的重写。
