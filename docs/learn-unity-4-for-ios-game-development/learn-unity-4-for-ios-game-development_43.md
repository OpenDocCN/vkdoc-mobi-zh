# 当前游戏对象的层菜单

当前游戏对象的层菜单位于右侧。与使用名称进行描述、使用标签进行识别不同，层（Layers）用于对游戏对象进行分组。你可以将这些层想象成类似 Photoshop 中的图层。注意不要将此层菜单与编辑器窗口右上角的“层”菜单混淆（不幸的是，在默认布局中，它们彼此相邻）。右上角的“层”菜单控制着哪些游戏对象在场景视图中可见（它实际上应该位于场景视图中）。

#### 变换组件

与其他所有游戏对象一样，主摄像机拥有一个**变换（Transform）组件**，用于指定该游戏对象的位置、旋转和缩放比例。缩放比例对于摄像机而言没有意义，但位置和旋转属性允许你放置和瞄准摄像机。

单击组件右上角的小问号图标。这将在网页浏览器窗口中打开变换组件的参考手册文档。

**提示** 每次遇到新组件时，你应该做的第一件事就是查阅文档。

每个组件的参考手册页面都会描述该组件的属性并解释其用法。每个参考手册页面还包含一个指向该组件对应脚本参考页面的链接，详细说明了如何从脚本中访问该组件。

### 摄像机组件

**摄像机（Camera）组件**是使此游戏对象成为摄像机的关键。严格来说，主摄像机是一个拥有`Camera`组件的游戏对象——该组件是组件的一个子类。但为了方便，我们可以说主摄像机是一个“摄像机”（因为这是它的功能），而使用“摄像机组件”来明确我们讨论的是该组件。

同样，点击此组件的问号图标将打开相应的参考手册页面，列出摄像机组件的属性。但我将在此处（按检查器视图中自上而下的顺序）简要介绍它们。

#### 清除标志

`清除标志（Clear Flags）`属性指定摄像机在每次渲染通道开始时如何初始化颜色缓冲区和深度缓冲区。颜色缓冲区包含摄像机渲染的像素，而深度缓冲区则保存每个像素到摄像机的距离（通过这种方式，它决定新绘制的像素是否应替换该位置的先前像素）。

-   固定颜色（Solid Color）：表示颜色缓冲区将被清除为指定的颜色——这基本上用作背景色，同时深度缓冲区也会被清除。
-   天空盒（Skybox）：同样会清除深度缓冲区，并在背景中渲染指定的天空盒纹理（如果没有提供天空盒，则默认为固定颜色）。
-   仅深度（Depth Only）：保持颜色缓冲区不变，仅清除深度缓冲区。这在添加一个在主摄像机之后渲染并覆盖其结果的摄像机时很有用，例如，在游戏世界的渲染之上渲染用户界面或 HUD（抬头显示）。在这种情况下，你不希望第二个摄像机在渲染前清屏。

#### 剔除遮罩

这正是层（Layers）的实用性所在。`剔除遮罩（Culling Mask）`指定了此摄像机将渲染哪些层。换句话说，只有当游戏对象的层被包含在摄像机的`剔除遮罩`中时，该对象才对摄像机可见。

默认值为`Everything`，对应所有层。但例如，如果你有一个 HUD 摄像机，你可以为所有 HUD 游戏对象定义一个 HUD 层，然后将 HUD 摄像机的`剔除遮罩`设置为仅渲染 HUD 层。接着，将主摄像机的`剔除遮罩`设置为包含除 HUD 层之外的所有层。

#### 投影

默认情况下，摄像机设置为使用透视投影（perspective projection），正如我在上一章描述场景视图时提到的，这意味着物体离摄像机越远，看起来越小（这种效果称为*透视缩短（foreshortening）*）。在此模式下，摄像机具有一个类似于四棱锥（不包括锥体底面）的视景体，从摄像机位置沿视线方向发出。这个视景体被称为*视锥体（frustum）*，由视野（FOV）、近平面和远平面定义。

视野类似于真实世界摄像机的视场角。FOV 是以度为单位表示的垂直视角。水平视角则由 FOV 和摄像机的宽高比共同隐式定义。近平面和远平面分别指定了屏幕在摄像机前方多远的位置，以及视锥体底座的距离。任何在视锥体之外的东西对摄像机都是不可见的。

为了更好地了解视锥体的外观，可以看看场景视图。图 3-7 展示了当你按下场景 Gizmo 上的 y 轴箭头，将视角切换到俯视图时，主摄像机视锥体的样子。按下 F 键（编辑菜单中“Frame Selected”命令的快捷键）将主摄像机居中显示在场景视图中，然后缩小直到你能看到整个视锥体。你还应该取消选中`Gizmos`菜单中的`3D Gizmos`复选框，这样当您缩小时，摄像机图标就不会缩小到看不见。

![9781430248750_Fig03-07.jpg](img/9781430248750_Fig03-07.jpg)

图 3-7. 场景视图中的摄像机视锥体

场景视图的右下角还会显示一个摄像机预览。这就是摄像机此时会渲染的内容，也就是游戏视图此时会显示的内容（你现在可以查看游戏视图来确认）。这个场景中还看不到任何东西，但稍后这将是一个很方便的功能。

请注意，视锥体轮廓和摄像机预览仅在摄像机被选中且检查器视图中的摄像机组件处于展开状态时显示（检查器视图中的每个组件都可以通过单击左上角的小三角形来展开或折叠）。这对于任何在场景视图中具有额外图形显示的组件来说通常是成立的。在检查器视图中展开或折叠组件将切换该显示。

透视投影的替代方案是正交投影（orthographic），它没有透视缩短效果，因此适用于 2D 游戏、等距游戏、用户界面和 HUD。正交模式下特有的摄像机属性是正交投影大小（Size），即正交视图在世界单位中的半高。

#### 视口

视口（Viewport）为`(0,0,1,1)`表示摄像机覆盖整个屏幕。这些数字是归一化的屏幕坐标，意味着屏幕宽度和高度上的坐标范围是从 0 到 1。对于分屏或屏幕内的窗口，你需要多个摄像机，每个摄像机设置不同的视口。例如，`(0,0,1,.5)`覆盖一半屏幕。然而，Unity iOS 不支持不覆盖全屏的视口。

#### 深度

当场景中存在多个摄像机时，深度（Depth）是另一个有用的属性。在每次渲染更新期间，摄像机将按照其深度值从小到大依次执行渲染通道。因此，举例来说，如果你有用于查看 HUD 的第二个摄像机，你可以指定主摄像机的深度为 0，HUD 摄像机的深度为 1，以确保游戏世界先渲染，游戏 HUD 渲染在其之上。

#### 渲染路径

`渲染路径（Rendering Path）`决定了场景的渲染方式。每个摄像机可以拥有自己的`渲染路径`，或默认使用在播放器设置中指定的路径。`顶点光照（Vertex Lit）`是最简单但最快的路径，`正向渲染（Forward Rendering）`支持更高级的图形，而`延迟渲染（Deferred）`是最华丽但最慢的，且在 Unity iOS 中不可用。

#### 目标纹理（专业版）



渲染到纹理功能仅在 Unity Pro 和 Unity iOS Pro 中可用。如果将纹理分配到此属性，摄像机将渲染到该纹理上，而非屏幕。随后，该纹理可以被放置到场景中，例如，用作电视显示器或汽车后视镜。

#### `HDR`

该属性启用高动态范围（`HDR`），它可以处理 `RGB` 分量超出正常 0 到 1 范围的像素，并有助于渲染具有极高对比度的场景。Unity iOS 中不提供 `HDR` 功能。

#### `FlareLayer` 组件

当你从 `GameObject` 菜单创建摄像机 `GameObject` 时，会自动附加一个摄像机 `Component` 和一个 `FlareLayer` `Component`。`FlareLayer` `Component` 允许摄像机渲染镜头光晕（本章稍后介绍）。

`FlareLayer` `Component` 有点不寻常，因为它不暴露给脚本，因此它有一个参考手册页面，但没有脚本参考页面。但是，仍然可以通过其类名从 `GameObject` 访问 `FlareLayer` `Component`，即调用 `GetComponent("FlareLayer")`。

#### `GUILayer` 组件

一个 `GUILayer` `Component` 也会自动附加到摄像机 `GameObject` 上。此组件允许摄像机渲染 `GUIText` 和 `GUITexture`，这些是常用于用户界面的 2D 元素。

**注意** `GUILayer`、`GUIText` 和 `GUITexture`（以及 `GUIElement`，它是 `GUIText` 和 `GUITexture` 的父类）与 UnityGUI（一个更新的内置 GUI 系统）无关。

你可能已经注意到，检查器视图将 `GUILayer` 显示为一个单词，而 `FlareLayer` 显示为两个单词。编辑器试图通过在大写字母跟随小写字母的地方引入空格，以更符合英语习惯的方式呈现类名。这对 `FlareLayer` 有效，但对 `GUILayer` 无效。无论如何，本书将坚持使用原始类名，因为这是在脚本中引用的内容。

#### `AudioListener` 组件

`AudioListener` `Component` 仅自动附加到主摄像机，因为一次只能有一个活动的 `AudioListener`。摄像机 `Component` 类似于你的眼睛，负责渲染其视野内的一切到你的电脑屏幕上，而 `AudioListener` `Component` 则像你的耳朵，负责将场景中所有可听的 `AudioSource` 路由到电脑扬声器。

### 向场景添加一个立方体

你已经注意到摄像机预览和游戏视图没有显示任何内容，当你点击播放按钮时，游戏视图也会出现同样的情况。当然，这是因为你的场景中除了摄像机之外没有任何物体。让我们遵循创建立方体这一历史悠久的传统来纠正这一点。

### 立方体剖析

当你在检查器视图中显示立方体时，看看它是由什么构成的。或者，换句话说，看看立方体的哪些 `Component` 使其成为一个立方体。

#### 变换组件

当然，与每个 `GameObject` 一样，这里有一个 `Transform` `Component`，它提供了立方体的位置、旋转和缩放。

#### 网格过滤器组件

`MeshFilter` `Component` 引用 3D 模型，在 Unity 中称为 `Mesh`，它由三角形和逐顶点信息（如纹理坐标）组成。当不引用像立方体这样的内置 `Mesh` 时，此属性会从项目资源中分配一个 `Mesh`。

#### 网格渲染器组件

与 `MeshFilter` 相辅相成的是 `MeshRenderer` `Component`，它决定了网格的渲染方式，并且很大程度上由材质（如果 `Mesh` 被分割成子网格，则是一系列材质）决定。你可以将材质想象成沙发上的布料——`Mesh` 提供形状，而材质包裹在 `Mesh` 周围以提供表面外观。

`MeshRenderer` 还控制 `Mesh` 是接收阴影还是投射阴影（我将在本章稍后介绍阴影）。

#### 盒碰撞器组件

`MeshFilter` 和 `MeshRenderer` `Component` 只让立方体变得可见。`BoxCollider` `Component` 提供了一个物理表面，用于 Unity 物理系统中的碰撞。由于此场景中没有可以与立方体碰撞的东西，因此你不需要此 `Component`。将其留在那里没有坏处，但你也可以通过取消选中其复选框来禁用该 `Component`。或者，你可以通过右键单击它并从出现的菜单中选择 `Remove Component` 将其从 `GameObject` 中完全移除（图 3-12）。

### 与视图对齐

既然你移动了立方体，你可以再次按 F 键使其在场景视图中居中。但这不会影响主摄像机，正如你在场景视图的摄像机预览中所看到的，以及如果你检查游戏视图的话也会看到。

幸运的是，有一种方便的方法可以将主摄像机与场景视图摄像机同步。选择主摄像机，然后在 `GameObject` 菜单下，调用 `Align With View`（图 3-13）。现在，主摄像机与场景摄像机具有相同的位置和旋转角度，并且你可以在摄像机预览中看到场景视图和游戏视图是相同的。

### 摄像机控制

现在如果你点击播放按钮，你可以看到立方体，但没有太多事情发生。立方体只是呆在那里，没有交互性。对于这种只有一个物体的场景，我喜欢做的第一件事是添加一个摄像机轨道脚本，这样我就可以从所有角度检查物体。方便的是，Unity 在标准资源中附带了几种摄像机控制脚本，其中包括一个 `MouseOrbit` 脚本。

将 *MouseOrbit* 脚本拖拽到 Hierarchy 视图中的 Main Camera GameObject 上。这将把该脚本附加到 GameObject 上，因此如果您现在选择 Main Camera，*MouseOrbit* 脚本将作为其组件之一显示出来（图 3-18）。

![9781430248750_Fig03-18.jpg](img/9781430248750_Fig03-18.jpg)

`MouseOrbit` 组件的第一个属性是对 `MouseOrbit` 脚本的引用（请注意 Unity 如何将组件类型美观地打印为 `Mouse Orbit`，但不要被误导，它实际上是 `MouseOrbit`！）。您实际上可以通过将 Project 视图中的另一个脚本拖拽到该字段，或单击其右侧的小圆圈（这将弹出一个可用脚本列表供您选择，目前列表中全都是您刚刚导入的标准资源脚本），来更改该属性引用的脚本。

脚本本身决定了后续的属性。请注意，MouseOrbit 组件中的每个属性都对应于 MouseOrbit 脚本顶部声明的公共变量（以 `var` 开头的每一行都是公共变量，而以 `private var` 开头的行是私有变量）。

**提示** 在 Inspector 视图菜单（右上角）中选择 Debug 选项，也会将私有变量显示为属性，这对调试很有用。现在试试，您会看到私有变量 `x` 和 `y` 显示出来。Debug 选项还会添加内置组件的诊断属性。

脚本定义的第一个属性是 Target，即摄像机将围绕其旋转的 GameObject。它默认为空（或在代码中为 `null`），因此如果您点击 Play，摄像机将没有围绕旋转的对象。要修正此问题，请将 Cube 从 Hierarchy 视图拖拽到 Target 字段中（图 3-19）。

### 添加光源

### 光源的构成

同样地，一个 GameObject 在附加了 Camera 组件时成为摄像机，在附加了 Light 组件时则作为光源工作。与 Camera 组件类似，Light 组件也具有若干属性，我们将按照 Inspector 视图中从上到下的顺序在此介绍它们。

#### 类型

光源的 Type 指定它是点光源、方向光、聚光灯还是区域光。由于此 GameObject 是作为点光源创建的，因此其 Type 自动设置为 Point。与作为无限远光源（有时称为*无限光*）的方向光不同，点光源具有位置（因此通常被称为位置光）并向各个方向辐射光线。聚光灯也有位置，但像圆锥体一样向特定方向辐射光线。区域光仅用于生成光照贴图（预先计算并“烘焙”到纹理中的光照）。

#### 范围

点光源的 Range 属性指定光源的半径。只有半径内的物体才会受到该光源的影响。

#### 颜色

Color 属性是光源发出的颜色。单击颜色小矩形会弹出一个颜色选择器来分配此属性。

#### 强度

Light Intensity 的取值范围是 0 到 1，其中 0 表示无光照，1 表示最大亮度。

#### 阴影类型

光源的 Shadow Type 属性指定它是否投射阴影，如果是，则指定是硬阴影还是软阴影。阴影仅在 Unity Pro 和 Unity iOS Pro 中可用，即便如此，也仅适用于方向光和点光源（在 Unity iOS Pro 中，仅适用于方向光）。

#### 遮罩纹理

方向光和聚光灯可以将遮罩纹理投射到表面，通常用于模拟阴影。

#### 剔除遮罩



`Culling Mask` 属性指定了光源将影响的层级。这与摄像机组件的 `Culling Mask` 决定摄像机可见对象的方式非常相似，光源的 `Culling Mask` 决定了哪些对象会被光源照亮。

#### Flare

`Flare` 属性在分配了 `Flare` 资源后，会产生一个从此光源发出的镜头光晕效果。

#### Draw Halo

`Draw Halo` 属性会在光源位置周围产生一个光晕效果。启用此属性时，将提供一个额外的光晕颜色属性。

#### Render Mode

光源的 `Render Mode` 指定该光源是 `Important`、`Not Important` 还是 `Auto`，其中 `Auto` 表示其重要性会根据光源的亮度和当前的质量设置自动确定。光源的重要性决定了它是否被应用为像素光。像素光是实现阴影和 `Cookie` 等效果所必需的。

#### Lightmapping

`Lightmapping` 属性决定此光源是用于动态光照、生成光照贴图，还是两者兼用。在这些示例中，你不会使用光照贴图（生成它们需要很长时间），因此 `RealtimeOnly` 或 `Auto` 对这些光源来说都没问题。

## 进一步探索

本章似乎涵盖了大量内容（绝非双关）来创建一个简单的静态场景，但在此过程中，我们探索了许多基本的 Unity 概念，特别是组件与游戏对象之间的关系。因此，你现在拥有了一个包含凹凸贴图和纹理的立方体场景，该立方体由一个带有光晕的点光源照亮，并被天空盒包围。该场景中的资源是从本地文件、Unity 标准资源和 Unity 资源商店导入的。由于主相机上附加了 `MouseOrbit` 脚本，场景也并非完全静止。脚本编写将在本书中扮演主导角色，从下一章开始，我们将超越静态场景。

在继续之前，休息一下，回顾一下 Unity 官方文档中关于目前已使用功能的部分，哪怕只是为了熟悉在哪里可以找到这些文档，以便将来参考。

#### Unity 手册

Unity 手册的 Unity 基础部分主要与上一章相关，描述了用户界面，但该部分确实有一个“创建场景”页面。更相关的是“构建场景”部分，它描述了游戏对象、组件和脚本之间的关系，如何使用检查器视图编辑组件属性，以及如何在场景视图中导航和移动游戏对象——这些内容在本章中都已涵盖。灯光和相机也做了讲解。本节列出但本书不会使用的一个功能是地形引擎（它在 Unity iOS 中可用，但速度较慢）。不过，地形是一个重要且令人印象深刻的 Unity 功能，值得一读。“资源导入与创建”部分描述了本章中使用的资源类型：网格、材质、纹理和脚本。导入资源的过程以及资源商店也在该部分做了说明。
