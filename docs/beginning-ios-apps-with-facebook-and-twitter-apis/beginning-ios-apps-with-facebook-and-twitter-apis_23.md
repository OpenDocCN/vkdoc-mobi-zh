# 隐私，隐私，还是隐私

就在不久之前，大多数人还通过电子邮件或即时通讯（IM）分享生活经历。在隐私和安全方面，那是一个更简单的时代——用户直接登录自己的邮箱或 IM 账户，从台式机或笔记本电脑直接向一个或多个特定收件人发送链接、照片等内容。

随着互联网的发展，用户分享信息的方式变得日益复杂且相互关联；信息已从用户的桌面迁移到了云端。然而，这种复杂性和关联性的增加导致了一个结果：对于单个用户来说，确保隐私和安全变得更加困难，因为恶意公司或个人有更多机会获取用户某个账户的凭证。

读完本章后，我们希望你能收获两个重要教训：

-   人们正通过*社交图谱*（Facebook 对用户在线好友网络的称呼）分享更多——也更有价值的信息。
-   安全和隐私的标准正在发生变化。

**注意：** 安全与隐私问题必须得到最高程度的重视。无论明智与否，用户都将极为敏感的个人信息托付给了 Facebook 和 Twitter。如果你的应用将他们的隐私或利益置于风险之中，他们将会憎恨你，在 App Store 评论中痛贬你的应用，并针对你的母亲说尽坏话。在使用 Facebook 和 Twitter API 时，务必将用户的隐私与安全放在首位。

## 旧有方式

用户生成的内容如今经过的环节比以往任何时候都多，这增加了人为或系统失误的风险。让我们看一个经典例子：使用在线服务打印数码照片。

过去，用户需要在照片冲印网站创建账户，登录后从电脑上传想要冲印的照片。从隐私角度看，在这种场景下，用户只需信任该照片冲印网站已采取适当措施，防止有人入侵网站获取用户名、密码、个人照片甚至信用卡信息。但这个例子中涉及的变量相对较少：仅涉及用户和照片冲印网站两方。

## 热点问题简史

在过去的几年里，Facebook 和 Twitter 都未能幸免于隐私和安全方面的纰漏。尽管大部分问题已得到缓解，但了解一点历史有助于你在发布应用前识别任何潜在的热点问题。

### Facebook 的前科

Facebook 历史上最引人注目的隐私失误或许是 Facebook Beacon，这是 Facebook 构建的一款默认加入的平台应用，旨在让用户分享自己的购物信息。Facebook 因未经许可收集用户数据并与广告商分享而遭到攻击。自 2007 年 Beacon 事件以来，众多软件服务推出了允许用户与社交图谱分享购物信息的工具，包括 Swipely、Blippy 和 Mint.com。这三家公司都再次利用了这些买家数据，但没有任何一家像 Facebook 那样轻率行事。

自 Beacon 事件后，用户、记者和分析师随时准备抓住 Facebook 中任何可发现的安全漏洞，每披露一个问题都会引发一波 Facebook 抗议和声讨。

教训：真正重要的可能不是你如何处理用户数据——而是你是否让你的服务采用"选择加入"模式，并在每一步都征求许可。正如后续类似 Beacon 的服务所证明的那样，只要用户觉得过程公开透明，他们非常愿意在隐私方面进行尝试。



#### Twitter 的过往记录

与 Facebook 相比，`Twitter` 的隐私失误记录显得更为笨拙，但其战略性也较弱。用户通常不像怀疑 `Facebook` 那样怀疑 `Twitter` 的动机；不过话说回来，大多数用户也不会在 `Twitter` 个人资料中注入同样多的隐私内容。`Twitter` 本质上几乎就是一个面向公众的工具，因此用户早已习惯于将他们的推文视为公共财产。（而且，由于多个搜索引擎现在已经开始索引 `Twitter` 的实时内容，这些推文确实已成为整个互联网的领地。）

尽管如此，`Twitter` 也有其敏感之处。每当 `Twitter` 上出现安全问题，它们都不可避免地指向该公司飞速的增长——以及随之而来的所有成长阵痛。2007 年，短信推文被发现容易受到欺骗攻击，这使得恶意行为者能够从用户的个人资料信息中获取其电话号码。2009 年，一名黑客通过字典攻击破解了一名 `Twitter` 员工的管理员密码，导致部分名人账号被入侵。其他漏洞则允许用户操纵其他用户关注自己；深夜节目主持人柯南·奥布莱恩的账户就曾沦为这类攻击的受害者。2010 年秋季，发现了一种 XSS 蠕虫病毒，它利用一个简单的 JavaScript 函数来搞恶作剧。

所有这些违规行为后来都已得到处理，但在此之前它们已让 `Twitter` 声誉受损。2010 年，美国联邦贸易委员会因其安全漏洞对 `Twitter` 提起了诉讼；不过，这些诉讼后来已达成和解。虽然 `Twitter` 没有像 `Facebook` 那样在用户中引发同等的怀疑，但它零散的 API 和正处于青少年期的快速增长意味着可能存在更多漏洞。你需要对用户的 `Twitter` 账户格外小心。同时也要记住，尽管推文流乍看之下似乎无足轻重，但你永远不知道你的用户希望在那里隐藏什么。

### OAuth 如何改变一切

不过，在当今时代，可以想象，前面提到的照片冲印网站现在拥有一个 API，能够为第三方网站、应用程序和服务提供从用户账户导入或分享照片的能力，只要用户授予第三方应用此权限。这通常发生在用户在第三方应用内输入其在照片网站的凭证（即其用户名和密码）时。

通过允许外部网站访问用户账户，照片分享网站正在创造一种局面，即第三方可能获得对用户账户和个人信息的完全访问权——甚至有可能更改用户的密码。不仅如此，该第三方应用现在还可以访问存储在照片网站上的其他账户信息。

那么，为什么用户相信这一切都会顺利呢？

一个原因（用户可能不知道）是 `OAuth`，这是一种双向安全协议，在社交 API 中正变得相当标准。`OAuth` 的设计初衷是让用户能与第三方共享其账户中的资源，而无需向第三方提供其用户名和密码，从而避免危及整个账户（以及任何共享这些凭证的其他账户）。

我们说 `OAuth` 是双向的，因为它有两个版本（`1.3` 和 `2.0`）正在被积极使用，但并非全面覆盖。`OAuth 2.0` 主要由 `Facebook` 推广。如果你打算将 `Facebook` 集成到你的应用中，你将使用后一个版本。`Twitter` 允许你使用 `OAuth 1.3`。`Facebook` 不允许使用 `OAuth 1.3` 的应用，而 `Twitter` 不允许使用 `OAuth 2.0`。

假设在照片冲印网站的场景中，一个第三方想要通过 `OAuth` 访问用户的账户；交互过程将如下所示：

1. 第三方将联系照片冲印网站，要求通过 `OAuth` 访问用户账户。
2. 用户将看到照片冲印网站提供的登录页面。该页面要求用户输入用户名和密码以授予权限。
3. 然后，第三方网站将收到一个 `OAuth` 令牌，该令牌可用于访问用户账户，而无需知道用户的用户名和密码。

#### 新标准的出现

`OAuth` 正迅速成为各网站允许第三方站点、应用和服务共享访问用户资源的默认标准。`Facebook`、`Twitter` 及大多数其他社交网站现在都鼓励或要求第三方使用 `OAuth`，这一趋势很可能会持续下去。

因此，我们用了第 5 章的大部分篇幅来详细讲解 `OAuth`，以帮助你将 iOS 应用与 `Facebook` 和 `Twitter` 集成。这并非巧合，因为这是本书的第二章；在处理社交 API 时，没有比安全性更重要的事情了。



### 用户真正“想要”什么

我们已经讨论了安全性，现在来谈谈隐私。对于用户如何看待隐私，存在着截然不同的观点。以下是各方观点的简要总结，以便您决定希望您的应用在隐私光谱中处于何种位置（无论对您还是您的用户而言）。

克里斯托弗·普尔，又名“Moot”，是 `4chan.org` 的创始人，历来是网络完全匿名的支持者。他在 2010 年 6 月的 TED 会议上这样说：

“我们正在走向社交网络，我们正在走向持久身份。我们正在走向隐私的缺失；事实上，我们牺牲了很多隐私，我认为这样做，在走向这些东西的过程中，我们正在失去一些有价值的东西。”后来，他总结道：“想说什么就说什么，这很强大。”

确实强大。普尔论点的核心是，用户被“倾听”的愿望可能与他们对归属权的渴望完全分离。因此，虽然您的 iOS 应用可能希望为公开应用内创作的内容（例如发布 iPad 绘画或游戏结果）做好准备，但必须牢记一点：只要谷歌和必应在抓取网页，利用社交图谱发布这些信息就有可能使其成为可搜索和可追溯的信息。

Facebook 的首席执行官马克·扎克伯格持有截然相反的观点。他认为，保持在线数据私密的冲动是一种我们最终都会抛弃的愚蠢的残留本能。以下是他在 2010 年 1 月的一次关于隐私规范变化的采访中所说的：

“……在过去的五六年里，博客大规模兴起，所有这些不同的服务让人们分享所有这些信息。人们不仅对分享更多信息和不同类型的信息感到自在，而且对更公开地与更多人分享也感到自在。这种社会规范只是随着时间的推移而演变。我们认为自己在系统中的角色是不断创新和更新我们的系统，以反映当前的社会规范是什么。

“许多公司会被他们自己建立的惯例和遗产所束缚——为 3.5 亿用户进行隐私变更并不是许多公司会做的事情。但我们将其视为一件非常重要的事情，始终保持初学者心态，思考如果我们现在创办这家公司会怎么做，我们断定这些就是现在的社会规范，然后我们就去做了。”¹

本书的作者在这场辩论中（也许是策略性地）持中间立场。是的，匿名有其价值，尤其是在涉及未成年人的时候（例如在 iOS 游戏中心应用中）。但是，将您的现实身份与在线身份联系起来也日益成为常态。由您来决定您的应用是会在社交图谱中丰富用户的个人形象，还是会成为一个用户可以毫无顾忌使用您的应用的避风港。

除了用户的声誉，还有什么是利害攸关的？他们数据的价值。Twitter 和 Facebook 都声称拥有其用户创建的数据的所有权，并且可以自由地以任何方式将这些数据货币化。这是否会让用户暴露在超精准广告之下？我们是否会因为披露了真实的个人统计信息而被细分并成为营销目标？当然会，而且这两家公司已经在细分和定位他们的用户受众。但许多用户会认为，为在网络上建立真实形象所带来的好处付出这点代价是值得的。

### 教育您的用户

无论您认为对用户来说何种隐私级别是合适的，我们强烈建议在处理社交图谱时遵循两条通用原则。

向用户通知所有从社交图谱发布或获取的内容。效仿苹果的做法：每次 iOS 访问设备位置时，他们都会弹出一个提示窗口。有了这个弹出窗口，绝大多数用户完全接受他们的设备知道他们的位置。然而，如果这个过程在后台以“选择退出”的方式发生，许多用户会感到愤怒。教训是：您在隐私方面有很大的自由度，用户也愿意尝试您的应用——前提是您的应用对其如何使用用户数据以及为什么要使用用户数据完全透明。

确保用户知晓您的应用所执行操作的后果。对于熟悉计算机的用户，告诉他们发生了 `POST` 或 `GET` 事件可能就足够了。但许多用户可能不熟悉这些事件的后果。如果您的应用有任何可能泄露个人或隐私信息，请务必在应用内某处清楚说明风险。在不破坏视觉设计且不使交互变得杂乱的情况下，将此类警告或帮助文本集成到 iOS 应用中可能很困难，但本书的第 5 章可以帮助您确定何时以及何处执行此操作。

### 关于动态消息的说明

冒着啰嗦的风险，我们觉得必须提及，许多由 Facebook 和 Twitter 的 API 支持的操作会产生一些不可逆转的后果。这些风险关乎生死吗？可能不是。但一旦信息发布到社交图谱上，就极难（如果不是不可能）删除。

在 Twitter 上，推文流会被搜索引擎立即编入索引，因此即使用户删除了推文，其文本内容也可能长期存在。Facebook 的状态无法被搜索引擎索引，但它们会被推送到用户朋友的 Facebook 新闻源应用中，并且无法从其他人的新闻源中擦除，即使原始帖子被删除。请记住这一点，不要对用户的信息掉以轻心。

### 如果遇到安全漏洞该怎么办

如果您在开发应用时发现 Facebook 或 Twitter 平台存在您认为可能的安全问题，应立即向相应实体报告该缺陷。

对于 Facebook，这意味着在其错误跟踪系统中提交工单，该系统位于 [`http://bugs.developers.facebook.net`](http://bugs.developers.facebook.net)。对于更大的问题，您可以填写位于 [`http://www.facebook.com/help/contact.php?show_form=dev_support`](http://www.facebook.com/help/contact.php?show_form=dev_support) 的表格，尽管该公司表示对该表格的响应速度不如错误跟踪器快。

Twitter 有一个更细致的报告系统。该公司拥有几个不同的报告系统，根据您发现的缺陷类型进行细分。要查看您的报告选项，请查看 [`http://support.twitter.com/groups/33-report-a-violation`](http://support.twitter.com/groups/33-report-a-violation)；您可以浏览 `@support` 动态消息，查看该问题是否已被处理。

### 总结

我们认为您已经明白了：隐私很重要，而安全性更重要。要反复进行原型设计、测试和再测试。不要依赖苹果来审查您应用的安全性。使用适当版本的 OAuth，并考虑所有能想到的用例来防止漏洞。在开发的每个阶段都这样做，并且在确定安全之前，不要推出成品。并且不要忘记：一旦某些内容发布到社交图谱上，就几乎不可能删改。发布需谨慎！
