# 第 1 章

## 社交图谱能为你的应用带来什么

曾几何时，“社交”网络帮助人们与朋友联系。而如今，每个应用和网络服务都可以被视为社交性的。为什么呢？简单来说，是因为人们喜欢分享。无论是在电子游戏中发布高分，还是发布一张朋友能看到照片，iOS 用户已经习惯于向他们的朋友、家人和同事的网络展示自己的数字生活。

这个人与人组成的网络被称为*社交图谱*。一个人的社交图谱描述了他认识的每个人，以及这些人之间的连接方式。自 2007 年 Facebook 首席执行官马克·扎克伯格提出这个术语以来，社交图谱已不再局限于你认识的人。其他被添加的“节点”包括地点、活动、品牌和多媒体。所有这些都是人们相互连接的媒介。

Facebook 和 Twitter 的存在是为了记录用户的社交图谱，并推动他们建立新的连接。这两家公司都有强大的动力来扩展用户的社交图谱：了解用户的连接和偏好，使它们能够投放定向广告、提供推荐，并围绕电子商务和实体商业开展合作。

对于应用开发者而言，机遇是相似的。为 iOS 应用添加 Facebook 或 Twitter 功能，可以为盈利和新功能开辟广阔的新机遇，而且还有更多酷炫的东西等待发掘。将你的应用连接到社交图谱，可以使用户更容易登录、管理账户以及传输信息。Facebook 和 Twitter 都构建了庞大的 API 和框架，可以使开发者免于“重新发明轮子”。（例如，Facebook 甚至将其自定义的 iOS 框架开源了。）

这两项服务都拥有数亿用户，他们渴望探索。现在，既然所有这些人都投入了时间建立 Facebook 个人资料或源源不断地发布推文，其中许多人很好奇还能以其他方式使用自己的账户。向他们展示吧！

### 这本书是写给谁的？

这本书向 iOS 开发者展示如何将 Facebook 和/或 Twitter 集成到你的应用中，从而构建更安全、更灵活、更易用的应用。但这里的内容远不止技术指导。本书的章节还将深入探讨利用社交图谱所涉及的一些哲学问题。例如，它将涉及设计和品牌，以便用户在进入你的应用时，能认出他们喜爱的 Facebook 和 Twitter 功能。

#### 你需要准备什么

本书不会试图教你从头构建一个完整的 iOS 应用，因此在你开始接触 Facebook 和 Twitter API 时，最好已经构建了一个初具雏形的应用。此外，虽然我们将在信赖的老伙计 Cocoa Touch 和 Objective-C 中工作，但也会涉及许多需要 JavaScript、HTML 和 CSS 的 Web 相关技术。如果你以前有过 Web 编程经验，学习本书将要讨论的 API 会更加顺利。

#### 你应该了解什么

社交图谱关乎人。它关乎他们的内容、他们的朋友、他们的生意。你将遇到的一些交互在社交层面是复杂的——你正在处理人们之间的关系。如果你从未花太多时间使用 Facebook 或 Twitter，你就会很难理解这些关系在线上的运作方式。如果你正考虑将其中一个 API 添加到你的应用，你会发现值得花时间熟悉这些服务。做到这一点，你将对这些隐私问题（很多）、平台（并不完美）以及最重要的是，这些东西实际有何用处，有更细致入微的理解。

#### 你将学到什么

当你读完这本书时，你将知道如何构建一个能够快速、安全且谨慎地连接到世界上最流行的社交网络服务的应用。你将理解如何利用社交图谱使你的软件更有用、更有趣、更受欢迎。你还将看到平台中的弱点所在，并更好地理解 API 未来将如何演变。

但也许最关键的是，你将理解 Web 和 iOS 发展史上一个重要时刻的开端：*线上生活*与*现实生活*的融合。随着人们将现实生活中的关系、经历、兴趣和情感带入社交图谱，网络正被赋予巨大的力量。Twitter 和 Facebook 构建的存储空间越大，你的应用可用的用户数据就越多。而你越了解用户，你的程序就越有用。

### 了解社交图谱

如果你还没看过电影《社交网络》，我们帮你省点事。在电影的高潮部分，肖恩·帕克对扎克伯格说：“你甚至还不了解这个东西是什么。”他完全正确：没人知道 Facebook 是什么，或者它会变成什么。

Facebook 和 Twitter，尽管体量庞大、资金雄厚，但可能仍处于初创阶段。随着商业和社会围绕这些平台来塑造其媒体、沟通和商业，很多事情都将发生变化。如果你在这个阶段还想不到在你的应用中为 Facebook 或 Twitter 设计一个“杀手级”用例，别担心——你才看到第三页。你可能需要思考（并进行大量原型设计）才能理解如何在你的应用中最好地利用社交图谱。但这没关系，因为大家都处于同一起跑线。

为了让你的大脑开始产生好点子，我们将介绍一些你可以通过操作它们的 API 在应用中使用 Facebook 和 Twitter 的基本功能。

#### 简要用例

iOS 应用可以从 Facebook 和 Twitter API 中获取许多东西。一些非常基本的用例包括但不限于以下各节所述内容。你将在本书中学习如何完成这些章节中描述的所有操作；你还将学习如何构思更复杂的用例。



#### Facebook

以下示例说明了开发者如何在假设的应用中使用 Facebook：

- 从相机应用上传照片或视频至用户个人主页
- 从新闻应用发布内容链接至用户留言墙
- 从购物应用内部发布“赞”至用户留言墙
- 发布状态更新至用户个人主页
- 在通讯录应用中显示用户的好友列表及其头像
- 让用户从应用内部将自己设置为活动参与者
- 在应用内显示当前参与活动的其他用户
- 显示公开 Facebook 数据的搜索结果，使用户能够搜索人物、地点或内容

#### Twitter

以下示例说明了开发者如何在假设的应用中使用 Twitter：

- 从基于位置的应用发布活动链接推文
- 从照片编辑应用发布照片推文
- 向特定 Twitter 用户发送私信
- 在新闻应用中显示与某主题相关的推文
- 在通讯录应用中显示用户的关注者与被关注者列表及其个人资料
- 从 GPS 应用自动发布用户位置推文
- 围绕你的应用组织群组或社区
- 在美食指南应用中显示关于某餐厅的推文
- 公开游戏中的高分
- 搜索最新新闻或照片
- 使用趋势或“热门”话题作为输入

## 平台与服务的简要概述

Facebook 和 Twitter 都是强大的平台，但它们并不总能让你随心所欲。如果你对应用要添加的功能已有初步想法，以下是这些平台基本能力的概要。

#### Facebook

Facebook API 当前正处于持续的过渡阶段。最初的 Facebook API 是表现层状态转移（REST）API，但该 API 正被逐步淘汰，且已正式弃用。

所有未来的 Facebook 开发都应使用 Facebook 新的 Graph API。你将在 Graph API 中找到对所有新功能和未来 Facebook 特性的支持，并且它会持续更新，包含 REST API 的全部原始功能。

请注意，Graph API 仅支持返回 JavaScript 对象表示法（JSON）对象的响应。

以下是这些 API 的基本概要。

#### 读取

该 API 提供对 Facebook 图中存储的基本信息的访问。

#### 发布

该 API 使你能够向 Facebook 图中添加评论、赞等内容。

#### 搜索

该 API 允许你搜索社交图谱中的公开对象，例如所有公开帖子、人物、活动、地点等。

所有 Facebook API 均基于 HTTP，因此通过`HTTP GET`检索数据，通过`HTTP POST`提交数据。

为方便 iOS 开发者，Facebook 还提供了 iOS 的 Objective-C Facebook SDK。该 SDK 是开源的，作为 Facebook 基于 HTTP 的 Graph API 的封装层使用。本书将使用 iOS 的 Objective-C Facebook SDK，但在适当或能提供额外见解时，也会提及 HTTP API。

#### Twitter

Twitter 的 API 已演变为略显割裂的状态——其大部分为内部开发，但通过从第三方购买的代码进行了大幅增强。最终的 API 包含两个表现层状态转移（REST）API（一个核心 API 和一个搜索 API），以及一个流式 API。Twitter 的 API 支持 XML 和 JSON 格式，但在讨论技术细节和展示示例代码时，我们将使用默认的 XML 格式。以下是这些 API 的基本概要。

#### 核心 API

该 API 提供 `twitter.com` 的基本功能：推文、关注和时间线。

#### 搜索 API

该 API 提供 Twitter 的实时搜索索引，以及全局和本地趋势。

#### 流式 API

该 API 目前主要设计用于通过 HTTP 长轮询连接实现服务器到服务器的集成，并提供实时推文。Twitter 正在通过该 API 尝试服务器到客户端的集成。

所有 API 均基于 HTTP 并受到速率限制。与 Facebook 类似，Twitter 通过`HTTP GET`检索数据，通过`HTTP POST`提交数据。

请注意，Twitter 在开发每个 API 时都尽力遵循以下原则：

- 极其简单
- 直观易懂
- 自描述

### iOS 上的社交图谱

早在被称为 iPhone OS 的时代，苹果的移动平台并未为社交图谱应用提供太多支持，这些应用无法达到与桌面体验相媲美的程度。但慢慢地，苹果开始为设备赋予更多能力，为开发者提供更多工具。如今，凭借多任务处理和新睡眠模式，iOS 4 使社交应用得以进化出更深层次的功能。在此过程中，苹果用相当优雅（尽管有时有限）的解决方案解决了一些非常深刻的可用性问题。

当然，你可以用其他平台完成本书将讨论的许多功能，但效果（或外观）不会像在 iOS 上那样出色。以下是 iOS 4 带来的一些新特性：

- 多任务处理允许你的应用在后台继续运行。无论你的应用做什么，它都可以持续进行，无需用户手动激活。
- 更好的拼写检查和文本替换选项使数据输入更轻松。
- WiFi 连接在睡眠模式下具有有限的持久性，这意味着 iOS 设备在未使用时仍可继续执行与 Web 相关的操作。

**注意：** 当应用在 iOS 后台运行时，它无法在该状态下执行所有功能。出于可靠性和电池寿命的考虑，苹果选择将后台处理限制为七个特定的 API（详见第 10 章）。

iOS 4 中引入的其他变化将使社交图谱编程更加稳健。其中一些变化包括：

#### 本地通知

iOS 早已支持推送通知，但现在苹果也引入了本地通知。这些提醒不经过苹果的推送服务器，而是驻留在设备本身，在后台等待直到弹出给用户。有人通过 Skype 呼叫你的通知就是本地通知的一个例子。

#### 任务完成

当用户在退出应用时仍有任务正在进行，iOS 可以注册该线程并在后台继续运行，即使用户已转向其他操作。保持单个线程打开允许用户关闭应用的其余部分，将大部分内存释放回系统。一旦任务完成，iOS 将完全关闭该应用。

#### 快速任务切换与保存状态

在 iOS 4 之前，构建一个能在退出时保存用户进度的持久化应用非常困难。现在，所有 iOS 应用都建议保存状态。这意味着当用户返回应用时，应用的当前状态已保存在内存中，并呈现为用户离开时的样子。此功能由双击 Home 键时出现的“任务切换器”管理。这种状态保存功能在应用调用其他应用时特别有用，例如当用户从应用内部选择撰写邮件时。邮件发送后，用户发起邮件时所使用的应用将返回屏幕，保持其离开时的状态。
