# 现在我们知道了如何让飞船逐帧移动。

但我们还没有探讨如何告诉飞船*往哪里*移动。回顾一下，我们在`Example01Controlller`的根`UIView`上注册了一个`UITapGestureRecognizer`。这个`UITapGestureRecognizer`被配置为调用`viewTapped:`方法，如代码清单 5–6 所示。

**代码清单 5–6.** *Example01Controller.m (viewTapped:)*

```
-(void)viewTapped:(UIGestureRecognizer*)aGestureRecognizer{

UITapGestureRecognizer* tapRecognizer = (UITapGestureRecognizer*)aGestureRecognizer; CGPoint tapPoint = [tapRecognizer locationInView:self.view];

[viper setMoveToPoint: tapPoint];

}
```

在代码清单 5–6 中，我们在`viewTapped:`中做的第一件事是将传入的`UIGestureRecognizer`强制转换为`UITapGestureRecognizer`。然后，我们通过调用`locationInView:`找到点击在根视图中发生的位置。一旦获得了点击发生的位置，我们只需设置`viper`对象上的`moveToPoint`属性。请注意，我们无需中断任何已有的动画——只需更新`viper`对象的状态，当它的`updateLocation`方法再次被调用时，它就会开始向这个新点移动。这是一个非常简单的例子，但我希望它能说明逐帧动画相较于上一章我们看到的预定义动画的优势。

我们已经了解了这个简单动画是如何设置的，以及它是如何通过代码清单 5–4 中重复调用`updateScene`来驱动的。我们应该花点时间理解负责实现这些重复调用的类。

### 理解 `CADisplayLink` 和 `NSRunLoop`

通常，要创建流畅的动画，画面每秒至少需要更新 25 次，以避免人眼看到单独的帧。这意味着我们的游戏必须找到一种方法，以每秒 25 次的频率调用`updateScene`才能实现流畅的动画。这可以通过创建一个以任意速率调用`updateScene`的`NSTimer`来实现。然而，如果我们这样做，更新场景的代码就不一定能与硬件的屏幕原生刷新率同步。Core Animation 框架提供了一个专门用来创建我们正在尝试实现的这种动画的类：`CADisplayLink`。

在代码清单 5–3 中，我们创建了一个`CADisplayLink`，指定它应该在屏幕准备好重绘时调用`updateScene`。然后，我们通过调用`NSRunLoop`类的`currentRunLoop`方法，将这个`CADisplayLink`添加到调用`viewDidLoad`的同一个运行循环中。`NSRunLoop`类管理一个线程，以产生图 5–1 所示的那种循环。`NSRunLoop`负责处理来自用户和`CADisplayLink`的输入，并调度何时应调用`Example01Controller`中对应的方法。我们不需要过多了解`NSRunLoop`的工作原理；只需理解`viewTapped:`和`updateScene`方法将会由同一个线程调用，这样我们就不必担心多线程的复杂性了。

如前所述，`CADisplayLink`与我们主线程的`NSRunLoop`协同工作，使得每次屏幕重绘时`updateScene`方法被调用一次。`CADisplayLink`的`duration`属性以秒为单位报告每次屏幕重绘之间的时间。稍微探究一下可以发现，该属性在每次调用`updateScene`时报告的值略有不同，但大致在`0.0166648757`秒附近波动。这个值略低于每秒 60 帧，所以远高于所需的每秒 25 帧标准。

在更复杂的应用中，游戏状态完全有可能无法在 1.5 毫秒内更新完成；可能只是维护的数据太多了。如果是这种情况，你可以将`CADisplayLink`的`frameInterval`属性设置为大于 1 的值。这将导致`CADisplayLink`跳过一些屏幕重绘。图 5–6 展示了使用`CADisplayLink`时可能出现的四种不同情况。

**图 5–6.** *屏幕刷新场景*

在图 5–6 中，我们看到左侧列出了四个场景。每个场景的右侧有一些点，代表了该场景中的屏幕刷新次数。看最上面的场景，在屏幕刷新之间没有需要处理的工作，每个点等距分布，这意味着随着时间的推移，屏幕以均匀的间隔进行刷新。第二个场景展示了屏幕刷新之间有一些工作要做的情况，用每个点向右延伸的线条表示。每条线的长度小于每个点之间的距离，表明工作是在屏幕刷新的间隔时间内完成的。第三个场景展示了屏幕刷新之间需要完成的工作太多，而程序员没有考虑到这一点的情况。每次工作耗时超过屏幕刷新的正常时间间隔时，下一帧的绘制就会被延迟。

由于帧间更新游戏所需的时间会变化，屏幕刷新的速率就会变得不规律。当屏幕刷新不规律时，动画看起来就会断断续续，给用户带来糟糕的体验。为了避免这种情况，我们通过将`frameInterval`设置为`2`，配置`CADisplayLink`每隔一帧才触发一次更新。这让我们完成工作的时间翻倍，从而能够实现流畅的动画。

这种策略对某些开发者来说可能反直觉，因为在第四个场景中，可能会出现长时间没有工作在进行的情况。实际上，在给定的时间段内，场景 3 渲染的帧数比场景 4 要多。但事实是，对用户而言，断断续续的更新几乎总是比一个运行不那么快的游戏更令人厌烦。还要记住，iOS 设备的原生刷新率是每秒 60 帧，所以即使每隔一帧被跳过，动画速率仍然能达到每秒 30 帧。这足以创造出令人信服的动画效果。

我们已经了解了一个简单的逐帧动画，现在你应该理解了基本原理。接下来的两个例子将基于这种模式进行构建，创建包含屏幕上多个物体的动画，并做一些重构工作，开始搭建一个适用于复杂游戏的框架。
