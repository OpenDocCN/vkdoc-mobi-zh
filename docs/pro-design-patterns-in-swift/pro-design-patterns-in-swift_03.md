# 1. 理解设计模式  

设计模式是软件开发的"保险单"。保险的运作原理是用当前的小额投入，来避免未来可能产生的高额代价。例如，为汽车购买防盗险所支付的保费，仅占车辆价值的几个百分点，但一旦车辆被盗，你的总体损失就能降到最低。虽然车辆被盗带来的不便仍无法避免，但至少你不需要同时承受经济损失。  

在软件开发中，设计模式是为解决难题所投入的时间提供保障。这份"保费"就是你现在为代码增加额外灵活性所花费的时间，而"理赔"则是避免日后因调整应用程序工作方式而进行痛苦且漫长的重写。就像真实的保险单一样，你支付了保费却未必会受益——因为你预想的问题可能永远不会发生。但软件开发很少能一帆风顺，问题总是层出不穷，因此额外的灵活性通常是一项明智的投资。  

这是一本面向实战型专业程序员的书籍。我专注于设计模式的实际应用，重点放在代码示例而非抽象描述上。我会介绍最重要的设计模式，并演示如何通过 Swift 将其应用到 iOS 开发中。我描述的某些模式其实已经在 Cocoa 框架类中实现了，我会向你展示如何利用它们来创建更健壮、适应性更强的应用程序。  

读完本书后，你将理解当代软件开发中最核心的设计模式、它们旨在解决的问题，以及如何将这些模式应用到你的 Swift 项目中。  

## 将设计模式置于具体情境中  

每位经验丰富的程序员都有一套塑造自身编码风格的非正式策略。这些策略就像是防止以往项目问题再次发生的"保险单"。例如，如果你曾花了一周时间处理数据库 schema 在最后一刻的变更，那么在未来的项目中，你就会多花一点时间确保对 schema 的依赖没有硬编码在整个应用程序中——即使你并不确定这次 schema 是否会更改。你现在支付一点"保费"，以避免未来潜在的更大代价。你或许仍需进行调整，但整个过程会更加愉快，就像保险公司赔付了被盗车辆后，你选购新车的过程也会更舒心一样。  

非正式策略存在两个问题。第一个问题是**不一致性**。经验相似的程序员可能对问题的本质有不同看法，在最佳解决方案上也会产生分歧。  

第二个问题是，非正式策略由个人经验驱动，而这些经验往往伴随着强烈的情感。描述解决问题的难度，很难传达出你曾经历的痛苦与煎熬，这就难以说服他人投资于预防措施。同时，这也让你难以客观地看待问题的重要性。糟糕的经历会挥之不去，你可能会发现很难接受这样的现实：在采取避免既往项目问题的变更措施时，几乎得不到支持。  

### 引入设计模式  

设计模式能够识别出常见的软件开发问题，并提供应对策略。这与我之前描述的非正式方法颇为相似，但设计模式是客观、一致且不带情感包袱的表达。  

设计模式所描述的策略已被证明是有效的，这意味着你可以将自己的方法与这些策略进行比较。而且，由于它们涵盖了最常见的问题，你会发现，对于那些你未曾亲身经历的问题，设计模式也能提供解决方案。  

本书中的大多数其他设计模式都源自一本经典著作——《设计模式：可复用面向对象软件的基础》（Erich Gamma、Richard Helm、Ralph Johnson 和 John Vlissides 合著，Addison-Wesley，1995 年出版）。这四位作者被称为"四人组"（GoF），他们所描述的模式是现代软件开发中最重要、最基础的一部分。  

"四人组"的这本书值得一读，但本质上偏学术化。设计模式以抽象方式表达，不涉及特定编程语言或平台。这种抽象性使得它们难以运用：你很难判断某个模式是否描述了你关心的问题，也难以确定自己是否正确实现了解决方案。  

我在本书中的目标，是将设计模式置于具体情境中，为你提供轻松识别和应用所需模式的一切信息，并附带可直接用于项目的 Swift 实现。  

### 理解设计模式的结构  

大多数设计模式适用于应用程序中的小规模对象组，解决的是当一个对象（称为"调用组件"）需要对应用程序中一个或多个其他对象执行操作时所出现的问题。  

对于本书中的每个设计模式，我会描述该模式解决的问题，解释其工作原理，并向你展示如何使用 Swift 实现该模式。我还会介绍常见的模式变体，并指出与该模式最密切相关的陷阱。  

**统一建模语言（UML）在哪里？**  

通常会用 UML 来描述模式，但本书不会使用。我不用 UML 有几个原因。首先，大多数开发者并不能完全理解 UML，从 UML 图中获得的信息很有限。当然也有例外，这些人往往在大型企业工作，这些企业在开发开始前会有详细的分析和设计阶段。但对于业界其他人来说，UML 不过是一堆定义不清、容易被误解的方框和线条。  

我发现，UML 适合表达某些类型的关系，但在表现其他关系时却糟糕透顶。在很大程度程度上，理解模式意味着要理解代表其他组件知识的逻辑存在于何处，而这用 UML 很难传达。  

最后，一个相当主观的原因：UML 是软件开发中许多我不喜欢方面的典型症状。UML 常常被用作强制实施静态僵化设计的武器，并阻碍开发流程调整以满足不断变化的客户需求——因为 UML 会成为一个不可变更的参考点。  

基于这些尽管主观的原因，本书中我不会使用 UML。相反，我将使用自由形式的图表来说明我想强调的要点。  



## 量化设计模式的价值

人们很容易接受设计模式是件好事。每个人都理解那些在无数项目中用来解决棘手问题的成熟方案的吸引力。但要说服团队中的其他程序员在项目中采用某种特定模式则困难得多。

你可以通过问自己以下问题来评估一份保险是否物有所值：

- 该保险是否针对我可能遭遇的不利事件？
- 这种不利事件发生的频率如何？
- 保费是否只占处理该事件成本的一小部分？

这些简单的问题让我们明白，如果你没有车，或者你所在的小镇没有偷车贼，那么购买车险就没有意义。它们也突显了为价值 1.1 万美元的汽车每年支付 1 万美元保险费是多么不值，除非你预计会发生多次盗窃（如果是这样，你可能还要考虑搬到另一个地方去住）。

即使这只是对保险的简单看法，但要点很清楚：不要买一份没有好处的保险。设计模式也是如此：除非它能提供你可以量化并清晰向他人说明的价值，否则不要采用。评估设计模式价值所需的问题类似：

- 该模式是否识别出我可能遇到的问题？
- 这个问题出现的频率有多高？
- 我是否足够在意避免未来修复该问题的风险，以至于现在愿意投入工作来实现这个设计模式？

回答这些问题可能很困难。软件开发领域没有精算表，而且很难估计未来修复问题所需的工作量（尤其是那些可能根本不会出现的问题）。

相反，更简单的方法是关注设计模式能立即提供的好处。例如，那些提高应用程序模块化的模式，通常是为了最小化未来变更的影响，但一个模块化的应用程序拥有更少紧耦合的组件，这意味着更容易隔离代码单元。能够隔离代码单元对于有效的单元测试至关重要，因此，采用一种"变更保险"模式能带来立即可见的好处——提高代码的可测试性。

同样，那些增加应用程序抽象程度的设计模式，允许以更少的代码重复和更少的精力添加新功能。几乎每个人都能认同更快、更简单的开发是件好事，即使他们不认同需要避免设计模式旨在防范的问题。

然而，没有简单的答案。最终是否采用设计模式的决定，将由开发团队的综合经验、对需求规格完整性的信心以及单个开发人员的能力共同决定。

## 在问题发生后使用设计模式

如果你是一个团队中唯一提倡设计模式的人，而团队成员没有相关经验且时间紧张，你将很难推动模式的采用。你很可能无法说服他人。不要灰心。

我的建议是不要过于强硬。如果你强迫团队遵循新的实践，那么每一次由此导致的延迟和问题都会被归咎于你，尤其是当你试图防范的问题从未发生时，情况会更加困难。可悲的是，设计模式的倡导者常常被视为会预言不祥之事的预言家。

不要失去希望，但请把这本书放下，等待时机。如果你担心的问题没有发生——例如，数据库模式没有改变——那么可以庆幸项目躲过了一劫，然后继续下一个任务。

如果问题确实发生了，也不用担心；你仍然可以从设计模式中受益。你的项目现在陷入了你原本希望避免的境地，但你可以利用这些模式作为框架，帮自己摆脱困境。选择最合适的设计模式，并以此为基础构建整洁的代码，围绕它来推动问题的解决。通过这种方式，你可以利用糟糕的局面，让团队了解并采用经过验证的问题解决方案。这当然不如一开始就避免问题那么好，但至少你能创建一个长期解决方案，并为你对设计模式的热情增添一些可信度。

## 理解设计模式的局限性

设计模式有很多优点，但它们也有局限性。从本质上讲，设计模式是其他项目上其他人遇到问题的解决方案。设计模式是避免或解决问题的起点，而非精确定制的解决方案。这并不意味着它们没用，但它们确实需要一些调整才能适应你的项目。

请将设计模式视为菜谱。对模式进行修补、调整和修改，你最终会得到适合你的东西。你可能需要多次完善你的实现，并且随着从多个项目中获得洞察，你对模式的使用可能会变得更好，但你最终会得到一个比起点有所改进的方案，并有助于减轻常见问题的影响。

有些程序员将设计模式视为不可改变的法则。他们是"模式狂热者"——将模式的使用鼓吹为一种僵化的"最佳实践"，认为必须始终遵循，不能改动。这完全是胡说八道。应用你不需要的模式，或者拒绝调整模式以适应项目，完全没有抓住重点。

试图与模式狂热者争论毫无意义。他们从引用越来越晦涩的源头中获得乐趣，没有有效的方法让他们的观点接地气。我的建议是忽略他们，专注于通过构建健壮、可扩展且足够灵活以应对变化的优秀软件，而本书描述的设计模式可以帮助你实现所有这些目标。

## 关于本书

在本书中，我将描述如何使用`Swift`和`Cocoa`框架来运用最重要的设计模式。`Swift`为苹果平台吸引了许多新的开发者，因此我撰写本书是为了提供关于`Swift`和我所用到的`Cocoa`类的所有必要信息。我还将向你展示如何使用`Xcode`创建示例项目，如果你刚接触`Swift`开发世界，`Xcode`可能会是一个令人困惑的工具。如果你不想自己输入代码，可以从`Apress.com`下载所有示例。

### 你需要具备哪些知识？

你需要是一名有经验的开发者才能理解书中的概念。不需要事先了解`Swift`，但你应该理解面向对象编程的基本概念，并且使用过诸如`Objective-C`、`C#`、`Java`或`JavaScript`等现代语言。

### 你需要哪些软件？

你需要一台运行`OS X 10.10 (Yosemite)`的 Mac 电脑，并已下载安装`Xcode 6.1`。要获取`Xcode`，你需要注册成为苹果开发者，你可以在`https://developer.apple.com`上完成注册。如果你对`Xcode`不熟悉，不用担心；在第 2 章中，我会向你展示跟随示例所需的一切知识。



#### 本书的结构是怎样的？

本书分为五个部分，每个部分涵盖一系列相关主题。第一部分包含本章以及一个针对你将需要遵循的示例技巧的 Xcode 入门指南，这些技巧将帮助你在本书的示例中应用。我还构建了一个名为 SportsStore 的示例应用程序，我将在本书中反复引用它，以演示如何在具体情境中应用设计模式。

本书的其他四个部分各自专注于一种特定类型的模式。第二部分涵盖创建型设计模式，这些模式关注的是应用程序中对象的创建方式。第三部分描述结构型设计模式，这些模式定义并管理应用程序中对象之间的关系。本书的第四部分重点介绍描述对象之间如何进行通信的模式。在本书的最后一部分，我描述了模型/视图/控制器（MVC）模式，该模式适用于整个应用程序的结构，并且常用于 Mac OS 和 iOS 的 UI 应用程序。
