## 序言

我开发 iOS 应用已经有一段时间了，并在 2019 年 SwiftUI 发布后几乎立刻就开始学习它。我记得自己迫不及待地想开始使用它，因为它对我来说似乎比 UIKit 直观得多。

起初 SwiftUI 的文档并不多，我们这些直接投入其中的人不得不自己摸索很多问题。于是我开始在博客上分享使用 SwiftUI 构建应用时学到的东西。

仅仅几个月后，苹果公司的 SwiftUI 团队就联系我，邀请我加入他们一起开发这个框架。这是一段无比宝贵的经历，让我对 SwiftUI 的内部机制以及 API 背后的决策有了深刻的理解。当然，SwiftUI 并不完美，还需要时间成熟，但它已经让许多开发者更快乐、更热衷于为苹果平台构建应用。为了使 SwiftUI API 如此简洁高效，团队投入了大量思考，以便我们能够默认获得平台正确的行为。

我于 2022 年 4 月离开苹果公司，作为联合创始人和软件工程师加入了 Nil Coalescing。虽然不能再直接为 SwiftUI 框架做贡献让我有点遗憾，但我非常高兴现在能够使用自己参与开发的 API，并与其他 iOS 开发者分享我的 SwiftUI 知识。

看到苹果公司通过推出像 *WidgetKit* 和 *Swift Charts* 这样要求使用 SwiftUI 的新框架来大力押注它，我相信它潜力巨大。同样令人欣慰的是，越来越多的开发者开始使用 SwiftUI，并且来自苹果和社区的优质 SwiftUI 相关内容也越来越多。

我相信，如果你打算使用 SwiftUI，*《使用 SwiftUI 和 Combine 进行异步编程》* 将是你藏书中的宝贵补充。我喜欢它独特地专注于从 SwiftUI 应用调用异步 API，因为这通常是实际项目中的主要组成部分。鉴于 Peter Friese 作为 Firebase 开发者倡导者的工作，他经常大量实验 Firebase 的异步 API 并教授如何将它们与 SwiftUI 结合使用，因此他无疑是谈论这个话题的最佳人选。本书充满了使用 Combine 和 Swift async/await API 的实用示例，将为开发具有网络功能或本地异步数据处理功能的 SwiftUI 应用奠定坚实的基础。

## 前言

我在 2009 年开始为 iPhone OS（当时是这么叫的）开发应用。我当时刚从 iPhone 3 升级到 iPhone 3GS，非常兴奋地想要尝试新功能，并用 `Xcode` 和 `Objective-C` 构建我的第一个应用！

我之前已经练习使用过各种其他的 IDE、语言和操作系统：从 `MS-DOS` 上的 `GW-BASIC`，到 `Windows` 上的 `Visual Basic` 和 `Delphi`，再到用于企业客户的 `Java`（与 `Jbuilder`、`Websphere Application Developer` 和 `Eclipse` 一起使用，用于构建 `J2EE` 和 `Spring` 应用）。

但使用 `Xcode` 和 `Objective-C` 有点令人震惊。我之前用过的所有其他 IDE 都强大得多（例如，它们都提供重构支持），而带有诸多方括号的 `Objective-C` 在我看来有点奇怪。

回顾多年构建软件的经历，`Delphi` 是我用过最高效的开发环境之一。它将成熟的面向对象语言（`Object Pascal`）与强大的 IDE 结合在一起，允许开发者快速构建 Windows 应用的 UI。此外，“快速应用开发”这个口号也确实很吸引人。

但 `Xcode` 和 `Objective-C` 是未来的方向，所以我振作精神，继续学习如何使用它们。

光阴似箭，十年过去了——我刚刚看完 WWDC 2019，为 SwiftUI 及其易用性感到兴奋。用 SwiftUI 构建 UI 感觉非常敏捷，在许多方面让我想起了 `Delphi`。我开始思考，如果能将 SwiftUI 与一个使构建后端系统更简单的框架结合起来，那该有多棒。

当谷歌 Firebase 团队寻找一位加入其开发者关系团队，专门关注 iOS 开发者生态系统的人时，我意识到这对我的意义：将我所热爱的语言（`Swift`）、一个能让开发者快速构建出色 UI 的 UI 工具包（SwiftUI），以及一个通过免除运行和维护自有后端系统需求来提高开发者效率的 SDK（Firebase）结合在一起。借助这些工具，我可以帮助人们更快地构建更好的应用。

在加入 Firebase DevRel 团队担任开发者倡导者的第一天，我就开始构建演示，以展示 Firebase iOS SDK 的功能。我很快发现，使用 SwiftUI 编写演示比使用 UIKit 容易得多。这种易用性意味着我可以专注于我真正想教授的内容：如何使用 Firebase SDK 及其 API。

SwiftUI 声明式的 UI 构建方法，在程序化构建 UI 时，对代码结构有更明确的意见，这使得其他开发者更容易理解你的代码意图。

而且，由于采用了基于函数式响应式编程的状态管理模型，SwiftUI 和 Combine 使得构建与底层数据模型和应用的其它部分始终保持同步的 UI 变得更加容易。我相信我们都遇到过并非如此的应用，并且可以证明这意味着糟糕的用户体验。
