# 第 8 章

## 设备驱动 I/O 与中断

任何计算机系统的本质都是接受输入数据、处理数据并产生输出。系统如何接受输入数据对于实时嵌入式系统至关重要。因为它需要在确定的时间间隔内处理输入数据并输出响应。实现这一点的最准确方法是持续轮询输入源以获取数据。然而，这给 CPU 带来了沉重的负担，将系统限制为仅执行一个任务。等待输入数据的到来并对其做出反应则可以减轻此负担并释放 CPU 资源。为了能够等待和响应，我们需要硬件支持，使 CPU 能够从正在执行的处理流程中转移注意力。

硬件触发的中断会导致 CPU 停止执行流程，并跳转到处理中断的代码中的特定入口点。中断的到达是不可预测的，因此它可能在任何时间发生，甚至在处理另一个中断时也是如此。一旦中断被处理完毕，CPU 将从其中断点恢复执行。

### 中断模型

每当 I/O 硬件需要 CPU 的注意时（因为其 I/O 寄存器状态发生变化），它会触发一个中断。中断可以分为可屏蔽中断、不可屏蔽中断、处理器间中断和软件中断。

- **可屏蔽中断** – 可以通过在中断屏蔽寄存器（`IMR`）中设置特定位来忽略的硬件中断。
- **不可屏蔽中断（NMI）** – 无法忽略的硬件中断。
- **处理器间中断（IPI）** – 在多处理器环境中，由一个处理器向另一个处理器触发的中断。主要用于实现缓存一致性。在 Windows Embedded Compact 7 中，由于它支持多核处理器，这种中断变得相关。
- **软件中断** – 由某些 CPU 指令触发的异常。用作执行系统调用的陷阱，因为 CPU 会切换到内核模式。
