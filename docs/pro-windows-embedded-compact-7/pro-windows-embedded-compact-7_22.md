#### 状态寄存器

至此，你已看到熟悉硬件对于为此特定硬件开发设备驱动程序的重要性。我们将保持一致，进一步探讨上一节开始的示例。在该示例中，状态寄存器名为 `ISR`。TWSI 通过 TWSI 中断状态寄存器向处理器中断控制器发出中断信号。设备驱动程序使用 `ISR` 的位来检查 TWSI 单元和总线的状态。`ISR` 位（位 9-5）在收到两线串行接口总线上的 ACK/NAK 位（位 1）后进行更新。表 2-2 以直观方式描述了 `ISR` 状态位。

- **RWM** – 读/写模式状态位。0 表示 TWSI 接口处于主设备发送或从设备接收模式，1 表示 TWSI 接口处于主设备接收或从设备发送模式。此为只读访问位，在停止状态后由硬件自动清零。
- **ACKNAK** – ACK/NACK 状态位。0 表示 TWSI 接口在总线上接收或发送了 ACK，1 表示 TWSI 接口在总线上接收或发送了 NAK。该位用于从设备发送模式，以确定所传输的字节是否为最后一个字节。此为只读访问位，在每字节及 ACK/NAK 信息接收后更新。
- **UB** – 单元忙状态位。0 表示 TWSI 接口不忙，1 表示 TWSI 接口忙。



`IBB` - TWSI 总线忙碌状态位，其中 0 表示 TWSI 总线空闲或 TWSI 接口正在使用总线（即单元忙碌），1 表示 TWSI 总线忙碌但处理器 TWSI 接口未参与事务。此为只读访问位。

`SSD` - 从机停止检测状态位，其中 0 表示未检测到停止条件，1 表示 TWSI 接口在从机接收或从机发送模式下检测到停止条件。此为用于获取状态的读取位，写入 1 可清除该位。

`ALD` - 仲裁丢失检测状态位，用于多主机操作，其中 0 表示赢得仲裁或从未发生仲裁，1 表示 TWSI 接口丢失仲裁。此为用于获取状态的读取位，写入 1 可清除该位。

`ITE` - IDBR 发送空状态位，其中 0 表示数据字节仍在发送中，1 表示 TWSI 接口已在 TWSI 总线上完成一个数据字节的发送。当在`ICR`中使能时，会触发中断。此为用于获取状态的读取位，写入 1 可清除该位。

`IRF` - IDBR 接收满状态位，其中 0 表示`IDBR`尚未接收新数据字节或 TWSI 接口空闲，1 表示`IDBR`寄存器已从总线接收新数据字节。当在`ICR`中使能时，会触发中断。此为用于获取状态的读取位，写入 1 可清除该位。

`GCAD` - 通用呼叫地址检测状态位，其中 0 表示未收到通用呼叫地址，1 表示 TWSI 接口收到通用呼叫地址。此为用于获取状态的读取位，写入 1 可清除该位。

`SAD` - 从机地址检测状态位，其中 0 表示未检测到从机地址，1 表示 TWSI 接口检测到与通用呼叫地址或`ISAR`匹配的七位地址。当在`ICR`中使能时，会触发中断。此为用于获取状态的读取位，写入 1 可清除该位。

`BAD` - 总线错误检测状态位，其中 0 表示未检测到错误，1 表示 TWSI 接口在检测到以下任一错误条件时设置此位：

- 作为主机发送器，字节发送后在接口上未检测到 ACK。
- 作为从机接收器，TWSI 接口产生 NAK 脉冲。
- 该寄存器中的所有其他位均无意义。

同一设备的另一个状态寄存器是 TWSI 总线监视器寄存器（`IBMR`），它跟踪`SCL`和`SDA`引脚的状态。

### 数据寄存器

示例中有两个数据寄存器。一个是从机地址寄存器（`ISAR`），用于定义 TWSI 接口的七位从机地址。在从机接收模式下，当七位地址与该寄存器的值匹配时，处理器会做出响应。处理器在启用 TWSI 操作之前会写入此寄存器。另一个数据寄存器是 TWSI 数据缓冲寄存器（`IDBR`）。处理器使用 TWSI 数据缓冲寄存器从 TWSI 总线发送和接收数据。`IDBR`的一侧由设备驱动程序访问，另一侧由 TWSI 移位寄存器访问。当完整字节被接收并确认后，`IDBR`接收进入 TWSI 单元的数据。处理器内核将要从 TWSI 接口发送的数据写入`IDBR`，并将其发送到串行总线。

仅使用此寄存器的低字节，因为发送和接收的数据是字节大小。这意味着只有位 0-7 是用于读写操作的字节。

### I/O 端口寄存器

根据系统的 CPU 架构，外围设备寄存器可以是特定的 I/O 空间地址。这些地址不属于 CPU 可寻址的内存空间。这些地址只能通过特殊的机器指令来访问。例如，英特尔的 x86 架构提供了一个独立于物理内存地址空间的 I/O 地址空间。I/O 地址空间包含 2¹⁶（64KB）个独立可寻址的 8 位 I/O 端口，编号为 0 到 FFFFH。任意两个连续的 8 位端口可视为一个 16 位端口，任意四个连续的端口可视为一个 32 位端口。通过这种方式，处理器可以对 I/O 地址空间中的设备传输 8、16 或 32 位数据。但您应查阅英特尔 x86 开发者文档，了解不同版本之间的差异。

### 内存映射寄存器

另一方面，内存映射 I/O 意味着外围设备的寄存器被视为系统物理内存的一部分，并像任何其他内存地址一样被访问。但应注意，必须防止对映射用于 I/O 操作的地址空间进行缓存。防止缓存与 CPU 相关，通常 BSP 开发者会负责提供正确的方法。

此外，这些寄存器成为虚拟地址空间的一部分。由于 Windows CE 软件（无论是用户模式还是内核模式）在将 I/O 寄存器映射到内存时使用虚拟地址，因此这种做法更受青睐。而且，即使 CPU 架构提供了特定的 I/O 空间，使用内存映射方案也能显著提高具有大量寄存器组的外围设备的性能。通常，特定的 I/O 机器指令比使用内存访问指令慢。

### 访问寄存器

无论采用何种访问 I/O 寄存器的方案，设备驱动程序都必须访问这些寄存器，以便控制设备、查询其状态以及检索或发送数据。为此，我们在开始开发设备驱动程序时，必须遵循一些基本规则。以下是最重要的规则列表：



- 对寄存器使用 `volatile` 限定符 - `volatile` 类型限定符声明了一个项，其值可能在程序控制之外被更改。即使编译器认为该操作没有效果，它也会忽略优化设置，并为每个对 `volatile` 变量的赋值或引用生成代码。

- 使用 “C” 语言结构覆盖 - 这将允许通过单个指针访问所有设备寄存器。该指针通过将设备的基物理地址映射到虚拟内存地址空间来获得。

- 使用操作系统定义的宏来访问端口和寄存器。例如 `READ_PORT_UCHAR` 或 `READ_REGISTER_UCHAR`。对于 16 位和 32 位宽的端口和寄存器，也存在类似的宏。

为了更好地理解，让我们查看 `TWSI` 示例并使用它。表 2-1 告诉我们 `TWSI` 使用了哪些寄存器及其系统物理地址偏移量。如前所述，I/O 寄存器需要映射到非缓存内存。由于每种架构都提供了自己的映射非缓存内存的方法，Microsoft 提供了一个依赖于架构的 OAL 函数，用于将物理地址映射到非缓存虚拟内存：`OALPAtoUA`。要了解其实现方式，可以查看以下文件：

### I/O 设备中断

任何计算机（嵌入式系统也不例外）的基本概念是，输入某些数据，系统（这里指整个软件包：硬件、操作系统和应用程序）按照设计者的意图处理这些输入数据，并可能据此输出一些数据。获取输入数据的基本方式有两种：要么轮询 I/O 设备的输入寄存器或端口，一旦数据出现在这些端口上，就取走数据并清除这些端口以接收下一个数据；要么等待 I/O 设备发出信号，告知系统有新数据到达。这两种方式各有优缺点，但现代系统主要依赖后者。I/O 设备触发的信号使系统暂停当前操作，并获取新的输入数据，也就是说，系统被中断以接受新的数据输入。

#### 中断优先级

各种 I/O 外设对系统的重要性不同。尽管系统的实时时钟对操作系统进行时间管理至关重要，但其他设备的重要性可根据其在整个系统中的功能价值，以及可能的设备需求进行配置。例如，监控汽车发动机的 I/O 设备远比播放多媒体的 I/O 设备重要。

可编程中断控制器是一种帮助管理来自外设的中断请求的芯片。它判断哪个传入请求具有最高优先级，确定该传入请求的优先级是否高于当前正在服务的级别，并据此向 CPU 发出中断。

Windows CE 成为实时操作系统的特性之一，就是使用了嵌套中断。这允许高优先级中断中断对较低优先级中断的处理。

使用嵌套中断可以防止高优先级中断的丢失和延迟。当更高优先级的中断发生时，内核负责保存 ISR 的状态，并在高优先级 ISR 完成后恢复该状态。在大多数情况下，被抢占的 ISR 无法检测到它已被抢占。

#### 中断向量

中断向量是中断处理程序的内存地址。中断向量被放置在一个称为中断向量表的数组中。在 Windows CE 中，内核中会调用一个由硬件控制的向量函数，因此其实现依赖于 CPU 家族。向量表由内核在启动时初始化。当你安装 Platform Builder 并接受安装 Windows CE 源代码的条款时，你可能希望查看 `C:\WINCE700\PRIVATE\WINCEOS\COREOS\CORE\DLL\exdsptch.c` 中的代码，该文件实现了分发中断向量的函数。

#### 信号机制

硬件设计者设计了两种产生中断的方案。一种是中断信号线上的状态从 0 变为 1，或从 1 变为 0。这被称为上升沿检测或下降沿检测触发。这种方案存在一个问题：它对电气噪声敏感。即使没有新数据需要产生中断，一个随机的尖峰信号也可能触发中断。另一种方案是电平触发，即设备将中断线保持在一个稳定的电平上，直到中断服务程序解除该中断。这种方案对电气噪声的抵抗力更强。

在本节中，我们只是浅尝辄止地介绍了硬件处理中断的方式以及它与 Windows CE 的关系。为了理解你所使用的处理器如何处理中断，你需要阅读与中断控制器相关的文档，例如《系统与定时器配置开发人员手册》的第 9 章，该手册是 Marvell PXA3xx 处理器系列开发人员文档的第一卷。

### I/O 设备内存

设备驱动程序必须在外部设备和系统内存之间移动数据。硬件采用三种机制来执行内存传输：

- 程序化 I/O (PIO)
- 设备专用内存
- 直接内存访问 – DMA

设备使用的内存传输机制取决于它需要移动的数据量大小、设备速度以及它可能所属设备类别的标准。

#### 程序化 I/O (PIO)

程序化输入/输出是一种在 CPU 和外设（如 ATA 存储设备）之间传输数据的方法。这种机制基本上是利用 CPU 来寻址内存并移动数据。

PIO 设备通常在每次数据传输后产生一次中断。一些 PIO 硬件设备具有内部缓冲区或 FIFO 来减少中断频率。即便如此，大量的传输需要大量的 CPU 活动，并且仍然会产生大量的中断，从而导致系统性能下降。

> **注意** 典型的 PIO：例如，如果你的设备使用 I/O 端口，那么它默认使用 PIO。但是，如果你使用内存操作指令从内存映射的数据寄存器中读取或写入数据，这同样被视为 PIO。

#### 设备专用内存

有些设备需要物理内存中的专用地址范围。其原因可能是该设备使用一块内存作为数据传输的缓冲区，例如视频采集设备。另一个原因是该设备可能有一个内部 ROM，其中保存了某些启动代码，为了让 CPU 执行它，它必须是内存地址空间的一部分。

#### 直接内存访问 – DMA

直接内存访问，简称 DMA，是一种无需 CPU 干预即可在 I/O 设备和内存之间移动数据的机制。DMA 有两种模式：系统 DMA 和更现代的**总线主控 DMA**。DMA 提供高性能的数据传输。有几种 DMA 传输类型。

- **公共缓冲区 DMA** - 在公共缓冲区 DMA 中，驱动程序和系统共享同一块内存区域用于数据传输。驱动程序负责同步对缓冲区的访问。该内存不被缓存，这使得驱动程序更容易进行同步。设置好公共缓冲区后，驱动程序和硬件都可以直接写入缓冲区中的地址。

- **数据包 DMA** - 当存在一个单一的现有缓冲区，必须将其映射供硬件使用时执行。例如，在将文件从内存传输到磁盘时使用数据包 DMA。为此操作使用公共缓冲区 DMA 将是一种浪费，因为文件必须先传输到公共缓冲区，然后硬件才能将其传输到磁盘。



#### 分散/聚集 DMA

`分散/聚集 DMA`是一种快捷方法，可以一次设置多个数据包`DMA`传输。例如，如果你要通过网络传输一个数据包，网络协议栈的每一层都会添加自己的头部（`TCP`、`IP`、`以太网`等等）。这些头部都从内存中的不同位置分配。

#### 系统 DMA

`系统 DMA`是通过对系统主板上的系统`DMA 控制器`（`DMAC`）进行编程来直接执行传输的。`DMAC`充当一个非常简单的处理器，能够在**外设**和主内存之间移动指定数量的字节。在`I/O`操作开始时，设备驱动会将内存地址和传输计数加载到`DMAC`中，其余工作由`DMAC`完成。当`DMAC`完成数据移动后，它会生成一个中断来通知完成。在`DMAC`传输数据期间，驱动被挂起，`CPU`可以自由处理其他任务。这是`ISA`总线上典型的`DMA`操作。现代外设大多使用`总线主控 DMA`。

##### 总线主控 DMA

一个内置了自身`DMAC`的`I/O`设备可以接管数据总线并启动`DMA`操作。完全的总线主控能力意味着该`I/O`设备能够在无需`CPU`干预的情况下执行更复杂的操作序列。`总线主控 DMA`设备的驱动可以使用以下类型的`DMA`支持：

- `基于数据包的 DMA`：如果总线主控适配器允许驱动确定`DMA`传输操作何时完成和/或何时开始另一个传输操作。`基于数据包的 DMA`是`分散/聚集 DMA`操作的基础。
- `公共缓冲区 DMA`：如果总线主控适配器没有提供方式让驱动确定传输操作何时开始或完成，或者如果单个缓冲区区域被持续或重复用于`DMA`传输。
