# 新特性：移动开发技术

在本书中，你将探索移动开发中常见的 C# 技术与模式。其中一些与我们习惯的传统 .NET 开发方法完全相同，一些则略有变化，而少数则发生了较大的改变。以下是关键主题：

- 移动 UI 是 C# 移动开发中最大的一块全新学习领域。`Xamarin.Forms` 提供了一个跨平台 UI 工具包，包含现成的表单、页面、布局、视图和控件。`Xamarin.iOS` 和 `Xamarin.Android` 则提供了各自原生 UI 的绑定。（参见第 2 章至第 6 章以及第 8 章。）
- 移动应用中的数据访问层通常将控件和页面绑定到数据模型上，这些数据模型由本地数据库填充，并通过 Web 服务与远程数据服务器同步。（参见第 7 章。）
- 通过 SQLite 访问本地数据库与通常的数据库供应商有所不同，不过 ADO.NET 访问提供了一条熟悉的途径，而 `SQLite.NET` 组件则是一个功能丰富的选择。（参见第 7 章。）
- 数据绑定是 `Xamarin.Forms` 开发的核心，通常通过模型-视图-视图模型（MVVM）模式来实现。（参见第 7 章。）
- 跨平台架构是一系列代码共享策略的集合，旨在进一步实现我们“编写一次，随处运行”的目标。这些策略包括可移植类库结合依赖注入、共享文件和项目，以及条件编译。（参见第 9 章。）

让我们逐一探讨这些主题。

## 移动 UI

从 Web 开发转向跨平台移动开发时，你需要吸收大量新知识，主要集中在用户界面领域。首先面临的是新操作系统 UI API 的问题。Xamarin 通过提供针对主流操作系统的平台特定 C# 绑定（即 `Xamarin.Android` 和 `Xamarin.iOS`）来帮助解决这一问题，而 `Xamarin.Forms` 则提供了针对这两个平台以及 Windows Phone 的跨平台绑定。另一组挑战则涉及 Web 应用与移动应用之间的设计差异。紧凑的屏幕、触控灵敏度以及手持设备的外形因素共同造就了一种全新的用户体验。这需要采用根本性的新设计和开发方法，促使我们探索移动 UI 设计。

### Xamarin.Forms 与平台特定 UI

使用 C# 进行移动 UI 开发主要有两种方法，我们可以分开、互换或结合使用：

- `Xamarin.Forms` 是一个面向 Android、iOS 和 Windows Phone 的跨平台 UI 工具包。
- 平台特定 UI 则使用 `Xamarin.Android`、`Xamarin.iOS` 和 Windows Phone SDK。

`Xamarin.Forms` 包含一个完全跨平台的工具包，提供了一套统一的 UI 控件、布局和页面，这些元素能巧妙地映射到 iOS、Android 和 Windows Phone 上各自的原生 UI 绑定。由于 `Xamarin.Forms` 比平台特定库更新，其功能也相对较少。每个版本都让我们更接近功能完备的跨平台完美体验，但有时我们需要超越开箱即用的 `Xamarin.Forms` 类所提供的功能。在这些情况下，我们使用平台特定库，要么针对整个页面，要么仅针对页面的某一部分，通过使用 `Xamarin.Forms` 中名为 `PageRenderers` 的自定义渲染器来实现。

平台特定的方法更为古老且更成熟，因此也更为详细且功能丰富。这涉及直接绑定到平台特定 UI API 的库：`Xamarin.Android` 用于 Android，`Xamarin.iOS` 用于 iOS。对于 Windows Phone，我们使用 Windows Phone SDK，这是一个原生 API，无需 Xamarin 绑定。这些平台特定的库使我们能够深入访问原生 UI，以提供视觉惊艳、交互丰富的用户体验。但这也是有代价的：平台特定的代码需要为每个平台设立独立的 UI 项目，且代码重用性极低。

注意

`Xamarin.Forms` 是本书的重点，并由使用平台特定 UI 的自定义渲染器加以补充。然而，使用 `Xamarin.iOS` 和 `Xamarin.Android` 而不使用 `Xamarin.Forms` 来创建平台特定 UI 项目的开发者，可以充分利用第 2 章至第 7 章中关于 iOS 和 Android 的章节。如果你采用平台特定的方法，请务必查阅其他资料（例如 Xamarin 在线文档）以了解 `Xamarin.Android` 和 `Xamarin.iOS` 的解决方案设置及基础知识。

### 移动 UI 设计

UI 技术是大多数移动软件开发的核心。小型设备当前的硬件限制促使我们将繁重的计算任务交由 Web 服务远端的 PC 和服务器处理。在移动业务应用中运行的大多数组件都是为了支持可见的用户界面。移动业务层和数据访问层通常是其服务器端对应部分的简化版本。这意味着我们最常需要的是 UI 组件，用于帮助我们使用布局设计屏幕、实现数据输入和选择的控件、构建用于数据显示和编辑的列表和表格、创建用户导航，以及使用图像作为背景和图标。本书中的 UI 主题涵盖了移动应用开发中最常用的功能。在每一章中，我们都从最简单、最跨平台的方法开始，然后深入探讨平台特定的细节和粒度。以下是本书涵盖的移动 UI 主题：

- 屏幕、视图或页面类似于 C# 中 Web 和桌面的对应部分，使用带有方法和属性的控件，并触发我们在控制器中处理的事件。（参见第 2 章。）
- 布局帮助我们组织控件的位置和格式，使我们能够构建和设计移动应用的屏幕。（参见第 3 章。）
- 控件促进用户交互和数据输入，这是移动用户界面所特有的，并且与 PC/鼠标界面有显著不同，这很大程度上归因于手势的使用。（参见第 4 章。）
- 列表是移动应用中最强大的数据显示和选择方法之一。（参见第 5 章。）
- 导航让用户能够遍历应用、在屏幕之间切换以及访问功能。分层导航、模态屏幕、导航抽屉、警报、下钻列表和其他关键模式构成了移动 UI 导航的核心。（参见第 6 章。）
- 状态管理是在用户浏览应用时，处理屏幕之间传递的数据。（参见第 6 章。）
- 图像在移动体验中至关重要，广泛应用于菜单、列表、网格、轮播图和其他布局中。（参见第 2 章、第 4 章、第 6 章。）



#### Xamarin.Forms 自定义渲染器

`自定义渲染器`使我们能够超越 Xamarin.Forms 开箱即用的 UI 控件，在保持跨平台方法的同时，利用特定于平台的 UI 功能集。Xamarin.Forms 应用本质上是跨平台的，使用相同的代码库在三个主要平台上运行。这对于基本设计和使用某些控件来说效果很好。然而，许多项目会需要更深入地处理 UI，例如单个控件上的设计细微差别、原生模态对话框、页面上的额外图形或动画，或者任何超出当前版本 Xamarin.Forms 所提供范围的需求。这可以通过子类化原生控件并实现 `PageRenderers` 来创建自定义控件来实现，这些控件允许使用 Xamarin.iOS 和 Xamarin.Android 完全访问特定于平台的 UI 功能。这些特定于平台的控件可以在 Xamarin.Forms 页面中使用，以帮助维护跨平台架构。

### 数据访问层

移动端数据访问层不同于我们在 Web 应用中所习惯的设计，更接近于桌面应用中的设计。其方法从流行的 MVVM 模式到模型-视图-控制器 (MVC)，再到基本的 CRUD（创建/插入/更新/删除），复杂度不一而足。数据绑定页面通常将数据输入设备上的本地数据库，该数据库通过 Web 服务与远程数据服务器同步。

C# 移动开发中的 Web 服务是代码复用的基础方面。对于移动应用，许多这些服务模式与我们构建 Web 应用时所习惯的保持一致。然而，移动 Web 服务更接近于桌面应用中的服务，与 Web 应用服务的主要区别在于数据同步和离线使用的重要性。创建、更新和删除接口在线公开，供来自多种平台和设备的 RESTful 调用使用。移动 Web 服务模式是跨平台、一次编写代码的绝佳之地。

与我们在 Web 应用中所习惯的相比，移动应用的 Web 服务端组件保持不变——除了增加了与本地移动数据存储的数据同步功能，以支持在线和离线使用。离线使用需要在本地数据库中保留一份基本数据集，并在有连接时进行同步。并非所有应用都支持离线使用。

回顾我们在旧版相关技术中使用数据访问层的经验，我们将探讨 C# 移动应用中数据访问层的架构选项。

### 使用 SQLite 进行本地数据访问

SQLite 被誉为移动开发中最稳定、最可靠的跨平台数据库产品，是一个在 iOS、Android 和 Windows 设备上运行的开源项目。Xamarin 推荐它而不是其他许多第三方项目，并且它是本书涵盖的内容。Xamarin 在开发环境中提供对 SQLite 数据库的访问和创建，并为 ADO.NET 提供对 SQLite 的支持。还有一个 SQLite.NET Xamarin 组件，它是一个围绕基于 C 的 SQLite 数据层的 C# 包装器，提供对 SQLite 数据库的低级访问，包括异步事务。所有这些使得连接数据库、创建和索引表以及读写行变得简单且无痛。

### 数据绑定

使用 Xamarin.Forms 时，数据绑定是跨平台且一致的。它模仿了 Windows Presentation Foundation (WPF) 中的数据绑定，MVVM 模式是其实现的核心。在代码中，我们将控件字段绑定到数据模型，绑定机制是自动的。手动实现 `PropertyChanged` 事件允许你的代码与数据源保持同步。绑定可以在代码中或可扩展应用程序标记语言 XAML 中完成，并且可以是单向或双向的。控件、列表和文本被绑定到数据源或彼此属性上。越来越多的第三方供应商提供 Xamarin.Forms 控件套件，其中包括数据绑定的图表和网格，例如 Telerik、Infragistics、Syncfusion 和 DevExpress。

数据绑定并未内置于 Xamarin.Android 和 Xamarin.iOS 中。特定于平台的数据绑定实现通常使用开源第三方库（如 MvvmCross 和 MVVM Light）来实现。

### 跨平台开发

正如 .NET 提供了跨越众多操作系统和语言的统一基础设施一样，Xamarin 弥合了移动操作系统及其各自开发语言之间的差距：iOS 和 Objective-C、Android 和 Java、Windows Phone 和 Windows 平板以及 C#。通过这种方式，Xamarin 将 .NET 扩展到移动领域，远远超出了 Windows 操作系统。除了这非常酷之外，真正的价值在于有机会在项目和平台之间共享和复用代码。Xamarin 工具的最大优势在于跨平台代码；因此，采用跨平台方法进行移动模式开发将产生最高的效益。Xamarin 工具为我们提供了一窥“一次编写，随处部署”这一独角兽级目标的方法。

在我们追求跨平台实现的过程中，最大的敌人是特定于平台的代码。这些代码必须根据平台（无论是 iOS、Android 还是 Windows Phone）进行不同的实现。跨平台模式与操作系统无关。跨平台代码有时被称为共享代码或核心代码，因为它是在不同移动操作系统的项目之间共享的。

Xamarin.Forms 解决了最棘手的跨平台挑战：用户界面。使用 Xamarin 的开发人员拥有一个完全跨平台的数据解决方案，即使用 ADO.NET 或 SQLite.NET 与 SQLite 进行本地数据访问，然后再通过 Web 服务进行访问。即便如此，始终会存在特定于平台的代码，如下所示：

* 在 UI 中
* 设备特定功能，例如相机和位置
* 图形和动画
* 安全性、文件和设备权限

一旦我们识别了特定于平台的代码和跨平台代码，问题就变成了如何将它们组织到一个跨平台架构中。

我们有很多选择，从 PCL、共享项目、链接文件、接口、抽象到条件编译。PCL 提供了一种方法，可以使用有限制的、针对特定平台的 .NET 库子集来构建 C# 组件，并将其编译为动态链接库 (DLL)，该 DLL 可用于由 PCL 配置文件指定的任何平台的 Xamarin 项目中。数据访问层、客户端 Web 服务以及独立于平台的业务逻辑可以在此很好地共存。通过使用带有接口的依赖注入，仍然可以将特定于平台的功能引入这些库中。一种更松散、更灵活的方法是使用共享文件或项目，其中包含为每个平台重新编译的核心文件。条件编译是一种古老的技术，非常适合较小的特定于平台的自定义，它允许共享文件中的代码块仅在特定平台的编译中包含进来。

我们将深入研究这些技术及其相关模式，因为它们带有独角兽的印记，有助于我们最大化代码的跨平台覆盖范围。
