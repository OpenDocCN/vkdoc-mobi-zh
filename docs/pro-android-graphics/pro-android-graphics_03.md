# 2. Android 数字视频：格式、概念与优化

### 摘要

在本第二章中，我们将更深入地探讨数字视频在 Android 操作系统内部的实现方式。我们将基于你在前一章学到的概念进行构建，因为数字图像的所有特性同样适用于数字视频。

毕竟，数字视频实际上只是一系列动态的数字图像。这种运动的特性为数字视频带来了高度的复杂性，因为数学方程中增加了一个第四维度（时间）。这使得处理数字视频比处理数字图像要复杂一个数量级，尤其是对于你即将学习的编解码器而言。

这一点在视频压缩方面尤为突出，因此本章将更加侧重于视频编解码器，并介绍如何以一种新的媒体数据文件格式（该格式传统上且至今仍是一个庞大的、以千兆字节计的数据实体）来获取尽可能小的数据体积。

我们将审视当前 Android 支持的各类数字视频文件格式，以及允许在屏幕上播放视频的 `MediaPlayer` 类。你将全面了解 `VideoView` 用户界面小部件，Android 创建它是为了在一个预构建的 UI 容器中实现 `MediaPlayer`，方便我们使用。

你将学习理解本章后续操作所必需的基本数字视频概念。我们将从数字视频资源的数据体积（应用大小）以及 Android 设备类型（市场覆盖范围）两个角度，来审视数字视频优化。在本章中，你将创建一个 `pro.android.graphics` Java 包和一个 `GraphicsDesign` Android 应用程序。



### Android 数字视频格式：MPEG4 H.264 与 WebM VP8

Android 支持与 HTML5 原生支持的两种相同开源数字视频格式，即 MPEG-4（动态图像专家组）H.264 以及 ON2 VP8 格式（该格式由谷歌从 ON2 Technologies 收购并更名为 WebM）。WebM 随后作为开源数字视频文件格式发布，现已被集成至 Android 操作系统及所有浏览器中。

从内容制作的角度来看，这非常便捷，因为开发者制作并优化的视频内容既可用于浏览器及基于 HTML5 的操作系统等 HTML5 引擎，也可用于 Android 应用。

这种开源数字视频格式的跨平台支持场景将为内容开发者带来“一次制作，随处交付”的内容生产局面。只要专业视频图形开发者充分利用这些规模经济效应，就能降低内容开发成本并增加开发者收入。

正如我在第 1 章中所做的那样，我将按从旧到新的顺序介绍 MPEG 和 WebM 视频文件格式。Android 支持的最旧格式是 MPEG H.263，该格式仅应用于较低分辨率，因为它采用最古老的技术，压缩质量比最差。

由于如今大多数 Android 设备都采用中等（854x480）到高（1280x720）分辨率的屏幕，如果你打算使用 MPEG 文件格式，则应采用 MPEG-4 H.264 格式，这是目前全球使用最广泛的数字视频文件格式。

MPEG-4 H.264 格式被商业广播公司、HTML5 网页浏览器软件、移动 HTML5 应用以及 Android 操作系统所采用。毫不夸张地说，所有这些应用领域加在一起正迅速接近多数市场份额的地位。

MPEG-4 H.264 AVC（高级视频编码）数字视频文件格式在所有 Android 操作系统版本中均受支持，可用于视频播放；在 Android 3.0 及更高版本中则支持视频录制。只有当 Android 设备具备摄像硬件功能时，才支持视频录制。

此外，还有一种 MPEG4 SP（简单画质）视频格式，用于支持商业视频文件的播放。该格式适用于所有 Android 操作系统版本，可播放广播类内容（如电影、电视节目、迷你剧、电视剧、健身视频及类似产品）。

然而，如果你是个人 Android 内容制作人，会发现 MPEG-4 H.264 AVC 格式具有远胜一筹的压缩能力，尤其是当你使用更高级（更好）的编码套件时，例如 Sorenson Squeeze Pro 9 软件（本章稍后你将用到它）。

如果你决定在 Android 应用中使用 MPEG-4 数字视频格式而非谷歌的 WebM (VP8) 视频格式，那么 MPEG-4 H.264 AVC 将是你制作 Android 视频内容时采用的格式。之所以支持 MPEG4 SP 格式，很可能是因为许多商业视频和电影最初都采用这种较旧的视频格式压缩，而如今在用户 Android 设备上播放这些商业视频内容已成为这些设备的常见用途——这是因为互动内容相对匮乏。但你将会改变这一现状，对吧？！太棒了！

MPEG-4 视频文件支持的文件扩展名包括 `.3GP`（SP）和 `.MP4`（AVC）。我个人倾向于使用后者（`.MP4` AVC），因为这也是我在 HTML5 应用中使用的格式，并且采用更优秀的 AVC 格式更为普遍，但两种文件（扩展名）类型在 Android 操作系统中都应能正常工作。

Android 现在支持的更现代（更先进）的数字视频格式称为 WebM 或 VP8 数字视频格式，这种格式能以更小的数据占用提供更高质量的结果。这很可能就是谷歌收购 ON2（开发了 VP8 编解码器的公司）的原因。本章稍后你将全面了解编解码器的相关知识。

从 Android 2.3.3 版本开始，系统对 WebM 视频播放提供了“原生”支持，因此市面上大多数 Android 设备（包括 Amazon Kindle Fire HD 及初代 Kindle Fire）都应能支持这种更高质量的数字视频文件格式。这是因为该格式已原生集成到智能手机或平板电脑上安装的操作系统（OS）软件中。

WebM 还支持视频流式传输，你将在本章后续部分学习相关内容。如果用户拥有 Android 4.0 或更高版本的操作系统，则支持 WebM 视频格式的流式播放功能。出于这个原因，我建议对于非流式视频素材使用 MPEG-4 H.264 AVC，而对于流式视频则采用 WebM。我们将在本书最后一节介绍流式传输等高级视频概念。



### Android 中的 VideoView 与 MediaPlayer 类：视频播放器

在 Android 中，有两个主要的类直接继承自 `java.lang.Object` 超类，专门处理数字视频格式的播放。它们分别是来自 `android.widget` 包的 `VideoView` 控件类，以及来自 `android.media` 包的 `MediaPlayer` 媒体类。

大多数基本的数字视频播放功能都可以通过使用 `<VideoView>` XML 标签及其参数来实现，并将其直接设计到用户界面中，正如本章后面你将进行的操作一样。

Android 的 `VideoView` 类直接继承自 Android 的 `SurfaceView` 类，后者是一个显示类，它提供了一个嵌入在专用 Android `View` 层次结构中的专用绘图表面，用于实现高级的、直接渲染到屏幕的图形绘制管线。

`SurfaceView` 是按照 Z 轴顺序排列的，因此它位于承载 `SurfaceView` 的 `View` 窗口之后，从而 `SurfaceView` 通过其 `View` 窗口切割出这个视口，以允许内容显示给用户。

Z 轴顺序是我们将在本书后面深入探讨的一个概念，届时你将大量处理图层合成。简而言之，z 轴顺序是给定图像或视频源在图层堆栈中的排列顺序。

图层在概念上是从顶层到底层排列的。这就是为什么图像（或视频）资源在此图层层次结构中的顺序被称为 z 轴顺序，因为包含 x 和 y 二维图像数据的图层是沿着 3D z 轴堆叠的。本来打算让你在本书后面再学习 3D z 轴和 z 轴顺序；但现在看来我好像已经解释过了！

正如你现在所知，2D 图像和视频仅使用 x 和 y 轴来“寻址”其数据，因此你需要从 3D 的角度来审视 z 轴顺序和合成，因为合成中的图层需要沿着第三个 z 轴存在。

`SurfaceView` 类是 `View` 类的子类，而 `View` 类，正如你所知，又是 `Java Object` 主类的子类。因此，`VideoView` 子类是 `SurfaceView` 子类的一个专门化体现，它在 `SurfaceView` 类方法的基础上增加了更多方法，因此在 `View` 类层次结构中处于更下层的位置。关于这个 Android `VideoView` 及其方法和构造函数的更多详细信息，请访问以下网站 URL：

[`http://developer.android.com/reference/android/widget/VideoView.html`](http://developer.android.com/reference/android/widget/VideoView.html)

Android 的 `VideoView` 类实现了 `android.widget` 包中的一个名为 `MediaController.MediaPlayerControl` 的 Java 接口。这样，使用 `VideoView` 控件的开发者就可以访问 Android `MediaController` 类中与 `MediaPlayerControl` 相关的数字视频播放方法。

如果你打算自己编写数字视频播放引擎，那么你应该直接访问 Android 的 `MediaPlayer` 类本身。在这种场景下，你本质上是在编写自己的（自定义）视频播放引擎（`MediaPlayer` 的子类），以用你自己（更高级的）数字视频播放功能来替代对 `VideoView` 控件的使用。

关于这个 Android `MediaPlayer` 类如何工作，以及它的状态引擎图，更多信息请访问以下开发者网站 URL：

[`http://developer.android.com/reference/android/media/MediaPlayer.html`](http://developer.android.com/reference/android/media/MediaPlayer.html)

我可以写一整本高级编程书来讲解如何使用这个 Android `MediaPlayer` 类来编写一个视频播放引擎。然而，既然这本中级书籍专门涵盖了 Android 图形设计以及图像、动画、数字视频、合成、混合等之间的相互关系，我们将使用 `VideoView` 类。实际上，这也是 Android 建议我们使用的，这也是它提供这个类的原因。

### 数字视频的基础：运动、帧与帧率

数字视频是数字图像向第四维度（时间）的延伸。数字视频实际上是一系列随时间快速显示的数字图像集合，很像旧时的翻页书，你可以快速翻动书页，使画在书页上的角色或场景动画起来，从而产生运动。数字视频序列中的每一张图像都称为一帧。这个术语可能源于电影时代，那时胶片帧以每秒 24 帧的速度通过电影放映机，从而创造出运动的错觉。

由于数字视频的每一帧都包含数字图像，你在第一章中学到的所有概念也同样适用于视频。相关概念包括像素、分辨率、宽高比、色深、Alpha 通道、像素混合、图像合成，甚至数据占用优化。所有这些都可以同样应用于你的数字视频内容开发工作，并在你的图形设计中实现。

由于数字视频由这些数字图像帧集合组成，因此数字视频帧率的概念（以帧每秒表示，或通常称为 FPS）在你进行数字视频数据占用优化工作流程时也变得非常重要。

数字视频中帧的优化概念与图像中像素的优化概念（数字图像的分辨率）非常相似，因为视频帧中使用的每一帧都会成倍增加数据占用。在数字视频中，不仅帧的（图像）分辨率会极大地影响文件大小，每秒的帧数（即帧率）也同样如此。

在第一章中，你看到如果我们将图像中的像素数乘以颜色通道数，就可以得到该图像的原始数据占用。对于数字视频，我们现在必须将这个数字再乘以数字视频运行的每秒帧数，以及数字视频的总秒数。

继续第一章中的 VGA 示例，我们知道一幅 24 位的 VGA 原始图像恰好是 900KB。这使得在本例中将其提升到下一个层次的计算变得简单。数字视频传统上以 30 FPS 运行，因此一秒钟的标准清晰度（SD）或 VGA 原始未压缩数字视频包含 30 个图像帧，每帧 900K，产生 27000KB 的数据占用。

要找出这相当于多少兆字节（MB），我们需要将 27000 除以 1024，得到一秒钟原始数字视频的数据量为 26.3671875MB。让我们将其乘以 60 秒，得到每分钟 24 位 VGA 数字视频的原始数据量为 1582.0313 兆字节。再将其除以 1024，我们就得到了每分钟的数据千兆字节（GB）数，等于 1.54495。

你觉得 VGA 分辨率的视频原始数据量很大，超过 1.5GB/分钟吗？高清（HD）视频分辨率为 1920x1080，乘以 3 个 RGB 通道，乘以 30 帧，再乘以 60 秒，结果是每分钟 10.43GB！

你可以看出，拥有一种能够压缩数字视频产生的这种巨大原始数据占用的视频文件格式是多么重要！这就是为什么 Google 收购了 ON2，以获取他们的 VP8 视频编解码器和创造它的人！VP8 在保持高水平视频图像质量的同时，还将文件大小降低了数量级或更多。如果你想知道，一个数量级相当于十倍（10 倍）。



### 数字视频规范：比特率、流、标清与高清

一旦你确切知道如何通过为数字视频内容使用正确的比特率、帧率和帧分辨率来最佳优化数字视频压缩工作流程，你将会对你使用 MPEG 视频文件格式所实现的一些数字视频数据压缩比感到惊讶（本章后面会介绍）。我们将在第 18 章中更深入地探讨 WebM 视频优化，届时我会用一整章的篇幅来介绍高级数字视频数据占用空间优化技术和工作流程。

既然上一节我们刚讨论完分辨率，那么我们就从商用视频中使用的主要分辨率开始讲起。在高清（HD）出现之前，视频被称为标清（SD），它使用标准的像素高度（垂直分辨率）即 480 像素。

商用或广播视频通常采用两种宽高比之一：老式显像管电视使用的 4:3，以及新型高清电视使用的 16:9。让我们用你在第 1 章学到的数学知识来计算 4:3 SD 广播视频的像素分辨率。

480 除以 3 等于 160，乘以 4 等于 640，所以 4:3 的 SD 视频就是 VGA 分辨率，这就是我在前一章用这个特定分辨率作为例子的原因。还有另一种 16:9 的宽屏 SD 分辨率，它已成为安卓触摸屏智能手机以及被称为迷你平板的小尺寸平板电脑的入门级屏幕分辨率。

让我们来计算宽屏 SD 分辨率是多少：这次用 480 除以 9，得到 53.4，再乘以 16 得到 854，所以新型宽屏 SD 智能手机和平板电脑采用了 854x480 的“宽屏 SD”像素屏幕尺寸。这很大程度上是为了让现在已经主流的 HD 内容能够直接从 16:9 的 1920x1080 向下采样到 1280x720，或者到 854x480。

高清视频有两种分辨率：1280x720，我称之为伪高清，以及 1920x1080，业界称之为真高清。两者都是 16:9 宽高比，现在不仅用于电视和智能电视，还用于智能手机（Razor HD 是 1280x720）和平板电脑（Kindle Fire HD 是 1920x1200，这是不那么宽或者说更长的 16:10 宽高比）。

还有 16:10 的伪高清分辨率，其像素为 1280x800。实际上，这是笔记本电脑、上网本和中等尺寸平板电脑的常见分辨率。通常，大多数开发者会尽量让他们的视频内容分辨率与每个观看视频的安卓设备的分辨率相匹配。

无论你为数字视频内容使用何种分辨率，你的应用程序可以通过几种不同的方式访问视频。由于我是个数据优化狂人，我采用的方式是将视频内置于应用程序中；也就是说，放在安卓应用 `APK` 文件内部的 `raw` 数据资源文件夹里。我们稍后会对此进行探讨。

在安卓应用中访问视频的另一种方式是使用远程视频数据服务器；在这种情况下，视频会通过互联网从远程服务器实时流式传输到用户的安卓设备上进行播放。希望你的服务器不会崩溃！

视频流比播放内置视频数据文件更复杂，因为安卓设备需要与远程数据服务器实时通信，并在视频播放的同时接收视频数据包。安卓 4.0 及更高版本的设备通过 WebM 格式支持视频流播放。本书直到第 19 章才会涉及流媒体视频，因为它需要远程视频服务器；在此之前，你将使用内置视频数据，以便能够测试你的应用！

在本概述中，我们需要介绍的最后一个概念是比特率。比特率是视频压缩过程中使用的一个关键设置，因为它代表了目标带宽数据管道的大小，该管道每秒能够容纳一定数量的比特流通过。

当今互联网 2.0（移动设备电信网络）`IP` 基础设施中的慢速数据管道为 768 kbits/s 或 768 KBPS，为了适配这种“较窄”的数据管道而压缩的视频文件质量水平会较低，并且可能还需要更低的分辨率来进一步减少其总数据占用空间。较旧的 3G 网络就具有这种较慢的视频数据传输速度。

需要注意的是，一个过饱和（拥挤）的数据管道可能会将快速数据管道变成中速甚至慢速数据管道，因为在高峰使用时段，越来越多的用户试图通过该数据管道访问数据。

当今互联网 2.0 `IP` 基础设施中的中等数据管道为 1536 kbits/秒（1.5 MBPS，即兆比特每秒）。请注意，我们这里使用的是比特，而不是字节，所以要计算这代表每秒多少字节，需要除以八（一个字节有八个比特）。因此，一兆比特每秒等于每秒传输 128 千字节的数据，那么 1.5 兆比特每秒就是 192 千字节每秒。

较旧的 3G 网络传输速度在 600 kbits/s 到 1.5 mbits/s 之间，因此平均而言，这些网络的传输速度为 1.5 mbits/s，被归类为中等数据管道。

快速的数据管道至少为 2048 kbits/s，即 2 mbits/s 或 2 MBPS，以这种较高比特率压缩的视频显示出更高的视觉质量水平。更现代的 4G 网络声称速度在 3 到 6 mbits/s 之间，不过我会为 4G 网络将资源优化到 2 mbits/s，以确保即使网络因某种原因只提供 2 MBPS 的速度，你的视频资源仍然能够非常流畅地播放（即流传输）。

请注意，家庭网络通常提供快得多的 MBPS 性能，通常在 6 MBPS 到 24 MBPS 的范围内。移动网络目前处于 4G 阶段，尚未达到这一带宽，因此你在进行安卓图形应用开发时，需要针对更受限的数据管道进行优化。

比特率还必须考虑到任何一款安卓手机中的 CPU 处理能力，这使得视频数据优化更具挑战性。这是因为一旦“比特”通过数据管道传输，它们还需要被处理并显示到设备屏幕上。事实上，包含在安卓应用 `.APK` 文件中的任何内置视频资源只需要针对处理能力进行优化。之所以如此，是因为如果你使用内置数字视频资源，就没有数据管道供你的视频资源传输，也没有数据被传输！你的视频资源就在那里，就在你的 `APK` 文件中，随时可以播放。

这意味着你的数字视频资源的比特率不仅需要针对带宽（或者如果你使用内置数字视频资源，则针对 `APK` 文件大小）进行良好优化，还需要考虑到 CPU 处理能力的变化。一些单核低功耗嵌入式 CPU 可能无法解码高分辨率、高比特率的数字视频资源而不会丢帧，因此为给定的 CPU 准备一个分辨率更低、比特率更低、数据量只有十分之一的数字视频资源是个好主意。



### Android 数字视频文件：分辨率密度目标

为了让数字视频资源能在市面上各种 Android 操作系统设备上使用，你需要满足多个不同的分辨率目标，这样才能覆盖市面上大多数屏幕密度。

关于 Android 中的默认屏幕密度（因为它们与图形相关），你将在下一章关于基于帧的动画中了解更多。如果你像我一样好奇，为什么 Android 不直接获取一个高分辨率资源，然后使用双三次插值（或者至少是双线性插值）进行缩小，原因在于，目前 Android 的软肋恰恰是缩放，尤其是图像和视频。

如你所知，Android 设备的屏幕尺寸涵盖从较小的翻盖手机和 Android 手表，到较大的平板电脑和 iTV 设备。在这个范围内，还有智能手机、电子书阅读器、迷你平板电脑和中型平板电脑，因此你至少需要四种（如果不是五种）不同的目标 DPI（每英寸点数，即密度像素图像）分辨率。对于数字视频，这还包括不同的目标比特率，这样你就能尽力适配各种不同的设备屏幕密度，以及从单核到四核 CPU 产品的不同处理能力。

就数字视频支持而言，与其说是逐像素地适配显示屏幕，不如说是拥有三到四个比特率目标，因为 `VideoView` 类可以放大（或者更理想的是缩小）视频，你将在本章和本书后面看到这一点。

提供这种均匀分布的比特率范围非常重要，因为较小的 Android 设备往往处理能力较弱，难以流畅解码数据量大的（高）比特率视频，因此你希望一个 Android 应用程序拥有一系列比特率，能够“曲线拟合”市面上所有潜在 Android 设备的处理能力。

例如，Android 手表或翻盖手机很可能使用单核或双核 CPU，而 iTV 则至少使用四核处理器，甚至可能是八核。Galaxy S3 使用四核处理器，而 Galaxy S4 则使用八核（或双四核）处理器。

因此，你应该拥有能在任何处理器上流畅解码的比特率，范围从低分辨率的 512 KBPS 到高分辨率的 2 MBPS。无论压缩程度如何，分辨率密度越高（即像素越小），视频看起来就越好。

这是因为，在任何给定的 Android 设备屏幕上，像素间距越小（或者说像素密度越高），编码伪影就越难被看到（基本上它们会隐藏起来）。一旦你掌握了视频编辑或压缩工具（例如开源软件 `EditShare Lightworks 11` 或 `Sorenson Squeeze Pro`）的使用方法，这种在大多数 Android 设备硬件上常见的精细像素间距，就能让你在相对较小的数据量下获得高质量的视频。

### 优化数字视频：编解码器与压缩

数字视频是通过一种称为编解码器（codec，即编码-解码的缩写）的软件进行压缩的。视频编解码器有两端，一端是编码视频数据流，另一端是解码视频数据流。解码器将是使用它的操作系统或浏览器的一部分。

解码器通常针对速度进行了优化，因为播放的流畅性是首要问题；而编码器则针对减少其生成的数字视频资源的数据量进行了优化。因此，编码过程可能需要很长时间，具体取决于你的工作站包含多少个处理器核心。如今一台专业工作站通常拥有八个核心。

编解码器（编码器端）类似于插件，可以安装到不同的数字视频编辑软件包中，使它们能够编码不同类型的数字视频文件格式。

由于 Android 支持 H.263 和 H.264 MPEG4 格式，以及 ON2 VP8 WebM 视频格式，你需要确保自己使用的是能够将视频数据编码成这些数字视频文件格式的视频编解码器之一。如果操作正确，Android 操作系统将能够解码该数字视频数据流，因为这三种编解码器的解码器端是直接内置于 Android 操作系统中的，这就是为什么你要了解这三种格式。

不止一家软件制造商生产 MPEG 编码软件，因此不同的 MPEG 编解码器（编码器软件）在文件大小方面会产生不同（更好或更差）的结果。

我强烈建议你在想要专业地制作视频时获取的专业解决方案叫做 `Sorenson Squeeze`，目前版本是 9。Squeeze 有一个专业版，我将在本书中使用它，其价格低于一千美元。

还有一个名为 `EditShare Lightworks 11` 的开源解决方案。它目前不原生支持输出到 WebM VP8 编解码器，因此在 `EditShare Lightworks` 增加对 Android 编解码器的支持之前（他们承诺很快就会增加），我打算在本书中使用 `Squeeze Pro 9`。

我仍然建议你访问 [editshare.com](http://editshare.com) 或 [lwks.com](http://lwks.com) 网站注册并下载这个软件，因为它是与 `Blender 3D`、`GIMP2` 和 `Audacity` 一样最强大的开源软件包之一。

在针对数字视频文件大小进行优化（设置压缩参数）时，有大量变量会直接影响视频数据量。我将按照它们对视频文件大小影响的从大到小顺序来介绍这些变量，以便你了解应该调整哪些参数来获得期望的结果。

与数字图像压缩一样，视频每一帧的分辨率（即像素数量）是开始优化过程的最佳起点。如果你的目标用户使用的是 854x480 或 1280x720 的智能手机和平板电脑，那么你不需要使用真正的 1920x1080 高清分辨率视频来让你的数字视频资源获得良好的视觉效果。凭借那些超高密度的显示屏，你可以将 1280 视频放大 33% [通过 `1 - (1280 / 1920)` 计算得出]，它看起来仍然很棒。例外情况是，如果你正在为 GoogleTV 用户提供 iTV 应用，那么你可能需要使用 1920x1080 分辨率。

下一级的优化在于每秒视频所使用的帧数，假设视频本身的秒数无法缩短。这就是帧率，与其使用视频标准 30 FPS，不如考虑使用电影标准帧率 24 FPS，或者多媒体标准帧率 20 FPS。根据你的内容，你甚至可能可以使用 10 FPS 的帧率。



请注意，15 FPS 的数据量是 30 FPS 的一半（即输入到编解码器的数据减少了 100%），并且某些视频内容播放时的效果可能与 30 FPS 的内容相同！找出这一点的唯一真正方法是在内容优化（编码）工作流程中尝试这些设置。

下一个对减小数据占用空间最有效的设置是您为编解码器设定的**比特率**。比特率等同于应用的压缩量，因此也等同于视频数据的质量水平。需要注意的是，您可以简单地使用 30 FPS、1920 分辨率的高清视频，并指定一个非常低的比特率上限；然而，与先尝试使用较低帧率和分辨率，同时配合更高质量比特率设置的方法相比，前者的效果会逊色不少。

由于每个视频数据流都完全不同，要找出任何一个给定编解码器会对您的视频数据产生什么影响，唯一实际的方法是将数据送入编解码器并查看最终结果。如果使用这些设置（`WebM`或`MPEG4 AVC`可实现）能够获得紧凑的数据占用空间，我会建议在低端使用 `512 KBPS` 或 `768 KBPS`，中端使用 `1.5 MBPS` 到 `2.0 MBPS`，高端使用 `2.5 MBPS` 到 `3.5 MBPS`。

下一个在获取更小数据占用空间方面最有效的设置是编解码器用于采样数字视频的**关键帧**数量。视频通过查看一个帧，然后仅编码接下来几帧中的变化（或偏移）来实现压缩，这样就不必对视频数据流中的每一帧都进行编码。这就是为什么“说话头像”视频的编码效果会优于每一帧中每个像素都在移动的视频（例如使用快速摇摄或快速缩放）。

关键帧是编解码器中的一个设置，它强制编解码器每隔一段时间对视频数据资产进行一次全新的采样。通常关键帧有一个自动设置，允许编解码器决定采样多少个关键帧，以及一个手动设置，允许您指定每隔多少帧进行一次关键帧采样，通常是每秒一定次数，或者在整个视频时长内（总帧数）进行一定次数。对于本章后面将要优化的 `480x800` 数字视频资产，我将设置一个 400 帧的 3D 渲染行星飞越动画，总时长对应的关键帧为每 10 帧一个，即共 40 次。

大多数编解码器通常都有一个**质量**或**锐度**设置（或滑块），用于控制在压缩前应用于视频的模糊量。如果您不知道这个技巧，对图像或视频应用非常轻微的模糊（`0.2` 设置）——尽管通常不理想——可以实现更好的压缩，因为图像中的锐利过渡（边缘）比柔和过渡更难编码（需要更多数据来重现）。话虽如此，我建议将质量或锐度滑块保持在 `80%` 到 `100%` 质量之间，并尝试使用我们在这里讨论的其他变量来减小数据占用空间。

归根结底，为了对任何给定的视频数据资产实现最佳的数据占用空间优化，您需要微调许多不同的变量，并且每个视频（从数学角度）对编解码器来说都是不同的。因此，不存在可以开发出来以实现特定结果的“标准”设置。话虽如此，随着时间的推移，调整各种设置的经验最终会让您对需要更改哪些设置以获得期望结果有所感悟。

URI 与你熟悉的使用 HTTP 互联网协议时的 URL 类似，在 Android 操作系统中也以类似的方式用于引用资源的数据路径。如图 2-35 所示，刚写好的最后两行代码中，对象名称下方都有波浪形的黄色警告高亮，这是因为你尚未使用这两个对象定义。下一行代码将使用 `.setVideoURI()` 方法调用把这两个对象“连接”起来，从而解决此问题。

### 总结

在本章中，你学习了数字视频的概念、格式、术语、设置及优化。你还创建了基础（也称为应用启动代码）基础设施，它将容纳你的 `pro.android.graphics` 包和 `GraphicsDesign` 应用程序。

我们首先研究了 Android 和 HTML5 中使用的两种主要视频格式：MPEG-4 H.264 AVC 和 Google 的 WebM VP8 格式。你了解到所有 Android 版本均支持 MPEG-4 播放，而 WebM 在 2.3.3 及更高版本中得到支持，在 Android 4.0 及更高版本中支持流式播放。你还了解到 H.263 主要用于视频会议。

接着，我们概述了 Android 的 `VideoView` 和 `MediaPlayer` 类。这些类用于通过 XML 标记和 Java 代码在应用程序内播放数字视频。你还在 `MainActivity` 用户界面设计中实现了一个 `VideoView` 用户界面元素，用于承载视频数据。

随后，我们研究了数字视频的基础：帧和帧率（以帧每秒即 FPS 表示）。我们学习了如何计算不同分辨率和帧率的原始数据量“影响”，以便你了解如何衡量进入压缩工作流程的数据量。这使你能够计算实际的数据量压缩比率。

我们考察了传统视频行业的标清和高清分辨率标准，以及流式视频与本地视频及比特率的概念。你了解到比特率不仅有助于将数据适配到给定网络带宽容量（数据管道大小）中，还需要考虑移动（嵌入式）设备的处理能力。这是因为用户能够流畅地解码每秒的数据量（比特率）对于获得良好用户体验至关重要。

接着，我们研究了 Android 中的视频缩放问题，以及为视频提供几个分辨率密度目标的需求，这样 Android 就不需要将视频缩放太多。目前，Android 没有双三次插值算法（但它有双线性滤波器）。

最后，我们探讨了编解码器的作用以及用于数字视频优化的压缩设置类型。我们研究了按影响数据量大小从大到小排列的数字视频特性和编解码器设置。你了解到优化数字视频没有万能药，每个数据流都是唯一的（至少从编解码器的角度来看），而优化数字视频数据量与播放质量的比例充其量是一个迭代过程，既需要时间也需要经验。

接着，你通过安装并使用 Terragen 3 3D 动画软件生成了 400 帧虚拟世界动画，然后安装并使用 VirtualDub 1.9 软件使用 AVI 数字视频格式容器加载这些帧，从而创建了视频资源。你将这个 AVI 文件导入 Sorenson Squeeze Pro 9 软件，并将其压缩到原始大小的 1% 以下，这样你就可以将其包含在应用程序的 APK 文件中（作为本地资源）。
