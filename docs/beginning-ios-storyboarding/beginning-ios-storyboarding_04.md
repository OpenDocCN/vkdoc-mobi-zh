# 第 2 章



## 基础

随着 iOS 5 的发布，苹果为 iOS 开发者提供了包含超过 1500 个 API 的更新版 SDK，其中**故事板（Storyboarding）** 是最引人入胜的功能之一。

那么，什么是故事板？它又如何让 iOS 开发者的工作变得更加轻松呢？

在介绍故事板时，我们将一步步探索这个出色的新工具。故事板允许你利用 Xcode 设计工具中内置的新故事板功能来布局应用的工作流程。故事板专为使用导航栏和标签栏在视图之间进行切换的应用而设计，通过为你管理视图控制器来简化开发过程。你可以指定在视图间切换时使用的转场和连线（segues），而无需手动编写代码。通过故事板，你可以无缝交互并连接应用中的所有屏幕，免去了在屏幕间转场以及触发转场的控件所需编写的繁琐代码。这让你能以图形化方式查看应用中所有可能的路径，极大地减少了为复杂的多视图应用编写代码的工作量。

在深入探讨使用故事板创建应用的所有细节之前，让我们先快速了解一下这个新开发者工具背后的原理。

你可能已经知道，自 Xcode 4 发布以来，界面构建器（IB）已完全集成到 Xcode IDE 中，因此不再需要单独的应用程序来构建用户界面（UI）。在项目中选择一个界面文件（`.nib`/`.xib`）即可在 Xcode 中打开 IB 编辑器，在那里你可以随心所欲地调整 UI。

故事板是 Xcode 4.2 中引入的一种新的 IB 文件，它允许你在一个地方查看应用的整个 UI，包括应用各部分之间的转场以及触发这些转场的控件，因此你可以非常方便地概览整个应用。它类似于 `.nib`/`.xib` 文件，区别在于它为你管理多个视图控制器，并允许你指定从一个视图切换到另一个视图时使用的转场，而无需手动编码。你可以将故事板视为一组迷你 `.xib` 文件及其之间的转场和/或关系的集合。

借助内置的故事板功能，你现在可以在编写任何代码之前，以可视化的方式布局应用的工作流程！它还能减少为复杂的多屏幕应用编写代码的工作量。

是否曾好奇神秘的 `.storyboard` 文件背后是什么？答案可能会让你惊讶：它是 XML（可扩展标记语言）——没错，就是那个在 Web 服务器通信和各种 Web 应用中广泛使用的老牌 XML。因此，在底层，故事板只是一个以特殊方式格式化的文本文档，它定义了应用 UI 的对象图，iOS 在运行时根据文档中指定的规则解释并渲染 UI 元素。这个 XML 文件记录了故事板中每个 UI 元素的所有属性，以及它们之间的转场和/或关系。

事实上，你可以亲自验证这一点。打开 Xcode 并创建一个新项目。在第一个对话框屏幕中选择“单视图应用”模板。点击“下一步”。为你喜欢的项目命名，但确保勾选了“使用故事板”选项。将其保存到你喜欢的位置。打开新项目后，你会看到 Xcode 为你创建了一个名为 `MainStoryboard.storyboard` 的文件。在项目导航器中选择该文件，如图 2.0A 所示。现在，点击主工具栏右上角的“版本编辑器”图标，然后点击右下角的“日志模式”图标，如图 2-0B 所示。你现在可以看到故事板文件背后的 XML 了。是不是不太美观？想象一下，当你向其中添加大量 UI 元素后，它会变得多么复杂和难以阅读？幸运的是，界面构建器会以图形元素的形式为你优雅地渲染它，与原始 XML 相比，交互体验要愉悦得多。

现在你已了解故事板实现背后的秘密，让我们快速一瞥它初看起来的样子。

你可能已经注意到，Xcode 为单视图应用模板创建的基本故事板与你之前见过的任何常规 `.xib` 文件几乎相同，只有少数例外：有一个指向主视图的传入箭头，并且在主视图下方有一个写着“视图控制器”的黑色容器。这两个元素都有非常明确的用途。由于故事板可以包含多个视图控制器，你需要一种方法来指定应用启动时应加载哪一个；箭头指向的视图控制器将是应用启动时呈现给用户的第一个视图。要更改应用的初始视图控制器，请将箭头拖放到故事板中你想要首先呈现的任何视图控制器上。

那么黑色容器呢？它是每个视图控制器场景所谓的“停靠栏”。如前所述，故事板可以被视为一组 nib 文件，每个 nib 文件都由一个视图控制器类（默认或自定义）和一个第一响应者控制。

**注意：** 第一响应者是一个占位符对象，代表响应者链中的第一个对象，由 UIKit 框架在运行时动态确定。该对象首先接收多种事件，例如触摸、按键事件、动作消息等。在 iOS 应用中，没有单一的第一响应者对象——第一响应者始终是触摸目标所在的视图，通常是当前选中的对象或最前面窗口中具有当前焦点的对象。

停靠栏让你可以轻松访问刚才描述的两个对象，在常规的 nib 文件中，你通常会在文档大纲的顶部看到它们。通过点击停靠栏，你可以看到代表视图控制器和第一响应者占位符对象的图标，如图 2-0C 所示。你可以使用这些引用图标快速连接 `IBOutlet` 并为你的 UI 元素指定 `IBAction`。故事板中的每个视图控制器都有一组这两个对象，因此你可以通过每次只专注于一个特定屏幕，方便地编辑每个故事板场景的所有必要细节。此外，停靠栏可能包含添加到场景中的其他对象，例如手势识别器、导航项等。当你进行实际编码时，你会更好地理解如何使用它们。

我们将为你简要概述在开始为 App 制作故事板之前需要了解的主要高级概念。

图 2-0D 展示了一个标准故事板文件的主要组件，当你在项目中选用“主-从应用程序”模板时，Xcode 会为你创建该文件。现在，我们将解释这些关键组件的作用以及它们是如何协同工作的。

![images](img/9781430242727_Fig02-0D.jpg)

**图 2-0D.** *故事板的主-从应用程序模板*

故事板中有两个主要概念：场景（Scene）和转场（Segue）。

*场景*是一个视图控制器，它可以表示 App 中的一个屏幕，也可以仅仅表示一个主要的组件区域（例如分割视图控制器中的一个分割视图面板）。你可以在这里进行所有之前在 nib 文件中操作过的事情：拖放并重新排列 UI 元素（如标签、文本字段和图片），调整它们的大小，在属性检查器中设置它们的属性，等等。

*转场*定义了故事板中的导航。它指示如何从点 A 到达点 B。你通常通过选择一个事件源（例如按钮或表格视图单元格）并选择一个要过渡到的场景来创建转场。你还可以选择想要使用的动画类型。只需进行简单的选择，即可使用所有 Apple 内置的动画。此外，你甚至可以为自定义转场创建自己的自定义动画。

因此，你将使用场景和转场来构建你的应用程序 UI。

**注意：** 你的 App 中可以有多个故事板。你只需要在 `[appName]-Info.plist` 文件中，于“Main storyboard file base name”（主故事板文件基本名称）键下指定主故事板的名称即可，该键的默认值是 `MainStoryboard`。

你从一个代表第一个场景（初始视图控制器）的视图控制器对象开始。要为你的故事板获取视图控制器，你需要从对象库中选择“对象和控制器”（就像你对 nib 文件中所有其他 UI 元素所做的那样），然后将你需要的视图控制器拖放到故事板画布上。每个视图控制器始终管理着你的 App 中的一个场景。

**注意：** 在 iPhone 上，每个场景代表单个屏幕的内容。对于 iPad 应用程序，一个屏幕可以由多个场景的内容组成（例如在分割视图控制器的情况下）。

其中最重要的概念之一是数据如何传递。这里有一个非常经典的场景：一个主-从应用程序，你在表格视图中显示一些数据，当用户点击某个单元格时，会显示一个详细视图，展示关于所选内容的附加信息。在这种情况下，你必须将对所选对象的引用发送给详细视图，以便它确切知道要显示什么。

故事板为此提供了相应的 API。每个 `UIViewController` 都有一个名为 `prepareForSegue:sender:` 的方法，该方法旨在被重写。你可以在这里编写代码，这些代码将在每次触发转场时生效。

转场和标签、图像视图一样，都是 UI 元素。你可以在故事板中选择一个转场并检查其属性。你可以通过检查每个转场的标识符属性来区分它们，该属性可以在属性检查器中设置。在本章中，我们将向你展示如何执行此操作，并涵盖其他几个主要的故事板方面。

### helloAlien：一个快速示例应用程序

首先，我们将向你介绍故事板的四个基本概念：

*   用很少甚至不用代码即可轻松创建视图间的过渡。
*   将信息从次级视图传回主视图。
*   将信息从主视图发送到次级视图。
*   使用用户创建的视觉效果在视图之间进行过渡。

在我们的示例应用程序中，我们通过以下方式演示这四个概念：

*   创建一个按钮，用于触发从主视图到次级视图的过渡。
*   将一个“是/否”值从次级视图（外星人视图）传递到主视图。
*   将用户输入的文本从主视图发送到次级视图（外星人视图）。
*   创建一个自定义的转场（读作“segway”），用于在两个视图之间实现淡出/淡入效果。

为此，我们将编写一个简单的 App，询问外星人是否存在。转到次级视图，你会看到一个开关，外星人可以将其打开——当然，前提是外星人存在。如果外星人将开关拨到“开”的位置，那么当你返回主视图时，会出现一条消息，表示他确实存在。然后，你可以对外星人说些什么，你会在次级视图的外太空空间中看到这条消息。最后，你将创建一个自定义转场，以一种美观的方式将你从主视图带到次级视图。那么，让我们开始吧！

#### 第 2 步：将信息从辅助视图（外星视图）传回主视图

在这一步中，我们将展示将信息从辅助视图传回主视图的典型模式。最常见的做法是将通信描述为一个协议，并让主视图设置为辅助视图的委托。我们来看看具体如何实现。

![images](img/9781430242727_Fig02-12.jpg)
