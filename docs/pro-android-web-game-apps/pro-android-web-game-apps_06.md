## 避免“炫酷但吓人”的编码风格

当你精通 JavaScript 并掌握了一些高级特性后，可能会忍不住编写看起来极其“硬核”和“高级”的代码段。例如，将方法的参数写入控制台的函数，可以用下面这样的一行代码实现：

```javascript
console.log([tag, "(", Array.prototype.join.call(arguments, "|"), ")"].join(""));
```

如果我是你，我不会这样做。最好多花一两行代码，将代码拆分成更小的片段，使其可读性强且清晰易懂，即使对 JavaScript 初学者来说也是如此。编写易读易理解的代码，比编写杂乱难懂的代码要困难得多。

## 第 1 章：入门指南

## 结构

代码看起来可能很整洁，但离完美还有很大差距。JavaScript 是一门单线程、动态、弱类型的语言。如果正确使用，这三个特性非常强大，但如果使用不当，它们可能会引发各种问题。

### 避免为对象设置“魔法”参数

JavaScript 允许你在运行时向任何对象添加任意属性——因为它是一门动态语言。当你即时创建像坐标点这样的简单数据结构，或者将对象用作哈希映射时，这个特性非常好用。下面的代码展示了如何正确使用它。

```javascript
function getCurrentCoordinates() {
  {
    x: getCurrentX(),
    y: getCurrentY();
  }
}

var userStatus = {}
userStatus["John"] = IDLE;
userStatus["Mary"] = ONLINE;
userStatus["Peter"] = AWAY;
```

然而，在某些情况下，你应该避免这种结构上的变更。例如，如果存在预定义的数据结构，你就不应该以这种方式扩展它。

为什么这是一个坏习惯？想象一下，你身处一个团队，大家都在协作同一份代码。你开始在对象中添加自定义字段，因为你的函数需要它们。其他开发者发现有了这些有用新属性，于是他们开始在自己的代码中依赖这些字段。

这里有一个简单的例子：一位程序员想要编写一个函数，用于按特定格式打印日期。他可能会写出如下代码：

```javascript
function printNiceDate(date) {
  date.niceFormat = "";
  date.niceFormat += date.getFullYear() + " ";
  date.niceFormat += date.getMonth() + " ";
  date.niceFormat += date.getDate() + " ";
  console.log(date.niceFormat);
}
```

尽管这段代码 100% 正确，但它却在对象上创建了一个意想不到的字段。这个被“额外加持”的 `Date` 对象随后可能会被传递到应用程序的不同层级，另一位程序员可能会认为这个额外字段是可以依赖的标准功能。他会编写使用这个新字段的代码，如下所示：

```javascript
// 太酷了，我喜欢这个额外字段，它可能是个新标准
console.log("日期是 " + date.niceFormat.substr(4))
```

到了某个时候，原作者决定改变这个自定义字段的格式、将其删除或重命名。他认为这个新字段是他自己的属性——它不是标准属性，可能甚至没有在文档中提及，而且是由他引入的：这意味着他可以随意处理这个自定义字段。

如果他决定删除这个字段，项目就会出问题。使用了 `niceFormat` 的代码将无法再运行。两位程序员都会在调试和恢复原始流程上浪费大量时间。总结：如果你真的、真的需要自定义字段，请确保它们不会泄露出去。

即使你是一个在自己的车库里独立工作的“独行侠”，你仍然需要考虑团队协作的问题。

**注意：** 较新版本的 JavaScript 引入了通过 `Object.freeze`、`Object.seal` 和 `Object.preventExtensions` 来正式保护对象状态的方法。如果你需要对对象的不变性提供更强有力的保障，可以使用这些方法之一。

相反的情况也同样重要。除非你清楚自己在做什么，否则不应删除对象的任何属性。对象的结构必须始终保持可预测性。

这条规则有一个例外。如果动态添加的方法和属性是框架设计的一部分，那么使用它们是完全可以的。例如，我们将在第 14 章尝试的 `Crafty.js` 框架实现了实体组件系统，这种方法通过在运行时自定义特定对象来达到最佳效果。

## 尽可能使用异步调用

JavaScript 是单线程的，这意味着代码在执行时，应用程序会“冻结”。应避免在这个单线程内执行阻塞性的长时间运行操作，因为它们会使你的应用程序陷入停顿。用户界面会变得无响应，从而导致糟糕的用户体验。

这对于 XHR 请求尤其重要；同步调用在你使用本地服务器时可能看起来非常快，但在实际生产环境中几乎肯定会成为问题。你绝不应该使用同步 XHR。

在服务器端，使用异步 API 甚至更为重要。向硬盘上的文件写入十几个字节听起来像是不会造成任何延误的操作；它实在太快了。如果你单独测试服务器，这可能是真的。但当你有 30,000 名访客，每个访客都要求你的服务器写入十几个字节时，如果编写不当，这段代码片段就会成为瓶颈。

## 环境

开发环境与生产环境在很多方面都存在差异。当一切都在你的本地 PC 上时，你的项目生活在一个不切实际的微型天堂里。在现实世界中，网络数据包的传输需要更多时间：有时它们会丢失，连接会中断，资源也并非始终可用。

## 通知用户有关长时间操作的信息

所以你做的每件事都是正确的：你的代码是异步的，用户界面也从未无响应。但是，假设你需要为下一关加载大量精灵图，加载关卡数据，生成随机地形，或者计算人工智能的最佳策略。又或者，你可能需要同时完成所有这些操作。在这种情况下，用户有时不得不等待相当长的时间，才能看到有趣的事情发生。

不要让用户坐在空屏幕前，心里琢磨：“这到底需要多长时间——一分钟、五分钟，还是更久？！”

当你怀疑某个操作会花费超过一秒钟时，请使用某种进度条或“活动”指示器。

**注意：** 以下是网页可用性专家雅各布·尼尔森在 [www.useit.com/papers/responsetime.html](http://www.useit.com/papers/responsetime.html) 上描述的关于*长时间操作*的一个非常好的衡量标准：

-   **0.1 秒** 大约是让用户感觉系统在即时响应的极限，这意味着除了显示结果之外，无需任何特殊的反馈。
-   **1.0 秒** 大约是使用户思绪流动保持不被打断的极限，尽管用户会察觉到延迟。通常，在延迟超过 0.1 秒但不到 1.0 秒的情况下，不需要特殊的反馈，但用户会失去直接操作数据的感觉。
-   **10 秒** 大约是保持用户注意力集中在对话框上的极限。
