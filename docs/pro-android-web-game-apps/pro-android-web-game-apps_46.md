# 计划

与本书中的其他项目相比，我们即将完成的项目无论在代码行数还是功能数量上都相当庞大。我们需要一个周密的计划来追踪进度，并理解为何要选择特定路径来实现目标。目标是构建一个能够渲染地形与物体，并能与用户交互的等距引擎。我们暂时略过具体的游戏玩法，专注于架构、渲染机制、优化以及玩家交互。我们将实现加载关卡、允许用户滚动浏览、点击地面瓦片改变地形类型（例如“修建道路”）、向地图添加新物体以及移除现有物体等功能。这是许多策略游戏的典型流程；其余部分只是游戏玩法的细节（例如“当木材少于 300 时，不允许建造马厩”）。

### 交互模式

用户将拥有三种与游戏世界进行“交互”的模式：移动、编辑地形和编辑物体。用户可以随时在模式之间切换——我们将为此创建一个小型 UI 元素。再次查看图 7-1，左上角带有“房子”图标的小圆圈显示了当前模式。当用户点击该圆圈时，模式会切换。

### 渲染层

渲染按层组织（我们在第六章讨论过该技术），每一层负责渲染特定类型的对象。本项目将包含三种层类型：一种用于等距瓦片，一种用于世界物体，一种用于 UI 元素。它们分别称为`IsometricTileLayer`、`ObjectLayer`和`UiLayer`，如图 7-2 所示。为了简化对多层的处理，我们将创建`LayerManager`。

### 构建骨架

我们首先构建游戏的骨架——即 HTML 文件和主要的`Game`类——启动事件处理并创建一个动画循环，方法与上一章类似。此时，我们将拥有一个可以“运行”的起点。下一步是添加等距地形。

等距地形本身并非难事，但有一些需要注意的事项和需要实施的优化。本章的第二部分专门讨论等距地形：渲染瓦片、实现离屏缓存、滚动地图以及其他相关问题。最终成果是一片空旷的、可滚动的地形，其上没有任何物体。

### 添加物体

接下来显而易见的步骤是向地图添加物体。我们将需要`ObjectLayer`，它比`IsometricTileLayer`更容易构建。我们还会引入上一章简要提及的一项优化——物体集群（用于限制“活跃”物体的数量，并且无需处理远离视口的物体）。最后，我们将研究像拱门和传送门这样的复杂物体——如何正确实现它们有一个小技巧。至此，我们将在等距地面上渲染出物体。其中一些物体将能够移动，以证明移动处理是正确的。

### 脏矩形算法

下一个主要主题是构建脏矩形算法——将屏幕的某些区域标记为“脏”，以便在下一帧仅渲染这些区域。例如，如果屏幕上有一个移动的小物体，则无需重绘整个区域——只需重绘该物体覆盖的部分。等距层和物体层都会受此变化影响，在本节结束时，游戏将不会在渲染未发生变化的屏幕部分上浪费任何周期。

### LayerManager

接下来，我们将引入`LayerManager`——这个类简化了对多层结构的处理——因为此时我们已经有了两个图层（地形和物体），并且即将添加第三个（UI 元素）。我们添加的图层越多，管理就变得越复杂。
