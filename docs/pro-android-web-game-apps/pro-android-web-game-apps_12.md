# 第 1 章：入门

汽车的颜色是在 `Car` 构造函数中设置的。由于构造函数本身并未执行，因此颜色变量为 `undefined`。这个问题也很容易解决。修改 `FireTruck` 构造函数，并在其中添加对 `Car` 构造函数的调用。这样，我们就可以让 `Car` 初始化其自身的对象变量。

```javascript
function FireTruck() {
  Car.call(this, "red");
}
```

重新运行代码，即可获得正确的输出结果。

尽管代码可以运行，但还存在一个小问题需要解决。当你创建新函数时，其 `prototype` 属性并非为空。它包含一个名为 `constructor` 的属性，该属性指向函数本身。当我们调用

```javascript
FireTruck.prototype = Object.create(Car.prototype);
```

时，这个属性会丢失，因为新创建的对象没有自己的属性。我们可以将缺失的属性作为第二个参数传递给 `create`。

```javascript
FireTruck.prototype = Object.create(Car.prototype, {
  constructor: {
    value: FireTruck, // 等同于 FireTruck.prototype.constructor
    enumerable: false,
    writable: true,
    configurable: true
  }
});
```

`constructor` 并非普通属性。当你在对象的键中循环遍历时，它不会出现，但如果你通过名称直接访问，它是可用的。我们可以通过将属性描述符传递给 `create()` 的第二个参数来实现相同的结果。描述符包含以下几个字段：

- `value`：初始值
- `enumerable`：该属性是否应出现在对象属性列表中（例如 `for… in` 或 `Object.keys` 循环中）
- `writable`：该属性是否可以被赋予不同的值
- `configurable`：如果为 true，则可以更改描述符定义的规则，或删除该属性

通过这种方式，我们模拟了真实构造函数的行为，保留了 ECMA（欧洲计算机制造商协会，负责 JavaScript 背后标准的组织）规范所定义的原型结构。这是前面描述的编码原则的实际实现：保留对象的结构。即使你不知道为什么可能需要这个属性，你也必须保留它。

让我们看看**清单 1-8**，这是该示例的最终版本。它展示了如何实现继承以及构建自定义原型链。继承代码被移到了独立的函数中。这样做有几个好处：至少，我们消除了代码重复（如果应用程序中有几十个对象）。此外，如果你将游戏移植到较旧的浏览器，可能需要实现不同的继承策略，因为较旧的浏览器可能不支持上述特性。

**清单 1-8.** *继承示例的最终版本*

```javascript
function extend(subConstructor, superConstructor) {
  subConstructor.prototype = Object.create(superConstructor.prototype, {
    constructor: {
      value: subConstructor,
      enumerable: false,
      writable: true,
      configurable: true
    }
  });
}

function Car(color) {
  this._color = color;
}

Car.prototype.move = function() {
  console.log(this._color + " car moves");
};

Car.prototype.turn = function(direction) {
  console.log(this._color + "car turns " + direction);
};

Car.prototype.stop = function() {
  console.log(this._color + " car stops");
};

function FireTruck() {
  Car.call(this, "red");
}

extend(FireTruck, Car);

FireTruck.prototype.turnCannon = function(direction) {
  console.log("Cannon moves to the " + direction);
};

var truck = new FireTruck();
truck.move();
truck.turnCannon("right");
```

`extend` 函数将在本书的每一个脚本中使用。

**注意：** JavaScript 中 OOP 的实现方式在过去几年中已经历过多次演变。解决同一任务有多种方法：有些方法更优，而另一些则略逊一筹或已过时。所描述的方法是迄今为止最简洁的一种。此外，`Node.js`（我们将在第 10 章学习的服务器端 JavaScript 引擎）也使用了相同的对象扩展方法。采用此方法可使对象从客户端移植到服务器端变得更加容易。事实上，我略微取巧地使用了 `Node` 核心模块中的函数来维持这种一致性。

关于命名的最后一点说明。在本章中，我尽量避免了使用“类”这个词。然而，事实证明，`class` 是一个很好的词，用来描述可用于构造新对象的蓝图这一概念。使用“构造函数”和“原型链”这样的词非常不便。在接下来的章节中，我将使用“类”一词来指代任何构造函数及其原型。换句话说，即定义该对象的全部属性集。既然我们已经澄清了 JavaScript OOP 背后的机制，这便不会引起任何误解。

OOP 为你组织代码结构提供了极大的可能性。尽管 JavaScript 没有类，但它仍然是一种面向对象的语言。在很长一段时间里，JavaScript 仅用于编写简单的脚本，OOP 方法似乎是一种不必要的复杂性。然而，当代码库规模增长时，OOP 便展现出了其全部潜力。

## 游戏的构成

你玩过游戏吗？我相信你玩过。让我们从一个简单的问题开始：电子游戏到底是关于什么的？主要是为了获得乐趣，对吧？每个游戏都为玩家呈现一个虚拟世界，这个世界由其自身的规则和机制支配。这个世界可以像俄罗斯方块一样简单——一个屏幕，上面有砖块从上方落下——也可以像现代网络游戏那样复杂，里面有骑士和巫师、公会和政治。

玩家试图完成一个目标：得分、将点连成一条线，或征服一两个星系。游戏本身是玩家与游戏世界之间的互动。玩家通过某种方式感知游戏状态：阅读文本、查看 3D 场景或浏览网页。玩家接收所有这些信息并采取行动。然后世界会更新，再次轮到玩家行动。他采取另一个行动，如此循环，直到游戏结束。

那么，游戏实现这一切的关键组成部分是什么？如今，游戏强烈侧重于图形内容。实际上，游戏图形可能是吸引玩家并让他们尝试你的产品的唯一机会。玩家希望游戏漂亮，Web 应用程序游戏也不例外。大多数游戏都有一个在屏幕上绘制世界的图形引擎。有些复杂，有些简单，但几乎没有完全没有任何图形的游戏。

图形并非游戏的全部，它们只是视觉效果。游戏的核心是交互。玩家希望移动屏幕上的星际飞船，并将它们送入战斗。每个游戏开发项目的第二个关键部分是用户输入处理。

我们已经习惯了控制器、鼠标和键盘，但移动平台提供了完全不同的交互方式：触摸屏、加速度计、陀螺仪和其他传感器。用户无法执行“鼠标右键单击”，但他们可以点击、滑动、摇晃或旋转他们的设备。直观的操作听起来可能很容易实现，至少在你开始编写实际代码之前是这样的。
