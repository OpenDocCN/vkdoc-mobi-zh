# 排版后内容

**图 4-10.** *在理想情况下，定时器从不会延迟，所有动画都能准时完成。*

然而在现实世界中，情况则有所不同。我们计划让每一帧在屏幕上停留 50 毫秒。一个以 60 FPS 运行的定时器大约每隔 16 毫秒（每秒 60 帧 = 1000 ÷ 60 毫秒/帧）就会触发一次我们的函数。这意味着定时器并不会在动画序列开始后的恰好第 50 毫秒触发。函数第一次执行是在第 16 毫秒，第二次在第 32 毫秒，第三次在第 48 毫秒，第四次在第 64 毫秒。那么帧会在何时切换呢？答案是在第 64 毫秒之后（而非计划中的第 50 毫秒）。图 4-11 阐释了这一概念。

按照同样的逻辑，下一帧会在下一次触发时渲染，即第 64 + 50 毫秒（当前帧的时间加上帧间间隔），也就是从动画开始后的第 128 毫秒。依此类推，直到动画结束。

**图 4-11.** *在现实世界中，定时器并不精确，动画也会出现延迟。*

如果这个动画有 6 帧，你可能会认为它将在 300 毫秒（每帧 50 毫秒）后结束。但由于每帧我们都会延迟 14 毫秒（实际耗时 64 毫秒而非 50 毫秒），动画最终将持续 384 毫秒。

这种方法的问题在于，我们通过计算自上一帧开始的时间来决定是否需要渲染新的一帧。由于定时器不精准，每一帧都会在计算中引入误差。帧数越多，预期结果与实际结果之间的差异就越大。

让我们尝试一种不同的模型。我们将使用自动画开始的时间，而非自帧开始的时间。由于我们知道动画的总时长，因此可以确定已经过去了多长时间的比例。当从动画开始经过了 100 毫秒，而整个动画需要 300 毫秒才能完成时，已完成的比例是 100 ÷ 300 = 1/3，即三分之一。此时，无论距离上一帧过去了多久，我们都应该绘制第 2 帧。

这样一来，我们就消除了误差累积的影响，并使动画尽可能接近预期的时间。为了更好地理解这个思路，让我们看看图 4-12，它展示了我们最终的解决方案。中间的坐标轴表示自动画开始以来已过去的时间比例。第一帧的渲染方式完全相同：它在屏幕上停留了 64 毫秒。当达到总时间的六分之一时，第一帧就必须切换。六分之一约等于 0.167；需要四次迭代才能达到这个阈值。第三次迭代开始时比例为 0.16 —— 非常接近下一帧的切换点，但还不够。第四次迭代发生在第 64 毫秒，比例为 0.21；此时帧才会实际切换。第二帧应该在比例达到 0.333 时结束，因此只需再经过三次迭代（从比例 0.21 到 0.37）即可到达。第二帧在屏幕上停留了 48 毫秒 —— 比应有的时间少了 2 毫秒。然而，这个数值比第一帧的情况更接近期望的 50 毫秒。

**图 4-12.** *一种更好的方法。我们不再依赖自上一帧以来的时间，而是依赖自动画序列开始以来经过的时间比例。*

在这种情况下，整体动画只延迟了 4 毫秒 —— 这比之前的 84 毫秒好得多。让我们实现这种方法，最终制作一个优秀的动画函数版本。代码如列表 4-26 所示。

**列表 4-26.** *动画函数的最终版本*

```javascript
var numFrames = 6;
var duration = numFrames*50;
var animationStartTime = new Date().getTime();

function animate(t) {
    var now = t || new Date().getTime();
    // 确保我们不会超出边界
    var timePassed = (now - animationStartTime)%duration;
    var fraction = timePassed/duration;
    var currentFrame = Math.floor(fraction*numFrames);
```
